## 稽核循環流程

每一輪循環執行以下三步：

### Step 1：排查（Detect）

從以下維度系統性掃描所有文件：

**A. 合規性**
- 契約要求的必要章節是否存在（對照交付項目一覽表的「基本內容」欄）
- 是否包含大陸品牌/產品（契約明確禁止陸籍人員與陸製資通訊產品）
- CNS 27001 合規要求是否滿足
- 個資保護（PIA）相關內容是否充分

**B. 數字一致性**
- 所有數字（KPI、人數、模組數、延遲指標等）是否與 decisions.yaml 一致
- 跨文件引用同一數字時是否一致（如 STT 延遲、FR 數量、模組數量）

**C. 結構完整性**
- 是否有空白/stub 章節（少於 5 行實質內容的章節）
- 是否有 placeholder 文字（TODO、TBD、待補、xxx、[填入]）
- 分章檔案（_ch1、_ch2...）與主檔的內容是否一致/完整
- 目錄（TOC）與實際章節標題是否對應

**D. 交叉引用**
- 引用其他文件/章節的地方，被引用的目標是否存在
- 圖表編號是否連續、是否被正文引用

**E. 格式規範**
- Header block（版本、案號、日期）是否每份文件都有
- 用語是否統一（廠商/本公司/立約商 只能用一種）
- Markdown 語法是否正確（表格對齊、標題層級）

**F. 攻擊面防禦**
- 品質報告中標記為「未修」的 ATK 項目是否已修復
- Prompt Injection 防禦是否有技術細節支撐

排查結果格式：
```
## 排查結果（第 N 輪）

### 致命級（必修）
- [ISSUE-001] 文件: #XX, 位置: 第 Y 行, 問題: 描述, 預期: xxx, 實際: yyy

### 重要級
- [ISSUE-002] ...

### 一般級
- [ISSUE-003] ...
```

### Step 2：驗證（Verify）

對排查出的每個問題：
1. **重新讀取**該文件的相關段落，確認問題真實存在（排除誤報）
2. **交叉驗證**：與 decisions.yaml、需求文件、其他相關文件比對
3. 標記：`[CONFIRMED]` 或 `[FALSE POSITIVE]`

### Step 3：修復（Fix）

對所有 `[CONFIRMED]` 的問題：
1. 按嚴重度排序：致命級 > 重要級 > 一般級
2. 逐一修復，每次修復後立即驗證修復是否正確
3. 修復時注意：
   - 只改有問題的部分，不要重寫整份文件
   - 數字修正必須引用 decisions.yaml 的具體 key
   - 新增內容必須與文件整體風格一致
   - 如果主檔和分章檔都存在，優先修改主檔

修復完成後，回到 Step 1 開始下一輪排查。

## 終止條件

當一輪排查結果為「0 個致命級 + 0 個重要級 + N 個一般級（N ≤ 3）」時，停止循環。

## 最終輸出

循環結束後，更新 `pipeline/output/品質報告.md`，包含：
1. 本次稽核共執行了幾輪循環
2. 每輪發現並修復了多少問題
3. 殘留問題清單（如有）
4. 更新後的品質評分

## 注意事項

- 不要憑空捏造內容，新增的技術描述必須基於 decisions.yaml 中的技術選型
- 不要刪除現有的正確內容
- 每輪排查最多處理 15 個問題，避免一次改太多導致連鎖錯誤
- 如果某個問題不確定如何修復，標記為 `[NEEDS_REVIEW]` 跳過，留給人工處理

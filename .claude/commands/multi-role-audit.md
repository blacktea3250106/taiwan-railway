## 稽核循環流程

### Phase 1：多角色平行審查（Detect）

以 10 位角色的視角，同時掃描所有 `pipeline/output/*.md` 文件（排除 `_ch*` 分章檔和品質報告）。

每個角色依各自審查焦點產出問題清單：

```
[ISSUE-{角色ID}-{序號}] 嚴重度：致命/重要/一般
  文件：#XX 文件名
  位置：§X.X 或具體行
  問題：描述
  預期：正確應為什麼
  實際：目前文件寫了什麼
  負責修復角色：R?
```

分配修復角色的規則：
- 架構/技術問題 → R1 或 R2
- 管理/時程問題 → R3
- 資安/個資問題 → R4
- 文字品質問題 → R5
- 合規問題 → R6
- 數字一致性問題 → R7
- 格式問題 → R8
- R9/R10 發現的問題 → 依問題性質分配給 R1~R8

### Phase 2：交叉驗證（Verify）

對 Phase 1 產出的每個問題：

1. **重新讀取**該文件的相關段落，確認問題真實存在
2. **交叉比對** decisions.yaml 和其他相關文件
3. 標記結果：
   - `[CONFIRMED]` — 問題確實存在
   - `[FALSE_POSITIVE]` — 誤報，說明原因
   - `[DUPLICATE]` — 與其他問題重複，標記主問題編號

去重規則：多角色指出同一問題時，保留最完整的描述，合併為一條。

### Phase 3：修復（Fix）

對所有 `[CONFIRMED]` 問題，按優先序修復：

**修復順序**：致命級（全部）→ 重要級（全部）→ 一般級（全部）

**修復規則**：
1. **只改有問題的部分**，不重寫整份文件
2. 數字修正必須引用 decisions.yaml 的具體 key
3. 新增內容必須與文件整體風格一致
4. 如果主檔（如 `10_程式設計書.md`）和分章檔（如 `10_程式設計書_ch1.md`）都存在，**優先修改主檔**
5. 每次修復後立即驗證修復正確性（讀取修改後的段落確認）
6. 修復完一個問題，立即標記為 `[FIXED]`

**修復方式**：
- 使用 Edit 工具直接修改 `pipeline/output/` 下的 .md 檔案
- 每個修復操作記錄：修了什麼、改了哪個檔案、改前改後對比

### Phase 4：回歸審查（Re-detect）

修復完成後，回到 Phase 1 重新審查：
- 確認已修復的問題不再出現
- 檢查修復是否引入新問題
- 檢查修復是否造成跨文件不一致

### 循環終止條件

當一輪審查結果滿足：**0 致命 + 0 重要 + 一般級 ≤ 3** → 停止循環。

最多執行 **5 輪**循環（防止無限迴圈）。

## 注意事項

- 不要憑空捏造內容，技術描述必須基於 decisions.yaml 中的技術選型
- 不要刪除現有的正確內容
- 每輪最多處理 **20 個問題**，避免連鎖錯誤
- 不確定如何修復的問題標記 `[NEEDS_REVIEW]` 跳過
- 修復跨文件數字不一致時，**先確認 decisions.yaml 的值，再統一修改所有文件**
- 每輪修復後 git diff 檢視變更量，確保改動合理
- 讀取已有的 `pipeline/review_results/` 審查結果作為第一輪的起點（如果存在）

# DOCX 交付文件稽核 — 掃描、排查、修復循環

你是台鐵 AI 客服系統建置案（案號 L0215P2010U）的 DOCX 交付文件稽核員。

## 任務

對 `pipeline/output_docx/` 目錄下的所有 `.docx` 檔案執行**掃描 → 排查 → 修復**循環，直到掃描不出新問題為止。

## 稽核範圍

### 目標檔案
`pipeline/output_docx/*.docx` — 共 13 份交付文件的 Word 版本。

### 對照基準
1. **需求合規**：`台鐵需求文件.docx`（契約需求原文，用 python-docx 讀取）
2. **數字正確性**：`pipeline/decisions.yaml`（Single Source of Truth）
3. **政府標案格式**：依行政院「文書處理手冊」及政府資訊系統交付慣例
4. **原始內容**：`pipeline/output/*.md`（Markdown 原稿，DOCX 應與其內容一致）

## 三大稽核維度

### 維度 A：需求文件合規性

用 python-docx 讀取 `台鐵需求文件.docx`，逐項比對每份交付文件是否滿足契約要求：

1. **必要章節覆蓋** — 契約規定的每份文件應包含的章節是否齊全
2. **KPI 指標正確** — IVR/Agent 辨識率 ≥85%、STT ≥85%、回應 ≤5000ms、可用率 ≥99% 等，是否與 decisions.yaml 一致
3. **硬體規格一致** — 伺服器型號、數量、GPU 配置是否與 decisions.yaml 的 SECTION E 一致
4. **合規紅線** — 12 條紅線全檢（見下方清單）
5. **交付時程** — 各文件的交付期限是否與契約/decisions.yaml 一致
6. **教育訓練** — 時數（≥54 小時）、課程類別、對象是否齊全

### 維度 B：政府標案格式規範

1. **頁面設定**
   - A4 直式（21.0cm × 29.7cm）
   - 邊界：上/下/右 2.54cm，左 ≥3.17cm（含裝訂邊）
2. **字型**
   - 中文：標楷體
   - 英文/數字：Times New Roman
   - 內文：12pt；標題一：16-22pt 粗體；標題二：14-18pt 粗體；標題三：12-16pt 粗體
3. **頁首/頁尾**
   - 頁首含專案名稱或案號（L0215P2010U）
   - 頁尾含頁碼
4. **目錄**
   - 前 30 段內應有目錄（TOC）
   - TOC 項目應與實際標題對應
5. **表格**
   - 所有表格必須有完整框線
   - 表格標題置於表格上方
   - 跨頁表格需重複標題列
6. **編號系統**
   - 章節編號連續（不跳號）
   - 圖表編號連續且被正文引用
7. **Header Block**
   - 每份文件首頁含：文件名稱、版本、案號、日期、廠商名稱

### 維度 C：內容與排版品質

1. **Markdown→DOCX 轉換完整性**
   - 對照 `pipeline/output/*.md` 原稿，檢查是否有內容遺漏
   - 表格是否完整轉換（行列數一致、內容無截斷）
   - 程式碼區塊是否正確呈現
2. **跑版問題**
   - 表格欄寬異常（過窄導致文字換行破碎、過寬超出頁面）
   - 圖片/圖表溢出頁面邊界
   - 連續空段落過多（>5 個）
   - 孤行/寡行（段落首尾行落在跨頁處）
3. **內容品質**
   - 無空白/stub 章節（<5 行實質內容的章節）
   - 無 placeholder（TODO、TBD、待補、xxx、[填入]、【】空白）
   - 用語統一（廠商/本公司/立約商 只用一種）
   - 全文繁體中文，無簡體字元

## 稽核循環流程

### Step 1：掃描（Scan）

使用 python-docx 程式化讀取每份 DOCX 檔案，從三大維度系統性掃描。

**執行方式**：
1. 先執行 `python pipeline/audit_docx_format.py` 取得格式基線報告
2. 用 python-docx 讀取 `台鐵需求文件.docx` 提取契約要求清單
3. 逐檔掃描，產出問題清單

**掃描結果格式**：
```
## 掃描結果（第 N 輪）

### 致命級（必修）
- [DOCX-001] 檔案: XX.docx, 維度: A/B/C, 類別: xxx, 問題: 描述, 預期: xxx, 實際: yyy

### 重要級
- [DOCX-002] ...

### 一般級
- [DOCX-003] ...
```

每輪最多列出 **20 個問題**，優先列出致命級。

### Step 2：排查驗證（Verify）

對掃描出的每個問題：
1. **重新讀取** DOCX 該段落/表格/頁面，確認問題真實存在
2. **交叉比對** decisions.yaml、台鐵需求文件.docx、對應的 .md 原稿
3. 標記結果：
   - `[CONFIRMED]` — 問題確實存在
   - `[FALSE_POSITIVE]` — 誤報，說明原因
   - `[DUPLICATE]` — 與其他問題重複

### Step 3：修復（Fix）

對所有 `[CONFIRMED]` 問題，依修復策略分類處理：

**策略 A — 修改 Markdown 原稿再重新轉換**（優先）：
- 適用於：內容遺漏、數字錯誤、用語不一致、placeholder、章節缺失
- 修改 `pipeline/output/*.md` 對應檔案
- 修完後執行 `bash pipeline/convert_to_docx.sh` 重新產生 DOCX
- 驗證新 DOCX 是否已修復

**策略 B — 直接修改 DOCX**：
- 適用於：純格式問題（邊界、字型、頁首頁尾、表格框線）
- 用 python-docx 寫腳本直接修改 DOCX
- 適用於無法透過 Markdown 轉換控制的格式

**策略 C — 修改轉換腳本**：
- 適用於：系統性轉換問題（如所有檔案都缺頁首、字型全錯）
- 修改 `pipeline/convert_to_docx.sh` 或相關轉換腳本/模板
- 重新執行轉換

**修復規則**：
1. 優先使用策略 A（改源頭），避免 DOCX 與 Markdown 不同步
2. 每次修復後立即驗證
3. 數字修正必須引用 decisions.yaml 的具體 key
4. 不要刪除正確內容
5. 如果主檔和分章檔都存在，優先修改主檔

修復完成後，回到 Step 1 重新掃描。

## 循環終止條件

當一輪掃描結果為：**0 致命 + 0 重要 + 一般級 ≤ 3** → 停止循環。

最多執行 **5 輪**（防止無限迴圈）。

## 注意事項

- 不要憑空捏造內容，技術描述必須基於 decisions.yaml
- 不要刪除現有的正確內容
- 格式修復（策略 B）時，保留原始內容不變，只調整格式屬性
- 修復前先確認 `pipeline/convert_to_docx.sh` 的轉換邏輯，避免重複修復被覆蓋的問題

- 每輪修復後用 `python pipeline/audit_docx_format.py` 驗證格式改善情況

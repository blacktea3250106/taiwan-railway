# 台鐵 AI 客服系統建置案
# Agent Team 與 Prompt 工程藍圖

---

## 一、核心問題：為什麼不能用一條 Prompt 打天下？

這份標案有 **13 項交付物**，橫跨 270 天，涵蓋五種完全不同的知識領域：

```
系統架構設計 ← 需要架構師思維
├── 政府採購合規 ← 需要法規專家思維
├── AI/ML 工程 ← 需要 ML 工程師思維
├── 資安合規 ← 需要資安顧問思維
├── 專案管理 ← 需要 PM 思維
└── 技術寫作 ← 需要技術作家思維
```

一條 Prompt 不可能同時扮演好六個角色。
正確做法是：**分角色、分階段、分文件、鏈式產出**。

## 三、13 項交付物的 Prompt Chain（鏈式產出流程）

### 關鍵原則：文件之間有嚴格的依賴關係

```
Phase 1 (D+30)        Phase 2 (D+80~180)           Phase 3 (D+240)        Phase 4 (D+270)
─────────────         ──────────────────           ──────────────         ──────────────
                      
#1 工作計畫書 ──┬──→ #2 系統分析報告書 ──┬──→ #9 測試報告書           #13 結案報告
               │     #3 程式設計規格 ────┤     #10 程式設計書
               │                        │     #11 原始碼光碟
               ├──→ #4 教育訓練計畫 ────┼──→ #12 訓練執行報告
               │     #5 測試計畫書 ──────┘
               │
               └──→ #6 軟硬體清單
                     #7 使用維護手冊
                     #8 上線執行計畫
```

**這代表：你不能跳過 #1 直接寫 #2。每份文件的輸出是下一份的輸入。**

## 四、完整 Prompt 設計：逐份文件

### ── Phase 1：工作計畫書（D+30）──

#### Step 1-1：建立共享知識庫（所有 Agent 共用）

這是最關鍵的一步。先把需求文件的「精華摘要」製作成一份結構化的 Context 文件，
讓所有後續 Prompt 都可以引用，避免每次都要重新餵完整文件。

```markdown
## Prompt: 建立專案知識庫

Role: 你是首席系統架構師，正在準備台鐵 AI 客服系統建置案（案號 L0215P2010U）
的投標/履約文件。

Context: [貼入前次深度解析報告全文]

Task: 請將以上解析報告精煉為一份「專案知識卡」，格式如下：

1. **專案 One-Liner**: 一句話說明本案在做什麼
2. **硬性約束清單**（紅線，不可違反）：列出所有強制/禁止事項
3. **交付物時程表**：13 項交付物 + 期限 + 付款比例
4. **技術關鍵數字**：所有 KPI 數值（85%、5秒、30埠...）
5. **硬體規格速查表**：正式 + 測試環境
6. **既有系統介接清單**：所有需要串接的外部系統
7. **待釐清項目**：14 項待確認事項

Output: 以 Markdown 格式輸出，每項控制在 3 行以內，總長度不超過 2000 字。
這份文件將作為所有後續 Prompt 的 Context 注入。
```

## 五、Agent 協作規則（Orchestration Protocol）

### 5.1 文件產出鏈（Pipeline）

```
每份文件的產出流程：

Step A: 主筆 Agent 產出初稿
         │
Step B: A2 合規官 審查 ──→ 發現違規 ──→ 回退 Step A 修正
         │
Step C: A6 技術作家 格式化（A4 直式橫書、頁次、版本號）
         │
Step D: 指揮官（你）最終審查
         │
Step E: 定稿，存入知識庫，供下游文件引用
```

### 5.2 Prompt 注入規則

```
每一條 Prompt 都必須包含以下四個區塊：

┌─────────────────────────────────────────┐
│  [Role]      → 指定是哪個 Agent         │
│  [Context]   → 注入專案知識卡 + 上游文件 │
│  [Constraint]→ 紅線清單 + 格式要求       │
│  [Task]      → 具體章節大綱 + 輸出格式   │
└─────────────────────────────────────────┘

禁止遺漏 Context 和 Constraint。
遺漏 Context = Agent 不知道台鐵的具體需求，會產出泛用內容。
遺漏 Constraint = Agent 可能寫出簡體中文、忘記資安條款、遺漏合規要求。
```

### 5.3 品質閘門（Quality Gate）

每份文件交付前，必須通過以下三道閘門：

| 閘門 | 檢查者 | 檢查項目 |
|------|--------|---------|
| **G1 合規閘門** | A2 合規官 | 無陸籍人士/無大陸品牌/無跨境傳輸/CNS 27001/繁體中文 |
| **G2 一致性閘門** | A6 技術作家 | 術語統一/編號連續/交叉引用正確/格式合規 |
| **G3 追溯性閘門** | A1 架構師 | 每項需求可追溯至需求書/每項測試可追溯至需求 |

```markdown
## Prompt: 合規閘門審查（G1）

Role: 你是政府採購合規顧問（Agent A2）。

Context: [待審查文件全文]

Checklist:
1. 搜尋全文，確認無「简体中文」字元或大陸慣用語（如「信息」應為「資訊」）
2. 搜尋全文，確認所有品牌/產品名均不在禁止清單中
3. 確認所有資料處理流程均在地端完成，無雲端/境外呼叫
4. 確認 CNS 27001 相關控制措施已被引用或實作
5. 確認人員相關條款已包含國籍限制聲明
6. 確認保密同意書/切結書流程已被納入

Output: 以表格輸出檢查結果：
| 檢查項 | 通過/不通過 | 問題位置 | 修正建議 |
```

## 六、實際執行建議：怎麼用 Claude 跑這套系統

### 方案 A：單一 Claude 對話，角色切換（最簡單）

```
適用情境：個人作業，預算有限

操作方式：
1. 開一個 Claude 長對話
2. 先用 Step 1-1 建立專案知識卡，釘選為 Context
3. 每切換一份文件時，重新宣告 Role
4. 每份文件完成後，產出摘要存入本地檔案
5. 下一份文件開始時，注入上游摘要

優點：Context 連續，不會遺漏前文
缺點：對話太長時 Context 壓縮可能丟失細節
建議：每完成一份文件，存檔並開新對話
```

### 方案 B：Claude Code + 多 Agent 並行（推薦）

```
適用情境：有 Claude Code 環境（就是你現在的環境）

操作方式：
1. 在工作目錄建立以下結構：
   台鐵/
   ├── 00_知識庫/
   │   ├── 專案知識卡.md          ← Step 1-1 產出
   │   ├── 深度解析報告.md        ← 已完成
   │   └── 合規紅線清單.md
   ├── 01_工作計畫書/
   ├── 02_系統分析報告書/
   ├── ...
   └── 13_結案報告/

2. 利用 Agent tool 並行啟動多個子 Agent：
   - Agent A1: 撰寫系統架構章節
   - Agent A3: 同時撰寫 AI 模型選型章節
   - Agent A5: 同時撰寫資安章節
   
3. 主 Agent（你的對話）負責 Orchestration：
   - 分派任務
   - 收集各 Agent 產出
   - 執行合規閘門審查
   - 整合為最終文件
```

### 方案 C：Claude Agent SDK（最強大，需開發）

```
適用情境：團隊作業，需要自動化流水線

架構：
  Orchestrator Agent
    ├── spawn(A1_Architect, task="撰寫系統分析第一章")
    ├── spawn(A3_ML_Engineer, task="撰寫系統分析第二章 AI 部分")
    ├── spawn(A5_InfoSec, task="撰寫系統分析第四章資安部分")
    ├── await_all()
    ├── spawn(A2_Compliance, task="審查合併後的系統分析報告")
    ├── spawn(A6_TechWriter, task="格式化最終版本")
    └── output("02_系統分析報告書/final.md")

優點：完全自動化，品質閘門內建
缺點：需要撰寫 Orchestrator 腳本
```

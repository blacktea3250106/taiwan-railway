## 第一章　專案總覽與執行摘要

### 1.1　專案背景與目標

國營臺灣鐵路股份有限公司（以下簡稱台鐵）為提升旅客服務品質與客服作業效率，辦理「AI 客服系統建置案」（案號 L0215P2010U），於既有客服體系中導入人工智慧技術，建置智慧語音應答（AI-IVR）、AI 虛擬客服（AI-Agent）、智慧質檢（AI-QI）等核心功能，介接既有 PSTN/WebCall/數位客服通道，全面採用開源技術堆疊，以地端高可用（HA）雙機架構部署於台鐵自有機房，實現 AI 承接第一線旅客諮詢、釋放人工座席處理複雜案件之目標。

本案專案目標及其達成情形如下：

| 目標面向 | 具體目標 | 量化指標 | 達成結果 |
|---------|---------|---------|---------|
| 服務效率 | AI 承接第一線旅客諮詢 | AI-IVR / AI-Agent 應答正確率 ≥85% | IVR 87.5%、Agent 88.0%，達標 |
| 回應速度 | VAD 偵測語音結束至系統回覆首字播出 | ≤5,000ms | P95=3,842ms，達標 |
| 通道容量 | 語音與文字通道同時服務能力 | 語音 ≥30 埠、文字 ≥30 埠 | 各 30 埠穩定運行，達標 |
| 語音辨識 | 中英文混合語音轉文字辨識 | STT 正確率 ≥85%（200 音檔，中英 4:1） | 89.0%，達標 |
| 品質管理 | 客服錄音自動轉寫、評分、異常偵測 | 15 分鐘錄音 → ≤5 分鐘完成質檢報告 | 平均 3.8 分鐘，達標 |
| 後送管理 | 後送案件自動分類與優先級判定 | 分類準確率 ≥85% | 88.0%，達標 |
| 系統穩定 | 全年不間斷營運 | 可用率 ≥99%/季、MTTR ≤4 小時 | 觀察期可用率 99.7%，達標 |

## 第二章　工作項目

### 2.1　十大子系統建置

本案 10 大子系統依六層式架構（AD-001）部署，將 AI 功能模組獨立於既有系統之外，確保零衝擊介接：

**基礎架構設計：**

- **基礎設施層：** 正式環境 2+2+2 伺服器 ＋ 測試環境 1+1 伺服器，42U 機櫃，RPS 冗餘電源
- **平台安全層：** CNS 27001 控制措施、RBAC + AD/LDAP 整合、TLS 1.3 傳輸加密、AES-256 儲存加密
- **資料管理層：** PostgreSQL 16 + TimescaleDB（關聯）、Qdrant（向量）、Redis 7（快取）
- **AI 服務層：** Llama 3.1 8B-Instruct AWQ + vLLM（LLM）、Whisper large-v3 + faster-whisper（STT）、Coqui XTTS-v2（TTS）、multilingual-e5-large（Embedding）
- **通道接入層：** PJSIP SIP/RTP stack、CRM Gateway REST/WebSocket、SMS Gateway
- **使用者層：** 值機平台（React 18 / Material UI）、管理後台、旅客端 Web/App

**GPU 資源分配：** 依 AD-002 規劃，GPU#0 專責 LLM 推論，GPU#1 負責 Embedding + 質檢 + 擴充預留；STT/TTS 以 CPU 推論（AD-003），避免與 LLM 搶奪 GPU 資源。

**HA 備援架構：** 正式環境採 Active-Active + Failover 雙機架構（AD-005），正常運行時雙機以 50/50 負載分攤流量，任一台故障時單機承載 100%。HA 機制透過 Keepalived + VIP（Virtual IP）實現，VRRP 心跳 ≤2 秒偵測、≤5 秒完成切換（#09 實測最長 3.8 秒）。資料庫採 PostgreSQL Streaming Replication 同步，確保資料零遺失。Redis 採 Sentinel 主從複寫共享 Session，確保故障切換時對話上下文不遺失。整體系統可用率設計目標 ≥99%/季。

**SIP 介接方式：** 採 SIP/RTP fork 方式介接既有語音系統（AD-007），不修改 MES Server 既有設定，AI-IVR 為新增獨立模組，被動接收語音流副本，故障時自動退場，對既有系統零衝擊。

### 3.2　Phase 1：啟動與規劃期（D+1 ~ D+30）

Phase 1 以需求釐清與規劃為核心，主要執行工作如下：

1. **專案啟動會議（D+3）：** 與台鐵資訊中心、客服中心召開專案啟動會議，確認專案組織、溝通機制及工作時程。
2. **需求訪談（D+5 ~ D+25）：** 辦理密集需求訪談共 8 場次，訪談對象涵蓋客服值機人員、督導、資訊中心系統管理員及外點後送人員。訪談重點包括現行客服作業流程、常見旅客諮詢類型、既有系統介接架構、人工客服痛點分析。
3. **既有系統介接調查（D+10 ~ D+30）：** 實地勘查台鐵機房，確認 PSTN E1 VoIP Gateway、WebCall SBC（雙備援）、MES Server、行政交換機等既有設備架構，訂定 SIP/RTP fork 介接方案（AD-007），確保不修改 MES Server 任何設定。
4. **系統分析與架構設計（D+15 ~ D+50）：** 完成六層式架構設計（AD-001）、GPU 資源分配策略（AD-002、AD-003）、HA 備援方案（AD-005）及 Docker Compose 部署架構（AD-006）。
5. **工作計畫書交付（D+30）：** 完成 #01 工作計畫書並提交台鐵審查，經一次修正後通過。

### 3.3　Phase 2：開發與整合期（D+31 ~ D+180）

Phase 2 為系統開發主要階段，主要執行工作如下：

1. **硬體採購與安裝（D+61 ~ D+100）：** 完成全部伺服器、GPU 卡、儲存設備及機櫃之採購、機房安裝及硬體驗測。設備到貨後進行 72 小時燒機測試，全數通過。
2. **AI 模型調校（D+70 ~ D+140）：**
   - LLM：Llama 3.1 8B-Instruct 模型以 AWQ 量化部署於 vLLM 引擎，建立台鐵業務知識庫並完成 RAG 管線調校。
   - STT：Whisper large-v3 搭配 faster-whisper（CTranslate2 加速版），建立台鐵專屬詞庫（站名、車種、票種等），以 hotword boosting 提升業務用語辨識率。
   - TTS：Coqui XTTS-v2 語音合成引擎部署，完成語速、語調調整及動態選單語音生成。
   - Embedding：multilingual-e5-large 模型部署於 GPU#1，支援向量資料庫 Qdrant 之語意檢索。
3. **子系統開發（D+80 ~ D+160）：** 10 大子系統依模組化架構進行迭代開發，每兩週為一個 Sprint，共完成 8 個 Sprint。各 Sprint 均執行單元測試與程式碼審查。
4. **SIP 介接整合（D+120 ~ D+160）：** 完成 AI-IVR 與既有語音系統之 SIP/RTP fork 介接，包括 E1 專線（30 B-channel）、WebCall SBC（雙備援 failover）及行政分機（類比 VoIP Gateway）三路整合測試。測試期間 MES Server 無任何設定修改，既有電話系統通話品質未受影響。
5. **HA 備援建置（D+140 ~ D+170）：** 完成 Active-Active + Failover 雙機架構部署（AD-005），包括 Keepalived VIP 管理、PostgreSQL Streaming Replication、Redis Sentinel 共享 Session、Docker Compose 自動重啟機制及 HA 切換腳本。故障切換演練三次，VRRP 偵測時間 1.1~1.3 秒，VIP 漂移完成切換時間 2.4~3.8 秒，均 ≤5 秒達標。
6. **第二期文件交付（D+180）：** 完成 #02 系統分析報告書、#03 程式設計規格報告書、#04 教育訓練計畫書、#05 測試計畫書、#06 軟硬體清單及系統架構圖、#07 軟體使用及維護手冊、#08 系統上線執行計畫書，共 7 份文件，全數通過審查。

## 第五章　系統上線執行報告

### 5.1　灰度上線策略

本案採**灰度上線**策略（10% → 30% → 100%），分三階段逐步擴大 AI 客服服務範圍，而非一次全量切換。核心考量如下：

1. **風險可控：** 每階段僅開放有限流量由 AI 處理，若出現正確率不足或系統異常，可立即將流量回歸人工客服，不影響既有服務運作。
2. **逐步驗證 KPI：** 各階段設定明確的 Go/No-Go 判定標準，確保各項 KPI 達標後方擴大範圍。
3. **避開尖峰時段：** 灰度上線期間不安排於春節疏運、連假收假等話務量暴增時段啟動新階段切換。
4. **人工客服零衝擊：** 全程維持既有人工客服系統正常運作，AI-IVR 透過 SIP/RTP fork 被動接收語音流（AD-007），不修改 MES Server 任何設定，故障時自動退場。

### 7.1　專案執行經驗

本案執行過程累積以下 5 項重要經驗：

1. **台語辨識之挑戰與因應：** 台鐵旅客來電中約有 15~20% 夾雜台語，Whisper large-v3 對台語段落之辨識率顯著低於純國語。透過建立台鐵專屬詞庫、hotword boosting 及信心度三級分流機制（≥0.85 直答、0.6~0.85 附加提示、<0.6 轉人工），有效降低台語辨識不足對旅客服務之影響。此經驗顯示：針對特定業務場景之專屬詞庫建設，對 STT 辨識率提升效果顯著。

2. **GPU 資源分工策略：** 採 GPU#0 專責 LLM、GPU#1 負責 Embedding + 質檢之雙卡分工策略（AD-002），有效避免資源搶奪。STT/TTS 以 CPU 推論（AD-003）進一步降低 GPU 壓力。此分工策略使單卡 L40S 即可穩定支撐 30 路語音 + 30 路文字之併發需求，為後續擴充保留餘裕。

3. **SIP 零侵入介接：** 採用 SIP/RTP fork 方式（AD-007）介接既有語音系統，全程不修改 MES Server 設定，AI-IVR 為新增獨立模組。此方式大幅降低對既有系統之風險，亦使灰度上線之回滾方案極為單純——停止 AI-IVR 服務即可，不需還原任何系統設定。

4. **密集需求訪談之效益：** Phase 1 安排 8 場次密集需求訪談，涵蓋第一線值機人員至管理層，提前釐清業務邏輯與邊界案例。此投入有效降低 Phase 2 開發階段之需求變更頻率，全案僅 2 次需求變更請求（CR），且均為非關鍵性調整。

5. **灰度上線策略之驗證：** 三階段灰度上線（10% → 30% → 100%）成功在真實話務環境中逐步驗證系統穩定性與 KPI 達標狀況，各階段 Go/No-Go 機制確保風險可控。此策略特別適用於客服場景——旅客無感知地由人工切換至 AI 服務，服務品質不受影響。

### 7.2　短期建議事項

1. **知識庫定期更新機制：** 建議台鐵指派專責人員（建議 1~2 人），每月至少一次更新 AI 知識庫內容，包括新增路線、票價異動、臨時營運變更等資訊，以維持 AI 回答之正確性與即時性。系統已內建知識庫版本管理功能，可回溯任一歷史版本。

2. **客服回饋持續校正：** 系統已建置客服標記錯誤答案 → 知識庫更新 → 重新 Embedding → 回答修正之閉環流程（FR-AGT-008）。建議值機人員於日常作業中積極標記 AI 錯誤回答，每月彙整分析，持續提升應答品質。

3. **報表客製需求收集：** 保固期內可免費客製 ≥10 個報表（FR-RPT-003）。建議台鐵於保固期前 6 個月內彙整報表需求，以充分利用此權益。

### 7.3　中長期建議事項

1. **台語 ASR 模型強化：** 隨著台語語音辨識技術持續進步，建議於保固期滿後評估引入專為台語設計之 ASR 模型（如 Formosa Foundation Model 系列），進一步提升台語混合語音之辨識率。

2. **多模態服務擴展：** 目前系統以語音與文字通道為主，未來可評估擴展至圖片辨識（如旅客拍攝時刻表照片查詢）、視訊客服等多模態服務，進一步提升旅客體驗。

3. **AI 質檢自動化程度提升：** 目前 AI-QI 系統已實現自動轉寫、規則評分與異常偵測。未來可進一步發展情緒分析之細緻度（如區分不滿、焦慮、憤怒等情緒層級），並結合客訴預測模型，於通話進行中即時提示值機人員。

## 第一章　系統摘述

### 1.1 系統設計理念與整體定位

本系統以「AI 輔助而非取代人工」為核心設計理念，透過智慧語音應答（AI-IVR）、AI 虛擬客服（AI-Agent）及智慧質檢（AI-QI）三大核心引擎，介接台鐵既有 PSTN/WebCall/數位客服通道，提升旅客服務品質與客服作業效率。系統全面採用開源技術堆疊，以地端高可用（HA）雙機架構部署於台鐵機房，所有資料不離開機房，確保資訊安全與旅客個資保護。

系統設計遵循以下原則：

- **零衝擊介接**：AI-IVR 以 SIP/RTP fork 方式被動接收語音流副本，不修改 MES Server 既有設定，不影響現有電話系統運作（AD-007）。
- **開源永續使用**：所有 AI 模型與軟體元件採用開源授權，軟體授權到期後可繼續使用到期前最後更新版本（AD-008）。
- **降低維運門檻**：部署採 Docker Compose，不使用 Kubernetes，正式環境 6~8 台伺服器規模無需容器叢集的複雜度（AD-006）。
- **異常即切人工**：系統在任何環節發生異常時，均能立即將通話或對話切換至人工客服，確保旅客服務不中斷。

### 1.2 六層式系統架構

本案採六層式架構設計（AD-001），將 AI 功能模組獨立於既有系統之外，各層職責明確，層間以標準化介面通訊，任一層的升級或替換均不影響其他層級：

#### 第一層：基礎設施層

承載所有運算與儲存資源。正式環境配置 AI-IVR 伺服器 2 台（16 核心 ≥2.2GHz×2，DDR5 64GB+，960GB SSD×3 RAID-5）、AI 伺服器 2 台（16 核心 ≥3.0GHz×2，DDR5 128GB+，960GB SSD×3 RAID-5，NVIDIA L40S 48GB×2）、管理/資安監控伺服器 2 台（16 核心 ≥3.0GHz×2，DDR5 64GB+）、SMS 簡訊收發系統 2 台，統一收納於 42U 機櫃。測試環境另設 AI-IVR 測試伺服器 1 台、AI 測試伺服器 1 台（L40S×1）、SMS 測試 1 台。所有設備採冗餘電源（RPS）設計，伺服器主機均具備 BMC 遠端管理功能。

#### 第二層：平台安全層

提供全系統的安全防護基礎。存取控制採 RBAC 角色型權限模型，整合台鐵既有 AD（Active Directory）進行帳號驗證（LDAP 協定）。傳輸層強制 TLS 1.3 加密，儲存層對旅客個資採用 AES-256 加密。所有操作日誌保留 ≥6 個月，個資存取日誌獨立管控。本層依 CNS 27001 資訊安全管理標準設計控制措施，詳見§4.3 資訊安全需求。

#### 第三層：資料管理層

統一管理結構化與非結構化資料。關聯式資料庫採用 PostgreSQL 16 搭配 TimescaleDB 擴展處理時序資料（日誌、監控指標）；向量資料庫採用 Qdrant（Rust 語言開發，HNSW 索引，Top-K 查詢延遲 <50ms），搭配 multilingual-e5-large 中文 Embedding 模型實現語意檢索；快取層採用 Redis 7 處理 Session 管理與即時對話上下文存取。

#### 第四層：AI 服務層

本系統核心運算層，涵蓋四大 AI 引擎：

| AI 引擎 | 技術選型 | 部署方式 | 核心功能 |
|---------|---------|---------|---------|
| LLM 大型語言模型 | Llama 3.1 8B-Instruct AWQ 量化 + vLLM | GPU#0 專責推論（AD-002） | 語意理解、問答生成、多輪對話、後送案件分析 |
| STT 語音識別 | Whisper large-v3 + faster-whisper | CPU 推論（AD-003） | 即時中英文語音轉文字、VAD 斷點偵測 |
| TTS 語音合成 | Coqui XTTS-v2（備案：MeloTTS / Piper） | CPU 推論（AD-003） | 中英文即時語音合成、動態選單生成 |
| Embedding + QI | multilingual-e5-large + 情緒分析模型 | GPU#1（AD-002） | 向量化語意檢索、質檢語調情緒分析 |

GPU 資源分配策略（AD-002）：GPU#0 專責 LLM 推論，單一 vLLM 引擎以 continuous batching + PagedAttention 服務 ≤30 路並行推論（Llama 3.1 8B AWQ 量化模型權重 ~8GB，KV Cache 動態分配，峰值 VRAM ~42GB，單卡 L40S 48GB 足以承載）；GPU#1 負責 Embedding 計算、質檢批次分析及未來擴充預留，避免即時推論與批次任務搶奪資源，確保語音回應 ≤5,000ms 之 SLA 目標。STT 與 TTS 選擇以 CPU 推論（AD-003），因串流語音處理屬低延遲任務，CPU 的每核心延遲優於 GPU 批次吞吐特性，30 路 faster-whisper 約使用 9 個 CPU 核心，AI-IVR 伺服器配備 32 核心足以承載。

#### 第五層：通道接入層

統一管理所有外部通訊通道。語音通道透過 PJSIP（pjsua2 Python binding）實現 SIP/RTP 協定處理，支援 E1 專線（現有 1 條，30 B-channel）、WebCall SBC（雙備援 failover）及行政交換機（類比 → VoIP Gateway → G.711 編碼）三種語音來源。文字通道透過 CRM Gateway 以 REST/WebSocket API 接入數位客服訊息。SMS 通道透過雙備援簡訊收發系統處理旅客簡訊服務。內部服務間通訊採 gRPC 協定（二進制序列化 + HTTP/2 多工），相較 REST 快 2~5 倍，確保 AI-IVR 至 LLM 推論的延遲預算。對外 REST API 依國發會共通性應用程式介面規範設計。

#### 第六層：使用者層

提供各角色專屬操作介面，前端框架採用 React 18 + Next.js + Material UI（MUI），支援 RWD 響應式設計，相容 Edge/Chrome/Firefox/Safari 瀏覽器。各介面功能於 1.4 節使用者角色說明中詳述。

#### 實體部署架構

正式環境以 Active-Active + Failover 架構部署（AD-005），正常運行時雙機以 50/50 負載分攤流量，任一台故障時單機承載 100%。HA 機制透過 Keepalived + VIP（Virtual IP）實現，VRRP 心跳 ≤2 秒偵測、≤5 秒完成切換。Session 資料存放於 Redis 共享，確保故障切換時對話上下文不遺失。

| 實體伺服器 | 正式環境 | 測試環境 | 運行服務 |
|-----------|---------|---------|---------|
| AI-IVR 伺服器 | 2 台（Active-Active） | 1 台 | SIP/RTP 處理、STT（CPU）、TTS（CPU）、對話管理 |
| AI 伺服器 | 2 台（Active-Active） | 1 台 | LLM 推論（GPU#0）、Embedding/QI（GPU#1）、RAG Pipeline |
| 管理/資安監控伺服器 | 2 台（Active-Active） | — | PostgreSQL、Qdrant、Redis、Prometheus、Grafana、後台服務 |
| SMS 簡訊收發系統 | 2 台（雙備援） | 1 台 | 4G LTE+ 簡訊收發 |

網路拓撲採 VLAN 隔離設計，區分管理 VLAN、服務 VLAN 與資料 VLAN，詳見 #06 軟硬體清單及系統架構圖。

### 1.4 使用者角色與系統互動（使用案例分析）

本節以使用案例（Use Case）觀點描述各類使用者與系統之互動情境。本系統服務五類使用者，各角色之系統權限與功能存取範圍如下：

#### （一）旅客

旅客為本系統之最終服務對象，透過三種管道與系統互動：

- **電話語音（PSTN/WebCall）**：撥打客服電話或透過網頁電話接入 AI-IVR，以自然語言對話查詢時刻、票價、乘車資訊等，系統以語音回覆（FR-IVR-002）。AI 無法解決時自動轉接真人客服（FR-IVR-004），轉接時系統同步傳送對話摘要予值機員，旅客無需重複說明。通話結束前提供滿意度調查（FR-IVR-006）。
- **數位文字客服（Web/App）**：透過網頁或行動 App 內嵌之對話視窗，以文字方式與 AI-Agent 互動（FR-AGT-003）。對話視窗設有熱門選單與宣導推播專區，由台鐵後台自行設定調整。AI 回應自動標註知識來源出處（FR-AGT-004），供旅客自行驗證。
- **SMS 簡訊**：旅客可透過簡訊查詢服務，系統以 AI 自動回覆，同步通知客服人員（FR-AGT-007）。

#### （二）值機員（25 席）

客服第一線人員，使用值機平台（WS）處理旅客來電與文字諮詢：

- 統一操作介面整合接聽、保留、轉接、掛斷等通話控制功能（FR-WS-001）
- 接聽時系統即時推送 AI 建議回覆與相關知識庫內容，輔助快速應答（FR-WS-002）
- 通話/對話結束後自動生成對話摘要，支援建立與追蹤後送工單（FR-WS-003）
- 支援簽入/簽出、就緒/忙碌/休息等狀態切換，統計各狀態時長（FR-WS-005）

目前台鐵客服同時在線約 15 人（60% 在線率，排班制），AI 系統預期可獨立處理 60~70% 的簡單查詢（時刻、票價、車站資訊），使值機員得以專注處理複雜案件與後送事項。

#### （三）督導（2 席）

客服督導人員，負責即時服務品質監控：

- 即時監看所有座席狀態、監聽通話、密語指導、必要時強制介入（FR-WS-004）
- 查看即時服務指標儀表板，監控 AI 轉人工比例，若超過閾值系統自動告警
- 接收智慧質檢系統之異常通報（FR-QI-006），針對標記異常之通話進行後續處理

#### （四）系統管理者

數位發展處 IT 人員，透過管理後台（ADM）執行系統維運：

- 系統參數設定、帳號權限管理（整合 AD/LDAP）（FR-ADM-001、FR-ADM-002）
- 系統監控儀表板（Prometheus + Grafana），即時掌握 CPU/GPU/記憶體/服務狀態（FR-ADM-003）
- 日誌稽核查詢、備份回復操作（FR-ADM-004、FR-ADM-005）
- 知識庫內容維護——新增、修改、刪除知識條目及版本管理（FR-LLM-005）

管理後台設計以「台鐵 IT 人員」為目標讀者，操作流程力求直覺，搭配維護手冊之 step-by-step 操作指引，確保保固期結束後台鐵可自行維運。

#### （五）外點後送人員（225 人，同時在線約 23 人）

各車站與外點單位之後送案件處理人員，使用既有後送案件系統。本案 AI 系統透過 API 將自動分類、優先級判定及情感分析結果推送至後送系統（FR-LLM-008），不修改現有後送系統操作介面，225 位外點人員之作業流程維持不變。

### 1.6 效益分析

#### 量化效益

| 效益指標 | 現況基線 | 導入 AI 後目標 | 改善幅度 |
|---------|---------|---------------|---------|
| 30 秒接聽率 | 64% | ≥85% | +21 百分點 |
| 整體接聽率 | 83% | ≥95%（AI 承接後溢流率大幅降低） | +12 百分點 |
| 轉接率 | 23% | ≤10%（AI 獨立解決 60~70% 簡單查詢） | -13 百分點 |
| 數位客服配對率 | 93% | ≥98%（AI-Agent 無需等待排隊配對） | +5 百分點 |
| 質檢覆蓋率 | 人工抽檢（推估 <5%） | 100%（AI 全量自動質檢） | 全面提升 |
| 質檢時效 | 人工聽取 1:1 時間 | 15 分鐘錄音 ≤5 分鐘完成分析 | 效率提升 3 倍 |

#### 質化效益

- **服務品質一致性**：AI 回覆內容基於知識庫與即時 API 查詢，避免人工客服因經驗差異產生回覆品質落差。
- **24 小時服務延伸**：AI-IVR 與 AI-Agent 可在非值班時段（夜間、假日）提供基本查詢服務，旅客無需等到上班時間。
- **知識累積與傳承**：所有對話紀錄、質檢結果、後送案件分析均系統化留存，新進客服人員可透過 AI 知識輔助快速上手。
- **管理決策支援**：多維度報表與儀表板（FR-RPT-001~003）提供話務趨勢、旅客關注議題、客服績效等數據，輔助管理層決策。

## 2.1　智慧語音應答系統（AI-IVR）

本節定義 AI-IVR 子系統之七項功能需求。AI-IVR 以 SIP/RTP fork 方式被動接收語音流（AD-007），不修改既有 MES Server 任何設定，確保導入過程對現行話務系統零衝擊。語音處理鏈路依序為：RTP 收音 → STT 轉寫 → LLM 推論 → TTS 合成 → RTP 回送，全程於地端 AI-IVR 伺服器（16 核心 ≥2.2GHz ×2、DDR5 64GB+）完成，無任何資料離開台鐵機房。

## 2.2　語音識別系統（STT）

STT 子系統採用 OpenAI Whisper large-v3 模型，以 faster-whisper（CTranslate2 加速版）部署，運行於 AI-IVR 伺服器之 CPU（AD-003）。CPU 推論策略之設計考量：串流語音辨識為「每收到一小段語音即需立即處理」的低延遲任務，CPU 每核心延遲優於 GPU（GPU 強項在批次吞吐量）；30 路 × faster-whisper ≈ 9 CPU 核心，AI-IVR 伺服器 32 核心綽綽有餘，同時將 GPU 資源完整保留予 LLM 推論。

- RTP 語音串流（G.711 PCM，8kHz / 16kHz）

**輸出：**
- 辨識文字（含時間戳記、信心度分數）
- 語言標籤（zh-TW / en）

**處理邏輯：**
1. 基於 VAD（FR-STT-003）切割語音 chunk，每 chunk 送入 faster-whisper 辨識
2. 建置台鐵專屬詞庫（hotword boosting）：站名（新左營、板橋）、車種（自強號、太魯閣、普悠瑪、區間車）、票種（去回票、電子票證）等，提升領域專用辭彙辨識率
3. 類比線路（行政交換機）語音品質較低，VAD 參數需針對背景雜訊調校；G.711 編碼在類比轉數位場景損失極小（64kbps PCM），辨識率預估影響 ≤2%
4. faster-whisper RTF（Real-Time Factor）約 0.3x，即 1 秒語音僅需 0.3 秒處理，滿足串流即時性要求

**例外流程：**
- 語音品質極差（背景噪音過大、訊號中斷）：辨識信心度低於閾值時，回傳低信心標記，FR-IVR-002 據此觸發確認話術
- faster-whisper 程序異常崩潰：HA 架構下由備機 STT 實例接手（AD-005），單機內亦配置多個 worker 互為備援

## 2.3　語音合成系統（TTS）

TTS 子系統採用 Coqui XTTS-v2 模型，運行於 AI-IVR 伺服器 CPU（AD-003），每路語音合成約佔用 0.2 CPU 核心，30 路並行僅需 6 核心，與 STT 合計 15 核心，32 核心之 AI-IVR 伺服器仍有充足餘裕。XTTS-v2 模型以 MPL 2.0 授權開源釋出，原始碼與模型權重可永久使用，符合契約§三(八)「授權到期後可繼續使用」之要求。鑑於 Coqui AI 公司已於 2023 年底停業，本案同步驗證備案方案 MeloTTS 及 Piper，確保長期維護無虞（詳見第一章 §1.5 技術選型風險分析）。

## 2.4　大型語言模型（LLM）

本節定義 LLM 子系統之八項功能需求。LLM 為本案 AI 核心引擎，同時服務 AI-IVR（語音通道）與 AI-Agent（文字通道）兩大場景。推論引擎採 vLLM 0.5+ 搭載 Llama 3.1 8B-Instruct 繁中微調版、AWQ 4-bit 量化（AD-004），運行於 GPU#0（NVIDIA L40S 48GB）專責推論（AD-002），量化後模型權重 ~8GB 常駐 VRAM，KV Cache 依並行數動態分配，峰值 VRAM ~42GB，單卡 L40S 48GB 足以承載 ≤30 路並行推論。vLLM 之 PagedAttention 技術使 KV Cache 利用率最大化，continuous batching 機制確保高併發下請求無需排隊。所有推論均於台鐵機房地端完成，無任何資料離開機房邊界。

GPU 資源分配策略（AD-002）：GPU#0 專責即時 LLM 推論（AI-IVR + AI-Agent），GPU#1 負責 Embedding 向量化、質檢分析（AI-QI）及未來擴充預留，兩卡工作負載隔離，避免批次任務與即時推論搶奪資源，確保 ≤5,000ms 回應時間 SLA。

## 2.5　AI 虛擬客服（AI-Agent）

本節定義 AI-Agent 子系統之八項功能需求。AI-Agent 為面向旅客之數位文字客服，透過 Web、App 及 SMS 三種通道提供服務。AI-Agent 之語意理解與問答生成能力來自 LLM 引擎（FR-LLM-002），知識檢索來自 Qdrant 向量資料庫，兩者結合實現 RAG（Retrieval-Augmented Generation）架構。文字通道之回應時間要求為旅客送出訊息至系統回覆首段文字呈現 ≤5,000ms（agent_response），應答正確率 ≥85%（agent_accuracy）。

## 2.6　智慧質檢系統（AI-QI）

本節定義智慧質檢子系統之七項功能需求。AI-QI 以批次模式處理 MES 錄音檔，執行語音轉寫、規則比對、情緒分析與異常偵測，產出質檢報告供督導人員審閱。質檢 STT 轉寫複用 Whisper large-v3 + faster-whisper 引擎（AD-003），運行於 AI-IVR 伺服器 CPU；情緒分析模型運行於 GPU#1（AD-002），與即時 LLM 推論（GPU#0）隔離，避免批次任務與即時服務搶奪資源。

質檢系統使用之客服錄音屬於台鐵既有「客服品質管理」目的範圍，轉寫文字於產出後即套用正則表達式去識別化（身分證→A******789、電話→0912***789），去識別化後之文字方可用於統計分析與質檢評分，原始語音與去識別化文字分離存放，存取權限限制為督導以上角色。

**描述：** 支援 WAV、MP3、OGG、FLAC 等常見音檔格式匯入，提供手動上傳與自動排程兩種模式。自動排程模式下，系統定時掃描 MES 錄音儲存目錄，自動匯入新增錄音並排入質檢佇列。15 分鐘錄音檔須於 5 分鐘內完成轉寫、規則比對、評分與報告產出。

- 台鐵專屬詞庫（同 FR-STT-001 §2.2 所述，詳見第二章(A)）

**輸出：**
- 逐字轉寫文字（含時間戳記，精確至 0.1 秒）
- 說話人分離標記（客服 / 旅客）
- 去識別化後之轉寫文字（供後續質檢分析使用）

**處理邏輯：**
1. 質檢場景 beam size 設定為 5（即時場景為 1），以準確率換取容許之批次延遲
2. 說話人分離（Speaker Diarization）：利用 pyannote-audio 開源模型，區分客服端與旅客端語音
3. 轉寫完成後立即執行去識別化：正則表達式遮罩個資欄位（身分證、電話、信用卡號），去識別化文字與原始轉寫分離儲存
4. 轉寫文字以 JSON 格式儲存至 PostgreSQL，關聯通話 ID、座席 ID、日期時間，保留 ≥6 個月（符合 log_retention 要求）

**例外流程：**
- 語音品質極差（訊噪比 <10dB）：標記為「轉寫品質不佳」，仍完成轉寫但質檢報告加註警示
- 錄音時長超過 60 分鐘：自動分段處理（每段 15 分鐘），各段獨立轉寫後合併

### FR-QI-004　靜音與插話偵測

| 欄位 | 內容 |
|------|------|
| **編號** | FR-QI-004 |
| **名稱** | 靜音與插話偵測 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(六) |
| **對應編號** | TC-QI-004 / M-QI-04 / P-QI-04 |
| **KPI 關聯** | — |

**描述：** 自動分析錄音中的靜音段與雙方同時說話（插話）事件，作為質檢評分之客觀依據。

**輸入：**
- 原始音檔（自 FR-QI-001）
- 說話人分離結果（自 FR-QI-002）

**輸出：**
- 靜音事件清單（起止時間、持續秒數、歸屬方）
- 插話事件清單（起止時間、持續秒數）
- 靜音率與插話率統計值

**處理邏輯：**
1. 靜音偵測：以 VAD 分析音檔，連續靜音 ≥3 秒標記為靜音事件（閾值可於管理後台調整）
2. 靜音歸屬：結合說話人分離結果，判定靜音發生時「應由誰說話」（旅客等待客服回覆 = 客服端靜音）
3. 插話偵測：雙通道（客服/旅客）同時存在語音活動且持續 ≥0.5 秒，標記為插話事件
4. 統計值計算：靜音率 = 客服端靜音總時長 ÷ 通話總時長；插話率 = 插話總次數 ÷ 通話總時長（分鐘）
5. 統計值可作為 FR-QI-003 自訂規則之輸入（例如「客服端靜音率 >15% 扣 N 分」）

**例外流程：**
- 單通道錄音（無法分離客服/旅客）：僅偵測整體靜音，不進行歸屬分析，報告標註「單通道限制」


### FR-QI-005　語調與情緒分析

| 欄位 | 內容 |
|------|------|

| **對應編號** | TC-QI-005 / M-QI-05 / P-QI-05 / AD-002 |
| **KPI 關聯** | — |

**描述：** 分析客服人員與旅客之語調情緒，識別負面情緒（憤怒、焦躁、不耐）並標記異常通話。情緒分析模型運行於 GPU#1（AD-002），以批次推論方式處理，不與 GPU#0 之即時 LLM 推論競爭資源。

**輸入：**

**輸出：**

**處理邏輯：**
1. 聲學情緒模型：採用開源 Speech Emotion Recognition（SER）模型，以梅爾頻譜與韻律特徵為輸入，每 5 秒窗口產出一個情緒標籤
2. 文本情緒輔助：以 LLM（GPU#1）分析轉寫文字之情緒傾向，與聲學結果加權融合（聲學 0.6 + 文本 0.4）
3. 情緒曲線：將逐段情緒分數繪製為時間軸曲線，於質檢報告中以圖表呈現
4. 異常標記規則：旅客連續 3 個以上窗口（≥15 秒）為負面情緒，或客服出現任何負面情緒窗口，即標記為異常通話
5. 異常通話自動觸發 FR-QI-006 警報機制

**例外流程：**
- GPU#1 負載過高（使用率 >90%）：情緒分析任務排入延遲佇列，優先保障 Embedding 向量化之即時需求
- 語音品質不佳導致聲學特徵不可靠：降級為純文本情緒分析（權重調整為文本 1.0）

## 2.8　SMS 簡訊服務

SMS 簡訊服務之核心功能需求已於第二章(B) FR-AGT-007 定義（雙備援收發裝置、AI 自動回覆、同步通知客服人員、紀錄保留 ≥5 年），本節補充 SMS 子系統之硬體架構與資料保存策略。

### SMS 硬體架構

正式環境配置 2 台 SMS 簡訊收發系統（Active-Active 雙備援，AD-005），測試環境配置 1 台。每台設備內建 256GB+ 儲存空間與 4G LTE+ 通訊模組。LTE 模組之品牌與產地須符合合規紅線要求，排除大陸地區品牌與大陸地區製造之通訊晶片組（含華為海思、中興、展訊等），優先選用 Sierra Wireless、Telit、u-blox 等歐美品牌模組。

### SMS 資料保存策略

| 項目 | 規格 |
|------|------|
| 保存期限 | ≥5 年（需求書§三(五)） |
| 保存內容 | 簡訊收發紀錄（發送/接收時間、收發號碼去識別化、簡訊內容、AI 自動回覆 / 人工回覆標記） |
| 儲存位置 | PostgreSQL 16 資料庫（台鐵機房地端） |
| 資料量估算 | 月均 2,500~7,000 則 × 60 個月（5 年）= 150,000~420,000 筆，每筆含 metadata 約 2KB，總量 ≤1GB |
| 備份策略 | 每日增量備份 + 每週全量備份，備份檔異機儲存（管理/資安監控伺服器） |
| 查詢功能 | 管理後台提供依時間區間、號碼（去識別化）、關鍵詞篩選之查詢介面 |

SMS 自動回覆之同步通知機制：AI-Agent 產生自動回覆後，透過 WebSocket 即時推播至值機平台（FR-WS-001），客服人員可於座席介面查看 AI 回覆內容與旅客資訊，判斷是否需人工介入接手後續對話。自動回覆紀錄同時寫入資料庫，供稽核追蹤。


## 2.9　知識庫管理

知識庫管理之核心功能需求已於第二章(B) FR-LLM-005（知識庫維護與自動學習）及 FR-AGT-002（向量資料庫整合與版本管理）定義，本節補充知識庫之資料結構、覆蓋率管理與生命週期策略。

### 知識庫資料結構

知識庫以 Qdrant 向量資料庫（AD-008）為核心，搭配 intfloat/multilingual-e5-large 中文 embedding 模型進行向量化。知識文件以 chunk 為單位儲存，每個 chunk 攜帶以下 metadata：

| 欄位 | 說明 |
|------|------|
| doc_id | 文件唯一識別碼 |
| doc_title | 文件標題 |
| doc_version | 文件版本號 |
| chunk_index | 該 chunk 在文件中的序號 |
| source_category | 來源分類（公告/規章/FAQ/時刻表/票務規定/其他） |
| publish_date | 文件發布日期 |
| expire_date | 文件有效期限（可為空，表示永久有效） |
| last_updated | 最後更新時間 |

此 metadata 結構支援 FR-AGT-004 RAG Citation 之引用來源標註，LLM 回應時可明確標示「本回覆參考自：{doc_title} 第 {chunk_index} 段，版本 {doc_version}」。

### 知識庫覆蓋率管理

為確保 AI 回答品質達 ≥85% 正確率（ivr_accuracy / agent_accuracy），知識庫須涵蓋台鐵客服常見問題之絕大多數場景。系統提供以下覆蓋率管理機制：

1. **未命中查詢追蹤：** RAG 檢索 Top-K 相似度分數均 <0.5 之旅客查詢，自動收集至「未命中清單」（FR-RPT-001 知識點分析報表），每週產出報告供知識庫管理者審閱
2. **覆蓋率指標：** 系統計算「有效命中率 = RAG 檢索相似度 ≥0.5 之查詢數 ÷ 總查詢數」，於管理後台儀表板即時顯示
3. **降級策略：** 當知識庫無法提供足夠參考資訊時（信心度 <0.6），系統自動觸發轉人工機制（詳見第二章(B) FR-LLM-002 信心度三層策略），避免以不完整資訊誤導旅客
4. **增量更新流程：** 管理者於管理後台上傳新文件後，系統自動 chunking → embedding → 寫入 Qdrant，全程無需停機，更新即時生效

### 知識庫資料來源與初始匯入

台鐵知識庫初始資料來源預估包含：

- 旅客服務常見問答（FAQ）
- 各級列車時刻表與票價資訊（透過 API 即時查詢，不靜態儲存）
- 旅客運送契約、營運規章
- 各站設施與服務資訊
- 優惠票種與購票規定
- 遺失物處理流程
- 行李託運規定
- 無障礙服務資訊

初始匯入時由台鐵提供上述文件之電子檔（PDF / Word / HTML），系統以自動化 pipeline 執行文件解析 → 清洗 → chunking → embedding → 寫入 Qdrant，預估知識庫規模 10,000~50,000 筆 chunk。

## 3.1　內部介面

內部介面指 AI 客服系統各子模組之間的通訊鏈路，部署於同一 Docker Compose 叢集內（AD-006），透過容器內部網路通訊，不經過外部網路。

### 3.1.1　AI-IVR ↔ LLM 推論介面（gRPC）

| 項目 | 規格 |
|------|------|
| **協定** | gRPC / Protocol Buffers over HTTP/2 |
| **端對端延遲預算** | ≤2,500ms（含序列化與反序列化） |
| **呼叫模式** | Unary RPC（單次請求—回應） + Server-side Streaming（逐 token 串流） |
| **相關模組** | M-IVR-01（語音通道）→ M-LLM-01（GPU 推論引擎） |
| **GPU 分配** | GPU#0 專責 LLM 推論（AD-002），vLLM 引擎 continuous batching |
| **超時處理** | 推論逾 2,500ms 觸發 deadline exceeded，AI-IVR 播放候補話術「請稍候」並重試一次；二次失敗轉人工 |

**延遲分配說明：** 需求書要求旅客說完至系統回覆首字 ≤5,000ms（ivr_response）。整段延遲鏈路分配如下：VAD 偵測尾端靜音 ~500ms → STT 轉寫 ~800ms → gRPC 呼叫含 LLM 推論 ≤2,500ms → TTS 首包合成 ~300ms → RTP 封包發送 ~100ms，合計 ~4,200ms，保留 ~800ms 緩衝。選用 gRPC 而非 REST 的原因正是其二進制序列化與 HTTP/2 多工特性，較 REST/JSON 快 2~5 倍，確保推論呼叫不成為瓶頸。

### 3.1.2　AI-Agent ↔ LLM 推論介面（gRPC）

| 項目 | 規格 |
|------|------|
| **協定** | gRPC / Protocol Buffers over HTTP/2 |
| **端對端延遲預算** | ≤3,500ms（文字通道無 STT/TTS 環節，延遲預算較寬裕） |

文字通道延遲鏈路：訊息接收 ~50ms → 向量檢索（Qdrant Top-K）≤200ms → gRPC LLM 推論 ≤3,500ms → 回應組裝與 Citation 抽取 ~200ms，合計 ~3,950ms，滿足 agent_response ≤5,000ms。

## 4.1　效能需求

### 4.1.1　端到端延遲預算

系統最核心的效能指標為旅客感知延遲。需求書要求語音通道（AI-IVR）旅客說完至系統回覆首字 ≤5,000ms（ivr_response，需求書§三(一)），文字通道（AI-Agent）旅客送出訊息至系統回覆首段文字 ≤5,000ms（agent_response，需求書§三(五)）。

為確保端到端延遲可控，系統將 5,000ms 預算分解至各處理階段，每一段皆設有獨立上限與超時機制：

**語音通道（AI-IVR）延遲分解：**

| 處理階段 | 元件 | 延遲預算 | 說明 |
|---------|------|---------|------|
| VAD 尾端偵測 | M-STT-03 | ~500ms | 偵測旅客停止說話，觸發後續處理 |
| STT 語音轉文字 | M-STT-01（faster-whisper） | ≤800ms | CPU 推論（AD-003），chunk-based streaming |
| LLM 推論 + RAG 檢索 | M-LLM-01（vLLM） | ≤2,500ms | GPU#0 專責推論（AD-002），含 Qdrant Top-K 檢索 <50ms |
| TTS 首包合成 | M-TTS-01（XTTS-v2） | ≤700ms | CPU 推論（AD-003），串流輸出首包 ≤300ms，含封包緩衝 |
| RTP 封包發送 | M-IVR-01（PJSIP） | ~100ms | 網路傳輸與 jitter buffer |
| **合計** | | **~4,600ms** | **保留 ~400ms 安全緩衝** |

**文字通道（AI-Agent）延遲分解：**

| 處理階段 | 元件 | 延遲預算 | 說明 |
|---------|------|---------|------|
| 訊息接收與前處理 | M-AGT-01 | ~100ms | WebSocket 接收、敏感詞初篩 |
| RAG 向量檢索 | M-AGT-02（Qdrant + multilingual-e5-large） | ≤200ms | HNSW 索引 Top-K 查詢 <50ms，含 embedding 計算 |
| LLM 推論 | M-LLM-01（vLLM） | ≤2,500ms | 與語音通道共用 GPU#0，continuous batching 排程 |
| 後處理與引用標註 | M-AGT-03 | ~200ms | RAG Citation 來源比對、格式化回應 |
| **合計** | | **~3,000ms** | **保留 ~2,000ms 緩衝供個性化處理與歷史查詢** |

**超時降級機制：** 任一階段超過預算時，系統不等待而直接進入降級流程：

- STT 逾 800ms → 截斷當前語音片段，以已辨識部分文字送入 LLM
- LLM 推論逾 2,500ms → 語音通道播放「請稍候」候補話術並重試一次；二次失敗轉人工（FR-IVR-004）
- TTS 逾 700ms → 切換至預錄候補語音（如「系統處理中，請稍候」）
- 文字通道 LLM 逾 3,000ms → 先回傳「正在查詢中...」佔位訊息，後續以串流方式補全回覆

### 4.1.2　正確率指標

| 指標 | 目標值 | 計算公式 | 驗收方式 | 來源 |
|------|-------|---------|---------|------|
| IVR 應答正確率 | ≥85% | (完全正確×1 + 部分正確×0.5) ÷ 總次數 ×100% | 200 題，台鐵提供 | 需求書§三(一) |
| STT 辨識率 | ≥85% | 100% − (錯誤或缺漏字數 / 總字數) ×100% | 200 音檔（中英 4:1，≤150s） | 需求書§三(二) |
| Agent 應答正確率 | ≥85% | 同 IVR | 200 題，台鐵提供 | 需求書§三(五) |
| 後送案件分類正確率 | ≥85% | (AI 正確分類數 / 總案件數) ×100% | 200 筆真實後送案件 | 需求書§三(四) |

正確率未達標時的應對策略：系統採三層信心度閾值設計（詳見第二章(B) FR-LLM-002 信心度三層策略），信心度 <0.6 時主動轉接人工客服，避免低信心回答影響旅客體驗。涉及票價、班次時刻等關鍵資訊，一律從訂票紀錄查詢系統與時刻表查詢系統 API 即時取得，不依賴 LLM 記憶性知識。

### 4.1.3　併發容量

| 通道類型 | 最低併發埠數 | 設計目標 | 來源 |
|---------|------------|---------|------|
| 語音（PSTN E1 + WebCall + 行政分機） | ≥30 埠 | 支援 2 條 E1（60 B-channel） | 需求書§三(一) |
| 數位文字（Web + App） | ≥30 埠 | 支援突發 50 埠 | 需求書§三(五) |

語音 30 埠與文字 30 埠共用同一組 GPU 推論資源。單一 vLLM 引擎以 continuous batching + PagedAttention 服務並行推論，模型權重 ~8GB 常駐 VRAM，KV Cache 依並行數動態分配，峰值 ~42GB；理論吞吐量 ~2,800 token/s（單卡），HA 雙機各承擔 30 路，足以支撐 60 路併發推論請求。

### 4.1.4　質檢效能

15 分鐘錄音 → ≤5 分鐘完成轉寫分析並產出質檢報告（qi_speed，需求書§三(六)）。faster-whisper 的即時處理倍率（RTF）約 0.3x（即 1 秒語音用 0.3 秒處理），15 分鐘錄音轉寫約需 4.5 分鐘，含質檢規則評分與報表生成，預估可在 5 分鐘內完成。質檢批次任務由 GPU#1 負責（AD-002），與即時推論 GPU#0 分離，確保批次質檢不影響線上服務延遲。

### 4.3.2　資料加密策略

| 層級 | 加密方式 | 適用範圍 |

### 4.3.3　存取控制與稽核

- **身分認證：** 整合台鐵既有 Active Directory（AD/LDAP），統一帳號認證，不另建帳號體系
- **角色權限（RBAC）：** 值機員 / 督導 / 系統管理者 / 知識庫管理者四級角色，各角色僅具最小必要權限（FR-ADM-002）
- **操作日誌：** 所有使用者操作（登入登出、知識庫異動、系統設定變更、個資存取）均記錄完整軌跡，含時戳、操作者、IP 位址、異動內容（FR-ADM-004）
- **時戳校時：** 所有伺服器以 NTP 同步台鐵內部時間源，確保日誌時序一致

## 4.4　擴展性需求

### 4.4.1　GPU 運算擴充路徑

本案 AI 伺服器每台配置 NVIDIA L40S 48GB ×2（AD-002），硬體規格要求「AI 伺服器應保留擴充空間與介面」。系統設計之擴充路徑：

| 擴充階段 | 觸發條件 | 擴充方式 | 影響範圍 |
|---------|---------|---------|---------|
| **Stage 0（現行）** | — | GPU#0 跑單一 vLLM 引擎（Llama 3.1 8B AWQ，權重 ~8GB + KV Cache 動態分配，峰值 ~42GB）；GPU#1 跑 Embedding + QI | 滿足 30+30 路併發 |
| **Stage 1** | 正確率需提升 | GPU#0 升級為 Llama 3.1 70B-Instruct AWQ 4-bit（~36GB），KV Cache 可用空間減少（並行路數降至 ~10~15 路），可搭配 GPU#1 分流 | 僅調整 vLLM 啟動參數與模型檔案，無需硬體異動 |
| **Stage 2** | 系統擴充第二條 E1 專線（60 B-channel） | 於 AI 伺服器擴充插槽加裝第 3 張 GPU | 需停機安裝，但軟體架構無需改動 |

模型替換完全透明：vLLM 支援熱載入模型，更換 LLM 模型（如未來台灣自研模型）僅需替換模型檔案與調整推論參數，應用層程式碼無需修改。

### 4.4.2　儲存擴充

- 每台伺服器 960GB SSD ×3（RAID-5），有效容量約 1.8TB
- 以月話務量 35,000 通、平均通話 3 分鐘估算：每月錄音約 105,000 分鐘 ≈ 190GB（G.711 格式）
- RAID-5 可用空間支撐約 9 個月錄音儲存；超過保留期限之錄音自動歸檔或清除
- SMS 紀錄保留 ≥5 年（sms_retention），以文字紀錄為主，每月約 7,000 則 ×5 年 ≈ 42 萬則，儲存需求 <1GB，無擴充壓力

### 4.4.3　軟體模組擴充

六層式架構（AD-001）之層間以 API 隔離，新增模組僅需實作對應 gRPC/REST 介面即可整合，不影響既有模組。Docker Compose 部署（AD-006）支援橫向新增容器服務，無需變更部署架構。

## 6.4　防護措施

### 6.4.1　即時去識別化（PIA-R01）

STT 模組（M-STT-03）在輸出文字時即時套用正則表達式遮罩：

| 個資類型 | 遮罩規則 | 遮罩範例 |
|---------|---------|---------|
| 身分證字號 | 保留首碼及末三碼 | A123456789 → A******789 |
| 電話號碼 | 保留首 4 碼 | 0912345678 → 0912****** |
| 信用卡號 | 保留末 4 碼 | ************1234 |
| Email | 保留域名 | ***@gmail.com |

去識別化後之文字存入 PostgreSQL，原始語音錄音由 MES Server 獨立儲存，兩者存取權限分離。

### 6.4.2　LLM 對話隔離（PIA-R02）

- 每通對話使用獨立 Session，LLM 無法跨 Session 存取其他旅客資料
- System Prompt 明確指示 LLM 不得記憶、複述或推論旅客個資
- Redis Session Cache 設定 TTL（通話結束後 30 分鐘自動清除）
- LLM 無直接資料庫存取權限，僅透過 API 取得已去識別化之查詢結果

### 6.4.3　存取控制與稽核（PIA-R03）

- RBAC 四級角色（FR-ADM-002）：系統管理員、督導、值機人員、唯讀稽核
- 質檢錄音存取需督導級以上權限，且每次存取記錄日誌
- 存取日誌包含：人員帳號、存取時間、存取資料範圍、操作類型
- 日誌保留 ≥6 個月，由 TimescaleDB 時序儲存，防竄改

### 6.4.4　報表匯出管控（PIA-R04）

- 統計報表預設去識別化呈現（FR-RPT-001~003）
- 含個資之明細匯出需督導級以上權限審核
- 匯出檔案自動加入使用者帳號浮水印
- 匯出操作記錄稽核日誌

### 6.4.5　簡訊資料保護（PIA-R05）

- SMS 簡訊紀錄加密儲存（AES-256），金鑰由台鐵資訊中心管控
- 傳輸過程使用 TLS 加密
- SMS 裝置實體安全由台鐵機房門禁管控
- SMS 紀錄保留 ≥5 年（契約要求），到期後安全刪除

### 6.4.6　備份資料防護（PIA-R06）

- PostgreSQL 備份檔加密（AES-256）後存放
- 備份媒體存放於台鐵機房受管控區域
- 備份還原作業需系統管理員權限，操作全程日誌記錄

### 6.4.7　供應鏈人員管控（PIA-R07）

- 全員簽署保密協定（含分包人員）
- 適任性查核（契約附件 10）
- 維運期間遠端存取限 VPN + 雙因子認證
- 契約終止後保密義務持續有效

## 6.5　當事人權利保障

依個人資料保護法第 3 條，本系統保障旅客以下權利：

| 權利 | 實作方式 |
|------|---------|
| 查詢或請求閱覽 | 旅客可透過客服人員查詢個人對話紀錄（經身分驗證） |
| 請求製給複製本 | 由台鐵客服中心受理，依作業程序提供 |
| 請求補充或更正 | 透過客服人員或後送工單辦理 |
| 請求停止蒐集、處理或利用 | 旅客可要求不錄音（轉人工服務）；AI 質檢排除該通紀錄 |
| 請求刪除 | 依法定保存期限屆滿後自動刪除；提前刪除需台鐵主管核准 |

## 6.6　評估結論

本系統經個資影響評估後，已針對所有辨識之風險建立對應之技術與管理防護措施。關鍵控制重點：

1. **最小蒐集原則**：僅蒐集服務所需之最低限度個資
2. **即時去識別化**：STT 輸出即套用遮罩，降低後續處理鏈之個資暴露面
3. **存取最小權限**：RBAC 四級角色 + AD/LDAP 整合，確保各級人員僅存取業務所需資料
4. **全程稽核軌跡**：操作/存取日誌保留 ≥6 個月，支持事後追溯
5. **加密防護**：傳輸 TLS 1.3 + 儲存 AES-256，金鑰由台鐵管控

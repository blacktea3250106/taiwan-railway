## 第一章　程式功能清單

### 1.1　清單說明

本章彙整台鐵 AI 客服系統全部程式之功能對照清單，作為程式設計書之總索引。清單中每一支程式均 1:1 對應系統分析報告書（#02）所定義之功能需求（FR），並歸屬於程式設計規格報告書（#03）所規劃之軟體模組（M）。三份文件之編號體系一致，確保需求可追溯至程式碼層級：

- **程式代號（P-{模組}-NN）**：本書之程式設計單元，每支程式含類別圖、核心函式、輸入輸出規格與關鍵演算法，詳見第二章以後各節。
- **功能需求編號（FR-{模組}-NNN）**：對應 #02 系統分析報告書第二章至第八章所定義之功能需求規格。
- **軟體模組編號（M-{模組}-NN）**：對應 #03 程式設計規格報告書第五章所定義之模組設計規格。
- **測試案例編號（TC-{模組}-NNN）**：與 FR 編號 1:1 對應，詳見測試計畫書（#05）。

全系統共計 **45 支程式**，涵蓋 10 大子系統（KB 知識庫在程式層面獨立列出，邏輯上屬 LLM 子系統之子模組），對應 48 項功能需求與 45 個軟體模組。程式代號中，部分模組因功能需求合併實作（如 FR-LLM-002 與 FR-LLM-003 共用 P-LLM-02），故程式數量略少於功能需求總數。所有程式均以 Python 3.11+（後端）或 React 18+ / Next.js（前端）開發，技術堆疊版本鎖定詳見 #06 軟硬體清單及系統架構圖 §6.4。


### 1.2　程式功能清單

#### 一、智慧語音應答系統（AI-IVR）— 6 支程式

本子系統部署於 AI-IVR 伺服器（16 核心 ≥2.2GHz ×2，DDR5 64GB+），以 SIP/RTP fork 方式被動接收語音流（AD-007），語音處理鏈路全程於地端完成。

| 程式代號 | 程式名稱 | 所屬功能需求 | 所屬模組 | 簡述 |
|---------|---------|-------------|---------|------|
| P-IVR-01 | SIP Gateway 與 CTI 整合程式 | FR-IVR-001 語音通道與 CTI 整合；FR-IVR-002 智慧來電應答與引導 | M-IVR-01 | PJSIP 通話管理、RTP 收發、≥30 埠通道控制、STT→LLM→TTS Pipeline 串接 |
| P-IVR-02 | 多輪對話引擎程式 | FR-IVR-003 多輪對話與引導機制 | M-IVR-02 | Redis 對話上下文管理、Prompt 模板設計、對話狀態機控制 |
| P-IVR-03 | 轉接管理程式 | FR-IVR-004 串接真人客服 | M-IVR-03 | 對話摘要生成、SIP REFER 轉接、知識輔助推送 |
| P-IVR-04 | DTMF 偵測與動態選單程式 | FR-IVR-005 按碼偵測與動態選單 | M-IVR-04 | RFC 2833 + SIP INFO 雙模 DTMF、TTS 選單合成 |
| P-IVR-05 | 滿意度調查程式 | FR-IVR-006 滿意度調查 | M-IVR-05 | 調查流程控制、結果寫入 PostgreSQL |
| P-IVR-06 | 通話防護程式 | FR-IVR-007 防護機制 | M-IVR-06 | 敏感詞正則引擎、API Rate Limiter、對話逾時 Timer |

> P-IVR-01 同時實作 FR-IVR-001 與 FR-IVR-002 兩項功能需求，因語音通道管理與智慧應答主流程在 SIP 信令層緊密耦合，拆分反而增加通訊開銷與延遲風險。

#### 二、語音識別系統（STT）— 3 支程式

STT 採 CPU 推論（AD-003），30 路 faster-whisper 串流辨識約使用 9 個 CPU 核心，部署於 AI-IVR 伺服器。

| 程式代號 | 程式名稱 | 所屬功能需求 | 所屬模組 | 簡述 |
|---------|---------|-------------|---------|------|
| P-STT-01 | 即時語音辨識引擎程式 | FR-STT-001 即時中英文語音轉文字 | M-STT-01 | faster-whisper 串流模式、音訊前處理（降噪/正規化）、辨識率 ≥85% |
| P-STT-02 | 意圖辨識程式 | FR-STT-002 語音內容分類與意圖辨識 | M-STT-02 | 意圖分類模型、路由分派邏輯 |
| P-STT-03 | VAD 語音活動偵測程式 | FR-STT-003 語音斷點與停頓偵測 | M-STT-03 | Silero VAD 參數配置、端點偵測觸發機制 |

#### 三、語音合成系統（TTS）— 2 支程式

TTS 同樣採 CPU 推論（AD-003），每路約 0.2 CPU 核心，首包延遲 ≤300ms。Coqui XTTS-v2 為主要引擎，備案 MeloTTS / Piper 已預留介面切換機制。

| 程式代號 | 程式名稱 | 所屬功能需求 | 所屬模組 | 簡述 |
|---------|---------|-------------|---------|------|
| P-TTS-01 | 語音合成引擎程式 | FR-TTS-001 中英文即時語音合成 | M-TTS-01 | XTTS-v2 串流合成、語音快取、備案引擎介面 |
| P-TTS-02 | 動態選單合成程式 | FR-TTS-002 動態選單語音生成 | M-TTS-02 | 選單模板引擎、SSML 標記處理 |

#### 四、大型語言模型服務（LLM）— 5 支程式

LLM 推論採 Llama 3.1 8B-Instruct AWQ 4-bit 量化，透過 vLLM 0.5+ 部署於 GPU#0（AD-002）。單一 vLLM 引擎以 continuous batching + PagedAttention 服務 ≤30 路並行推論，模型權重 ~8GB + KV Cache 動態分配，峰值 VRAM ~42GB（#09 實測）。

| 程式代號 | 程式名稱 | 所屬功能需求 | 所屬模組 | 簡述 |
|---------|---------|-------------|---------|------|
| P-LLM-01 | vLLM 推論引擎程式 | FR-LLM-001 GPU 運算與高併發穩定性 | M-LLM-01 | vLLM 啟動配置（GPU#0 綁定、continuous batching ≤30 路）、健康檢查、自動重啟 |
| P-LLM-02 | 語意理解與對話引擎程式 | FR-LLM-002 語意理解與問答生成；FR-LLM-003 上下文關聯與連續對話 | M-LLM-02 | RAG Pipeline（檢索→Rerank→Prompt 組裝→生成→Citation 抽取）、信心度計算與三層防線邏輯（詳見 #02 §2.4 FR-LLM-002）、多輪對話上下文 Prompt 管理 |
| P-LLM-03 | 文字處理引擎程式 | FR-LLM-004 關鍵詞識別與文字修正 | M-LLM-03 | 同義詞擴展字典、敏感詞過濾規則引擎、錯別字修正（Levenshtein + 字典） |
| P-LLM-04 | 對話紀錄管理程式 | FR-LLM-007 對話紀錄管理 | M-LLM-04 | 對話儲存 Schema、五格式匯出引擎（Word/PDF/Excel/HTML/CSV） |
| P-LLM-05 | 後送案件分析程式 | FR-LLM-008 後送案件分析 | M-LLM-05 | 自動分類 Prompt 設計、優先級判定規則、情感分析 Pipeline、正確率 ≥85% |

> P-LLM-02 合併實作 FR-LLM-002 與 FR-LLM-003，因語意理解與上下文關聯在推論階段共用同一 Prompt 組裝與 KV Cache 管理流程，拆分將導致重複建立對話上下文。

### 2.1　P-LLM-01　vLLM 推論引擎程式

**對應編號：** FR-LLM-001 / TC-LLM-001 / M-LLM-01
**KPI 關聯：** ivr_response ≤5,000ms、agent_response ≤5,000ms

#### 程式功能

管理 vLLM 推論引擎的生命週期，包含啟動配置、GPU 資源綁定、健康檢查與自動重啟。本程式為所有 LLM 推論請求的統一入口，以 gRPC 介面（內部高頻通訊）接收來自 AI-IVR 與 AI-Agent 的推論請求，並透過 vLLM 的 continuous batching 機制併發處理。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | gRPC 推論請求（prompt 文字、max_tokens、temperature、session_id） |
| **輸出** | 生成文字（streaming token 或完整回應）、token 使用量統計 |
| **處理程序** | ① 接收 gRPC 請求 → ② 將請求投入 vLLM AsyncEngine 的請求佇列 → ③ vLLM 以 PagedAttention 管理 KV Cache，continuous batching 合併多路請求 → ④ AWQ 4-bit 量化推論（模型權重 ~8GB + KV Cache 動態分配，峰值 VRAM ~42GB） → ⑤ 回傳 streaming token 或完整生成結果 |
| **存取資料** | 模型權重檔（本機磁碟，唯讀）、vLLM 推論指標（Prometheus metrics endpoint） |
| **計算條件** | 單卡 L40S 48GB 承載單一 vLLM 引擎（Llama 3.1 8B AWQ），以 continuous batching + PagedAttention 服務 ≤30 路並行推論；推論延遲預算 ≤2,500ms（佔 5,000ms SLA 之 50%） |

#### 健康檢查與自動重啟

- 每 10 秒對 vLLM 引擎發送心跳探測，檢查推論佇列深度與 GPU 記憶體使用率
- 連續 3 次心跳逾時（>5 秒無回應），觸發 vLLM 引擎自動重啟
- GPU 記憶體使用率 >90% 時，暫停接受新請求並發送告警至管理後台（FR-ADM-003）
- 所有推論請求與異常事件寫入結構化日誌，保留 ≥6 個月

#### 使用元件

| 元件 | 用途 |
|-----|------|
| vLLM 0.5+（PagedAttention + continuous batching） | LLM 推論引擎 |
| Llama 3.1 8B-Instruct AWQ 4-bit | 推論模型 |
| gRPC（Python grpcio） | 內部服務間通訊 |
| Prometheus client | 推論指標匯出（QPS、延遲分佈、GPU 使用率） |


### 2.2　P-LLM-02　語意理解與對話引擎程式

**對應編號：** FR-LLM-002 + FR-LLM-003 / TC-LLM-002、TC-LLM-003 / M-LLM-02
**KPI 關聯：** ivr_accuracy ≥85%、agent_accuracy ≥85%

#### 程式功能

本程式為 LLM 模組的核心，實作完整的 RAG Pipeline 與多輪對話管理。接收經 STT 轉寫或文字通道傳入的旅客查詢，經向量檢索、Rerank、Prompt 組裝、LLM 生成、Citation 抽取，產出附帶來源標註的回應文字。FR-LLM-002（語意理解與問答生成）與 FR-LLM-003（上下文關聯與連續對話）合併實作，因兩者共用同一 Prompt 組裝流程與 KV Cache 管理，拆分將導致重複建立對話上下文（詳見第一章 §1.3）。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 旅客查詢文字（經個資遮罩處理）、session_id、對話歷史摘要（自 Redis 讀取） |
| **輸出** | 回應文字、信心度分數（0~1）、Citation 來源清單（文件名稱/頁碼/段落編號）、意圖分類標籤 |
| **處理程序** | 見下方 RAG Pipeline 七階段 |
| **存取資料** | Qdrant 向量資料庫（語意檢索）、Redis（Session 對話上下文）、PostgreSQL（歷史對話摘要、知識庫 metadata） |
| **計算條件** | 信心度門檻驅動三層回應策略（見下方）；Qdrant 檢索 Top-K ≤200ms；LLM 推論 ≤2,500ms |

#### RAG Pipeline 七階段

| 步驟 | 階段名稱 | 說明 |
|------|---------|------|
| ① | 查詢改寫 | 旅客查詢進入，改寫為標準化檢索形式 |
| ② | 向量檢索 | 改寫後查詢送入向量檢索 |
| ③ | Rerank | 候選文件重排序 |
| ④ | Prompt 組裝 | 組裝結構化 Prompt |
| ⑤ | LLM 生成 | 大型語言模型生成回應 |
| ⑥ | Citation 抽取 | 抽取回應中的來源標記 |
| ⑦ | 信心度評估與回應路由 | 依信心度分數決定回應策略 |

1. **查詢改寫（Query Rewriting）**：將旅客口語化查詢改寫為適合向量檢索的標準化形式。結合 Redis 中的前序對話上下文，解析指代詞（「那班車」→「14:30 自強號 1234 次」）與省略語意。處理時間預算 ≤50ms。

2. **向量檢索（Vector Retrieval）**：以 multilingual-e5-large Embedding 模型（約 1.3GB，部署於 GPU#1）將改寫後查詢向量化，於 Qdrant 執行 HNSW 近鄰搜尋，取回 Top-K 候選文件片段（預設 K=10）。每筆候選保留完整 metadata（文件名稱、頁碼、段落編號、更新時間戳），供後續 Citation 標註使用。查詢延遲 <50ms。

3. **Rerank 重排序**：以交叉編碼器（Cross-Encoder）對 Top-K 候選依查詢相關性重新排序，取前 3~5 筆作為 LLM 生成的參考上下文。此步驟提升檢索精準度，降低無關文件干擾 LLM 生成品質。

4. **Prompt 組裝**：組裝結構化 Prompt，依序注入：
   - System Instruction（限定回答範圍為台鐵業務、繁體中文回覆、禁止編造資訊）
   - 前序對話摘要（自 Redis 讀取，維持多輪對話連貫性）
   - Rerank 後的知識片段（含 metadata 標記，供 Citation 抽取）
   - 旅客當次查詢

5. **LLM 生成**：透過 P-LLM-01 呼叫 vLLM 推論引擎，以 streaming token 方式回傳生成結果。推論延遲預算 ≤2,500ms（128 input + 64 output token 之 P95 延遲約 1.2 秒）。

6. **Citation 抽取**：解析 LLM 生成文字中的來源標記，對應至步驟 2 保留的 metadata，輸出結構化 Citation 清單（文件名稱、頁碼、段落編號）。前端以 footnote 或 tooltip 方式呈現來源連結（FR-AGT-004）。

7. **信心度評估與三層回應路由**：依 LLM 生成結果與檢索匹配程度計算信心度分數（0~1），驅動三層回應策略：

| 信心度區間 | 回應策略 | 說明 |
|-----------|---------|------|
| ≥0.85 | 直接回答 | 回應文字附帶 Citation 來源標註 |
| 0.6 ~ 0.85 | 回答 + 人工確認提示 | 回應文字末尾加註「建議洽人工客服確認」 |
| <0.6 | 轉接人工客服 | 不輸出 AI 生成回答，直接觸發轉接流程（P-IVR-03 / P-AGT-05），附帶對話摘要 |

涉及票價、時刻等關鍵營運資訊時，不論信心度高低，一律透過 API 即時查詢台鐵既有系統取得資料，不依賴 LLM 參數記憶，避免過時或錯誤資訊誤導旅客。所有 AI 回覆一律標註「本回覆由 AI 生成，如有疑問請按 0 轉接專人」（語音通道）或「如需人工協助請點選轉接」（文字通道）。

#### 多輪對話上下文管理

- 以 Redis 儲存當次對話 Session Context（TTL = 通話時長 + 5 分鐘緩衝）
- 每輪交互結束後，將當輪對話摘要（旅客意圖 + AI 回應要點）壓縮後追加至 Session Context
- 當 Session Context 超過 Prompt 容量上限時，以滑動窗口保留最近 5 輪對話，並以 LLM 摘要壓縮更早的歷史
- 通話/對話結束後，完整對話紀錄寫入 PostgreSQL，Session Context 自 Redis 清除

#### 使用資料庫表格

| 資料庫 | 表格/Collection | 關鍵欄位 | 用途 |
|-------|----------------|---------|------|
| Qdrant | `kb_vectors` | vector（1024 維）、doc_id、doc_name、page、paragraph、updated_at | 向量語意檢索 |
| Redis | `session:{session_id}` | dialog_history、intent_state、user_context | 即時對話上下文 |
| PostgreSQL | `conversations` | id、session_id、channel（voice/text）、started_at、ended_at | 對話主記錄 |
| PostgreSQL | `conversation_turns` | id、conversation_id、role（user/assistant）、content、confidence、citations_json、created_at | 逐輪對話內容 |
| PostgreSQL | `user_history_summary` | user_id、summary_vector、last_updated | 個性化歷史摘要（供 P-AGT-04） |


### 2.3　P-LLM-03　文字處理引擎程式

**對應編號：** FR-LLM-004 / TC-LLM-004 / M-LLM-03
**KPI 關聯：** ivr_accuracy ≥85%、agent_accuracy ≥85%

#### 程式功能

對旅客輸入文字與 LLM 生成回應執行前處理與後處理，包含同義詞擴展、敏感詞過濾、錯別字修正三大功能。本程式作為 P-LLM-02 RAG Pipeline 的前置與後置處理器，提升查詢檢索品質與回應安全性。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 旅客原始查詢文字（前處理）；LLM 生成回應文字（後處理） |
| **輸出** | 正規化查詢文字（前處理）；過濾後安全回應文字（後處理） |
| **處理程序** | 前處理：① 錯別字修正（Levenshtein 距離 + 台鐵專屬字典比對）→ ② 同義詞擴展（「自強」↔「自強號」、「左營」↔「新左營」等） → ③ 敏感詞偵測（個資正則遮罩：身分證/電話/信用卡號）；後處理：④ 回應文字敏感詞過濾 → ⑤ 繁體中文校驗（攔截簡體字元） |
| **存取資料** | PostgreSQL `synonym_dict`（同義詞對照表）、`sensitive_words`（敏感詞規則庫）、`correction_dict`（錯別字修正字典） |
| **計算條件** | 前處理 + 後處理合計 ≤100ms |

#### 台鐵專屬字典

系統內建台鐵業務領域字典，涵蓋：
- 車站名稱（含簡稱、舊名、易混淆站名）
- 車種名稱（自強號、太魯閣、普悠瑪、區間車）
- 票種名稱（去回票、電子票證、定期票）
- 常見口語與錯別字對照（「板喬」→「板橋」、「自強好」→「自強號」）

字典維護透過管理後台（FR-ADM-001）由台鐵系統管理者自行新增修改，修改後即時生效，無需重啟服務。

### 2.5　P-LLM-05　後送案件分析程式

**對應編號：** FR-LLM-008 / TC-LLM-008 / M-LLM-05
**KPI 關聯：** classify_accuracy ≥85%

#### 程式功能

對客服後送案件執行自動分類、優先級判定、情感分析與趨勢預測，分析結果透過 API 推送至既有後送案件系統，225 位外點後送人員之作業流程維持不變。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 後送案件文字內容（來自對話摘要或客服人員手動建立）、案件 metadata |
| **輸出** | 分類標籤（一級/二級分類）、優先級（緊急/高/中/低）、情感標籤（正面/中性/負面 + 強度分數）、趨勢標籤 |
| **處理程序** | ① 案件文字前處理（P-LLM-03 同義詞擴展 + 敏感詞過濾）→ ② 呼叫 P-LLM-01 執行分類推論（以 few-shot Prompt 設計，注入分類定義與範例）→ ③ 優先級判定（規則引擎：含關鍵詞「安全」「事故」自動升為緊急）→ ④ 情感分析（LLM 判定情感極性與強度）→ ⑤ 結果寫入 PostgreSQL 並推送至後送案件系統 API |
| **存取資料** | PostgreSQL `backlog_cases`（案件主表）、`backlog_analysis`（分析結果）、`classification_rules`（分類定義與規則） |
| **計算條件** | 分類正確率 ≥85%（以 200 筆真實後送案件驗收，TC-LLM-008）；單筆分析 ≤3 秒 |

#### 使用介面

- **輸入介面：** 值機平台（P-WS-03）建立後送工單時自動觸發，或管理後台批次匯入
- **輸出介面：** REST API 推送至既有後送案件系統（API 規格待得標後與台鐵確認）；分析結果呈現於值機平台工單詳情頁與報表儀表板（P-RPT-01）


### 2.6　P-KB-01　知識庫管理引擎程式

**對應編號：** FR-LLM-005 / TC-LLM-005 / M-KB-01
**關聯 AD：** AD-004、AD-008

#### 程式功能

管理台鐵 AI 客服知識庫的完整生命週期，包含知識條目之新增、修改、刪除（CRUD）、版本管理（git-like snapshot）、增量 Embedding 更新與 Qdrant Collection 管理。知識庫是 RAG 問答品質的基礎——系統於得標後 D+50 前取得台鐵提供的種子資料（FAQ、SOP、規章），並透過本程式持續擴充與維護。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 知識條目（標題、內容、分類、附件）；批次匯入檔案（Word/PDF/Excel）；版本操作指令（commit/rollback/diff） |
| **輸出** | 操作結果回應；版本差異報告；Embedding 更新狀態 |
| **處理程序** | ① 知識條目 CRUD → ② 文件解析與 chunking（依段落/頁面切分，保留 metadata）→ ③ multilingual-e5-large Embedding 生成（GPU#1，模型約 1.3GB）→ ④ 向量與 metadata 寫入 Qdrant `kb_vectors` Collection → ⑤ 版本快照記錄（記錄變更差異，支援回溯至任一歷史版本） |
| **存取資料** | PostgreSQL `kb_documents`（文件主表）、`kb_chunks`（切分片段）、`kb_versions`（版本歷史）；Qdrant `kb_vectors` |
| **計算條件** | 單篇文件 Embedding 更新 ≤10 秒（≤50 chunks）；批次匯入以非同步任務執行，不阻塞線上服務 |

#### 增量更新策略

- 知識條目修改時，僅重新計算受影響 chunks 的 Embedding，不全量重建向量索引
- Qdrant Collection 使用 HNSW 索引，新增/更新向量後索引自動漸進式重建，不影響線上查詢
- 版本管理以 snapshot 模式記錄每次變更的差異（新增/修改/刪除的 chunk 清單），支援一鍵回溯至指定版本

#### 使用資料庫表格

| 資料庫 | 表格/Collection | 關鍵欄位 | 用途 |
|-------|----------------|---------|------|
| PostgreSQL | `kb_documents` | id、title、category、content、status、created_by、created_at、updated_at | 知識條目主表 |
| PostgreSQL | `kb_chunks` | id、doc_id、chunk_index、chunk_text、char_offset、page、paragraph | 文件切分片段 |
| PostgreSQL | `kb_versions` | id、version_tag、changes_json、created_by、created_at | 版本快照歷史 |
| Qdrant | `kb_vectors` | vector（1024 維）、doc_id、chunk_id、doc_name、page、paragraph、updated_at | 向量語意索引 |

#### 使用介面

- **管理後台（FR-ADM-001）：** 系統管理者透過 Web UI 進行知識條目 CRUD、批次匯入、版本管理
- **REST API：** 供外部系統或排程任務呼叫（符合國發會共通性應用程式介面規範）

## 第三章　STT / TTS 模組程式說明

本章說明語音識別（STT）3 支程式與語音合成（TTS）2 支程式之程式功能、處理邏輯、使用資料庫及元件。STT 與 TTS 均部署於 AI-IVR 伺服器（CPU 16 核心 ≥2.2GHz ×2，DDR5 64GB+），採 CPU 推論而非 GPU（AD-003），理由為：串流語音處理屬「每收到一小段語音就要立即處理」的低延遲任務，CPU 的每核心延遲優於 GPU 的批次吞吐特性；30 路並行 STT 約需 9 CPU 核心、TTS 約需 6 CPU 核心，合計 15 核心，AI-IVR 伺服器 32 核心綽綽有餘，GPU 資源留給 LLM 推論（AD-002）。

語音互動完整信令流程與延遲預算分配詳見 #03 程式設計規格報告書第四章 §4.1；資料庫完整 Schema 詳見 #03 第六章。


### 3.1　P-STT-01　即時語音辨識引擎程式

**對應編號：** FR-STT-001 / TC-STT-001 / M-STT-01
**KPI 關聯：** stt_accuracy ≥85%

#### 程式功能

接收 PJSIP RTP 接收器解碼後的 PCM 音訊流，以 faster-whisper（CTranslate2 加速版 Whisper large-v3）執行即時中英文混合語音辨識，輸出辨識文字供後續意圖辨識（P-STT-02）與 LLM 推論（P-LLM-02）使用。本程式以 chunk-based streaming 模式運作，由 VAD 模組（P-STT-03）觸發辨識時機，確保辨識結果與旅客發言節奏同步。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | PCM 音訊流（G.711 解碼後 16kHz 16-bit mono）、VAD 觸發信號（來自 P-STT-03） |
| **輸出** | 辨識文字（含時間戳記、語言標籤 zh/en、辨識信心度分數） |
| **處理程序** | ① 音訊前處理（重採樣至 16kHz、振幅正規化、背景降噪） → ② 接收 VAD 觸發信號，將語音段落送入 faster-whisper 辨識佇列 → ③ CTranslate2 引擎執行 Whisper large-v3 推論（CPU 模式，int8 量化加速） → ④ 後處理（繁體中文正規化、標點符號插入、台鐵專屬詞彙 hotword boosting） → ⑤ 輸出辨識結果至意圖辨識模組 |
| **存取資料** | 模型權重檔（本機磁碟，唯讀）；PostgreSQL `hotword_dict`（台鐵專屬詞彙表） |
| **計算條件** | 單路辨識延遲 ≤800ms（佔 5,000ms SLA 之 16%）；30 路並行約需 9 CPU 核心；辨識率 ≥85%（200 音檔測試，中英文比例 4:1，每檔 ≤150 秒） |

#### 音訊前處理 Pipeline

音訊品質直接影響辨識率。系統來源包含 PSTN E1（G.711 64kbps）、WebCall（Opus/G.722）與行政交換機（類比→G.711），其中行政交換機經類比→數位轉換後，背景雜訊可能高於其他來源。前處理 Pipeline 針對不同來源採用差異化參數：

1. **重採樣**：統一轉換為 16kHz 16-bit mono PCM（Whisper 原生輸入格式）
2. **振幅正規化**：將音量正規化至 -3dBFS，避免過輕或過響影響辨識
3. **背景降噪**：以 RNNoise（開源即時降噪）過濾環境雜訊，類比線路來源自動啟用更積極的降噪參數

#### 台鐵專屬詞彙 Hotword Boosting

Whisper large-v3 對標準國語辨識率可達 88~92%，但台鐵業務特有詞彙（車站名、車種名、票種名）需額外處理。系統透過 hotword boosting 機制，在解碼階段提升特定詞彙的輸出機率：

- **車站名稱：** 「新左營」「板橋」「南港」「瑞芳」等（含易混淆站名）
- **車種名稱：** 「自強號」「太魯閣」「普悠瑪」「區間車」
- **票種名稱：** 「去回票」「電子票證」「定期票」
- **常見口語修正：** 「自強好」→「自強號」、「板喬」→「板橋」

詞彙表透過管理後台（FR-ADM-001）維護，修改後即時載入，無需重啟服務。得標後以台鐵客服錄音進行 fine-tune，可進一步提升業務場景辨識率。

#### 使用元件

| 元件 | 用途 |
|-----|------|
| faster-whisper（CTranslate2 引擎） | Whisper large-v3 推論（CPU 模式），比原版 PyTorch 快 4 倍、記憶體用量降低 50% |
| Whisper large-v3 模型 | 開源中英文混合語音辨識，繁中實測準確率 88~92% |
| RNNoise | 即時背景降噪（開源，CPU 運算） |
| NumPy / SoundFile | 音訊前處理（重採樣、正規化） |


### 3.2　P-STT-02　意圖辨識程式

**對應編號：** FR-STT-002 / TC-STT-002 / M-STT-02
**KPI 關聯：** stt_accuracy ≥85%

#### 程式功能

接收 P-STT-01 輸出的辨識文字，執行意圖分類與路由決策，判定旅客查詢屬於哪一類業務意圖（如時刻查詢、票價查詢、掛失、投訴等），並決定後續路由（AI 自動回答 / 轉接人工 / 觸發 API 查詢）。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | STT 辨識文字、辨識信心度、session_id |
| **輸出** | 意圖分類標籤（一級/二級分類）、路由決策（ai_answer / transfer_human / api_query）、分類信心度 |
| **處理程序** | ① 接收辨識文字 → ② 文字前處理（呼叫 P-LLM-03 同義詞擴展與錯別字修正） → ③ 意圖分類（以 LLM few-shot 分類，透過 P-LLM-01 推論，注入意圖定義與範例） → ④ 路由決策（依意圖類別與信心度決定後續處理路徑） → ⑤ 將意圖標籤寫入 Redis Session Context，供多輪對話追蹤 |
| **存取資料** | PostgreSQL `intent_definitions`（意圖定義與範例）；Redis `session:{session_id}`（對話上下文） |
| **計算條件** | 分類延遲 ≤200ms（含 LLM few-shot 推論，使用短 prompt 快速回應） |

#### 路由決策規則

| 意圖類別 | 路由 | 說明 |
|---------|------|------|
| 時刻/票價查詢 | api_query → ai_answer | 先呼叫台鐵既有 API 取得即時資料，再由 LLM 組裝回答 |
| FAQ 類問題 | ai_answer | 直接走 RAG Pipeline（P-LLM-02） |
| 掛失/退票/投訴 | transfer_human | 涉及交易操作，直接轉接人工客服（P-IVR-03） |
| 非業務問題 | ai_answer（引導） | LLM System Instruction 限定範圍回答，連續 2 輪非業務則結束對話（P-IVR-06） |

#### 使用介面

- **上游：** 接收 P-STT-01 辨識結果
- **下游：** 將意圖與路由決策傳遞至 P-LLM-02（RAG Pipeline）、P-IVR-03（轉接管理）或 P-IVR-06（防護機制）


### 3.3　P-STT-03　VAD 語音活動偵測程式

**對應編號：** FR-STT-003 / TC-STT-003 / M-STT-03
**KPI 關聯：** ivr_response ≤5,000ms

#### 程式功能

持續監聽 RTP 音訊流，偵測旅客發言的起始與結束時機。VAD 的準確度直接影響兩項使用者體驗：過早判定結束會截斷旅客語句；過晚判定則延長等待時間，壓縮 STT→LLM→TTS 的回應預算。本程式採用 Silero VAD 模型，相較 WebRTC VAD 的能量門檻法，Silero 以神經網路辨識語音特徵，對背景雜訊與氣音的抗干擾能力顯著較佳。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | PCM 音訊流（即時 RTP 解碼輸出） |
| **輸出** | VAD 事件（speech_start / speech_end + 時間戳記）、語音段落音訊切片 |
| **處理程序** | ① 以 30ms 窗口持續掃描音訊流 → ② Silero VAD 模型判定每個窗口是否包含語音 → ③ 連續 N 個窗口偵測到語音觸發 speech_start 事件 → ④ 連續 M 個窗口偵測到靜音觸發 speech_end 事件 → ⑤ 將 speech_start 至 speech_end 之間的音訊切片送入 P-STT-01 辨識佇列 |
| **存取資料** | Silero VAD 模型權重（本機磁碟，唯讀） |
| **計算條件** | 端點偵測延遲 ≤200ms（佔 5,000ms SLA 之 4%）；CPU 佔用極低（每路 <0.05 核心） |

#### 參數調校策略

VAD 參數需依音訊來源品質差異化配置：

| 參數 | PSTN E1 / WebCall | 行政交換機（類比線路） | 說明 |
|-----|-------------------|---------------------|------|
| speech_threshold | 0.5 | 0.6 | 語音判定門檻（0~1），類比線路因背景雜訊較高需提高門檻 |
| min_speech_duration | 250ms | 300ms | 最短語音段落長度，避免雜訊誤觸發 |
| min_silence_duration | 500ms | 600ms | 靜音持續多久判定發言結束 |
| speech_pad | 100ms | 150ms | 語音前後保留的緩衝區，避免截斷首尾音節 |

上述參數可透過管理後台（FR-ADM-001）調整。系統上線初期（D+180 ~ D+240）將依實際客服錄音統計分析，逐步收斂至最佳參數組合。

#### 使用元件

| 元件 | 用途 |
|-----|------|
| Silero VAD | 神經網路語音活動偵測模型（開源，ONNX 格式，CPU 推論） |
| ONNX Runtime | Silero VAD 模型推論引擎 |


### 3.4　P-TTS-01　語音合成引擎程式

**對應編號：** FR-TTS-001 / TC-TTS-001 / M-TTS-01
**KPI 關聯：** ivr_response ≤5,000ms

#### 程式功能

將 LLM 生成的文字回應即時合成為自然語音，以串流方式輸出至 PJSIP RTP 發送器播放給旅客。主引擎採用 Coqui XTTS-v2 模型，支援中英文混合語音合成與 voice cloning（可依台鐵指定的聲音風格客製化）。所有合成運算於 AI-IVR 伺服器 CPU 完成，完全地端運行，無任何外部網路連線。

Coqui AI 公司已於 2023 年底停業，但 XTTS-v2 模型以 MPL-2.0 開源授權釋出，原始碼與模型權重可永久使用。系統同時內建 MeloTTS 與 Piper 作為備案引擎，三者共用統一的 TTS 抽象介面，管理者可透過後台切換，無需修改上下游程式碼。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | LLM 生成文字（streaming token 或完整文句）、語音參數（語速、語調、聲音風格） |
| **輸出** | PCM 音訊流（16kHz → G.711 編碼後交由 PJSIP 發送） |
| **處理程序** | ① 文字前處理（數字轉國字、英文單字音譯標注、SSML 標記解析） → ② 查詢語音快取（常用語句如「歡迎致電台鐵客服」「請稍候為您轉接」已預先合成並快取） → ③ 快取未命中時，以 XTTS-v2 模型執行串流合成（首包延遲 ≤300ms） → ④ 音訊後處理（音量正規化、採樣率轉換） → ⑤ 串流輸出至 RTP 發送器 |
| **存取資料** | XTTS-v2 模型權重（本機磁碟，唯讀）；Redis `tts_cache:{text_hash}`（語音快取） |
| **計算條件** | 單路合成延遲 ≤800ms（首包 ≤300ms + 串流持續輸出）；每路約需 0.2 CPU 核心；30 路並行合計約 6 CPU 核心 |

#### 語音快取機制

高頻使用的固定語句（系統歡迎語、轉接提示、滿意度調查引導語等）預先合成並儲存於 Redis，以文字內容 SHA-256 hash 作為快取鍵。快取命中時直接回傳音訊流，延遲 <10ms，大幅降低 CPU 運算負載。快取容量預設 500 條語句，LRU 策略淘汰低頻項目。管理後台修改語句模板時自動觸發快取更新。

#### 備案引擎切換

| 引擎 | 特點 | 切換方式 |
|-----|------|---------|
| Coqui XTTS-v2（主） | 中文自然度最佳，支援 voice cloning | 預設引擎 |
| MeloTTS | 多語言支援佳，推論速度快，Apache-2.0 授權 | 管理後台一鍵切換 |
| Piper | 超輕量（每路 <0.05 核心），適合極端高併發場景 | 管理後台一鍵切換 |

三套引擎共用 `TTSEngine` 抽象介面（`synthesize_stream(text, voice_config) → AudioStream`），上游呼叫端無需感知底層引擎差異。切換引擎僅需修改一個環境變數並重啟 TTS 容器（Docker Compose 滾動更新，零停機）。

#### 使用元件

| 元件 | 用途 |
|-----|------|
| Coqui TTS（XTTS-v2 模型） | 主語音合成引擎（CPU 推論，MPL-2.0 授權） |
| MeloTTS / Piper | 備案語音合成引擎 |
| Redis | 語音快取（常用語句預合成結果） |
| SSML Parser | SSML 標記解析（停頓、強調、語速控制） |


### 3.5　P-TTS-02　動態選單合成程式

**對應編號：** FR-TTS-002 / TC-TTS-002 / M-TTS-02
**KPI 關聯：** —（無直接 KPI，支援 FR-IVR-005 按碼偵測與動態選單）

#### 程式功能

依管理後台設定動態生成語音選單內容，無需預錄音檔。系統管理者可透過後台（FR-ADM-001）自行編輯選單結構、選項文字與排列順序，修改後系統自動調用 P-TTS-01 合成新語音並更新快取。本程式與 P-IVR-04（DTMF 偵測）協同運作，構成完整的語音選單互動流程。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 選單定義（JSON 格式：選單 ID、各選項文字、DTMF 對應按鍵）；選單觸發事件（來自 P-IVR-04） |
| **輸出** | 合成後的語音選單音訊流（交由 P-TTS-01 串流播放） |
| **處理程序** | ① 載入選單定義（自 PostgreSQL `menu_definitions` 表讀取） → ② 依模板組裝選單文字（含 SSML 標記：各選項間插入 300ms 停頓、按鍵編號以強調語調播放） → ③ 呼叫 P-TTS-01 合成語音 → ④ 合成結果寫入快取（選單內容變更前持續使用） |
| **存取資料** | PostgreSQL `menu_definitions`（選單定義表：menu_id、menu_name、items_json、updated_at） |
| **計算條件** | 選單首次合成 ≤2 秒（含模板組裝 + TTS 合成）；後續使用快取，延遲 <10ms |

#### 選單模板範例

```
歡迎致電台鐵客服，<break time="300ms"/>
查詢時刻表請按 <emphasis>1</emphasis>，<break time="300ms"/>
查詢票價請按 <emphasis>2</emphasis>，<break time="300ms"/>
掛失或退票請按 <emphasis>3</emphasis>，<break time="300ms"/>
轉接人工客服請按 <emphasis>0</emphasis>，<break time="300ms"/>
或直接說出您的問題。
```

SSML 標記由 P-TTS-01 的 SSML Parser 解析，控制停頓時長（`<break>`）、語調強調（`<emphasis>`）與語速調節（`<prosody>`）。

#### 使用介面

- **上游：** 管理後台（FR-ADM-001）編輯選單定義 → 本程式自動觸發合成
- **下游：** P-IVR-04（DTMF 偵測）觸發播放選單 → 本程式回傳快取音訊

### 4.1　P-AGT-01　數位通道管理程式

**對應編號：** FR-AGT-001 / TC-AGT-001 / M-AGT-01

管理 AI-Agent 數位文字通道的連線生命週期。系統透過 WebSocket 與 CRM Gateway 建立即時雙向通訊，同時支援 ≥30 埠數位通道併發連線。本程式負責連線池管理、Session 建立與銷毀、訊息路由分派及連線異常處理。

### 4.2　P-AGT-02　向量檢索引擎程式

**對應編號：** FR-AGT-002 / TC-AGT-002 / M-AGT-02
**KPI 關聯：** agent_accuracy ≥85%

#### 程式功能

整合 Qdrant 向量資料庫實現語意檢索，為 RAG Pipeline 提供知識庫檢索能力。本程式接收查詢文字，透過 multilingual-e5-large 模型生成 Embedding 向量，在 Qdrant 中執行 HNSW 近鄰搜尋，回傳 Top-K 相關文件片段及其 metadata（文件名稱、頁碼、段落編號）供後續 Citation 標註使用。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 查詢文字（經 P-LLM-03 同義詞擴展與錯別字修正後）、Top-K 參數（預設 5）、過濾條件（知識庫版本、文件類別） |

| **存取資料** | Qdrant Collection（知識庫向量索引）；multilingual-e5-large 模型權重（GPU#1） |
| **計算條件** | Embedding 生成 + Top-K 查詢延遲 <200ms（Qdrant Rust 引擎 HNSW 查詢 <50ms）；知識庫規模預估 10,000~50,000 筆文件，單機 Qdrant 完全足夠（AD-008） |

#### 知識庫版本管理

Qdrant Collection 以 `kb_v{version}` 命名，每次知識庫全量更新建立新 Collection，舊版本保留 3 個版本供回溯。版本切換為原子操作（修改 alias 指向），零停機。增量更新機制詳見第二章 §2.6（P-KB-01）。

### 4.3　P-AGT-03　智慧問答與 Citation 程式

**對應編號：** FR-AGT-003 + FR-AGT-004 / TC-AGT-003、TC-AGT-004 / M-AGT-03
**KPI 關聯：** agent_accuracy ≥85%、agent_response ≤5,000ms

#### 程式功能

AI-Agent 文字對話的核心處理程式，合併實作智慧問答（FR-AGT-003）與 Citation 來源標註（FR-AGT-004）。接收旅客文字訊息後，經個資遮罩、RAG 檢索、LLM 推論生成回答，並自動從回應中抽取引用來源，以 footnote 格式標註供旅客與客服人員驗證。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 旅客文字訊息、session_id、對話歷史上下文 |
| **輸出** | AI 回應文字（含 Citation footnote）、信心度分數、引用來源清單 |
| **處理程序** | ① 個資遮罩（正則遮罩身分證/電話/信用卡號） → ② 呼叫 P-AGT-02 向量檢索，取得 Top-K 知識片段 → ③ Prompt 組裝（System Instruction + 檢索結果 + 對話歷史 + 旅客問題） → ④ 透過 P-LLM-01 執行 LLM 推論 → ⑤ Citation 抽取（解析 LLM 輸出中的來源引用標記，對照 metadata 生成 footnote） → ⑥ 信心度三層路由（詳見第二章 §2.2） |
| **存取資料** | Redis `session:{session_id}`（多輪對話上下文）；PostgreSQL `conversation_logs`（對話紀錄） |
| **計算條件** | 端到端回應 ≤5,000ms（API Gateway ≤100ms + 意圖辨識 ≤200ms + 個資遮罩 ≤50ms + 向量檢索 ≤200ms + Prompt 組裝 ≤50ms + LLM 推論 ≤2,500ms + Citation 抽取 ≤100ms + 回覆發送 ≤100ms = 3,300ms，餘裕 1,700ms） |

#### Citation 標註機制

RAG Pipeline 在檢索階段保留每個 chunk 的來源 metadata（文件名稱、頁碼、段落編號）。LLM 生成回答時，Prompt 明確要求以 `[ref:N]` 格式標註引用來源。本程式於生成完畢後解析引用標記，對照 metadata 生成結構化 Citation 列表，前端以 footnote 或 tooltip 呈現來源連結，旅客可點擊查看原始文件段落。

#### 信心度三層路由

採用與語音通道相同的回應策略（詳見第二章 §2.2）：信心度 ≥0.85 直接回答；0.6~0.85 回答並附註「建議洽詢專人確認」；<0.6 回覆「此問題建議洽詢客服人員」並提供轉接按鈕。此策略確保 AI 不會在不確定時提供錯誤資訊，涉及票價、時刻等關鍵資訊一律從台鐵既有 API 即時查詢，不依賴 LLM 記憶。

#### 使用介面

- **上游：** P-AGT-01（數位通道管理）分派訊息
- **下游：** P-AGT-02（向量檢索）、P-LLM-01（LLM 推論）、P-AGT-05（轉接管理）


### 4.4　P-AGT-04　個性化引擎程式

**對應編號：** FR-AGT-005 / TC-AGT-005 / M-AGT-04
**KPI 關聯：** agent_accuracy ≥85%

#### 程式功能

基於使用者歷史對話與需求，為 AI-Agent 提供個性化回應能力。系統以 Redis 儲存當次對話的即時上下文（session-level），以 PostgreSQL 儲存歷史對話摘要（user-level），透過 LLM Prompt 注入使用者歷史摘要實現個性化回應。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | user_id（匿名化識別碼）、session_id、當次對話上下文 |
| **輸出** | 個性化上下文片段（注入至 P-AGT-03 的 Prompt 中） |
| **處理程序** | ① 以 user_id 查詢 PostgreSQL 歷史對話摘要（最近 10 次對話） → ② 擷取使用者偏好標籤（常查詢路線、慣用車種、語言偏好） → ③ 組裝個性化上下文片段（≤200 token，避免佔用過多 Prompt 空間） → ④ 將上下文片段傳遞至 P-AGT-03 注入 Prompt |
| **存取資料** | PostgreSQL `user_profiles`（使用者歷史摘要，去識別化存取）；Redis `session:{session_id}` |
| **計算條件** | 上下文查詢與組裝 ≤50ms；歷史資料須遵守個資保護規範，以匿名化 user_id 存取，存取行為寫入個資存取日誌 |

#### 個資保護

歷史對話摘要僅儲存去識別化後的意圖標籤與查詢類別（如「常查詢台北→高雄時刻」），不保留旅客原始對話內容。存取權限限定為 AI-Agent 服務帳號，操作日誌寫入獨立的個資存取日誌表（詳見第二章 §2.7 P-KB-02）。


### 4.5　P-AGT-05　Web/App 整合與轉接程式

**對應編號：** FR-AGT-006 / TC-AGT-006 / M-AGT-05
**KPI 關聯：** —（無直接 KPI，支援 agent_accuracy 之整體服務品質）

#### 程式功能

整合 Web 及 App 數位客服介面，當 AI 無法解決旅客問題時（信心度 <0.6 或旅客主動要求），自動轉接真人客服並附帶完整對話摘要。轉接時透過 WebSocket 推播通知值機平台（FR-WS-001），客服人員可即時接手對話，無需旅客重述問題。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 轉接觸發事件（信心度不足 / 旅客按「轉接專人」/ 連續 3 輪無法解決）、session_id |
| **輸出** | 轉接通知（含對話摘要、旅客意圖標籤、通道來源）→ 值機平台 |
| **處理程序** | ① 觸發轉接 → ② 呼叫 P-LLM-02 生成對話摘要（≤100 字） → ③ 透過 WebSocket 推播轉接通知至值機平台，附帶摘要與完整對話歷史 → ④ 值機員接手後，旅客端即時切換為「真人客服模式」提示 → ⑤ 完整對話紀錄（含 AI 段與真人段）寫入 PostgreSQL |

| **計算條件** | 轉接延遲 ≤2 秒（含摘要生成 + WebSocket 推播） |


### 4.6　P-SMS-01　SMS 簡訊服務程式

**對應編號：** FR-AGT-007 / TC-AGT-007 / M-SMS-01
**KPI 關聯：** sms_retention ≥5 年

#### 程式功能

整合雙備援 SMS 收發裝置（4G LTE+），處理旅客簡訊收發與 AI 自動回覆。收到旅客簡訊後，將訊息轉發至 P-AGT-03 進行智慧問答處理，AI 回覆經 SMS 裝置發送給旅客，同時透過 WebSocket 推播至值機平台，通知客服人員有 AI 自動回覆紀錄，客服人員可判斷是否需人工介入接手。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 旅客 SMS 訊息（來自 SMS Gateway 的 HTTP callback）、裝置狀態信號 |
| **輸出** | AI 回覆 SMS（經 SMS Gateway 發送）、值機平台同步通知 |
| **處理程序** | ① SMS Gateway 收到旅客簡訊，以 HTTP callback 通知本程式 → ② 訊息前處理（個資遮罩、編碼轉換） → ③ 轉發至 P-AGT-03 取得 AI 回覆 → ④ 回覆文字裁切（SMS 單則 ≤70 字，超長自動分則） → ⑤ 透過 SMS Gateway 發送回覆 → ⑥ WebSocket 推播「AI 自動回覆」通知至值機平台（含旅客號碼遮罩、原始訊息、AI 回覆內容） → ⑦ 完整收發紀錄寫入 PostgreSQL |
| **存取資料** | PostgreSQL `sms_logs`（簡訊收發紀錄，保留 ≥5 年）；Redis（SMS 發送頻率限制） |
| **計算條件** | SMS 回覆端到端延遲 ≤30 秒（含 AI 推論 + SMS 發送）；雙裝置 Active-Standby 備援，主裝置故障自動切換 |

#### 雙備援機制

兩台 SMS 收發裝置以 Active-Standby 模式運作。主裝置每 30 秒回報心跳，連續 3 次心跳逾時（90 秒），自動將收發路由切換至備援裝置。切換期間佇列中的待發送簡訊由備援裝置接續處理，不遺失。SMS Gateway 的 4G LTE 模組選用非大陸品牌及非大陸製造產品，硬體清單逐項標註品牌與產地。

#### SMS 紀錄保存

所有簡訊收發紀錄（時間戳記、來源號碼遮罩、訊息內容、AI 回覆、處理狀態）以結構化格式寫入 PostgreSQL + TimescaleDB 時序分割表，按月分割、自動壓縮，5 年保留期滿後由排程任務歸檔至離線儲存。

### 4.8　P-QI-01　多格式數據匯入與批次處理程式

**對應編號：** FR-QI-001 / TC-QI-001 / M-QI-01
**KPI 關聯：** qi_speed（15 分鐘錄音 → ≤5 分鐘完成轉換分析並產出質檢報告）

#### 程式功能

智慧質檢系統的入口程式，負責接收與管理質檢任務。支援多種音檔格式匯入（WAV、MP3、OGG、FLAC），自動排程批次處理。MES 錄音系統產出的錄音檔，透過指定目錄監控或 API 上傳方式匯入，系統自動建立質檢任務並分派至後續處理 Pipeline。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 音檔（多格式）、質檢任務參數（質檢規則集 ID、優先級、排程時間） |
| **輸出** | 質檢任務 ID、任務狀態（排隊中/處理中/完成/失敗） |
| **處理程序** | ① 音檔格式偵測與轉換（統一轉為 16kHz 16-bit mono WAV） → ② 音檔 metadata 擷取（錄音時長、通道數、取樣率） → ③ 建立質檢任務（寫入 PostgreSQL `qi_tasks`） → ④ 依優先級排入批次處理佇列（Redis Sorted Set） → ⑤ Worker 依序取出任務，觸發 P-QI-02（語音轉文字）→ P-QI-03（規則評分）→ P-QI-04/05（行為與情緒分析）→ P-QI-06（異常偵測）→ P-QI-07（報表生成） |
| **存取資料** | PostgreSQL `qi_tasks`（質檢任務表）；Redis Sorted Set（批次佇列）；本機磁碟（音檔暫存） |
| **計算條件** | 15 分鐘錄音之完整質檢 Pipeline 須於 ≤5 分鐘內完成；批次排程支援 cron 設定（預設每小時執行一次）與即時手動觸發 |

#### 批次排程策略

質檢為非即時批次任務，排程於離峰時段（22:00~06:00）執行大量積壓任務，尖峰時段僅處理高優先級與手動觸發任務，確保 AI 伺服器 GPU#1 資源不與即時 Embedding 查詢競爭（AD-002）。


### 4.9　P-QI-02　語音轉文字處理程式

**對應編號：** FR-QI-002 / TC-QI-002 / M-QI-02
**KPI 關聯：** qi_speed、stt_accuracy ≥85%

#### 程式功能

將客服錄音批次轉換為文字，與即時 STT（P-STT-01）共用 faster-whisper 引擎但獨立排程，避免影響即時語音辨識效能。質檢 STT 針對台鐵業務場景術語進行額外優化，啟用完整語句後處理（標點符號、段落分割、說話者分離）。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | WAV 音檔（來自 P-QI-01 轉換後） |
| **輸出** | 結構化轉寫結果（含時間戳記、說話者標識 Agent/Customer、辨識文字、信心度） |
| **處理程序** | ① 音檔載入與前處理（降噪、正規化，流程同 P-STT-01） → ② faster-whisper 批次辨識（非串流模式，CPU 推論，可充分利用長音檔的批次吞吐優勢） → ③ 說話者分離（基於音訊能量與頻率特徵，區分客服人員與旅客） → ④ 台鐵專屬詞彙 hotword boosting（共用 P-STT-01 詞彙表） → ⑤ 輸出結構化轉寫結果 |
| **存取資料** | 模型權重檔（本機磁碟）；PostgreSQL `qi_transcripts`（轉寫結果） |
| **計算條件** | 批次模式 RTF（Real-Time Factor）約 0.3x（即 15 分鐘錄音約 4.5 分鐘完成轉寫），留 0.5 分鐘給後續分析評分 |


### 4.10　P-QI-03　質檢規則與評分引擎程式

**對應編號：** FR-QI-003 / TC-QI-003 / M-QI-03
**KPI 關聯：** qi_speed

#### 程式功能

依管理者自訂的質檢規則對轉寫結果執行自動評分。系統支援多維度評分項目（開場白/結尾語規範、敬語使用、問題解決率、回應速度等），每項規則可設定權重與通過門檻，計算加權總分並判定是否通過質檢標準。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 結構化轉寫結果（來自 P-QI-02）、質檢規則集 ID |
| **輸出** | 各項評分明細、加權總分、通過/不通過判定、扣分原因說明 |
| **處理程序** | ① 載入質檢規則集（自 PostgreSQL `qi_rule_sets` 讀取） → ② 逐項規則比對：正則比對（開場白/結尾語關鍵字）、LLM 語意評分（問題解決品質）、統計指標（平均回應時間） → ③ 各項乘以權重計算加權總分 → ④ 與門檻值比對，判定通過或不通過 → ⑤ 寫入評分結果至 PostgreSQL `qi_scores` |
| **存取資料** | PostgreSQL `qi_rule_sets`（規則定義）、`qi_scores`（評分結果） |
| **計算條件** | 單通評分 ≤10 秒（含 LLM 語意評分） |

#### 規則自訂機制

管理者透過管理後台（FR-ADM-001）設定質檢規則，支援以下規則類型：

| 規則類型 | 範例 | 評分方式 |
|---------|------|---------|
| 關鍵字比對 | 開場白需包含「台鐵客服您好」 | 正則比對，命中得分/未命中扣分 |
| 語意評分 | 客服回答是否解決旅客問題 | LLM few-shot 評分（0~10 分） |
| 行為指標 | 平均回應時間 ≤15 秒 | 統計計算，門檻判定 |
| 禁忌語偵測 | 禁止使用不當用語 | 正則 + 敏感詞庫比對 |

分析質檢音檔中的靜音段與插話行為。長時間靜音可能代表客服人員查詢資料或態度消極，頻繁插話可能影響旅客體驗，兩者均為質檢評分的重要行為指標。

### 4.12　P-QI-05　語調與情緒分析程式

**對應編號：** FR-QI-005 / TC-QI-005 / M-QI-05
**KPI 關聯：** —（無直接 KPI，支援質檢異常偵測）

#### 程式功能

分析客服人員與旅客的語調變化與情緒狀態。以語音特徵（基頻、語速、能量變化）結合文字情感分析（LLM 情感分類），綜合判定每段對話的情緒標籤（正面/中性/負面/激動），標記情緒異常段落供督導重點檢視。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | WAV 音檔、結構化轉寫結果（來自 P-QI-02） |
| **輸出** | 逐段情緒標籤、情緒時序曲線、異常標記（旅客負面情緒升高 / 客服語調異常） |
| **處理程序** | ① 語音特徵擷取（基頻 F0、語速 syllable/sec、能量 RMS，以 librosa 計算） → ② 文字情感分析（以專用情緒分類模型於 GPU#1 獨立推論，不經由 P-LLM-01 vLLM 引擎） → ③ 特徵融合：語音特徵 70% + 文字情感 30%（語音為主要判斷依據） → ④ 時序分析：偵測情緒急劇變化點（如旅客從中性突然轉為負面） → ⑤ 異常段落標記，寫入 PostgreSQL `qi_emotion_results` |
| **存取資料** | PostgreSQL `qi_emotion_results`；LLM 推論（GPU#1，批次模式） |
| **計算條件** | 15 分鐘音檔情緒分析 ≤60 秒（含語音特徵擷取 + LLM 批次情感分類） |

#### GPU 資源隔離

情緒分析的 LLM 推論部署於 GPU#1（AD-002），以批次模式執行（非即時串流），排程於質檢批次任務中。GPU#1 同時負責 Embedding 查詢，兩者透過 GPU 記憶體使用率監控協調：Embedding 查詢優先（即時需求），情緒分析批次任務在 GPU 記憶體使用率 <60% 時方可執行。


### 4.13　P-QI-06　異常偵測與警報程式

**對應編號：** FR-QI-006 / TC-QI-006 / M-QI-06
**KPI 關聯：** —（無直接 KPI，支援服務品質即時管控）

#### 程式功能

彙整 P-QI-03（評分）、P-QI-04（行為偵測）、P-QI-05（情緒分析）結果，依預設警報規則判定是否觸發異常警報，通知督導人員及時介入。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 質檢評分結果、行為統計、情緒分析結果 |
| **輸出** | 警報事件（等級：一般/重要/緊急）→ 管理後台 + 督導值機平台 |
| **處理程序** | ① 彙整單通質檢完整結果 → ② 逐項比對警報規則（評分低於門檻 / 靜音超時 / 旅客情緒激動 / 禁忌語觸發） → ③ 判定警報等級 → ④ 緊急警報即時推播（WebSocket + 系統通知）；一般警報寫入警報佇列，督導登入時集中顯示 |

| **計算條件** | 警報判定 ≤1 秒；緊急警報推播 ≤3 秒 |

#### 警報規則範例

| 警報條件 | 等級 | 動作 |
|---------|------|------|
| 質檢總分 <60 分 | 重要 | 通知督導覆核 |

| 步驟 | 程式 | 時間上限 | 使用資源 | 可重疊階段 |
|------|------|---------|---------|-----------|
| ① 音檔轉換 | P-QI-01 | ≤10s | CPU | — |
| ② STT 轉寫 | P-QI-02 | ≤270s | CPU（Whisper.cpp） | 與 ⑤ 情緒語音特徵擷取並行（不同 CPU 執行緒） |
| ③ 規則評分 | P-QI-03 | ≤10s | GPU#0（LLM） | — |
| ④ 行為偵測 | P-QI-04 | ≤15s | CPU | 與 ③ 並行（CPU vs GPU） |
| ⑤ 情緒分析 | P-QI-05 | ≤60s | CPU 特徵擷取 + GPU#1 情感分類 | 語音特徵擷取（CPU）與 ② 並行；LLM 情感分類（GPU#1）與 ③ 並行 |
| ⑥ 異常判定 | P-QI-06 | ≤1s | CPU | — |
| ⑦ 報表生成 | P-QI-07 | ≤5s | CPU | — |

**階段重疊分析：** STT 轉寫（②，CPU）與情緒語音特徵擷取（⑤ 前半，CPU 不同執行緒）可同步執行，情緒語音特徵擷取 ≤15s 完全隱藏於 STT 的 270s 內。規則評分（③，GPU#0）與行為偵測（④，CPU）及情緒 LLM 分類（⑤ 後半，GPU#1）三者分別使用不同硬體資源，可同步執行，取最長者 ≤60s。逐字稿格式化處理內嵌於 ② 完成後立即輸出，不另佔時間。

**有效單通時間預算：** 音檔轉換 10s → STT 轉寫 270s（含隱藏之語音特徵擷取）→ max（規則評分 10s, 行為偵測 15s, 情緒 LLM 分類 60s）= 60s → 異常判定 1s → 報表 5s → **合計 ≤300s（5.0 分鐘），符合 ≤5 分鐘 SLA**。多通錄音另透過 Worker 平行化同時處理，進一步提升整體吞吐量。

### 5.1　MES Server 整合（SIP/RTP）

**整合對象：** Media Exchange Server（既有話務分派與錄音伺服器）
**整合方式：** RTP fork + SIP 事件訂閱
**相關程式：** P-IVR-01（FR-IVR-001、FR-IVR-002）
**相關決策：** AD-007（SIP/RTP fork 方式介接既有語音系統）

#### 整合機制

AI-IVR 以被動方式接收 MES Server 的 RTP fork 語音流副本，不攔截、不修改、不影響原始通話路徑。MES Server 在建立通話時，將 RTP 語音流複製一份傳送至 AI-IVR 伺服器，AI-IVR 僅對副本進行 STT→LLM→TTS 處理，原始通話路徑完全不受影響。

#### 介面呼叫說明

| 介面項目 | 規格 |
|---------|------|
| **通訊協定** | SIP（信令）+ RTP（語音流），PJSIP pjsua2 Python binding |
| **資料方向** | MES → AI-IVR（RTP fork 單向複製）；AI-IVR → MES（SIP REFER 轉接指令） |
| **音訊格式** | G.711 μ-law / A-law，64kbps PCM |
| **事件訂閱** | 透過 SIP SUBSCRIBE/NOTIFY 訂閱通話狀態（來電通知、保留、掛斷），推送至值機平台（P-WS-01） |
| **轉接機制** | AI-IVR 判定需轉接真人時，發送 SIP REFER 至 MES，由 MES 執行實際的通話路由切換（P-IVR-03） |

#### 整合約束

- MES Server 端僅需設定 RTP fork 目的 IP（AI-IVR 伺服器 VIP），不修改話務分派邏輯或錄音設定
- AI-IVR 伺服器雙機以 Active-Active 架構部署（AD-005），透過 Keepalived VIP 接收 RTP fork 流量，任一台故障時自動切換，MES 端無需任何調整
- RTP fork 為語音流複製，不佔用 E1 B-channel 通道容量：現有 1 條 E1 的 30 個 B-channel 供人工客服與 AI-IVR 共用的是 SIP 信令通道，RTP fork 副本透過 LAN 傳送，不經過 E1 線路


### 5.2　E1 VoIP Gateway 整合（SIP Trunk）

**整合對象：** PSTN E1 VoIP Gateway（公眾電話語音來源）
**整合方式：** SIP Trunk 註冊
**相關程式：** P-IVR-01（FR-IVR-001）
**相關決策：** AD-007

#### 整合機制

AI-IVR 透過 PJSIP 建立 SIP UA（User Agent），以 SIP Trunk 方式向 VoIP Gateway 註冊，接收從 E1 專線進入的 PSTN 語音通話。現有 1 條 E1 提供 30 個 B-channel（每 channel 64kbps），系統架構支援最多 2 條 E1 擴充。

#### 介面呼叫說明

| 介面項目 | 規格 |
|---------|------|
| **通訊協定** | SIP Trunk（UDP/TCP 5060）、SRTP 加密（強制） |
| **信令流程** | E1 來電 → VoIP Gateway 轉 SIP INVITE → MES 分派 → RTP fork 至 AI-IVR |
| **DTMF 偵測** | 支援 RFC 2833（RTP 事件）與 SIP INFO 雙模式（P-IVR-04），自動偵測 Gateway 支援模式 |
| **通道監控** | 即時回報通道佔用數至管理後台（P-ADM-01），達 27 路（90%）觸發容量告警 |

#### 行政交換機路徑

行政交換機（02-2381-5226）為類比線路，經 VoIP Gateway 以 G.711 編碼轉換為數位信號後，經 MES 路由至 AI-IVR。類比→數位轉換採用 G.711 PCM 編碼（64kbps），語音品質損失極小。P-STT-01 的音訊前處理 Pipeline 針對類比線路來源設定差異化的 VAD 參數與降噪門檻值，以因應背景雜訊高於數位線路的特性（詳見第三章 §3.1）。

#### 異常處理

- SIP 註冊失敗時每 30 秒自動重試，連續 3 次失敗後發送告警至值班人員
- 系統日誌記錄所有 SIP 信令事件（SIP INVITE/BYE/REFER/REGISTER），保留 ≥6 個月
- AI-IVR 異常時，通話由 MES 依原有規則分派至人工客服，旅客不受影響


### 5.3　WebCall SBC 整合（SIP Trunk Failover）

**整合對象：** WebCall Session Border Controller（雙備援）
**整合方式：** SIP Trunk 雙註冊 + 自動 Failover
**相關程式：** P-IVR-01（FR-IVR-001）
**相關決策：** AD-005、AD-007

#### 整合機制

台鐵既有系統配置兩台 WebCall SBC 確保網頁電話服務穩定性。AI-IVR 的 SIP UA 同時向 Primary 與 Secondary SBC 進行雙重 SIP Trunk 註冊，正常時僅接收 Primary SBC 的語音流，Primary 故障時自動切換至 Secondary，failover 時間目標 ≤5 秒。

#### 介面呼叫說明

| 介面項目 | 規格 |
|---------|------|
| **通訊協定** | SIP Trunk over TLS（WebCall 為網頁來源，強制加密） |
| **Failover 策略** | PJSIP 每 5 秒向雙 SBC 發送 SIP OPTIONS 心跳探測；Primary 連續 2 次無回應即標記為不可用，流量切至 Secondary |
| **切換影響** | 切換期間已建立的通話由 MES 維持，AI-IVR 僅遺失該通的 AI 輔助功能，通話本身不中斷 |
| **回切機制** | Primary 恢復後，新通話自動回到 Primary，既有通話維持在 Secondary 直到結束 |

#### 故障切換機制

WebCall SBC 採用 Active-Standby 雙機備援，透過 Keepalived VRRP 實現自動故障切換，切換時間 ≤3 秒。AI-IVR 端的 PJSIP SIP Trunk 同時向雙 SBC 註冊，故障切換對 AI-IVR 層透明，無需額外配置變更。


### 5.4　CRM Gateway 整合（REST / WebSocket）

**整合對象：** CRM Gateway（既有數位文字客服閘道器）
**整合方式：** REST API + WebSocket
**相關程式：** P-AGT-01（FR-AGT-001）、P-AGT-03（FR-AGT-003）、P-AGT-05（FR-AGT-006）
**相關決策：** AD-005

#### 整合機制

AI-Agent 透過 WebSocket 與 CRM Gateway 建立即時雙向通訊通道，接收旅客文字訊息並回傳 AI 回應。同時以 REST API 處理非即時操作（Session 歷史查詢、對話紀錄匯出等）。對外 REST API 依國發會「共通性應用程式介面規範」設計，採用 JSON 格式、統一錯誤碼、版本化 URI（`/api/v1/`）。

#### 介面呼叫說明

| 介面項目 | 規格 |
|---------|------|
| **即時通訊** | WebSocket（wss://），支援 ≥30 埠併發連線，心跳間隔 30 秒 |
| **訊息格式** | JSON（`session_id`、`channel`（Web/App）、`message`、`timestamp`） |
| **AI 回應** | JSON（`response_text`、`citations[]`（來源文件名稱、頁碼）、`confidence_score`） |
| **轉接通知** | AI-Agent 判定需轉真人時，透過 WebSocket 發送轉接事件，CRM Gateway 將對話路由至值機平台（P-WS-01），附帶對話摘要 |

| **認證方式** | API Key + HMAC 簽章，每個介接系統配發獨立 Key |

#### SMS 簡訊整合

SMS 簡訊收發系統（雙備援，4G LTE+）透過本機 REST API 與 AI-Agent 整合（P-SMS-01，FR-AGT-007）。旅客簡訊經 SMS 裝置接收後轉為 REST 請求送入 AI-Agent 處理，AI 自動回覆結果同步透過 WebSocket 推播至值機平台通知客服人員。簡訊紀錄寫入 PostgreSQL，保留 ≥5 年。雙備援 SMS 裝置採 Active-Standby 架構，Primary 裝置故障時系統自動切換至 Standby 裝置。

|---------|------|
| **通訊協定** | REST API（HTTPS），依國發會共通性應用程式介面規範 |
| **呼叫時機** | LLM RAG Pipeline 中的「工具呼叫」（Tool Use）階段：LLM 識別旅客意圖涉及即時資料時，觸發 API 查詢，將查詢結果注入 Prompt 作為 grounding context |
| **查詢類型** | 時刻表查詢（車次、起訖站、發車時間）、票價查詢（車種、區間、票種）、訂票紀錄查詢（身分證/手機號碼，需旅客授權） |
| **回應格式** | JSON，系統解析後納入 LLM Prompt，生成自然語言回覆並標註資料來源（RAG Citation） |
| **逾時處理** | API 呼叫逾時上限 2,000ms；逾時或失敗時 LLM 回覆「目前無法查詢即時資料，建議撥打專線或至台鐵官網查詢」，不以推測資料回覆 |
| **個資保護** | 訂票紀錄查詢需旅客口頭或文字授權確認後方可發起 API 呼叫；查詢結果不寫入對話紀錄之個資欄位，僅保留查詢時間與查詢類型 |

#### API 介接方式

透過台鐵既有訂票系統 RESTful API 介接，採用 OAuth 2.0 認證機制，API 規格依台鐵提供之介面規格書（ICD）實作。系統架構設有 API Adapter 抽象層，確保介面變更時僅需調整 Adapter 實作，不影響上層業務邏輯。

## 第二章　LLM / RAG 模組程式說明

本章說明大型語言模型服務（LLM）與知識庫管理（KB）共 7 支程式之程式功能、處理邏輯、使用資料庫及元件。LLM 推論部署於 AI 伺服器 GPU#0（NVIDIA L40S 48GB），採 Llama 3.1 8B-Instruct AWQ 4-bit 量化，透過 vLLM 0.5+ 提供推論服務（AD-004）；知識庫向量化與 RAG 檢索部署於 GPU#1（AD-002），確保即時推論與批次 Embedding 不互相搶奪資源。所有運算於台鐵機房地端完成，無任何資料離開機房。

模組間介面規格與系統流程圖詳見 #03 程式設計規格報告書第四章；資料庫完整 Schema 詳見 #03 第六章。

### 2.1　P-LLM-01　vLLM 推論引擎程式

**對應編號：** FR-LLM-001 / TC-LLM-001 / M-LLM-01
**KPI 關聯：** ivr_response ≤5,000ms、agent_response ≤5,000ms

#### 程式功能

管理 vLLM 推論引擎的生命週期，包含啟動配置、GPU 資源綁定、健康檢查與自動重啟。本程式為所有 LLM 推論請求的統一入口，以 gRPC 介面（內部高頻通訊）接收來自 AI-IVR 與 AI-Agent 的推論請求，並透過 vLLM 的 continuous batching 機制併發處理。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | gRPC 推論請求（prompt 文字、max_tokens、temperature、session_id） |
| **輸出** | 生成文字（streaming token 或完整回應）、token 使用量統計 |
| **處理程序** | ① 接收 gRPC 請求 → ② 將請求投入 vLLM AsyncEngine 的請求佇列 → ③ vLLM 以 PagedAttention 管理 KV Cache，continuous batching 合併多路請求 → ④ AWQ 4-bit 量化推論（模型權重 ~8GB + KV Cache 動態分配，峰值 VRAM ~42GB） → ⑤ 回傳 streaming token 或完整生成結果 |
| **存取資料** | 模型權重檔（本機磁碟，唯讀）、vLLM 推論指標（Prometheus metrics endpoint） |
| **計算條件** | 單卡 L40S 48GB 承載單一 vLLM 引擎（Llama 3.1 8B AWQ），以 continuous batching + PagedAttention 服務 ≤30 路並行推論；推論延遲預算 ≤2,500ms（佔 5,000ms SLA 之 50%） |

#### 健康檢查與自動重啟

- 每 10 秒對 vLLM 引擎發送心跳探測，檢查推論佇列深度與 GPU 記憶體使用率
- 連續 3 次心跳逾時（>5 秒無回應），觸發 vLLM 引擎自動重啟
- GPU 記憶體使用率 >90% 時，暫停接受新請求並發送告警至管理後台（FR-ADM-003）
- 所有推論請求與異常事件寫入結構化日誌，保留 ≥6 個月

#### 使用元件

| 元件 | 用途 |
|-----|------|
| vLLM 0.5+（PagedAttention + continuous batching） | LLM 推論引擎 |
| Llama 3.1 8B-Instruct AWQ 4-bit | 推論模型 |
| gRPC（Python grpcio） | 內部服務間通訊 |
| Prometheus client | 推論指標匯出（QPS、延遲分佈、GPU 使用率） |

### 2.2　P-LLM-02　語意理解與對話引擎程式

**對應編號：** FR-LLM-002 + FR-LLM-003 / TC-LLM-002、TC-LLM-003 / M-LLM-02
**KPI 關聯：** ivr_accuracy ≥85%、agent_accuracy ≥85%

#### 程式功能

本程式為 LLM 模組的核心，實作完整的 RAG Pipeline 與多輪對話管理。接收經 STT 轉寫或文字通道傳入的旅客查詢，經向量檢索、Rerank、Prompt 組裝、LLM 生成、Citation 抽取，產出附帶來源標註的回應文字。FR-LLM-002（語意理解與問答生成）與 FR-LLM-003（上下文關聯與連續對話）合併實作，因兩者共用同一 Prompt 組裝流程與 KV Cache 管理，拆分將導致重複建立對話上下文（詳見第一章 §1.3）。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 旅客查詢文字（經個資遮罩處理）、session_id、對話歷史摘要（自 Redis 讀取） |
| **輸出** | 回應文字、信心度分數（0~1）、Citation 來源清單（文件名稱/頁碼/段落編號）、意圖分類標籤 |
| **處理程序** | 見下方 RAG Pipeline 七階段 |
| **存取資料** | Qdrant 向量資料庫（語意檢索）、Redis（Session 對話上下文）、PostgreSQL（歷史對話摘要、知識庫 metadata） |
| **計算條件** | 信心度門檻驅動三層回應策略（見下方）；Qdrant 檢索 Top-K ≤200ms；LLM 推論 ≤2,500ms |

#### RAG Pipeline 七階段

```
旅客查詢 → ① 查詢改寫 → ② 向量檢索 → ③ Rerank
         → ④ Prompt 組裝 → ⑤ LLM 生成 → ⑥ Citation 抽取
         → ⑦ 信心度評估與回應路由
```

1. **查詢改寫（Query Rewriting）**：將旅客口語化查詢改寫為適合向量檢索的標準化形式。結合 Redis 中的前序對話上下文，解析指代詞（「那班車」→「14:30 自強號 1234 次」）與省略語意。處理時間預算 ≤50ms。

2. **向量檢索（Vector Retrieval）**：以 multilingual-e5-large Embedding 模型（約 1.3GB，部署於 GPU#1）將改寫後查詢向量化，於 Qdrant 執行 HNSW 近鄰搜尋，取回 Top-K 候選文件片段（預設 K=10）。每筆候選保留完整 metadata（文件名稱、頁碼、段落編號、更新時間戳），供後續 Citation 標註使用。查詢延遲 <50ms。

3. **Rerank 重排序**：以交叉編碼器（Cross-Encoder）對 Top-K 候選依查詢相關性重新排序，取前 3~5 筆作為 LLM 生成的參考上下文。此步驟提升檢索精準度，降低無關文件干擾 LLM 生成品質。

4. **Prompt 組裝**：組裝結構化 Prompt，依序注入：
   - System Instruction（限定回答範圍為台鐵業務、繁體中文回覆、禁止編造資訊）
   - 前序對話摘要（自 Redis 讀取，維持多輪對話連貫性）
   - Rerank 後的知識片段（含 metadata 標記，供 Citation 抽取）
   - 旅客當次查詢

5. **LLM 生成**：透過 P-LLM-01 呼叫 vLLM 推論引擎，以 streaming token 方式回傳生成結果。推論延遲預算 ≤2,500ms（128 input + 64 output token 之 P95 延遲約 1.2 秒）。

6. **Citation 抽取**：解析 LLM 生成文字中的來源標記，對應至步驟 2 保留的 metadata，輸出結構化 Citation 清單（文件名稱、頁碼、段落編號）。前端以 footnote 或 tooltip 方式呈現來源連結（FR-AGT-004）。

7. **信心度評估與三層回應路由**：依 LLM 生成結果與檢索匹配程度計算信心度分數（0~1），驅動三層回應策略：

| 信心度區間 | 回應策略 | 說明 |
|-----------|---------|------|
| ≥0.85 | 直接回答 | 回應文字附帶 Citation 來源標註 |
| 0.6 ~ 0.85 | 回答 + 人工確認提示 | 回應文字末尾加註「建議洽人工客服確認」 |
| <0.6 | 轉接人工客服 | 不輸出 AI 生成回答，直接觸發轉接流程（P-IVR-03 / P-AGT-05），附帶對話摘要 |

涉及票價、時刻等關鍵營運資訊時，不論信心度高低，一律透過 API 即時查詢台鐵既有系統取得資料，不依賴 LLM 參數記憶，避免過時或錯誤資訊誤導旅客。所有 AI 回覆一律標註「本回覆由 AI 生成，如有疑問請按 0 轉接專人」（語音通道）或「如需人工協助請點選轉接」（文字通道）。

#### 多輪對話上下文管理

- 以 Redis 儲存當次對話 Session Context（TTL = 通話時長 + 5 分鐘緩衝）
- 每輪交互結束後，將當輪對話摘要（旅客意圖 + AI 回應要點）壓縮後追加至 Session Context
- 當 Session Context 超過 Prompt 容量上限時，以滑動窗口保留最近 5 輪對話，並以 LLM 摘要壓縮更早的歷史
- 通話/對話結束後，完整對話紀錄寫入 PostgreSQL，Session Context 自 Redis 清除

#### 使用資料庫表格

| 資料庫 | 表格/Collection | 關鍵欄位 | 用途 |
|-------|----------------|---------|------|
| Qdrant | `kb_vectors` | vector（1024 維）、doc_id、doc_name、page、paragraph、updated_at | 向量語意檢索 |
| Redis | `session:{session_id}` | dialog_history、intent_state、user_context | 即時對話上下文 |
| PostgreSQL | `conversations` | id、session_id、channel（voice/text）、started_at、ended_at | 對話主記錄 |
| PostgreSQL | `conversation_turns` | id、conversation_id、role（user/assistant）、content、confidence、citations_json、created_at | 逐輪對話內容 |
| PostgreSQL | `user_history_summary` | user_id、summary_vector、last_updated | 個性化歷史摘要（供 P-AGT-04） |

### 2.3　P-LLM-03　文字處理引擎程式

**對應編號：** FR-LLM-004 / TC-LLM-004 / M-LLM-03
**KPI 關聯：** ivr_accuracy ≥85%、agent_accuracy ≥85%

#### 程式功能

對旅客輸入文字與 LLM 生成回應執行前處理與後處理，包含同義詞擴展、敏感詞過濾、錯別字修正三大功能。本程式作為 P-LLM-02 RAG Pipeline 的前置與後置處理器，提升查詢檢索品質與回應安全性。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 旅客原始查詢文字（前處理）；LLM 生成回應文字（後處理） |
| **輸出** | 正規化查詢文字（前處理）；過濾後安全回應文字（後處理） |
| **處理程序** | 前處理：① 錯別字修正（Levenshtein 距離 + 台鐵專屬字典比對）→ ② 同義詞擴展（「自強」↔「自強號」、「左營」↔「新左營」等） → ③ 敏感詞偵測（個資正則遮罩：身分證/電話/信用卡號）；後處理：④ 回應文字敏感詞過濾 → ⑤ 繁體中文校驗（攔截簡體字元） |
| **存取資料** | PostgreSQL `synonym_dict`（同義詞對照表）、`sensitive_words`（敏感詞規則庫）、`correction_dict`（錯別字修正字典） |
| **計算條件** | 前處理 + 後處理合計 ≤100ms |

#### 台鐵專屬字典

系統內建台鐵業務領域字典，涵蓋：
- 車站名稱（含簡稱、舊名、易混淆站名）
- 車種名稱（自強號、太魯閣、普悠瑪、區間車）
- 票種名稱（去回票、電子票證、定期票）
- 常見口語與錯別字對照（「板喬」→「板橋」、「自強好」→「自強號」）

字典維護透過管理後台（FR-ADM-001）由台鐵系統管理者自行新增修改，修改後即時生效，無需重啟服務。

### 2.4　P-LLM-04　對話紀錄管理程式

**對應編號：** FR-LLM-007 / TC-LLM-007 / M-LLM-04
**KPI 關聯：** log_retention ≥6 個月

#### 程式功能

管理所有語音與文字對話之完整紀錄，提供五種格式匯出（Word / PDF / Excel / HTML / CSV），並支援依時間、通道、客服人員、意圖分類等條件進行查詢篩選。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 匯出請求（格式類型、篩選條件、日期範圍）；對話完成事件（自 P-LLM-02 / P-AGT-03 接收） |
| **輸出** | 指定格式檔案（Word/PDF/Excel/HTML/CSV）；對話紀錄查詢結果（分頁 JSON） |
| **處理程序** | ① 對話完成時，將 Redis Session Context 序列化寫入 PostgreSQL `conversations` + `conversation_turns` → ② 匯出時依篩選條件查詢 PostgreSQL → ③ 以模板引擎渲染指定格式 → ④ 旅客個資欄位匯出前強制去識別化處理 |
| **存取資料** | PostgreSQL `conversations`、`conversation_turns`、`export_templates` |
| **計算條件** | 單次匯出 ≤1,000 筆對話，超過時以非同步任務處理並通知下載 |

#### 使用元件

| 元件 | 用途 |
|-----|------|
| python-docx | Word 格式匯出 |
| ReportLab | PDF 格式匯出 |
| openpyxl | Excel 格式匯出 |
| Jinja2 | HTML / CSV 模板渲染 |

### 2.5　P-LLM-05　後送案件分析程式

**對應編號：** FR-LLM-008 / TC-LLM-008 / M-LLM-05
**KPI 關聯：** classify_accuracy ≥85%

#### 程式功能

對客服後送案件執行自動分類、優先級判定、情感分析與趨勢預測，分析結果透過 API 推送至既有後送案件系統，225 位外點後送人員之作業流程維持不變。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 後送案件文字內容（來自對話摘要或客服人員手動建立）、案件 metadata |
| **輸出** | 分類標籤（一級/二級分類）、優先級（緊急/高/中/低）、情感標籤（正面/中性/負面 + 強度分數）、趨勢標籤 |
| **處理程序** | ① 案件文字前處理（P-LLM-03 同義詞擴展 + 敏感詞過濾）→ ② 呼叫 P-LLM-01 執行分類推論（以 few-shot Prompt 設計，注入分類定義與範例）→ ③ 優先級判定（規則引擎：含關鍵詞「安全」「事故」自動升為緊急）→ ④ 情感分析（LLM 判定情感極性與強度）→ ⑤ 結果寫入 PostgreSQL 並推送至後送案件系統 API |
| **存取資料** | PostgreSQL `backlog_cases`（案件主表）、`backlog_analysis`（分析結果）、`classification_rules`（分類定義與規則） |
| **計算條件** | 分類正確率 ≥85%（以 200 筆真實後送案件驗收，TC-LLM-008）；單筆分析 ≤3 秒 |

#### 使用介面

- **輸入介面：** 值機平台（P-WS-03）建立後送工單時自動觸發，或管理後台批次匯入
- **輸出介面：** REST API 推送至既有後送案件系統（API 規格待得標後與台鐵確認）；分析結果呈現於值機平台工單詳情頁與報表儀表板（P-RPT-01）

### 2.6　P-KB-01　知識庫管理引擎程式

**對應編號：** FR-LLM-005 / TC-LLM-005 / M-KB-01
**關聯 AD：** AD-004、AD-008

#### 程式功能

管理台鐵 AI 客服知識庫的完整生命週期，包含知識條目之新增、修改、刪除（CRUD）、版本管理（git-like snapshot）、增量 Embedding 更新與 Qdrant Collection 管理。知識庫是 RAG 問答品質的基礎——系統於得標後 D+50 前取得台鐵提供的種子資料（FAQ、SOP、規章），並透過本程式持續擴充與維護。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 知識條目（標題、內容、分類、附件）；批次匯入檔案（Word/PDF/Excel）；版本操作指令（commit/rollback/diff） |
| **輸出** | 操作結果回應；版本差異報告；Embedding 更新狀態 |
| **處理程序** | ① 知識條目 CRUD → ② 文件解析與 chunking（依段落/頁面切分，保留 metadata）→ ③ multilingual-e5-large Embedding 生成（GPU#1，模型約 1.3GB）→ ④ 向量與 metadata 寫入 Qdrant `kb_vectors` Collection → ⑤ 版本快照記錄（記錄變更差異，支援回溯至任一歷史版本） |
| **存取資料** | PostgreSQL `kb_documents`（文件主表）、`kb_chunks`（切分片段）、`kb_versions`（版本歷史）；Qdrant `kb_vectors` |
| **計算條件** | 單篇文件 Embedding 更新 ≤10 秒（≤50 chunks）；批次匯入以非同步任務執行，不阻塞線上服務 |

#### 增量更新策略

- 知識條目修改時，僅重新計算受影響 chunks 的 Embedding，不全量重建向量索引
- Qdrant Collection 使用 HNSW 索引，新增/更新向量後索引自動漸進式重建，不影響線上查詢
- 版本管理以 snapshot 模式記錄每次變更的差異（新增/修改/刪除的 chunk 清單），支援一鍵回溯至指定版本

#### 使用資料庫表格

| 資料庫 | 表格/Collection | 關鍵欄位 | 用途 |
|-------|----------------|---------|------|
| PostgreSQL | `kb_documents` | id、title、category、content、status、created_by、created_at、updated_at | 知識條目主表 |
| PostgreSQL | `kb_chunks` | id、doc_id、chunk_index、chunk_text、char_offset、page、paragraph | 文件切分片段 |
| PostgreSQL | `kb_versions` | id、version_tag、changes_json、created_by、created_at | 版本快照歷史 |
| Qdrant | `kb_vectors` | vector（1024 維）、doc_id、chunk_id、doc_name、page、paragraph、updated_at | 向量語意索引 |

#### 使用介面

- **管理後台（FR-ADM-001）：** 系統管理者透過 Web UI 進行知識條目 CRUD、批次匯入、版本管理
- **REST API：** 供外部系統或排程任務呼叫（符合國發會共通性應用程式介面規範）

### 2.7　P-KB-02　知識庫權限控管程式

**對應編號：** FR-LLM-006 / TC-LLM-006 / M-KB-02
**KPI 關聯：** log_retention ≥6 個月

#### 程式功能

依 RBAC（角色型存取控制）模型管理知識庫的存取權限，整合台鐵既有 AD（Active Directory）進行帳號驗證，所有操作均記錄完整稽核日誌。個資相關知識條目之存取日誌獨立管控，符合旅客個資保護之最小權限原則。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 使用者操作請求（含 JWT Token）、AD/LDAP 驗證憑證 |
| **輸出** | 授權結果（允許/拒絕）、操作日誌寫入確認 |
| **處理程序** | ① 解析 JWT Token 取得使用者身分 → ② 透過 LDAP 向台鐵 AD 驗證帳號有效性 → ③ 查詢 RBAC 權限表，判斷使用者對目標知識條目是否具有所需權限（讀取/編輯/刪除/管理） → ④ 授權通過則放行並寫入操作日誌；拒絕則回傳 403 並寫入異常存取日誌 |
| **存取資料** | PostgreSQL `rbac_roles`（角色定義）、`rbac_permissions`（權限對應）、`rbac_user_roles`（使用者角色指派）、`audit_logs`（操作日誌）、`pii_access_logs`（個資存取日誌，獨立表格） |
| **計算條件** | 權限驗證 ≤20ms（RBAC 查詢結果快取於 Redis，TTL = 5 分鐘）；日誌寫入採非同步方式，不影響請求延遲 |

### 2.8　模組間介面總覽

以下為本章 7 支程式與其他模組之主要呼叫關係：

```
P-IVR-01/02 ──gRPC──→ P-LLM-02 ──gRPC──→ P-LLM-01（vLLM 推論）
P-AGT-03    ──gRPC──→ P-LLM-02 ──查詢──→ Qdrant（P-KB-01 管理）
                         │
                    P-LLM-03（前/後處理）
                         │
P-WS-03  ──REST──→ P-LLM-05（後送分析）──API──→ 既有後送案件系統
P-ADM-01 ──REST──→ P-KB-01（知識庫 CRUD）──LDAP──→ AD（P-KB-02 驗證）
P-LLM-02 ──寫入──→ P-LLM-04（對話紀錄持久化與匯出）
```

所有內部高頻通訊（AI-IVR/AI-Agent → LLM 推論）使用 gRPC 協定（二進制序列化 + HTTP/2 多工），延遲比 REST 低 2~5 倍；對外 API（管理後台、既有系統介接）使用 REST，符合國發會共通性應用程式介面規範。介面規格詳見 #03 第五章模組設計規格。

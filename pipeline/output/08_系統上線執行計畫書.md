## 第一章　上線策略與總體規劃

### 8.1.1　上線策略概述

本案採**灰度上線**策略（10% → 30% → 100%），分三階段逐步擴大 AI 客服服務範圍，而非一次全量切換。此策略之核心考量如下：

1. **風險可控：** 每階段僅開放有限流量由 AI 處理，若出現正確率不足或系統異常，可立即將流量回歸人工客服，不影響既有服務運作。
2. **逐步驗證 KPI：** 各階段設定明確的 Go/No-Go 判定標準，確保 IVR 正確率 ≥85%（200 題驗證）、回應時間 ≤5,000ms、AI-Agent 正確率 ≥85% 等關鍵指標達標後方擴大範圍。
3. **避開尖峰時段：** 灰度上線期間不安排於春節疏運、連假收假等話務量暴增時段啟動新階段切換，確保系統在正常話務基線下完成穩定性驗證。
4. **人工客服零衝擊：** 全程維持既有人工客服系統正常運作，AI-IVR 透過 SIP/RTP fork 被動接收語音流（AD-007），不修改 MES Server 任何設定，AI 模組為新增獨立架構，故障時自動退場即可。

### 8.1.4　Go/No-Go 判定標準

每一灰度階段結束時，由專案經理召集台鐵承辦人員、客服中心主管及系統開發團隊召開**階段檢討會議**，依下表判定是否進入下一階段：

| 指標 | Go 條件 | No-Go 觸發值 | 處置方式 |
|-----|---------|-------------|---------|
| AI 回答正確率 | ≥85% | <70% | 暫停灰度，回歸人工客服，啟動知識庫補強與模型校正 |
| 回應時間 | ≤5,000ms（P95） | >8,000ms（P95） | 降低 vLLM 並行推論上限、排查 GPU 負載瓶頸 |
| 系統可用性 | ≥99%/季 | <90% | 暫停灰度，全面排查後重啟 |
| MTTR | ≤4hr | >4hr | 強化監控告警與值班機制 |
| 轉人工比例 | ≤40% | >60% | 分析轉接原因，補充知識庫高頻問題 |

> **閾值說明：** Go/No-Go 閾值與契約 KPI 一致（正確率 ≥85%、可用性 ≥99%/季），確保灰度期間即以正式標準驗證系統品質。任一階段未達標即暫停並啟動回滾程序（§8.1.5）。

### 8.1.5　回滾方案

當任一灰度階段觸發 No-Go 條件時，執行以下回滾程序：

1. **即時切換：** AI-IVR 停止接聽新來電，SIP 路由將全部流量導回 MES Server 原始路徑，由人工客服接手。此切換僅需調整 SIP 路由規則，不涉及 MES Server 設定變更，預估切換時間 ≤5 分鐘。
2. **AI-Agent 退場：** CRM Gateway 停止轉發新訊息至 AI-Agent，文字客服回歸人工作業模式。進行中的 AI 對話以「系統維護中，為您轉接專人」訊息結束。
3. **資料保全：** 回滾期間所有對話紀錄、系統日誌完整保留，供後續問題分析使用。
4. **通報機制：** 回滾觸發後 1 小時內通報台鐵承辦人員，72 小時內完成損害控制與根因分析報告。

回滾後的 AI 系統不影響既有人工客服運作（合規紅線 #7），因 AI-IVR 為獨立於 MES Server 之外的新增模組，停用 AI 等同回復至建置前狀態。


## 第二章　上線範圍與工作項目

### 8.2.1　上線範圍總覽

本案上線範圍涵蓋六大子系統，完整覆蓋語音、文字、簡訊三大客服通道，以及後端質檢、報表與管理功能。各子系統對應之功能需求編號、灰度階段與上線條件如下表：

| # | 子系統 | 功能需求編號 | 灰度階段 | 上線前提條件 |
|---|-------|------------|---------|------------|
| 1 | **AI-IVR 智慧語音應答** | FR-IVR-001 ~ FR-IVR-007 | 階段一（10%）起 | SIP/RTP 介接驗證通過、STT/TTS 模型部署完成、≥30 埠壓力測試通過 |
| 2 | **AI-Agent 虛擬客服** | FR-AGT-001 ~ FR-AGT-008 | 階段二（30%）起 | CRM Gateway API 介接驗證通過、向量資料庫（Qdrant）知識庫匯入完成、≥30 埠壓力測試通過 |
| 3 | **AI-QI 智慧質檢** | FR-QI-001 ~ FR-QI-007 | 階段三（100%）起 | 質檢規則設定完成、15 分鐘錄音 → ≤5 分鐘產出報告之效能驗證通過 |
| 4 | **SMS 簡訊服務** | FR-AGT-007 | 階段三（100%）起 | 雙備援 SMS 裝置上線測試通過、簡訊紀錄保留機制驗證（≥5 年儲存容量確認） |
| 5 | **查詢及報表** | FR-RPT-001 ~ FR-RPT-003 | 階段二（30%）起 | 資料來源串接完成、儀表板呈現正確性驗證 |
| 6 | **值機平台（WS）與管理後台（ADM）** | FR-WS-001 ~ FR-WS-005、FR-LLM-006 | 階段一（10%）起 | AD 帳號整合測試通過、RBAC 角色設定完成、全員帳號開通 |

### 8.2.2　各子系統上線內容

**一、AI-IVR 智慧語音應答系統**

上線內容涵蓋語音通道接入（E1 專線 30 B-channel + WebCall SBC 雙備援 + 行政分機類比轉 SIP）、STT 即時語音辨識（faster-whisper，CPU 推論，辨識率 ≥85%）、LLM 智慧應答（vLLM + Llama 3.1 8B-Instruct AWQ，GPU #0 專責推論，AD-002）、TTS 語音合成（XTTS-v2，CPU 推論）、多輪對話管理、DTMF 按碼偵測、轉人工客服機制及滿意度調查。AI 回應採三層信心度策略：≥0.85 直接回答、0.6~0.85 回答並提示「建議洽人工確認」、<0.6 直接轉接人工，確保錯誤回答不致無控制地到達旅客端。涉及票價、時刻等關鍵資訊一律由 API 即時查詢，不依賴 LLM 記憶。

**二、AI-Agent 虛擬客服系統**

上線內容涵蓋 ≥30 埠數位文字通道（Web + App）、RAG 檢索增強生成（Qdrant 向量資料庫 + multilingual-e5-large embedding）、互動式多輪文字問答、回應引用來源標註（RAG Citation）、個性化問答機制、AI 轉真人客服（附對話摘要）、系統持續優化與校正機制。AI-Agent 經由 CRM Gateway 以 REST/WebSocket API 介接既有數位客服平台，回覆內容限定於台鐵指定知識庫範圍，並標註「本回覆由 AI 生成，如有疑問請洽專人」。

**三、AI-QI 智慧質檢系統**

上線內容涵蓋多格式音檔匯入與批次排程、客服錄音語音轉文字（針對台鐵業務術語優化）、自訂質檢規則與評分機制、靜音與插話偵測、語調與情緒分析、異常偵測警報、多維度質檢報表。效能目標為 15 分鐘錄音 ≤5 分鐘完成轉換分析並產出質檢報告。質檢功能部署於 GPU #1（AD-002），與 LLM 推論互不搶奪運算資源。

**四、SMS 簡訊服務**

上線內容涵蓋雙備援 SMS 收發裝置（4G LTE+，裝置品牌與 LTE 模組均為非大陸品牌/非大陸製造）、AI 自動回覆簡訊同步通知客服人員、簡訊紀錄保留 ≥5 年（裝置儲存容量 256GB+，詳見 #06 §6.10 容量估算）。SMS 裝置規格詳見第六章（#06）§6.2.1。

**五、查詢及報表功能**

上線內容涵蓋對話服務類、AI 管理類、知識點分析、滿意度等統計分析報表（FR-RPT-001），視覺化儀表板支援自訂篩選條件與時間區間（FR-RPT-002），以及保固期內 ≥10 個客製報表服務（FR-RPT-003）。報表模組整合 Prometheus + Grafana 監控數據（AD-008），提供即時營運指標。

**六、值機平台（WS）與管理後台（ADM）**

值機平台上線內容涵蓋座席操作介面與通話控制（接聽/保留/轉接/掛斷）、AI 知識輔助即時建議推送、對話摘要與後送工單管理、督導即時監控與密語指導介入、座席狀態管理（簽入/簽出/就緒/忙碌/休息）。管理後台上線內容涵蓋 AD 整合帳號驗證（LDAP）、RBAC 多層級權限控管（FR-LLM-006）、知識庫維護介面（新增/修改/刪除/版本管理）、系統參數設定、監控儀表板與日誌稽核（日誌保存 ≥6 個月）。

## 第三章　實施方法

### 8.3.1　漸進式上線實施架構

本案上線實施採**漸進式部署（Canary Release）**方法論，透過 SIP 路由層與 CRM Gateway 訊息路由層分別控制語音與文字通道的流量比例，逐步擴大 AI 服務範圍。此方法之核心機制如下：

**語音通道流量控制（AI-IVR）**

AI-IVR 透過 SIP/RTP fork 被動接收語音流（AD-007），不修改 MES Server 任何設定。漸進比例透過 SIP 路由規則控制：

| 漸進階段 | 路由規則 | 流量來源 | AI 處理範圍 |
|---------|---------|---------|-----------|
| 10% | 僅 WebCall SBC → AI-IVR | 網頁電話 | 簡單 FAQ（時刻/票價/車站資訊） |
| 30% | WebCall + PSTN E1 部分 B-channel fork | 網頁電話 + 部分公眾電話 | 全類型查詢 |
| 100% | 全通道（含行政分機類比轉 SIP） | 所有語音來源 | 全服務範圍 |

階段一選擇 WebCall 通道作為首波漸進上線對象，理由為 WebCall 使用者多為熟悉數位工具之旅客，且 WebCall SBC 雙備援架構（詳見 #06 §6.9.2）可在 AI 退場時無縫回歸，風險最低。

**文字通道流量控制（AI-Agent）**

AI-Agent 經由 CRM Gateway 以 REST/WebSocket API 介接既有數位客服平台（詳見 #02 §2.14），漸進控制透過 CRM Gateway 的訊息路由設定，依百分比將新進對話分派至 AI 或人工佇列。階段二起啟用，階段三達成 ≥30 埠全量服務（FR-AGT-001）。

**流量比例動態調整機制**

漸進上線期間，專案經理依每日 KPI 監控數據（§8.1.4 Go/No-Go 五項指標），可在階段內微調流量比例。例如階段二期間若 GPU 使用率持續 >85%，可將 PSTN 通道的 fork 比例從 30% 降至 20%，待 vLLM 引擎的推論優化調校完成後再回升。調整操作透過 SIP 路由管理介面執行，無需重啟任何服務。

### 8.3.3　回滾計畫

回滾計畫區分為**局部回滾**（單一子系統退場）與**全面回滾**（AI 全數退場）兩種情境：

**局部回滾**

當單一子系統觸發 No-Go 條件，但其餘子系統運作正常時，僅將該子系統流量導回人工處理：

| 情境 | 回滾操作 | 預估時間 | 影響範圍 |
|-----|---------|---------|---------|
| AI-IVR 正確率 <70% | SIP 路由切回 MES 原路徑，停止 RTP fork | ≤5 分鐘 | 語音通道回歸人工接聽 |
| AI-Agent 回應異常 | CRM Gateway 停止轉發至 AI | ≤3 分鐘 | 文字客服回歸人工 |
| AI-QI 效能不達標 | 停止品檢批次排程 | ≤1 分鐘 | 品檢回歸人工抽檢 |
| SMS 收發失敗 | 切換至備援裝置或停用 AI 自動回覆 | ≤5 分鐘 | SMS 回歸人工處理 |

局部回滾不影響其餘正常運作之子系統，此為六層式架構（AD-001）的隔離優勢——各子系統獨立部署於 Docker 容器中（AD-006），停用任一服務不波及其他。

**全面回滾**

當系統可用性 <90% 或 MTTR >4 小時時，執行全面回滾：

1. PM 下達回滾指令，通知台鐵承辦人
2. SIP 路由全部切回 MES 原路徑（≤5 分鐘）
3. CRM Gateway 全部對話回歸人工（≤3 分鐘）
4. 進行中的 AI 通話以「系統維護中，為您轉接專人」語音/訊息結束
5. 全面回滾後 1 小時內完成事件通報（符合服務水準表 incident_notify ≤1hr 要求）
6. 72 小時內完成損害控制與根因分析報告（符合 incident_control ≤72hr 要求）

全面回滾後，既有人工客服系統完全不受影響（合規紅線 #7），因 AI 模組為獨立於 MES Server 之外的新增架構，停用 AI 等同回復至系統建置前狀態。回滾後之修復措施與重新上線時程，由 PM 與台鐵承辦人協商決定。

### 8.3.4　降級策略

除回滾外，系統設計內建三級降級機制，在不完全退場的情況下維持部分 AI 服務：

| 降級等級 | 觸發條件 | 降級行為 | 影響 |
|---------|---------|---------|------|
| L1 輕度降級 | GPU 使用率 >85% 持續 5 分鐘 | 降低 vLLM 引擎的並行推論上限（如從 30 降至 15），延長排隊但不拒絕 | 回應時間略增，不超過 5,000ms SLA |
| L2 中度降級 | 單台 AI 伺服器故障 | HA Failover（AD-005），存活機承載 100% 流量 | Keepalived VIP 漂移 ≤5 秒，服務短暫中斷後自動恢復 |
| L3 重度降級 | 外部 API（訂票/時刻表）不可用 | LLM 回覆「此功能暫時維護中，請按 0 洽專人」 | 該類查詢轉人工，其餘 FAQ 類 AI 服務不受影響 |

降級機制由 Prometheus 警示自動觸發，無需人工介入。L2 降級（HA 切換）為自動化流程，詳見 #06 §6.8.2。

### 8.4.2　版本管理策略

**映像檔標籤規則**

所有 Docker 映像檔採語意化版本標籤（Semantic Versioning）：

```
tra-ai-{服務名稱}:{主版本}.{次版本}.{修訂版}-{建置日期}
```

範例：`tra-ai-ivr:1.0.0-20260901`、`tra-ai-agent:1.0.0-20260901`

上線版本統一標記為 `v1.0.0`，灰度期間的修正以修訂版號遞增（v1.0.1、v1.0.2），保固期間的功能增補以次版號遞增（v1.1.0、v1.2.0）。

**版本紀錄與追溯**

每次版本異動須記錄於版本變更日誌（CHANGELOG），內容包含：變更項目、變更原因、影響範圍、測試結果、核准人。版本變更日誌隨 #11 原始程式碼光碟交付台鐵，確保完整追溯性。

**升級與回滾操作**

Docker Compose 部署架構（AD-006）使版本升級與回滾操作簡化為：

```bash
# 升級：拉取新版映像檔並重啟服務
docker compose pull {服務名稱}
docker compose up -d {服務名稱}

# 回滾：切換至前一版本標籤
docker compose stop {服務名稱}
# 修改 docker-compose.yaml 中的映像檔標籤為前一版本
docker compose up -d {服務名稱}
```

單一服務升級/回滾不影響其餘服務運作，預估操作時間 ≤5 分鐘。此操作流程將納入 #07 軟體使用及維護手冊，以台鐵 IT 人員為讀者撰寫 step-by-step 操作指引。

1. **知識庫覆蓋率補強：** 統計每日「AI 無法回答」之問題類型，優先補充高頻問題至知識庫，目標為覆蓋率每日提升 1~2%
2. **信心度閾值校正：** 依實際回答正確率微調三層信心度閾值（詳見 #02 §2.4 FR-LLM-002），必要時暫時提高轉人工閾值以降低錯誤回答風險
3. **STT 術語辨識優化：** 收集台鐵特有站名、車種代號等高頻辨識錯誤，加入 faster-whisper 的 hotword 清單
4. **TTS 自然度調校：** 收集旅客滿意度調查中「語音品質」相關回饋，調整 XTTS-v2 語速與語調參數

**第二階段：穩定觀察期（D+231 ~ D+250，共 20 天）**

| 項目 | 內容 |
|-----|------|
| 監控頻率 | 每週 KPI 報表（改為週頻） |
| 會議頻率 | 每週一次階段檢討會議 |
| 駐點安排 | PM 每週駐點 2 天，工程師遠端待命 |
| 調校重點 | 長期穩定性觀察、邊際 case 修正、效能基線建立 |
| 驗收準備 | 收集 KPI 數據供正式驗收使用 |

此階段重點為驗證系統在連續運作下的穩定性，觀察指標包含：

- 系統可用性是否穩定維持 ≥99%（以週為單位統計）
- 每日最大停機時間是否 ≤4 小時
- GPU 記憶體是否有洩漏趨勢（VRAM 使用量逐日上升）
- Redis Session 數量是否正常回收（無累積）
- 記錄檔儲存空間消耗速率是否符合 #06 §6.10 估算

**第三階段：驗收過渡期（D+251 ~ D+270，共 20 天）**

| 項目 | 內容 |
|-----|------|
| 主要工作 | 初驗 → 缺失改善 → 複驗 → 正式驗收 |
| KPI 驗證 | 以台鐵提供之 200 題驗證 IVR 正確率 ≥85%、Agent 正確率 ≥85%、200 音檔驗證 STT ≥85%、200 筆後送案件驗證分類正確率 ≥85% |
| 效能驗證 | IVR 回應 ≤5,000ms、Agent 回應 ≤5,000ms、QI 15 分鐘錄音 → ≤5 分鐘完成 |
| 併發驗證 | IVR ≥30 埠、Agent ≥30 埠同時在線 |
| 文件交付 | D+240 第三期交付完成、D+270 結案報告交付 |

驗收流程詳見第五章。正式驗收通過後，系統進入保固期，24/7 監控與 4 小時故障修復機制（含假日）正式啟動，詳見第六章。

### 8.4.4　觀察期異常處理流程

觀察期間發生異常事件時，依下列分級處理：

| 事件等級 | 定義 | 處理時限 | 處理方式 |
|---------|------|---------|---------|
| P1 緊急 | 系統全面不可用，或影響 ≥50% 使用者 | 立即回應，MTTR ≤4hr | 全面回滾 → 根因分析 → 修復 → 重新漸進上線 |
| P2 重要 | 單一子系統異常，影響 <50% 使用者 | 1 小時內通報，4hr 內修復 | 局部回滾 → 修復 → 重新啟用 |
| P3 一般 | 非關鍵功能異常（如報表延遲） | 次營業日處理 | 納入修正清單，下次維護窗口修復 |
| P4 輕微 | UI 顯示問題、非功能性建議 | 5 個工作日內處理 | 納入版本更新計畫 |

所有 P1/P2 事件須於知悉 1 小時內通報台鐵承辦人（符合 incident_notify ≤1hr），並於 72 小時內（重大事件 36 小時內）完成損害控制與復原（符合 incident_control 要求）。事件調查報告於損害控制後 1 個月內提交。


## 第五章　驗收流程

### 8.5.1　初驗作業

初驗期間為 D+241 ~ D+250，共 10 天。台鐵組成驗收委員會，依契約附件驗收清單逐項確認系統功能、效能及文件完整性。初驗作業流程如下：

1. **驗收委員會組成：** 由台鐵承辦單位召集資訊、客服、稽核等相關部門代表組成驗收委員會，必要時得邀請外部專家學者參與。
2. **驗收排程：** 委員會於初驗首日（D+241）召開啟動會議，確認驗收範圍、分工與排程，預計分 3 個工作日完成功能驗證、2 個工作日完成效能驗證、2 個工作日完成文件審查、3 個工作日彙整報告。
3. **現場展示：** 承商依驗收清單逐項進行系統功能展示，委員會當場確認是否符合契約規格。

### 8.5.2　核心 KPI 驗證

驗收委員會依下列 10 項核心 KPI 逐項驗證，全數達標方可通過初驗：

| # | KPI 項目 | 達標標準 | 驗證方式 |
|---|---------|---------|---------|
| 1 | IVR 語音回答正確率 | ≥85% | 台鐵提供 200 題驗證 |
| 2 | AI-Agent 文字回答正確率 | ≥85% | 台鐵提供 200 題驗證 |
| 3 | STT 語音辨識正確率 | ≥85% | 200 音檔驗證 |
| 4 | 回應時間（IVR/Agent） | ≤5,000ms（P95） | 壓力測試報告 |
| 5 | 併發處理能力 | ≥30 埠同時在線 | 併發測試報告 |
| 6 | 系統可用性 | ≥99%/季 | 觀察期監控數據 |
| 7 | MTTR（平均修復時間） | ≤4hr | 事件處理紀錄 |
| 8 | QI 品質稽核效率 | 15 分鐘錄音 → ≤5 分鐘完成 | 現場操作驗證 |
| 9 | 後送案件分類正確率 | ≥85% | 200 筆後送案件驗證 |
| 10 | 轉人工比例 | ≤40% | 觀察期統計數據 |

### 8.5.3　交付文件審查

驗收委員會同步審查下列 13 份交付文件之完整性與正確性：

| # | 文件名稱 | 交付期程 | 審查重點 |
|---|---------|---------|---------|
| 1 | #01 工作計畫書 | 第一期 | 專案範圍、時程、組織架構 |
| 2 | #02 系統分析報告書 | 第一期 | 需求追溯、架構設計 |
| 3 | #03 程式設計規格報告書 | 第二期 | API 規格、資料庫 Schema |
| 4 | #04 教育訓練計畫書 | 第二期 | 課程規劃、教材完整性 |
| 5 | #05 測試計畫書 | 第二期 | 測試策略、案例覆蓋率 |
| 6 | #06 軟硬體清單及系統架構圖 | 第二期 | 設備清單、架構圖正確性 |

### 8.5.4　缺失改善與複驗

初驗發現缺失後，依下列程序處理：

1. **缺失通知：** 驗收委員會於初驗完成後 3 個工作日內出具缺失改善通知書，載明缺失項目、嚴重等級及改善期限。
2. **改善期限：** 承商於接獲通知後 **10 個工作日內** 完成全部缺失改正，並提交缺失改善報告。
3. **複驗作業（D+251 ~ D+260）：** 驗收委員會針對缺失改善項目逐項複驗，確認每一缺失項目均已改正且不影響既有功能。
4. **複驗標準：** 所有缺失項目改正完成，且改正後系統功能與效能仍符合原契約規格。

### 8.5.5　正式驗收

複驗通過後，進入正式驗收程序（D+261 ~ D+270）：

1. **驗收報告：** 驗收委員會彙整初驗、複驗結果，出具正式驗收報告。
2. **驗收證明：** 台鐵依驗收報告出具正式驗收證明書，作為契約履約完成之依據。
3. **結案文件：** 承商於正式驗收通過後交付 #13 結案報告，內容涵蓋專案執行總結、效益評估、維運建議及保固期服務說明。
4. **保固期啟動：** 正式驗收通過次日起，系統進入保固期（詳見第六章）。


## 第六章　保固期服務

### 8.6.1　保固期間與服務範圍

**服務範圍：**

| 硬體維護 | 伺服器、GPU 卡、網路設備之故障排除與更換 |
| 安全性更新 | 作業系統安全性修補、容器映像檔弱點修補、開源套件安全性更新 |
| 客製報表 | 保固期間提供 **≥10 個** 客製報表免費開發（超出部分另議） |
| 知識庫維護 | 協助台鐵更新知識庫內容，每月至少 1 次知識庫健檢 |

### 8.6.2　服務水準協議（SLA）

| SLA 項目 | 標準 | 說明 |
|----------|------|------|
| 監控模式 | 24/7 全年無休 | 含假日、國定假日 |
| 故障修復時間（MTTR） | ≤4 小時 | 含假日，自接獲通報起算 |
| 系統可用性 | ≥99%/季 | 以季為單位統計，排除計畫性維護時段 |
| 通報回應時間 | ≤30 分鐘 | 接獲 P1/P2 通報後之首次回應 |
| 計畫性維護窗口 | 每月第二個週日 02:00~06:00 | 維護前 7 天通知台鐵 |

### 8.6.3　事件分級與通報機制

沿用 §4.4 觀察期異常處理之事件分級制度（P1~P4），保固期間通報機制如下：

| 事件等級 | 通報對象 | 通報時限 | 修復時限 |
|---------|---------|---------|---------|
| P1 緊急 | 台鐵承辦人 + 主管 | 知悉後 30 分鐘內 | MTTR ≤4hr |
| P2 重要 | 台鐵承辦人 | 知悉後 1 小時內 | 4hr 內修復 |
| P3 一般 | 台鐵承辦人 | 次營業日 | 5 個工作日內 |
| P4 輕微 | 月報彙整 | 月報提交時 | 下次維護窗口 |

所有事件處理紀錄納入季度維運報告，確保完整追溯性。

### 8.6.4　季度維運報告

承商每季提交維運報告，內容涵蓋：

1. **系統運行概況：** 可用性統計、效能趨勢分析、資源使用率（CPU/GPU/記憶體/儲存）
2. **事件處理統計：** 各等級事件數量、平均修復時間、根因分析摘要
3. **KPI 達成狀況：** 10 項核心 KPI 之季度統計數據與趨勢圖
4. **安全性報告：** 安全性更新清單、弱點掃描結果、資安事件摘要
5. **知識庫健檢：** 知識庫覆蓋率變化、新增/修正項目統計
6. **改善建議：** 系統優化建議、效能調校建議、下季工作規劃

### 8.6.5　保固期結束前知識移轉

保固期結束前 **2 個月**，承商啟動知識移轉作業，確保台鐵 IT 團隊具備獨立維運能力：

| 項目 | 內容 | 時程 |
|-----|------|------|
| 維運教育訓練 | 系統架構總覽、日常維運操作、故障排除 SOP | 保固結束前 60 天起，共 24hr |
| 文件交接 | 更新版 #07 維護手冊、維運 SOP、帳號權限清冊 | 保固結束前 30 天 |
| 系統移交 | 管理者帳號移交、監控系統移交、備份排程確認 | 保固結束前 14 天 |
| 聯繫窗口 | 提供保固期後技術諮詢窗口（有償），有效期 6 個月 | 保固結束日 |

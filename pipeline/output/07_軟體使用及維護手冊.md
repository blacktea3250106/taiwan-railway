# 第一章　系統架構摘述

## 1.1　架構設計總覽

本系統採用六層式架構設計（AD-001），將 AI 客服功能與台鐵既有系統完全隔離，確保導入過程對現行客服作業零衝擊。六層由下至上依序為：

| 層級 | 名稱 | 主要職責 |
|:---:|------|---------|
| L1 | 基礎設施層 | 伺服器硬體、Docker 容器運行環境、網路與儲存 |
| L2 | 平台安全層 | AD/LDAP 認證、RBAC 權限控管、日誌稽核、加密傳輸 |
| L3 | 資料管理層 | PostgreSQL 關聯資料庫、Qdrant 向量資料庫、Redis 快取 |
| L4 | AI 服務層 | LLM 推論（vLLM）、STT 語音辨識（faster-whisper）、TTS 語音合成（Coqui XTTS-v2）、RAG 檢索 |
| L5 | 通道接入層 | SIP/RTP 語音接入、CRM Gateway 文字接入、SMS 簡訊接入 |
| L6 | 使用者層 | 值機平台（WS）、管理後台（ADM）、旅客端對話介面 |

各層之間透過明確定義之 API 介面溝通：外部介接採用 REST API（符合國發會共通性應用程式介面規範），內部高頻服務間通訊採用 gRPC 以降低延遲。

## 1.2　硬體部署配置

正式環境部署於台鐵機房，所有資料地端儲存，不進行跨境傳輸。主要設備配置如下：

| 設備角色 | 數量 | 關鍵規格 | 部署服務 |
|---------|:---:|---------|---------|
| AI-IVR 伺服器 | 2 台 | 16 核心 ≥2.2GHz ×2、DDR5 64GB+、960GB SSD ×3（RAID-5） | STT、TTS、SIP/RTP 語音處理（CPU 推論） |
| AI 伺服器 | 2 台 | 16 核心 ≥3.0GHz ×2、DDR5 128GB+、NVIDIA L40S 48GB ×2 | LLM 推論、Embedding 運算、質檢分析 |
| 管理/資安監控伺服器 | 2 台 | 16 核心 ≥3.0GHz ×2、DDR5 64GB+ | PostgreSQL、Qdrant、Redis、Prometheus、Grafana |
| SMS 簡訊收發裝置 | 2 台 | 256GB+儲存、4G LTE+ | 雙備援簡訊收發 |

GPU 資源分配策略（AD-002）：每台 AI 伺服器配備 2 張 L40S，GPU#0 專責 LLM 推論（vLLM + Llama 3.1 8B-Instruct AWQ，模型權重約 8GB，單一 vLLM 引擎以 continuous batching 服務 ≤30 路並行推論，KV Cache 動態分配，峰值 ~42GB），GPU#1 負責 Embedding 運算、質檢分析及擴充預留。STT 與 TTS 均採 CPU 推論（AD-003），部署於 AI-IVR 伺服器，30 路串流推論約需 9 CPU 核心，不佔用 GPU 資源。

## 1.3　高可用性架構

正式環境採 Active-Active + Failover 架構（AD-005）：

- **正常運行：** 兩台同類伺服器各承擔 50% 負載，透過負載均衡器分配請求。
- **故障切換：** 任一台故障時，存活機自動承載 100% 流量，以 Keepalived VRRP 實現虛擬 IP 漂移，偵測間隔 ≤2 秒、切換完成 ≤5 秒。
- **人工降級：** 當系統發生異常無法自動恢復時，系統立即切換為人工客服模式，所有來電直接路由至值機員，確保服務不中斷。

所有軟體服務以 Docker Compose 編排（AD-006），每台伺服器各自運行獨立的 `docker-compose.yaml`，維運人員可透過標準 Docker 指令操作服務啟停與狀態查看，詳細操作步驟請參閱 §4.2（備份操作步驟）及 §4.5（Docker 映像安全與供應鏈管理）。

## 1.4　既有系統介接總覽

本系統以「零衝擊」原則介接台鐵既有環境，不修改既有系統之任何設定（AD-007）。各既有系統之設計限制與解決方案的詳細分析，請參閱系統分析報告書（#02）第一章 §1.7「現有系統限制與解決方案」。

**本案 AI 系統介接現況簡覽：**

| 通道 | 源系統 | 接入模式 | AI 系統對應模組 |
|------|--------|---------|-----------------|
| 語音 | PSTN E1 Gateway / WebCall SBC（雙備援） | SIP Trunk + RTP fork（被動複製） | AI-IVR 語音應答 |
| 語音錄音 | MES Server | RTP fork 接收語音串流複本 | AI-QI 智慧質檢 |
| 文字 | CRM Gateway | REST API / WebSocket | AI-Agent 文字客服 |
| 管理 | — | 本系統內部界面 | 管理後台（ADM） |

各介接通道之詳細技術設計（SIP Failover 邏輯、RTP fork 信號流程、CRM Gateway API 規格等），請參閱程式設計規格報告書（#03）第四章 §4.1~4.3。


# 第二章　系統流程圖與使用者權限

## 2.1　AI-IVR 語音互動流程

旅客透過 PSTN、WebCall 或行政分機撥入後，AI-IVR 之處理流程如下：

| 步驟 | 動作 | 系統元件 | 輸入 | 輸出 |
|:---:|------|---------|------|------|
| 1 | SIP/RTP 接收、G.711 解碼 | — | 旅客撥入（MES Server RTP fork，被動接收，不攔截原始通話） | 原始音訊串流（延遲 ≤10ms） |
| 2 | VAD 語音偵測 | M-STT-03 | 原始音訊串流 | 偵測旅客發言結束時機（延遲 ≤200ms） |
| 3 | STT 語音辨識 | M-STT-01 | 語音片段 | faster-whisper 串流辨識（CPU 推論）文字結果（延遲 ≤800ms） |
| 4 | 意圖辨識 | M-STT-02 | 辨識文字 | 文字意圖分類與路由決策（延遲 ≤200ms） |
| 5 | 個資遮罩 | — | 辨識文字 | 身分證/電話/信用卡號正則遮罩後文字（延遲 ≤50ms） |
| 6 | LLM 推論 | M-LLM-01~02 | 遮罩後文字 | 信心度 ≥0.85 → 直接回答；信心度 0.6~0.85 → 回答＋提示洽人工；信心度 <0.6 → 轉接真人客服（延遲 ≤2,500ms） |
| 7 | TTS 語音合成 | M-TTS-01 | LLM 回覆文字 | Coqui XTTS-v2 串流合成（CPU 推論）語音（首包延遲 ≤300ms，總延遲 ≤800ms） |
| 8 | RTP 發送播放 | — | 合成語音 | 編碼封包後播放回覆語音（延遲 ≤50ms） |

合計延遲 ≤4,600ms（SLA 上限 5,000ms，餘裕 400ms）

多輪對話透過 Redis 維護當次通話之 Session Context；通話結束時觸發滿意度調查，完整對話紀錄寫入 PostgreSQL。AI 無法解決之問題自動轉接真人客服，並附帶對話摘要與知識輔助資訊供值機員參考。

## 2.2　AI-Agent 文字對話流程

旅客透過 Web 或 App 發起文字對話後，AI-Agent 之處理流程如下：

| 步驟 | 動作 | 系統元件 | 輸入 | 輸出 |
|:---:|------|---------|------|------|
| 1 | API Gateway | — | 旅客送出訊息 | 訊息接收、Session 識別（延遲 ≤100ms） |
| 2 | 意圖辨識 | — | 接收之文字訊息 | 文字意圖分類、路由決策（延遲 ≤200ms） |
| 3 | 個資遮罩 | — | 分類後文字 | 正則遮罩去識別化（延遲 ≤50ms） |
| 4 | RAG 語意檢索 | M-AGT-02~03 | 遮罩後文字 | Qdrant 向量資料庫 Top-K 檢索結果（延遲 ≤200ms） |
| 5 | Prompt 組裝 | — | RAG 結果 | System Instruction + RAG 結果 + 歷史摘要（延遲 ≤50ms） |
| 6 | LLM 推論 | M-LLM-01~02 | 組裝後 Prompt | vLLM + Llama 3.1 8B AWQ 推論回覆（延遲 ≤2,500ms） |
| 7 | Citation 標註 + 回應輸出 | — | LLM 回覆 | 標註引用之知識來源出處（延遲 ≤100ms） |

合計延遲 ≤3,200ms（SLA 上限 5,000ms，餘裕 1,800ms）

AI 回應自動標註知識來源出處（RAG Citation），前端以 footnote 方式呈現。SMS 簡訊通道之 AI 自動回覆會同步透過 WebSocket 推播通知客服人員。

## 2.3　AI-QI 質檢批次流程

# 第三章　系統操作說明

本章說明 AI 客服系統各功能模組之日常操作方式，涵蓋系統登入、告警監控、知識庫管理、報表查詢及 AI 模型管理等作業。各操作均以管理後台（ADM）之 Web 介面為主要操作途徑，操作人員依據第 2.4 節所定義之角色權限執行對應功能。

## 3.1　系統登入與角色說明

### 登入方式

本系統整合台鐵既有 Active Directory（AD/LDAP）進行帳號驗證（FR-ADM-002 / M-ADM-02），並透過企業資訊入口平台實現 SSO 單一登入，操作人員無需另行建立系統帳號。

**登入步驟：**

1. 開啟瀏覽器，輸入管理後台網址（由系統管理員於部署時設定，預設為 `https://<管理伺服器IP>:8443/admin`）
2. 頁面自動導向台鐵企業資訊入口平台之 SSO 登入頁
3. 輸入台鐵 AD/LDAP 帳號與密碼
4. 驗證通過後，系統依據 AD 群組對應之角色自動載入對應功能選單
5. 首次登入時，系統提示閱讀並確認資安聲明與個資保護告知事項

> **Session 管理：** 登入後之 Session 有效期限為 8 小時（可由系統管理員調整），閒置超過 30 分鐘自動登出。所有登入與登出行為均寫入系統日誌。

### 角色功能總覽

系統定義四種角色，各角色之職責與主要操作如下：

| 角色 | 職責說明 | 主要操作畫面 |
|------|---------|------------|
| 系統管理員 | 負責系統維運、參數設定、帳號管理、備份回復 | 系統管理、備份管理、監控儀表板、知識庫管理 |
| 督導 | 負責客服品質監控、質檢規則設定、座席管理 | 督導監控台、質檢報表、知識庫管理、統計報表 |
| 值機員 | 負責來電接聽、文字客服回應、工單處理 | 值機平台（WS）、個人報表 |
| 外點人員 | 負責後送案件之查看與處理 | 後送案件管理 |

### 值機平台登入（值機員/督導專用）

值機員與督導除管理後台外，另需登入值機平台（WS）方可進行通話與文字客服操作：

> **[圖 3-1]** 〔值機平台主畫面——系統截圖待系統開發完成後補入〕

**操作步驟：**

1. 開啟值機平台網址（預設為 `https://<AI-IVR伺服器IP>:8443/ws`）
2. 以 AD/LDAP 帳號驗證登入
3. 系統自動帶入座席分機號碼（依 AD 帳號對應設定）
4. 點選「簽入」按鈕，座席狀態切換為「就緒」，開始接聽來電
5. 下班或休息時，點選「簽出」或切換為「暫離」狀態

> **多點登入限制：** 同一帳號僅允許於一個裝置登入值機平台，重複登入時前一 Session 自動中斷並通知使用者。

## 3.2　告警與監控操作

本系統之監控架構以 Prometheus 負責指標收集與告警規則評估，Grafana 負責視覺化呈現，兩者皆部署於管理/資安監控伺服器，以 Docker 容器運行（AD-006）。告警通知支援 Email 與 Webhook 兩種管道，僅系統管理員角色具備告警規則之新增、修改、刪除權限。

### 3.2.1　Prometheus 告警規則管理

**操作路徑：** 管理後台 → 系統管理 → 監控設定 → 告警規則

系統預設之告警規則涵蓋以下類別：

| 效能指標 | `llm_latency_high` | LLM 推論延遲 P95 超過 2,500ms（持續 5 分鐘） | Warning |

| 效能指標 | `tts_latency_high` | TTS 合成延遲 P95 超過 1,000ms（持續 5 分鐘） | Warning |
| 資源用量 | `cpu_usage_high` | CPU 使用率 >85%（持續 10 分鐘） | Warning |
| 資源用量 | `memory_usage_high` | 記憶體使用率 >90%（持續 5 分鐘） | Critical |
| 資源用量 | `disk_usage_high` | 磁碟使用率 >80% | Warning |
| 資源用量 | `disk_usage_critical` | 磁碟使用率 >90% | Critical |
| 資源用量 | `gpu_memory_high` | GPU VRAM 使用率 >90%（持續 5 分鐘） | Warning |
| 備份異常 | `backup_failed` | 排程備份連續 2 次執行失敗 | Critical |
| 安全事件 | `login_fail_burst` | 同一帳號 5 分鐘內登入失敗 ≥5 次 | Warning |
| 服務品質 | `ai_confidence_low` | AI 回應信心度 <0.6 之比例 >30%（持續 30 分鐘） | Warning |

**新增自訂告警規則：**

1. 進入「告警規則」頁面，點選「新增規則」
2. 填寫規則名稱、PromQL 查詢運算式、觸發持續時間、嚴重等級
3. 設定告警通知管道（Email / Webhook，可複選）
4. 點選「儲存」，規則立即生效（Prometheus 自動重新載入設定）
5. 可於「規則列表」中啟用/停用個別規則

**修改與刪除：**

- 點選規則列表中之「編輯」圖示，可修改規則參數
- 點選「刪除」圖示，系統提示確認後刪除（系統預設規則僅可停用，不可刪除）

## 3.3　知識庫管理操作

知識庫為 AI 客服系統之核心資產，AI-IVR 與 AI-Agent 之 RAG 語意檢索均依賴知識庫內容進行回答。知識庫文件以結構化方式儲存於 PostgreSQL（原始文件與中繼資料），並透過 Embedding 模型（multilingual-e5-large）產生向量後存入 Qdrant 向量資料庫，供即時語意檢索使用。

### 3.3.1　知識文件維護（CRUD）

**操作路徑：** 管理後台 → 知識庫管理 → 文件管理

> **[圖 3-4]** 〔知識庫管理介面——系統截圖待系統開發完成後補入〕

**角色權限：** 系統管理員與督導可新增、編輯、刪除；值機員僅可查看。

**文件格式支援：** 純文字（TXT）、Markdown（MD）、Word（DOCX）、PDF、CSV/Excel（XLSX）。上傳後系統自動解析為純文字內容，並依據設定之分段規則切割為知識片段（Chunk）。

**新增知識文件：**

1. 進入「文件管理」頁面，點選「新增文件」
2. 選擇文件類別（如：時刻表、票價規定、服務辦法、常見問題等）
3. 上傳檔案或於線上編輯器直接輸入內容
4. 設定中繼資料：標題、關鍵字、有效期限（選填）、適用通道（AI-IVR / AI-Agent / 全部）
5. 系統自動執行文件解析與分段預覽，操作人員可調整分段結果
6. 點選「發布」，系統自動觸發 Embedding 運算，將新文件之向量寫入 Qdrant
7. 發布完成後，新知識立即可被 AI 引用回答

**編輯既有文件：**

1. 於文件列表搜尋或篩選目標文件
2. 點選「編輯」進入線上編輯器
3. 修改內容後點選「儲存並發布」
4. 系統自動重新計算受影響片段之 Embedding，並更新 Qdrant 中對應向量
5. 舊版本自動歸檔，可於版本歷程中查看與回溯

**刪除文件：**

1. 於文件列表選取目標文件，點選「刪除」
2. 系統顯示確認對話框，提示該文件被引用之次數
3. 確認刪除後，系統移除 Qdrant 中對應之向量資料
4. 已刪除文件移至回收站，保留 30 天可還原

> **批次匯入：** 支援 ZIP 壓縮檔批次上傳多份文件，系統自動逐一解析與處理。批次匯入之進度可於「工作佇列」頁面追蹤。

### 3.3.2　Embedding 重建與版本管理

**Embedding 重建：**

當知識庫進行大規模更新（如：年度時刻表改點）或 Embedding 模型升級時，需執行全量 Embedding 重建：

**操作路徑：** 管理後台 → 知識庫管理 → Embedding 管理

1. 點選「全量重建」按鈕
2. 系統顯示預估處理時間（依知識庫規模而定，一般約 10~30 分鐘）
3. 確認後系統於背景執行重建作業，期間 AI 服務持續使用舊版本向量回答
4. 重建完成後，系統自動切換至新版本向量，舊版本保留一份以供緊急回溯
5. 重建結果顯示處理文件數、片段數、耗時、成功/失敗統計

> **注意：** 全量 Embedding 重建僅限系統管理員執行（FR-LLM-006）。重建期間 GPU#1 負載提升，建議於離峰時段（如凌晨）執行，以避免影響線上 AI 推論效能。

**版本管理：**

知識庫支援版本管理機制（M-KB-02），每次發布變更自動產生新版本：

| 操作 | 版本行為 |
|------|---------|
| 新增/編輯/刪除單一文件 | 產生增量版本，僅更新受影響之向量 |
| 全量 Embedding 重建 | 產生全量版本，重新計算所有向量 |
| 版本回溯 | 將知識庫還原至指定歷史版本 |

**版本回溯操作：**

1. 進入「Embedding 管理」→「版本歷程」
2. 列表顯示所有歷史版本，含版本號、建立時間、文件數、操作者
3. 點選目標版本之「回溯」按鈕
4. 系統確認後，將 Qdrant 中之向量資料切換至該版本
5. 回溯完成後，AI 立即使用該版本之知識回答

### 3.5.2　模型更新與熱載入

當需要更新 AI 模型（如 LLM 模型微調後重新部署、STT 模型升級）時，系統支援熱載入機制，無需停機即可完成模型切換：

**操作步驟：**

1. 將新模型檔案上傳至指定目錄（透過管理後台「AI 模型管理」→「上傳模型」，或以 SCP/SFTP 直接傳輸至伺服器）
2. 系統自動驗證模型檔案完整性（SHA-256 雜湊比對）
3. 於「AI 模型管理」頁面，新模型顯示於「待部署」清單
4. 點選「部署」按鈕，系統執行以下流程：
   - 於備援容器載入新模型
   - 執行自動化測試（預設測試集包含 20 組標準問答）
   - 測試通過後，負載均衡器將流量切換至新模型容器
   - 舊模型容器保留 10 分鐘後自動移除
5. 部署完成後，頁面顯示新模型版本與測試結果摘要

**模型回溯：**

若新模型部署後發現異常，可執行快速回溯：

1. 於「AI 模型管理」→「部署歷程」中選取前一版本
2. 點選「回溯」，系統以相同熱載入流程切換回舊模型
3. 回溯操作全程不影響線上服務

> **模型檔案管理：** 系統保留最近 3 個版本之模型檔案，超過者自動清理以節省磁碟空間。LLM 模型檔案（AWQ 量化後約 4~5GB）、STT 模型檔案（約 3GB）、TTS 模型檔案（約 2GB）及 Embedding 模型檔案（約 2GB），合計約 11~12GB，均納入 Docker Volume 每週備份範圍（詳見第四章）。

# 第四章　備份與回復

## 4.1　備份策略總覽

本系統之備份機制涵蓋四大資料類別，各類別採用不同的備份工具與排程週期，以確保資料完整性與災難復原能力。備份操作透過管理後台之「備份回復管理介面」（FR-ADM-005 / M-ADM-05）統一管理，僅系統管理員角色具備存取權限（詳見第 2.4 節角色權限表）。

| 備份對象 | 工具 | 備份方式 | 排程週期 | 預估單次容量 |
|---------|------|---------|---------|------------|
| PostgreSQL 關聯資料庫 | `pg_dump` 全備 + WAL 歸檔 | 全量備份 + 持續歸檔 | 全量：每日凌晨 02:00；WAL：持續寫入 | 依資料量成長，初期約 5~10GB |
| Qdrant 向量資料庫 | Qdrant Snapshot API | 快照備份 | 每日凌晨 03:00 | 依知識庫規模，初期約 2~5GB |
| Redis 快取 | RDB 快照 + AOF 日誌 | RDB 定時快照 + AOF 持續追加 | RDB：每小時；AOF：即時 | RDB 約 100~500MB |
| Docker Volume（設定檔、模型檔、日誌） | `tar` 打包 | 全量備份 | 每週日凌晨 04:00 | 依模型與日誌量，約 20~50GB |

> **儲存位置：** 所有備份檔案儲存於本機 RAID-5 磁碟陣列（960GB SSD ×3），並透過 `rsync` 同步至另一台同類伺服器，實現異機備援。所有資料均儲存於台鐵機房內，不進行跨境傳輸。

> **保留策略：** 系統日誌與操作紀錄備份保留 ≥6 個月（依資安查核表要求）；SMS 簡訊相關資料保留 ≥5 年（依需求書§三(五)要求）。每日備份保留 7 天，每週備份保留 4 週，每月備份保留 12 個月，逾期自動清理。

## 4.2　備份操作步驟

### 4.2.1　自動排程備份（預設啟用）

系統安裝完成後，以下備份排程預設啟用，無需人工介入：

1. **PostgreSQL 每日全量備份**：由 cron 排程於每日 02:00 執行 `pg_dump`，產出備份檔自動以日期命名（如 `pgbackup_20260406.sql.gz`），並壓縮儲存。WAL 歸檔持續運行，確保全量備份之間的異動亦可復原。
2. **Qdrant 每日快照**：於每日 03:00 透過 Qdrant Snapshot API 建立快照，快照檔存於指定備份目錄。
3. **Redis RDB 快照**：每小時自動產出 RDB 快照；AOF 日誌持續追加寫入，確保快取資料不因重啟遺失。
4. **Docker Volume 每週備份**：每週日 04:00 將關鍵 Volume（設定檔、模型權重、TLS 憑證等）打包為 `.tar.gz`。

### 4.2.2　手動觸發備份

當系統管理員需在非排程時間進行備份（例如重大更新前、模型更換前），可透過管理後台執行：

**操作路徑：** 管理後台 → 系統管理 → 備份管理

1. 登入管理後台（AD/LDAP 帳號驗證，詳見第 2 章）
2. 點選左側選單「系統管理」→「備份管理」
3. 畫面顯示各備份項目之最近一次備份時間與狀態
4. 選取欲備份之項目（可多選：PostgreSQL / Qdrant / Redis / Docker Volume）
5. 點選「立即備份」按鈕
6. 系統顯示備份進度條，完成後顯示備份檔名稱、大小與儲存路徑
7. 備份完成後，系統自動將備份檔同步至備援機

> **注意事項：** 手動備份期間系統仍可正常運作，不影響線上服務。PostgreSQL 備份採用非鎖定模式（`pg_dump` 不會阻塞讀寫），Qdrant 快照亦為非阻塞式操作。

### 4.2.3　備份狀態監控

管理後台之備份管理頁面提供以下監控資訊：

- **備份歷程表：** 列出近 30 天所有備份紀錄，含時間、類型、檔案大小、執行結果
- **磁碟空間使用：** 顯示備份目錄之已用空間與剩餘空間
- **異常告警：** 若排程備份連續 2 次失敗，系統透過 Prometheus 告警規則自動通知系統管理員（Email / Webhook），告警機制之設定方式詳見第 3.2 節

## 4.3　災難復原 SOP

當系統發生重大故障需進行資料復原時，依下列標準作業程序執行。本系統採 Active-Active + Failover 架構（AD-005），多數故障場景由備援機自動接管（切換時間 ≤5 秒），服務不中斷；僅當雙機同時故障或資料毀損時，方需執行以下復原程序。

系統之平均修復時間（MTTR）承諾為 ≤4 小時（含假日），備品策略為關鍵零件（SSD、RAM、電源）備 1 套放置於台鐵機房。

### 4.3.1　復原優先順序

災難復原時，依服務影響程度由高至低復原：

| 優先序 | 復原項目 | 理由 | 預估時間 |
|:-----:|---------|------|:-------:|
| 1 | Docker 容器環境 | 所有服務之運行基礎 | 10~20 分鐘 |
| 2 | PostgreSQL 資料庫 | 核心業務資料（對話紀錄、工單、帳號、設定） | 15~30 分鐘 |
| 3 | Redis 快取 | 即時對話上下文與 Session | 5~10 分鐘 |
| 4 | Qdrant 向量資料庫 | 知識庫語意檢索 | 10~20 分鐘 |
| 5 | Docker Volume（模型檔、設定檔） | LLM/STT/TTS 模型權重與系統設定 | 20~40 分鐘 |

> **說明：** 復原期間，系統自動切換為人工客服模式，所有來電直接路由至值機員，確保旅客服務不中斷。待系統復原完成並通過健康檢查後，再恢復 AI 輔助功能。

### 4.3.2　逐步復原操作

以下以單台伺服器完全毀損為例，說明完整復原流程：

**步驟一：確認故障範圍**

1. 透過 Prometheus/Grafana 監控面板確認故障伺服器與受影響服務
2. 確認備援機是否已自動接管（正常情況下 HA 機制會在 ≤5 秒內完成切換）
3. 若備援機已接管，線上服務不受影響，可從容進行故障機修復

**步驟二：還原作業系統與 Docker 環境**

1. 更換故障硬體（使用機房備品或原廠 NBD 到場服務）
2. 安裝作業系統（依安裝手冊操作）
3. 安裝 Docker Engine 與 NVIDIA Container Toolkit（若為 AI 伺服器）
4. 從備援機複製 `docker-compose.yaml` 與相關設定檔

**步驟三：還原 PostgreSQL 資料庫**

1. 從備份目錄取得最近一次全量備份檔（如 `pgbackup_20260406.sql.gz`）
2. 解壓並執行還原：透過管理後台「備份管理」→「還原」頁面操作，或以命令列執行 `pg_restore`
3. 套用 WAL 歸檔，將資料復原至故障前最後一筆交易（Point-in-Time Recovery）

**步驟四：還原 Qdrant 向量資料庫**

1. 從備份目錄取得最近一次 Snapshot 檔案
2. 透過 Qdrant Snapshot Restore API 還原至指定 Collection

**步驟五：還原 Redis 快取**

1. 將最近一次 RDB 快照檔複製至 Redis 資料目錄
2. 重啟 Redis 容器，自動載入 RDB 資料

**步驟六：還原 Docker Volume**

1. 解壓 Volume 備份檔至對應目錄
2. 確認 LLM 模型檔（Llama 3.1 8B-Instruct AWQ）、STT 模型檔（Whisper large-v3）、TTS 模型檔（XTTS-v2）完整無損
3. 確認 Embedding 模型（multilingual-e5-large）完整

**步驟七：啟動服務與驗證**

1. 執行 `docker compose up -d` 啟動所有容器
2. 透過管理後台確認各服務健康狀態（綠燈）
3. 執行測試通話與測試對話，確認 AI-IVR 與 AI-Agent 功能正常
4. 確認 Prometheus/Grafana 監控指標恢復正常
5. 將復原紀錄寫入系統日誌，供後續稽核

### 4.3.3　RPO 與 RTO 說明

| 指標 | 目標值 | 達成方式 |
|-----|-------|---------|
| RPO（復原點目標） | PostgreSQL：接近 0（WAL 持續歸檔）；Qdrant/Redis：≤1 小時 | WAL 即時歸檔 + RDB 每小時快照 |
| RTO（復原時間目標） | 單機故障：≤5 秒（HA 自動切換）；雙機故障：≤4 小時 | Active-Active 架構 + 備品策略 + MTTR ≤4hr |

## 4.5　Docker 映像安全與供應鏈管理

本節說明 Docker 容器映像之安全管理措施，涵蓋映像來源管控、簽名驗證、漏洞掃描及供應鏈風險管理，以確保部署之容器映像未遭竄改且無已知安全漏洞。

### 4.5.1　映像來源與基礎映像選擇

| 項目 | 規範 |
|------|------|
| 基礎映像來源 | 僅使用 Docker Official Images 或 Verified Publisher 映像 |
| 基礎映像選擇 | Python 服務採用 `python:3.11-slim`（Debian-based），減少攻擊面 |
| GPU 服務映像 | 採用 NVIDIA 官方 `nvidia/cuda:12.x-runtime` 映像 |
| 映像倉庫 | 部署私有 Docker Registry（Harbor），部署於管理伺服器，禁止直接從外部 Registry 拉取 |
| 映像標籤 | 禁止使用 `latest` 標籤，一律使用語意化版本（如 `v1.2.3`） |

### 4.5.2　映像簽名與完整性驗證

- 所有自建映像於 CI/CD 流程中使用 Docker Content Trust（DCT）進行簽名
- 部署前驗證映像簽名，未簽名映像禁止部署至正式環境
- 簽名金鑰由台鐵資訊中心與廠商共同管控（雙人授權）

### 4.5.3　映像漏洞掃描

| 掃描時機 | 掃描工具 | 處理方式 |
|---------|---------|---------|
| 映像建構後（CI 階段） | Trivy（開源，Apache 2.0） | Critical/High 漏洞阻擋部署 |
| 私有 Registry 儲存時 | Harbor 內建掃描 | 定期重掃已存放映像 |
| 部署前（CD 階段） | Trivy | 再次確認無新增漏洞 |
| 定期排程 | 每週一凌晨 03:00 | 掃描所有運行中映像，產出報告 |

漏洞處理優先級：
- **Critical**：24 小時內修補並重新部署
- **High**：72 小時內修補
- **Medium**：下次版本更新時修補
- **Low**：評估後列入追蹤清單

### 4.5.4　軟體組成分析（SCA）

- 使用 Trivy 對所有容器映像執行 SCA 掃描，產出軟體物料清單（SBOM）
- SBOM 格式採用 SPDX 標準，納入交付文件（#11 原始程式碼光碟）
- 掃描範圍涵蓋：作業系統套件、Python 套件（pip）、Node.js 套件（npm）
- 驗證所有第三方元件均非大陸地區品牌（契約§8(24)）
- 驗證所有元件授權與開源授權相容（Apache 2.0 / MIT / BSD / Llama Community）

### 4.5.5　容器運行時安全

| 安全措施 | 實作方式 |
|---------|---------|
| 非 root 執行 | 所有容器以非特權使用者執行（`USER appuser`） |
| 唯讀檔案系統 | 容器根檔案系統設為唯讀（`read_only: true`），僅掛載必要 Volume |
| 資源限制 | 設定 CPU/Memory 上限，防止單一容器耗盡主機資源 |
| 網路隔離 | 各服務使用獨立 Docker Network，僅開放必要連線埠 |
| 健康檢查 | 每 30 秒執行容器 Health Check，異常自動重啟 |

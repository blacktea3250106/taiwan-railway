## 8.1　上線策略與總體規劃

### 8.1.1　上線策略概述

本案採**灰度上線**策略（10% → 30% → 100%），分三階段逐步擴大 AI 客服服務範圍，而非一次全量切換。此策略之核心考量如下：

1. **風險可控：** 每階段僅開放有限流量由 AI 處理，若出現正確率不足或系統異常，可立即將流量回歸人工客服，不影響既有服務運作。
2. **逐步驗證 KPI：** 各階段設定明確的 Go/No-Go 判定標準，確保 IVR 正確率 ≥85%（200 題驗證）、回應時間 ≤5,000ms、AI-Agent 正確率 ≥85% 等關鍵指標達標後方擴大範圍。
3. **避開尖峰時段：** 灰度上線期間不安排於春節疏運、連假收假等話務量暴增時段啟動新階段切換，確保系統在正常話務基線下完成穩定性驗證。
4. **人工客服零衝擊：** 全程維持既有人工客服系統正常運作，AI-IVR 透過 SIP/RTP fork 被動接收語音流（AD-007），不修改 MES Server 任何設定，AI 模組為新增獨立架構，故障時自動退場即可。

### 8.1.4　Go/No-Go 判定標準

每一灰度階段結束時，由專案經理召集台鐵承辦人員、客服中心主管及系統開發團隊召開**階段檢討會議**，依下表判定是否進入下一階段：

| 指標 | Go 條件 | No-Go 觸發值 | 處置方式 |
|-----|---------|-------------|---------|
| AI 回答正確率 | ≥80%（灰度期暫定，正式驗收 ≥85%） | <70% | 暫停灰度，回歸人工客服，啟動知識庫補強與模型校正 |
| 回應時間 | ≤5,000ms（P95） | >8,000ms（P95） | 降低 vLLM 並行實例、排查 GPU 負載瓶頸 |
| 系統可用性 | ≥95%（灰度期暫定，正式營運 ≥99%/季） | <90% | 暫停灰度，全面排查後重啟 |
| MTTR | ≤4hr | >4hr | 強化監控告警與值班機制 |
| 轉人工比例 | ≤40%（灰度期暫定） | >60% | 分析轉接原因，補充知識庫高頻問題 |

> **灰度期暫定閾值說明：** 灰度期間 Go/No-Go 閾值較正式驗收標準略寬（正確率 80% vs. 85%、可用性 95% vs. 99%），目的為容許系統在真實環境中進行調校。待灰度三階段全數通過後，進入 D+220~D+240 正式 KPI 驗收期間，以契約規定之正式標準驗證。

### 8.1.5　回滾方案

當任一灰度階段觸發 No-Go 條件時，執行以下回滾程序：

1. **即時切換：** AI-IVR 停止接聽新來電，SIP 路由將全部流量導回 MES Server 原始路徑，由人工客服接手。此切換僅需調整 SIP 路由規則，不涉及 MES Server 設定變更，預估切換時間 ≤5 分鐘。
2. **AI-Agent 退場：** CRM Gateway 停止轉發新訊息至 AI-Agent，文字客服回歸人工作業模式。進行中的 AI 對話以「系統維護中，為您轉接專人」訊息結束。
3. **資料保全：** 回滾期間所有對話紀錄、系統日誌完整保留，供後續問題分析使用。
4. **通報機制：** 回滾觸發後 1 小時內通報台鐵承辦人員，72 小時內完成損害控制與根因分析報告。

回滾後的 AI 系統不影響既有人工客服運作（合規紅線 #7），因 AI-IVR 為獨立於 MES Server 之外的新增模組，停用 AI 等同回復至建置前狀態。

## 8.2　上線範圍與工作項目

### 8.2.1　上線範圍總覽

本案上線範圍涵蓋六大子系統，完整覆蓋語音、文字、簡訊三大客服通道，以及後端質檢、報表與管理功能。各子系統對應之功能需求編號、灰度階段與上線條件如下表：

| # | 子系統 | 功能需求編號 | 灰度階段 | 上線前提條件 |
|---|-------|------------|---------|------------|
| 1 | **AI-IVR 智慧語音應答** | FR-IVR-001 ~ FR-IVR-007 | 階段一（10%）起 | SIP/RTP 介接驗證通過、STT/TTS 模型部署完成、≥30 埠壓力測試通過 |
| 2 | **AI-Agent 虛擬客服** | FR-AGT-001 ~ FR-AGT-008 | 階段二（30%）起 | CRM Gateway API 介接驗證通過、向量資料庫（Qdrant）知識庫匯入完成、≥30 埠壓力測試通過 |
| 3 | **AI-QI 智慧質檢** | FR-QI-001 ~ FR-QI-007 | 階段三（100%）起 | 質檢規則設定完成、15 分鐘錄音 → ≤5 分鐘產出報告之效能驗證通過 |
| 4 | **SMS 簡訊服務** | FR-AGT-007 | 階段三（100%）起 | 雙備援 SMS 裝置上線測試通過、簡訊紀錄保留機制驗證（≥5 年儲存容量確認） |
| 5 | **查詢及報表** | FR-RPT-001 ~ FR-RPT-003 | 階段二（30%）起 | 資料來源串接完成、儀表板呈現正確性驗證 |
| 6 | **值機平台（WS）與管理後台（ADM）** | FR-WS-001 ~ FR-WS-005、FR-LLM-006 | 階段一（10%）起 | AD 帳號整合測試通過、RBAC 角色設定完成、全員帳號開通 |

各子系統之功能規格詳見第二章系統分析報告書（#02）§5.1~§5.8，硬體配置詳見第六章軟硬體清單及系統架構圖（#06）§6.2~§6.3。

### 8.2.2　各子系統上線內容

**一、AI-IVR 智慧語音應答系統**

上線內容涵蓋語音通道接入（E1 專線 30 B-channel + WebCall SBC 雙備援 + 行政分機類比轉 SIP）、STT 即時語音辨識（faster-whisper，CPU 推論，辨識率 ≥85%）、LLM 智慧應答（vLLM + Llama 3.1 8B-Instruct AWQ，GPU #0 專責推論，AD-002）、TTS 語音合成（XTTS-v2，CPU 推論）、多輪對話管理、DTMF 按碼偵測、轉人工客服機制及滿意度調查。AI 回應採三層信心度策略：≥0.85 直接回答、0.6~0.85 回答並提示「建議洽人工確認」、<0.6 直接轉接人工，確保錯誤回答不致無控制地到達旅客端。涉及票價、時刻等關鍵資訊一律由 API 即時查詢，不依賴 LLM 記憶。

**二、AI-Agent 虛擬客服系統**

上線內容涵蓋 ≥30 埠數位文字通道（Web + App）、RAG 檢索增強生成（Qdrant 向量資料庫 + multilingual-e5-large embedding）、互動式多輪文字問答、回應引用來源標註（RAG Citation）、個性化問答機制、AI 轉真人客服（附對話摘要）、系統持續優化與校正機制。AI-Agent 經由 CRM Gateway 以 REST/WebSocket API 介接既有數位客服平台，回覆內容限定於台鐵指定知識庫範圍，並標註「本回覆由 AI 生成，如有疑問請洽專人」。

**三、AI-QI 智慧質檢系統**

上線內容涵蓋多格式音檔匯入與批次排程、客服錄音語音轉文字（針對台鐵業務術語優化）、自訂質檢規則與評分機制、靜音與插話偵測、語調與情緒分析、異常偵測警報、多維度質檢報表。效能目標為 15 分鐘錄音 ≤5 分鐘完成轉換分析並產出質檢報告。質檢功能部署於 GPU #1（AD-002），與 LLM 推論互不搶奪運算資源。

**四、SMS 簡訊服務**

上線內容涵蓋雙備援 SMS 收發裝置（4G LTE+，裝置品牌與 LTE 模組均為非大陸品牌/非大陸製造）、AI 自動回覆簡訊同步通知客服人員、簡訊紀錄保留 ≥5 年（裝置儲存容量 256GB+，詳見 #06 §6.10 容量估算）。SMS 裝置規格詳見第六章（#06）§6.2.1。

**五、查詢及報表功能**

上線內容涵蓋對話服務類、AI 管理類、知識點分析、滿意度等統計分析報表（FR-RPT-001），視覺化儀表板支援自訂篩選條件與時間區間（FR-RPT-002），以及保固期內 ≥10 個客製報表服務（FR-RPT-003）。報表模組整合 Prometheus + Grafana 監控數據（AD-008），提供即時營運指標。

**六、值機平台（WS）與管理後台（ADM）**

值機平台上線內容涵蓋座席操作介面與通話控制（接聽/保留/轉接/掛斷）、AI 知識輔助即時建議推送、對話摘要與後送工單管理、督導即時監控與密語指導介入、座席狀態管理（簽入/簽出/就緒/忙碌/休息）。管理後台上線內容涵蓋 AD 整合帳號驗證（LDAP）、RBAC 多層級權限控管（FR-LLM-006）、知識庫維護介面（新增/修改/刪除/版本管理）、系統參數設定、監控儀表板與日誌稽核（日誌保存 ≥6 個月）。

### 8.2.5　既有系統介接上線順序

本案 AI 系統與 10 套既有系統介接，上線順序依風險由低至高安排：

| 優先序 | 既有系統 | 介接方式 | 灰度階段 | 說明 |
|-------|---------|---------|---------|------|
| 1 | AD（Active Directory） | LDAP | 階段一前 | 帳號驗證為所有操作前提，優先完成 |
| 2 | WebCall SBC（雙備援） | SIP Trunk（failover） | 階段一 | 風險最低的語音通道，作為首波灰度對象 |
| 3 | CRM Gateway | REST/WebSocket API | 階段二 | AI-Agent 文字客服的訊息通道 |
| 4 | PSTN E1 VoIP Gateway | SIP Trunk + RTP fork | 階段二 | 主要語音通道，AI-IVR 被動接收，不修改 MES 設定 |
| 5 | MES Server（話務/錄音） | RTP fork + API | 階段二 | 通話分配與錄音來源 |
| 6 | 企業資訊入口平台 | SSO/API | 階段二 | 單一登入整合 |
| 7 | 行政交換機 | 類比 VoIP GW → MES → AI-IVR | 階段三 | 類比線路轉 SIP，需額外語音品質調校 |
| 8 | 後送案件系統 | API | 階段三 | 225 位外點人員使用中，影響面較廣 |
| 9 | 訂票紀錄查詢系統 | API（規格待取得） | 階段三 | LLM 串接查詢，API 規格須於 D+30 前取得 |
| 10 | 時刻表查詢系統 | API（規格待取得） | 階段三 | 同上，API 規格延遲時啟用 Mock API 先行策略 |

> **API 規格風險控管：** 訂票紀錄查詢系統與時刻表查詢系統之 API 規格尚待台鐵提供，開發團隊已於介接層採用 Adapter Pattern，API 規格變更時僅需修改 Adapter，不影響核心邏輯。正式 API 到位前以 Mock API 進行開發與測試，確保時程不受阻。

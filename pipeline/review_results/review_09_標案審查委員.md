### 5. **#03 程式設計規格報告書 §2 / #10 程式設計書** | 信心度計算演算法未定義

**問題：**
```
全案統一描述「三層信心度防線」：
  ≥0.85：直接回答
  0.6~0.85：回答 + 提示「建議洽人工確認」  
  <0.6：直接轉接人工客服

但零份文件定義：
  ❌ 信心度如何從 LLM 輸出計算
  ❌ 是否採用 token log-probability ？
  ❌ 閾值 0.85 / 0.6 的理論依據（ECE 校準 ？）
  ❌ 多頭結合的權重公式（RAG retrieval score + NLI entailment + perplexity ？）
  ❌ 台鐵 200 題驗收集上的校準流程
```

**影響：**
- 開發人員無法據此實作信心度計算，三層防線淪為虛設
- 系統上線後，若 AI 給出錯誤回答但信心度顯示 0.88，審查委員會質疑「信心度是否有實際決策作用」
- 審查無法驗收此核心安全機制

**修正建議：** 
在 #03 §4.2（P-LLM-02）新增「信心度計算演算法」專節：

```
confidence_score = α × rag_retrieval_score 
                  + β × nli_entailment_score 
                  + γ × llm_perplexity_normalized

其中：
  rag_retrieval_score = Top-1 similarity / similarity_baseline
  nli_entailment_score = NLI(response, context) 三分類 softmax
  llm_perplexity_normalized = min(1, exp(-perplexity) / baseline)
  
  α=0.3, β=0.4, γ=0.3（經台鐵 200 題 ECE 校準，目標 ECE ≤0.1）

閾值校準流程：
  1. 台鐵 200 題驗收集 + AI 輸出 + 人工標註（0/0.5/1）
  2. 計算各 confidence bin 的準確率曲線  
  3. 選擇使 F1-score 最高的閾值對（0.60, 0.85）
  4. 文件化校準報告
```

在 #05 增加校準測試案例，#10 P-LLM-02 補充實作細節。

**驗收方式：** 
於 UAT 時，驗收台鐵 200 題測試集上的信心度曲線與實際正確率的 ECE（Expected Calibration Error）。

```
     "doc_id": "KB-0001",
     "doc_title": "時刻表查詢規則",
     "page_no": 2,
     "section_no": "2.3.1",

3. Embedding 重建：
   - 初始化：批次 embedding（batch_size=128，GPU 並行）

   - Reindex：知識庫版本升級時完整重建
```

**驗收方式：**

- 若後續系統調整依照 decisions.yaml，會走錯方向

```
AI 服務層部署：
```

```
    伺服器被入侵時無防禦縱深
    90 天輪換無自動化機制

(3) 容器安全：
    #07 維護手冊提及「避免 root 執行」，
    但未納入 #03 設計規格（應從設計階段定義，非運維補救）
```

**影響：**
- CISSP 審查委員會直接拒絕此方案
- 旅客個資加密無實質保護

**修正建議：** 

**(1) SDES 安全前提（#03 §3.8.4）：**
```
SIP 信令通道強制使用 SIPS (TLS on port 5061)，禁止 UDP 5060。
金鑰交換遵循 RFC 4568 (Security Descriptions)，
但前提是 SIP 信令完整加密。

若現有基礎設施不支援 TLS，備案方案：DTLS-SRTP（RFC 5763）。
```

**(2) 金鑰管理（#03 新增 §3.8.6 專節）：**
```
導入 HashiCorp Vault 作為 KMS（Key Management Service）：
  - 金鑰不落地於應用伺服器
  - 應用透過 Transit Secret Engine 進行 encrypt/decrypt RPC 調用
  - 雙人授權 (Dual Control)：解封機密需兩個不同人員授權
  - 自動輪換：90 天自動生成新金鑰，舊金鑰 30 天內回收
  
AWS KMS 成本估算：$0.03/請求，月均 10 萬筆 ≈ $3,000
→ 列入預算追加項（超出原 2,400 萬 0.1% 以內）
```

**(3) 容器安全設計規格（#03 新增 §3.8.7 專節）：**
```
所有應用容器運行配置：
  - USER: appuser (UID 1000)，禁止 root
  - Filesystem: Read-only rootfs，/tmp 掛載 tmpfs 可寫
  - Capabilities: DROP all，按需 ADD（如 NET_BIND_SERVICE）
  - Security Context: runAsNonRoot: true / allowPrivilegeEscalation: false
  - Image Signing: Docker Content Trust (DCT)，推入 Harbor 私有 Registry 前驗簽
  - Image Scan: Trivy 掃描，嚴重度 HIGH 以上禁止推入
```

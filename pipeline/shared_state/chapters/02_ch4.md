# 第四章　非功能需求規格

**案號：L0215P2010U**
**國營臺灣鐵路股份有限公司 AI 客服系統建置案**

---

本章定義系統在效能、可用性、資訊安全、擴展性、相容性及資料保存等六大面向之非功能需求。所有數值指標均源自需求書與服務水準表，並對應第二章功能需求之驗收基準。

## 4.1　效能需求

### 4.1.1　端到端延遲預算

系統最核心的效能指標為旅客感知延遲。需求書要求語音通道（AI-IVR）旅客說完至系統回覆首字 ≤5,000ms（ivr_response，需求書§三(一)），文字通道（AI-Agent）旅客送出訊息至系統回覆首段文字 ≤5,000ms（agent_response，需求書§三(五)）。

為確保端到端延遲可控，系統將 5,000ms 預算分解至各處理階段，每一段皆設有獨立上限與超時機制：

**語音通道（AI-IVR）延遲分解：**

| 處理階段 | 元件 | 延遲預算 | 說明 |
|---------|------|---------|------|
| VAD 尾端偵測 | M-STT-03 | ~500ms | 偵測旅客停止說話，觸發後續處理 |
| STT 語音轉文字 | M-STT-01（faster-whisper） | ≤800ms | CPU 推論（AD-003），chunk-based streaming |
| LLM 推論 + RAG 檢索 | M-LLM-01（vLLM） | ≤2,500ms | GPU#0 專責推論（AD-002），含 Qdrant Top-K 檢索 <50ms |
| TTS 首包合成 | M-TTS-01（XTTS-v2） | ≤700ms | CPU 推論（AD-003），串流輸出首包 ≤300ms，含封包緩衝 |
| RTP 封包發送 | M-IVR-01（PJSIP） | ~100ms | 網路傳輸與 jitter buffer |
| **合計** | | **~4,600ms** | **保留 ~400ms 安全緩衝** |

**文字通道（AI-Agent）延遲分解：**

| 處理階段 | 元件 | 延遲預算 | 說明 |
|---------|------|---------|------|
| 訊息接收與前處理 | M-AGT-01 | ~100ms | WebSocket 接收、敏感詞初篩 |
| RAG 向量檢索 | M-AGT-02（Qdrant + bge-large-zh-v1.5） | ≤200ms | HNSW 索引 Top-K 查詢 <50ms，含 embedding 計算 |
| LLM 推論 | M-LLM-01（vLLM） | ≤2,500ms | 與語音通道共用 GPU#0，continuous batching 排程 |
| 後處理與引用標註 | M-AGT-03 | ~200ms | RAG Citation 來源比對、格式化回應 |
| **合計** | | **~3,000ms** | **保留 ~2,000ms 緩衝供個性化處理與歷史查詢** |

**超時降級機制：** 任一階段超過預算時，系統不等待而直接進入降級流程：

- STT 逾 800ms → 截斷當前語音片段，以已辨識部分文字送入 LLM
- LLM 推論逾 2,500ms → 語音通道播放「請稍候」候補話術並重試一次；二次失敗轉人工（FR-IVR-004）
- TTS 逾 700ms → 切換至預錄候補語音（如「系統處理中，請稍候」）
- 文字通道 LLM 逾 3,000ms → 先回傳「正在查詢中...」佔位訊息，後續以串流方式補全回覆

### 4.1.2　正確率指標

| 指標 | 目標值 | 計算公式 | 驗收方式 | 來源 |
|------|-------|---------|---------|------|
| IVR 應答正確率 | ≥85% | (完全正確×1 + 部分正確×0.5) ÷ 總次數 ×100% | 200 題，台鐵提供 | 需求書§三(一) |
| STT 辨識率 | ≥85% | 100% − (錯誤或缺漏字數 / 總字數) ×100% | 200 音檔（中英 4:1，≤150s） | 需求書§三(二) |
| Agent 應答正確率 | ≥85% | 同 IVR | 200 題，台鐵提供 | 需求書§三(五) |
| 後送案件分類正確率 | ≥85% | (AI 正確分類數 / 總案件數) ×100% | 200 筆真實後送案件 | 需求書§三(四) |

正確率未達標時的應對策略：系統採三層信心度閾值設計（詳見第五章 AI 回應策略），信心度 <0.6 時主動轉接人工客服，避免低信心回答影響旅客體驗。涉及票價、班次時刻等關鍵資訊，一律從訂票紀錄查詢系統與時刻表查詢系統 API 即時取得，不依賴 LLM 記憶性知識。

### 4.1.3　併發容量

| 通道類型 | 最低併發埠數 | 設計目標 | 來源 |
|---------|------------|---------|------|
| 語音（PSTN E1 + WebCall + 行政分機） | ≥30 埠 | 支援 2 條 E1（60 B-channel） | 需求書§三(一) |
| 數位文字（Web + App） | ≥30 埠 | 支援突發 50 埠 | 需求書§三(五) |

語音 30 埠與文字 30 埠共用同一組 GPU 推論資源。以 Llama 3.1 8B AWQ 量化後 ~8GB VRAM 計算，單張 L40S（48GB）可同時運行 6 個 vLLM 實例，雙卡合計 12 個實例，理論吞吐量 ~2,800 token/s（單卡），足以支撐 60 路併發推論請求。

### 4.1.4　質檢效能

15 分鐘錄音 → ≤5 分鐘完成轉寫分析並產出質檢報告（qi_speed，需求書§三(六)）。faster-whisper 的即時處理倍率（RTF）約 0.3x（即 1 秒語音用 0.3 秒處理），15 分鐘錄音轉寫約需 4.5 分鐘，含質檢規則評分與報表生成，預估可在 5 分鐘內完成。質檢批次任務由 GPU#1 負責（AD-002），與即時推論 GPU#0 分離，確保批次質檢不影響線上服務延遲。

## 4.2　可用性需求

### 4.2.1　服務水準指標

| 指標 | 目標值 | 違約計點 | 來源 |
|------|-------|---------|------|
| 系統可用率 | ≥99%/季 | 每不足 1% 計 3 點 | 服務水準表 |
| 單日最大中斷 | ≤4 小時/日 | 逾 4hr 每小時計 1 點 | 服務水準表 |
| 平均修復時間（MTTR） | ≤4 小時（含假日） | 逾 4hr 每小時計 1 點 | 契約§7(六) |
| 事件通報時間 | ≤1 小時 | 逾 1hr 每小時計 1 點 | 服務水準表 |
| 事件控制時間 | ≤72 小時（重大 ≤36hr） | — | 服務水準表 |
| 事件調查報告 | ≤1 個月 | — | 服務水準表 |

**可用率計算：** 季可用率 = (季總分鐘數 − 計畫外停機分鐘數) ÷ 季總分鐘數 ×100%。計畫性停機維護（每月定期保養、每季停機維護、年度電力檢測）經台鐵核准後不計入停機時數。

### 4.2.2　HA 雙機備援機制

系統採 Active-Active + Failover 架構（AD-005），具體實現：

| 項目 | 規格 |
|------|------|
| 正常運作模式 | 雙機 50/50 負載分攤，SIP 層 Round-Robin 分配新通話，HTTP 層 Nginx 反向代理 |
| 故障偵測 | Keepalived VRRP 心跳，≤2 秒偵測異常 |
| 自動切換 | ≤30 秒完成 VIP 漂移與服務接管，故障機的負載由存活機 100% 承接 |
| Session 保持 | Redis 共享 Session，任一台故障時進行中的對話上下文不遺失 |
| 網路隔離 | 管理 VLAN / 服務 VLAN / 資料 VLAN 三層隔離 |

**人工降級程序：** 當雙機同時異常或 AI 服務全面中斷時，系統自動將所有來電直接轉接人工客服佇列（FR-IVR-004），並透過 Prometheus 告警通知維運人員。此機制確保「系統異常時應能立即切換人工客服且不影響正常運作」之合規要求。

### 4.2.3　定期維護排程

| 週期 | 內容 | 預估停機 |
|------|------|---------|
| 每月 | 系統巡檢、日誌清理、備份驗證 | 線上滾動更新，免停機 |
| 每季 | 安全性修補、Docker 映像更新、GPU 驅動確認 | ≤2 小時（離峰時段） |
| 每年 | 電力檢測配合、硬體巡檢、全備份恢復演練 | 依台鐵機房排程 |

### 4.3.2　資料加密策略

| 層級 | 加密方式 | 適用範圍 |
|------|---------|---------|
| 傳輸層 | TLS 1.3 | 所有 API 呼叫、管理後台存取、WebSocket 連線 |
| 語音傳輸 | SRTP（AES-128） | RTP 語音串流加密（PJSIP 原生支援） |
| 儲存層 | AES-256 | PostgreSQL 個資欄位加密、備份檔案加密 |
| 應用層 | 正則遮罩 + Token 化 | STT 輸出之身分證字號、電話號碼即時去識別化 |

### 4.3.3　存取控制與稽核

- **身分認證：** 整合台鐵既有 Active Directory（AD/LDAP），統一帳號認證，不另建帳號體系
- **角色權限（RBAC）：** 值機員 / 督導 / 系統管理者 / 知識庫管理者四級角色，各角色僅具最小必要權限（FR-ADM-002）
- **操作日誌：** 所有使用者操作（登入登出、知識庫異動、系統設定變更、個資存取）均記錄完整軌跡，含時戳、操作者、IP 位址、異動內容（FR-ADM-004）
- **時戳校時：** 所有伺服器以 NTP 同步台鐵內部時間源，確保日誌時序一致

## 4.4　擴展性需求

### 4.4.1　GPU 運算擴充路徑

本案 AI 伺服器每台配置 NVIDIA L40S 48GB ×2（AD-002），硬體規格要求「AI 伺服器應保留擴充空間與介面」。系統設計之擴充路徑：

| 擴充階段 | 觸發條件 | 擴充方式 | 影響範圍 |
|---------|---------|---------|---------|
| **Stage 0（現行）** | — | GPU#0 跑 Llama 3.1 8B AWQ（~8GB），6 實例；GPU#1 跑 Embedding + QI | 滿足 30+30 路併發 |
| **Stage 1** | 併發需求超過 60 路或正確率需提升 | GPU#0 升級為 Llama 3.1 13B FP16（~26GB），減至 2 實例但推論品質提升 | 僅調整 vLLM 啟動參數，無需硬體異動 |
| **Stage 2** | 系統擴充第二條 E1 專線（60 B-channel） | 於 AI 伺服器擴充插槽加裝第 3 張 GPU | 需停機安裝，但軟體架構無需改動 |

模型替換完全透明：vLLM 支援熱載入模型，更換 LLM 模型（如未來台灣自研模型）僅需替換模型檔案與調整推論參數，應用層程式碼無需修改。

### 4.4.2　儲存擴充

- 每台伺服器 960GB SSD ×3（RAID-5），有效容量約 1.8TB
- 以月話務量 35,000 通、平均通話 3 分鐘估算：每月錄音約 105,000 分鐘 ≈ 190GB（G.711 格式）
- RAID-5 可用空間支撐約 9 個月錄音儲存；超過保留期限之錄音自動歸檔或清除
- SMS 紀錄保留 ≥5 年（sms_retention），以文字紀錄為主，每月約 7,000 則 ×5 年 ≈ 42 萬則，儲存需求 <1GB，無擴充壓力

### 4.4.3　軟體模組擴充

六層式架構（AD-001）之層間以 API 隔離，新增模組僅需實作對應 gRPC/REST 介面即可整合，不影響既有模組。Docker Compose 部署（AD-006）支援橫向新增容器服務，無需變更部署架構。

## 4.5　相容性需求

### 4.5.1　瀏覽器相容性

值機平台（FR-WS-001~005）、管理後台（FR-ADM-001~005）及旅客端數位客服介面須支援以下瀏覽器：

| 瀏覽器 | 最低版本 | 備註 |
|--------|---------|------|
| Microsoft Edge | 最新穩定版 -2 | Windows 預設瀏覽器 |
| Google Chrome | 最新穩定版 -2 | 市佔率最高 |
| Mozilla Firefox | 最新穩定版 -2 | 政府機關常用 |
| Apple Safari | 最新穩定版 -2 | macOS / iOS 使用者 |

「最新穩定版 -2」指支援當前最新穩定版本及前兩個主要版本，確保合理的向下相容範圍。

### 4.5.2　響應式設計（RWD）

所有使用者介面採 React 18+ / Next.js 搭配 Ant Design 元件庫開發，原生支援 RWD：

- **桌面版（≥1280px）：** 值機平台完整功能，多面板同時呈現（通話控制、對話紀錄、AI 輔助建議）
- **平板（768~1279px）：** 管理後台報表瀏覽、督導監控
- **行動裝置（<768px）：** 旅客端數位客服對話介面、SMS 通知檢視

值機平台為操作密集型介面，建議搭配 ≥22 吋螢幕使用；行動裝置版為唯讀查詢與對話功能，不含完整管理操作。

### 4.5.3　API 相容性

對外 REST API 遵循國發會「共通性應用程式介面規範」（契約§2(四)），採用版本控制（/v1/）機制，新版本上線時舊版本至少維持 6 個月相容期，確保介接端有充足調整時間。API 文件以 OpenAPI 3.0 規格撰寫並提供 Swagger UI 互動式測試介面。

- SMS 紀錄滿 5 年後依台鐵資訊安全管理規定執行安全刪除（覆寫式刪除）
- 所有資料刪除操作均記錄於稽核日誌，不可逆刪除須經管理者雙重確認

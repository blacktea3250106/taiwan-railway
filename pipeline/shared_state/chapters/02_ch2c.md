## 2.6　智慧質檢系統（AI-QI）

本節定義智慧質檢子系統之七項功能需求。AI-QI 以批次模式處理 MES 錄音檔，執行語音轉寫、規則比對、情緒分析與異常偵測，產出質檢報告供督導人員審閱。質檢 STT 轉寫複用 Whisper large-v3 + faster-whisper 引擎（AD-003），運行於 AI-IVR 伺服器 CPU；情緒分析模型運行於 GPU#1（AD-002），與即時 LLM 推論（GPU#0）隔離，避免批次任務與即時服務搶奪資源。

質檢系統使用之客服錄音屬於台鐵既有「客服品質管理」目的範圍，轉寫文字於產出後即套用正則表達式去識別化（身分證→A1234*****、電話→0912***789），去識別化後之文字方可用於統計分析與質檢評分，原始語音與去識別化文字分離存放，存取權限限制為督導以上角色。

### FR-QI-001　多格式數據匯入與批次處理

| 欄位 | 內容 |
|------|------|
| **編號** | FR-QI-001 |
| **名稱** | 多格式數據匯入與批次處理 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(六) |
| **對應編號** | TC-QI-001 / M-QI-01 / P-QI-01 / AD-002 |
| **KPI 關聯** | qi_speed：15min 錄音 → ≤5min 完成轉換分析並產出質檢報告 |

**描述：** 支援 WAV、MP3、OGG、FLAC 等常見音檔格式匯入，提供手動上傳與自動排程兩種模式。自動排程模式下，系統定時掃描 MES 錄音儲存目錄，自動匯入新增錄音並排入質檢佇列。15 分鐘錄音檔須於 5 分鐘內完成轉寫、規則比對、評分與報告產出。

**輸入：**
- MES 錄音檔（自錄音伺服器取得，WAV/PCM 格式為主）
- 手動上傳之音檔（管理後台 FR-ADM-001 上傳介面）
- 排程設定（執行頻率、時段、優先級）

**輸出：**
- 質檢任務佇列（含任務狀態：排隊中/處理中/完成/失敗）
- 音檔 metadata（檔名、時長、取樣率、通話 ID、座席 ID、日期時間）

**處理邏輯：**
1. 自動排程以 cron 機制觸發，預設每小時掃描一次 MES 錄音目錄，掃描頻率可於管理後台調整
2. 音檔匯入前自動偵測格式與取樣率，非 16kHz/mono 之音檔自動轉換（ffmpeg）以符合 Whisper 輸入要求
3. 佇列採先進先出（FIFO），但督導可手動調整個別錄音之優先級
4. 批次處理時依 CPU 可用核心數動態調配 faster-whisper 工作執行緒，避免影響同機之即時 STT 服務
5. 15 分鐘錄音之處理時間預算分配：STT 轉寫 ≤2.5min（Whisper RTF ~0.3x，含格式轉換）、規則比對 + 評分 ≤1.5min、情緒分析 ≤0.5min、報告生成 ≤0.5min

**例外流程：**
- 音檔損毀或格式不支援：標記為「匯入失敗」，記錄錯誤原因，通知管理員
- 質檢佇列積壓（待處理 >100 筆）：觸發告警至管理後台（FR-ADM-003），督導可暫停低優先級任務

### FR-QI-002　語音轉文字處理（業務場景術語優化）

| 欄位 | 內容 |
|------|------|
| **編號** | FR-QI-002 |
| **名稱** | 語音轉文字處理（業務場景術語優化） |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(六)、§三(二) |
| **對應編號** | TC-QI-002 / M-QI-02 / P-QI-02 / AD-003 |
| **KPI 關聯** | qi_speed：15min→≤5min、stt_accuracy ≥85% |

**描述：** 以 Whisper large-v3 + faster-whisper（CTranslate2 加速）將客服錄音批次轉寫為繁體中文文字。質檢 STT 與即時 STT（FR-STT-001）共用相同引擎與台鐵專屬詞庫，但質檢場景為離線批次處理，可使用較大 beam size 提升準確率。轉寫結果自動套用台鐵專屬詞庫進行 hotword boosting，涵蓋站名（新左營、板橋等）、車種（自強號、太魯閣、普悠瑪、區間車）、票種（電子票證、去回票）等業務術語。

**輸入：**
- 質檢佇列中之音檔（自 FR-QI-001）
- 台鐵專屬詞庫（同 FR-STT-001 §2.2 所述，詳見第二章(A)）

**輸出：**
- 逐字轉寫文字（含時間戳記，精確至 0.1 秒）
- 說話人分離標記（客服 / 旅客）
- 去識別化後之轉寫文字（供後續質檢分析使用）

**處理邏輯：**
1. 質檢場景 beam size 設定為 5（即時場景為 1），以準確率換取容許之批次延遲
2. 說話人分離（Speaker Diarization）：利用 pyannote-audio 開源模型，區分客服端與旅客端語音
3. 轉寫完成後立即執行去識別化：正則表達式遮罩個資欄位（身分證、電話、信用卡號），去識別化文字與原始轉寫分離儲存
4. 轉寫文字以 JSON 格式儲存至 PostgreSQL，關聯通話 ID、座席 ID、日期時間，保留 ≥6 個月（符合 log_retention 要求）

**例外流程：**
- 語音品質極差（訊噪比 <10dB）：標記為「轉寫品質不佳」，仍完成轉寫但質檢報告加註警示
- 錄音時長超過 60 分鐘：自動分段處理（每段 15 分鐘），各段獨立轉寫後合併

### FR-QI-003　自訂質檢規則與評分機制

| 欄位 | 內容 |
|------|------|
| **編號** | FR-QI-003 |
| **名稱** | 自訂質檢規則與評分機制 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(六) |
| **對應編號** | TC-QI-003 / M-QI-03 / P-QI-03 |
| **KPI 關聯** | qi_speed：15min→≤5min |

**描述：** 提供管理後台介面供督導人員自訂質檢評分規則、各規則權重及通過標準。系統依設定之規則自動評分，無需人工逐通聽取。

**輸入：**
- 去識別化轉寫文字（自 FR-QI-002）
- 質檢規則範本（系統預設 + 管理者自訂）

**輸出：**
- 各規則項目評分明細
- 總分與通過/未通過判定
- 扣分原因標記（對應至轉寫文字之具體段落）

**處理邏輯：**
1. 系統預設質檢規則範本包含：開場白標準用語、結束語標準用語、禁用語偵測、回覆完整性、問題解決率
2. 管理者可新增自訂規則，每條規則設定：規則名稱、比對方式（關鍵詞/正則/LLM 語意判斷）、權重（0~100）、扣分值
3. LLM 語意判斷型規則：呼叫 GPU#1 上之推論服務（AD-002），以 prompt 請 LLM 判斷轉寫文字是否符合規則描述，適用於無法以關鍵詞/正則捕捉之語意層面規則（例如「客服人員是否表達同理心」）
4. 評分結果自動寫入 PostgreSQL 質檢報告表，關聯通話 ID 與座席 ID
5. 質檢規則支援版本管理，規則修改後不影響已完成之歷史評分

**例外流程：**
- 新增規則與既有規則衝突（如同一關鍵詞在不同規則中加分又扣分）：系統提示衝突並要求管理者確認
- LLM 語意判斷逾時：該項規則評分標記為「待人工覆核」，不影響其他規則評分

### FR-QI-004　靜音與插話偵測

| 欄位 | 內容 |
|------|------|
| **編號** | FR-QI-004 |
| **名稱** | 靜音與插話偵測 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(六) |
| **對應編號** | TC-QI-004 / M-QI-04 / P-QI-04 |
| **KPI 關聯** | — |

**描述：** 自動分析錄音中的靜音段與雙方同時說話（插話）事件，作為質檢評分之客觀依據。

**輸入：**
- 原始音檔（自 FR-QI-001）
- 說話人分離結果（自 FR-QI-002）

**輸出：**
- 靜音事件清單（起止時間、持續秒數、歸屬方）
- 插話事件清單（起止時間、持續秒數）
- 靜音率與插話率統計值

**處理邏輯：**
1. 靜音偵測：以 VAD 分析音檔，連續靜音 ≥3 秒標記為靜音事件（閾值可於管理後台調整）
2. 靜音歸屬：結合說話人分離結果，判定靜音發生時「應由誰說話」（旅客等待客服回覆 = 客服端靜音）
3. 插話偵測：雙通道（客服/旅客）同時存在語音活動且持續 ≥0.5 秒，標記為插話事件
4. 統計值計算：靜音率 = 客服端靜音總時長 ÷ 通話總時長；插話率 = 插話總次數 ÷ 通話總時長（分鐘）
5. 統計值可作為 FR-QI-003 自訂規則之輸入（例如「客服端靜音率 >15% 扣 N 分」）

**例外流程：**
- 單通道錄音（無法分離客服/旅客）：僅偵測整體靜音，不進行歸屬分析，報告標註「單通道限制」

### FR-QI-005　語調與情緒分析

| 欄位 | 內容 |
|------|------|
| **編號** | FR-QI-005 |
| **名稱** | 語調與情緒分析 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(六) |
| **對應編號** | TC-QI-005 / M-QI-05 / P-QI-05 / AD-002 |
| **KPI 關聯** | — |

**描述：** 分析客服人員與旅客之語調情緒，識別負面情緒（憤怒、焦躁、不耐）並標記異常通話。情緒分析模型運行於 GPU#1（AD-002），以批次推論方式處理，不與 GPU#0 之即時 LLM 推論競爭資源。

**輸入：**
- 原始音檔之聲學特徵（梅爾頻譜、基頻 F0、語速）
- 說話人分離結果（自 FR-QI-002）
- 去識別化轉寫文字（文本情緒輔助判斷）

**輸出：**
- 逐段情緒標籤（正面/中性/負面）與信心度分數
- 通話整體情緒曲線（時間軸視覺化）
- 異常情緒標記（旅客持續負面 ≥60 秒、客服出現負面情緒等）

**處理邏輯：**
1. 聲學情緒模型：採用開源 Speech Emotion Recognition（SER）模型，以梅爾頻譜與韻律特徵為輸入，每 5 秒窗口產出一個情緒標籤
2. 文本情緒輔助：以 LLM（GPU#1）分析轉寫文字之情緒傾向，與聲學結果加權融合（聲學 0.6 + 文本 0.4）
3. 情緒曲線：將逐段情緒分數繪製為時間軸曲線，於質檢報告中以圖表呈現
4. 異常標記規則：旅客連續 3 個以上窗口（≥15 秒）為負面情緒，或客服出現任何負面情緒窗口，即標記為異常通話
5. 異常通話自動觸發 FR-QI-006 警報機制

**例外流程：**
- GPU#1 負載過高（使用率 >90%）：情緒分析任務排入延遲佇列，優先保障 Embedding 向量化之即時需求
- 語音品質不佳導致聲學特徵不可靠：降級為純文本情緒分析（權重調整為文本 1.0）

**輸入：**

**例外流程：**
- WebSocket 連線中斷：降級為 Email 通知，並記錄推播失敗事件
- 大量警報湧入（短時間內 >20 則）：自動彙整為摘要通知，避免通知疲勞

## 2.7　查詢及報表功能（RPT）

本節定義系統層級之查詢及報表功能。質檢專屬報表已於 §2.6 FR-QI-007 定義，本節聚焦對話服務、AI 管理、知識點分析、滿意度等橫跨所有子系統之統計報表，以及保固期內之客製報表服務承諾。報表資料儲存於 PostgreSQL 16 + TimescaleDB（AD-008），前端採 React 18+ / Ant Design 儀表板元件呈現。

---

### FR-RPT-001　統計分析報表

| 欄位 | 內容 |
|------|------|
| **編號** | FR-RPT-001 |
| **名稱** | 統計分析報表 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(七) |
| **對應編號** | TC-RPT-001 / M-RPT-01 / P-RPT-01 |
| **KPI 關聯** | — |

**描述：** 提供四大類統計分析報表：對話服務類、AI 管理類、知識點分析類、滿意度類，涵蓋語音與文字雙通道之完整服務數據。

**輸入：**
- AI-IVR 通話紀錄（通數、時長、轉人工率、獨立解決率）
- AI-Agent 對話紀錄（對話數、回應時間、轉人工率）

- 知識庫命中紀錄（各知識點被引用次數、未命中查詢）
- 滿意度調查結果（FR-IVR-006 語音通道評分）

**輸出：**
- **對話服務類：** 日/週/月話務量、接聽率、平均處理時長、AI 獨立解決率、轉人工率、尖峰時段分析
- **AI 管理類：** LLM 推論正確率趨勢、回應延遲 P50/P95/P99、GPU 使用率、模型版本與效能對比
- **知識點分析類：** 熱門知識點排行、未命中查詢清單（知識庫缺口分析）、各知識點正確率
- **滿意度類：** 整體滿意度分數、各服務通道滿意度對比、不滿意通話關聯分析

**處理邏輯：**
1. 資料每 5 分鐘從各子系統之 PostgreSQL 表彙整至報表資料表，以 TimescaleDB 連續聚合減輕查詢壓力

3. 報表均支援自訂時間區間篩選，預設提供「今日 / 本週 / 本月 / 自訂」四種快捷
4. 所有報表支援匯出 PDF / Excel / CSV 格式

**例外流程：**
- 資料來源延遲（某子系統日誌寫入延遲 >10 分鐘）：報表頁面顯示「資料更新至 HH:MM」時間戳，不顯示可能不完整之數據

## 2.8　SMS 簡訊服務

SMS 簡訊服務之核心功能需求已於第二章(B) FR-AGT-007 定義（雙備援收發裝置、AI 自動回覆、同步通知客服人員、紀錄保留 ≥5 年），本節補充 SMS 子系統之硬體架構與資料保存策略。

### SMS 硬體架構

正式環境配置 2 台 SMS 簡訊收發系統（Active-Active 雙備援，AD-005），測試環境配置 1 台。每台設備內建 256GB+ 儲存空間與 4G LTE+ 通訊模組。LTE 模組之品牌與產地須符合合規紅線要求，排除大陸地區品牌與大陸地區製造之通訊晶片組（含華為海思、中興、展訊等），優先選用 Sierra Wireless、Telit、u-blox 等歐美品牌模組。

### SMS 資料保存策略

| 項目 | 規格 |
|------|------|
| 保存期限 | ≥5 年（需求書§三(五)） |
| 保存內容 | 簡訊收發紀錄（發送/接收時間、收發號碼去識別化、簡訊內容、AI 自動回覆 / 人工回覆標記） |
| 儲存位置 | PostgreSQL 16 資料庫（台鐵機房地端） |
| 資料量估算 | 月均 2,500~7,000 則 × 60 個月（5 年）= 150,000~420,000 筆，每筆含 metadata 約 2KB，總量 ≤1GB |
| 備份策略 | 每日增量備份 + 每週全量備份，備份檔異機儲存（管理/資安監控伺服器） |
| 查詢功能 | 管理後台提供依時間區間、號碼（去識別化）、關鍵詞篩選之查詢介面 |

SMS 自動回覆之同步通知機制：AI-Agent 產生自動回覆後，透過 WebSocket 即時推播至值機平台（FR-WS-001），客服人員可於座席介面查看 AI 回覆內容與旅客資訊，判斷是否需人工介入接手後續對話。自動回覆紀錄同時寫入資料庫，供稽核追蹤。

## 2.9　知識庫管理

知識庫管理之核心功能需求已於第二章(B) FR-LLM-005（知識庫維護與自動學習）及 FR-AGT-002（向量資料庫整合與版本管理）定義，本節補充知識庫之資料結構、覆蓋率管理與生命週期策略。

### 知識庫資料結構

知識庫以 Qdrant 向量資料庫（AD-008）為核心，搭配 BAAI/bge-large-zh-v1.5 中文 embedding 模型進行向量化。知識文件以 chunk 為單位儲存，每個 chunk 攜帶以下 metadata：

| 欄位 | 說明 |
|------|------|
| doc_id | 文件唯一識別碼 |
| doc_title | 文件標題 |
| doc_version | 文件版本號 |
| chunk_index | 該 chunk 在文件中的序號 |
| source_category | 來源分類（公告/規章/FAQ/時刻表/票務規定/其他） |
| publish_date | 文件發布日期 |
| expire_date | 文件有效期限（可為空，表示永久有效） |
| last_updated | 最後更新時間 |

此 metadata 結構支援 FR-AGT-004 RAG Citation 之引用來源標註，LLM 回應時可明確標示「本回覆參考自：{doc_title} 第 {chunk_index} 段，版本 {doc_version}」。

### 知識庫覆蓋率管理

為確保 AI 回答品質達 ≥85% 正確率（ivr_accuracy / agent_accuracy），知識庫須涵蓋台鐵客服常見問題之絕大多數場景。系統提供以下覆蓋率管理機制：

1. **未命中查詢追蹤：** RAG 檢索 Top-K 相似度分數均 <0.5 之旅客查詢，自動收集至「未命中清單」（FR-RPT-001 知識點分析報表），每週產出報告供知識庫管理者審閱
2. **覆蓋率指標：** 系統計算「有效命中率 = RAG 檢索相似度 ≥0.5 之查詢數 ÷ 總查詢數」，於管理後台儀表板即時顯示
3. **降級策略：** 當知識庫無法提供足夠參考資訊時（信心度 <0.6），系統自動觸發轉人工機制（詳見第二章(B) FR-LLM-002 信心度三層策略），避免以不完整資訊誤導旅客
4. **增量更新流程：** 管理者於管理後台上傳新文件後，系統自動 chunking → embedding → 寫入 Qdrant，全程無需停機，更新即時生效

### 知識庫資料來源與初始匯入

台鐵知識庫初始資料來源預估包含：

- 旅客服務常見問答（FAQ）
- 各級列車時刻表與票價資訊（透過 API 即時查詢，不靜態儲存）
- 旅客運送契約、營運規章
- 各站設施與服務資訊
- 優惠票種與購票規定
- 遺失物處理流程
- 行李託運規定
- 無障礙服務資訊

初始匯入時由台鐵提供上述文件之電子檔（PDF / Word / HTML），系統以自動化 pipeline 執行文件解析 → 清洗 → chunking → embedding → 寫入 Qdrant，預估知識庫規模 10,000~50,000 筆 chunk。

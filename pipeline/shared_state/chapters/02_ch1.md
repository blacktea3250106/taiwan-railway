## 第一章　系統摘述

### 1.1 系統設計理念與整體定位

本系統以「AI 輔助而非取代人工」為核心設計理念，透過智慧語音應答（AI-IVR）、AI 虛擬客服（AI-Agent）及智慧質檢（AI-QI）三大核心引擎，介接台鐵既有 PSTN/WebCall/數位客服通道，提升旅客服務品質與客服作業效率。系統全面採用開源技術堆疊，以地端高可用（HA）雙機架構部署於台鐵機房，所有資料不離開機房，確保資訊安全與旅客個資保護。

系統設計遵循以下原則：

- **零衝擊介接**：AI-IVR 以 SIP/RTP fork 方式被動接收語音流副本，不修改 MES Server 既有設定，不影響現有電話系統運作（AD-007）。
- **開源永續使用**：所有 AI 模型與軟體元件採用開源授權，軟體授權到期後可繼續使用到期前最後更新版本（AD-008）。
- **降低維運門檻**：部署採 Docker Compose，不使用 Kubernetes，正式環境 6~8 台伺服器規模無需容器叢集的複雜度（AD-006）。
- **異常即切人工**：系統在任何環節發生異常時，均能立即將通話或對話切換至人工客服，確保旅客服務不中斷。

### 1.2 六層式系統架構

本案採六層式架構設計（AD-001），將 AI 功能模組獨立於既有系統之外，各層職責明確，層間以標準化介面通訊，任一層的升級或替換均不影響其他層級：

#### 第一層：基礎設施層

承載所有運算與儲存資源。正式環境配置 AI-IVR 伺服器 2 台（16 核心 ≥2.2GHz×2，DDR5 64GB+，960GB SSD×3 RAID-5）、AI 伺服器 2 台（16 核心 ≥3.0GHz×2，DDR5 128GB+，960GB SSD×3 RAID-5，NVIDIA L40S 48GB×2）、管理/資安監控伺服器 2 台（16 核心 ≥3.0GHz×2，DDR5 64GB+）、SMS 簡訊收發系統 2 台，統一收納於 42U 機櫃。測試環境另設 AI-IVR 測試伺服器 1 台、AI 測試伺服器 1 台（L40S×1）、SMS 測試 1 台。所有設備採冗餘電源（RPS）設計，伺服器主機均具備 BMC 遠端管理功能。

#### 第二層：平台安全層

提供全系統的安全防護基礎。存取控制採 RBAC 角色型權限模型，整合台鐵既有 AD（Active Directory）進行帳號驗證（LDAP 協定）。傳輸層強制 TLS 1.3 加密，儲存層對旅客個資採用 AES-256 加密。所有操作日誌保留 ≥6 個月，個資存取日誌獨立管控。本層依 CNS 27001 資訊安全管理標準設計控制措施，詳見第八章資訊安全與合規專章。

#### 第三層：資料管理層

統一管理結構化與非結構化資料。關聯式資料庫採用 PostgreSQL 16 搭配 TimescaleDB 擴展處理時序資料（日誌、監控指標）；向量資料庫採用 Qdrant（Rust 語言開發，HNSW 索引，Top-K 查詢延遲 <50ms），搭配 bge-large-zh-v1.5 中文 Embedding 模型實現語意檢索；快取層採用 Redis 7 處理 Session 管理與即時對話上下文存取。

#### 第四層：AI 服務層

本系統核心運算層，涵蓋四大 AI 引擎：

| AI 引擎 | 技術選型 | 部署方式 | 核心功能 |
|---------|---------|---------|---------|
| LLM 大型語言模型 | Llama 3.1 8B-Instruct AWQ 量化 + vLLM | GPU#0 專責推論（AD-002） | 語意理解、問答生成、多輪對話、後送案件分析 |
| STT 語音識別 | Whisper large-v3 + faster-whisper | CPU 推論（AD-003） | 即時中英文語音轉文字、VAD 斷點偵測 |
| TTS 語音合成 | Coqui XTTS-v2（備案：MeloTTS / Piper） | CPU 推論（AD-003） | 中英文即時語音合成、動態選單生成 |
| Embedding + QI | bge-large-zh-v1.5 + 情緒分析模型 | GPU#1（AD-002） | 向量化語意檢索、質檢語調情緒分析 |

GPU 資源分配策略（AD-002）：GPU#0 專責 LLM 推論，確保 30 路並行時每路均有獨立推論資源（Llama 3.1 8B AWQ 量化後約 8GB VRAM，單卡 L40S 48GB 可同時運行 6 個實例）；GPU#1 負責 Embedding 計算、質檢批次分析及未來擴充預留，避免即時推論與批次任務搶奪資源，確保語音回應 ≤5,000ms 之 SLA 目標。STT 與 TTS 選擇以 CPU 推論（AD-003），因串流語音處理屬低延遲任務，CPU 的每核心延遲優於 GPU 批次吞吐特性，30 路 faster-whisper 約使用 9 個 CPU 核心，AI-IVR 伺服器配備 32 核心足以承載。

#### 第五層：通道接入層

統一管理所有外部通訊通道。語音通道透過 PJSIP（pjsua2 Python binding）實現 SIP/RTP 協定處理，支援 E1 專線（現有 1 條，30 B-channel）、WebCall SBC（雙備援 failover）及行政交換機（類比 → VoIP Gateway → G.711 編碼）三種語音來源。文字通道透過 CRM Gateway 以 REST/WebSocket API 接入數位客服訊息。SMS 通道透過雙備援簡訊收發系統處理旅客簡訊服務。內部服務間通訊採 gRPC 協定（二進制序列化 + HTTP/2 多工），相較 REST 快 2~5 倍，確保 AI-IVR 至 LLM 推論的延遲預算。對外 REST API 依國發會共通性應用程式介面規範設計。

#### 第六層：使用者層

提供各角色專屬操作介面，前端框架採用 React 18 + Next.js + Ant Design，支援 RWD 響應式設計，相容 Edge/Chrome/Firefox/Safari 瀏覽器。各介面功能於 1.4 節使用者角色說明中詳述。

#### 實體部署架構

正式環境以 Active-Active + Failover 架構部署（AD-005），正常運行時雙機以 50/50 負載分攤流量，任一台故障時單機承載 100%。HA 機制透過 Keepalived + VIP（Virtual IP）實現，VRRP 心跳 ≤2 秒偵測、≤5 秒完成切換。Session 資料存放於 Redis 共享，確保故障切換時對話上下文不遺失。

| 實體伺服器 | 正式環境 | 測試環境 | 運行服務 |
|-----------|---------|---------|---------|
| AI-IVR 伺服器 | 2 台（Active-Active） | 1 台 | SIP/RTP 處理、STT（CPU）、TTS（CPU）、對話管理 |
| AI 伺服器 | 2 台（Active-Active） | 1 台 | LLM 推論（GPU#0）、Embedding/QI（GPU#1）、RAG Pipeline |
| 管理/資安監控伺服器 | 2 台（Active-Active） | — | PostgreSQL、Qdrant、Redis、Prometheus、Grafana、後台服務 |
| SMS 簡訊收發系統 | 2 台（雙備援） | 1 台 | 4G LTE+ 簡訊收發 |

網路拓撲採 VLAN 隔離設計，區分管理 VLAN、服務 VLAN 與資料 VLAN，詳見第九章網路拓撲設計。

### 1.4 使用者角色與系統互動

本系統服務五類使用者，各角色之系統權限與功能存取範圍如下：

#### （一）旅客

旅客為本系統之最終服務對象，透過三種管道與系統互動：

- **電話語音（PSTN/WebCall）**：撥打客服電話或透過網頁電話接入 AI-IVR，以自然語言對話查詢時刻、票價、乘車資訊等，系統以語音回覆（FR-IVR-002）。AI 無法解決時自動轉接真人客服（FR-IVR-004），轉接時系統同步傳送對話摘要予值機員，旅客無需重複說明。通話結束前提供滿意度調查（FR-IVR-006）。
- **數位文字客服（Web/App）**：透過網頁或行動 App 內嵌之對話視窗，以文字方式與 AI-Agent 互動（FR-AGT-003）。對話視窗設有熱門選單與宣導推播專區，由台鐵後台自行設定調整。AI 回應自動標註知識來源出處（FR-AGT-004），供旅客自行驗證。
- **SMS 簡訊**：旅客可透過簡訊查詢服務，系統以 AI 自動回覆，同步通知客服人員（FR-AGT-007）。

#### （二）值機員（25 席）

客服第一線人員，使用值機平台（WS）處理旅客來電與文字諮詢：

- 統一操作介面整合接聽、保留、轉接、掛斷等通話控制功能（FR-WS-001）
- 接聽時系統即時推送 AI 建議回覆與相關知識庫內容，輔助快速應答（FR-WS-002）
- 通話/對話結束後自動生成對話摘要，支援建立與追蹤後送工單（FR-WS-003）
- 支援簽入/簽出、就緒/忙碌/休息等狀態切換，統計各狀態時長（FR-WS-005）

目前台鐵客服同時在線約 15 人（60% 在線率，排班制），AI 系統預期可獨立處理 60~70% 的簡單查詢（時刻、票價、車站資訊），使值機員得以專注處理複雜案件與後送事項。

#### （三）督導（2 席）

客服督導人員，負責即時服務品質監控：

- 即時監看所有座席狀態、監聽通話、密語指導、必要時強制介入（FR-WS-004）
- 查看即時服務指標儀表板，監控 AI 轉人工比例，若超過閾值系統自動告警
- 接收智慧質檢系統之異常通報（FR-QI-006），針對標記異常之通話進行後續處理

#### （四）系統管理者

數位發展處 IT 人員，透過管理後台（ADM）執行系統維運：

- 系統參數設定、帳號權限管理（整合 AD/LDAP）（FR-ADM-001、FR-ADM-002）
- 系統監控儀表板（Prometheus + Grafana），即時掌握 CPU/GPU/記憶體/服務狀態（FR-ADM-003）
- 日誌稽核查詢、備份回復操作（FR-ADM-004、FR-ADM-005）
- 知識庫內容維護——新增、修改、刪除知識條目及版本管理（FR-LLM-005）

管理後台設計以「台鐵 IT 人員」為目標讀者，操作流程力求直覺，搭配維護手冊之 step-by-step 操作指引，確保保固期結束後台鐵可自行維運。

#### （五）外點後送人員（225 人，同時在線約 23 人）

各車站與外點單位之後送案件處理人員，使用既有後送案件系統。本案 AI 系統透過 API 將自動分類、優先級判定及情感分析結果推送至後送系統（FR-LLM-008），不修改現有後送系統操作介面，225 位外點人員之作業流程維持不變。

### 1.6 效益分析

#### 量化效益

| 效益指標 | 現況基線 | 導入 AI 後目標 | 改善幅度 |
|---------|---------|---------------|---------|
| 30 秒接聽率 | 64% | ≥85% | +21 百分點 |
| 整體接聽率 | 83% | ≥95%（AI 承接後溢流率大幅降低） | +12 百分點 |
| 轉接率 | 23% | ≤10%（AI 獨立解決 60~70% 簡單查詢） | -13 百分點 |
| 數位客服配對率 | 93% | ≥98%（AI-Agent 無需等待排隊配對） | +5 百分點 |
| 質檢覆蓋率 | 人工抽檢（推估 <5%） | 100%（AI 全量自動質檢） | 全面提升 |
| 質檢時效 | 人工聽取 1:1 時間 | 15 分鐘錄音 ≤5 分鐘完成分析 | 效率提升 3 倍 |

#### 質化效益

- **服務品質一致性**：AI 回覆內容基於知識庫與即時 API 查詢，避免人工客服因經驗差異產生回覆品質落差。
- **24 小時服務延伸**：AI-IVR 與 AI-Agent 可在非值班時段（夜間、假日）提供基本查詢服務，旅客無需等到上班時間。
- **知識累積與傳承**：所有對話紀錄、質檢結果、後送案件分析均系統化留存，新進客服人員可透過 AI 知識輔助快速上手。
- **管理決策支援**：多維度報表與儀表板（FR-RPT-001~003）提供話務趨勢、旅客關注議題、客服績效等數據，輔助管理層決策。

# 第三章　系統介面需求

**案號：L0215P2010U**
**國營臺灣鐵路股份有限公司 AI 客服系統建置案**

---

本章定義系統介面三大類別：**內部介面**（AI 子系統間通訊）、**外部介面**（與台鐵既有系統介接）、**使用者介面**（人員操作與旅客互動）。所有介面設計遵循六層式架構（AD-001）之層間隔離原則，內部採 gRPC 高效二進制通訊（AD-006），外部 REST API 依國發會「共通性應用程式介面規範」辦理，傳輸層一律 TLS 1.3 加密。

---

## 3.1　內部介面

內部介面指 AI 客服系統各子模組之間的通訊鏈路，部署於同一 Docker Compose 叢集內（AD-006），透過容器內部網路通訊，不經過外部網路。

### 3.1.1　AI-IVR ↔ LLM 推論介面（gRPC）

| 項目 | 規格 |
|------|------|
| **協定** | gRPC / Protocol Buffers over HTTP/2 |
| **端對端延遲預算** | ≤2,500ms（含序列化與反序列化） |
| **呼叫模式** | Unary RPC（單次請求—回應） + Server-side Streaming（逐 token 串流） |
| **相關模組** | M-IVR-01（語音通道）→ M-LLM-01（GPU 推論引擎） |
| **GPU 分配** | GPU#0 專責 LLM 推論（AD-002），vLLM 引擎 continuous batching |
| **超時處理** | 推論逾 2,500ms 觸發 deadline exceeded，AI-IVR 播放候補話術「請稍候」並重試一次；二次失敗轉人工 |

**延遲分配說明：** 需求書要求旅客說完至系統回覆首字 ≤5,000ms（ivr_response）。整段延遲鏈路分配如下：VAD 偵測尾端靜音 ~500ms → STT 轉寫 ~800ms → gRPC 呼叫含 LLM 推論 ≤2,500ms → TTS 首包合成 ~300ms → RTP 封包發送 ~100ms，合計 ~4,200ms，保留 ~800ms 緩衝。選用 gRPC 而非 REST 的原因正是其二進制序列化與 HTTP/2 多工特性，較 REST/JSON 快 2~5 倍，確保推論呼叫不成為瓶頸。

### 3.1.2　AI-Agent ↔ LLM 推論介面（gRPC）

| 項目 | 規格 |
|------|------|
| **協定** | gRPC / Protocol Buffers over HTTP/2 |
| **端對端延遲預算** | ≤3,500ms（文字通道無 STT/TTS 環節，延遲預算較寬裕） |
| **呼叫模式** | Server-side Streaming（逐段回覆，前端即時顯示） |
| **相關模組** | M-AGT-03（智慧問答）→ M-LLM-02（語意理解），經 M-AGT-02（Qdrant 向量檢索）預檢索後注入 context |
| **附加資料** | RAG 檢索結果之 chunk metadata（文件名稱、頁碼、段落編號）隨 gRPC message 傳遞，供 Citation 標註（FR-AGT-004） |

文字通道延遲鏈路：訊息接收 ~50ms → 向量檢索（Qdrant Top-K）≤200ms → gRPC LLM 推論 ≤3,500ms → 回應組裝與 Citation 抽取 ~200ms，合計 ~3,950ms，滿足 agent_response ≤5,000ms。

### 3.1.3　AI-QI ↔ LLM 批次介面（gRPC）

| 項目 | 規格 |
|------|------|
| **協定** | gRPC / Protocol Buffers |
| **呼叫模式** | Unary RPC（批次提交） |
| **排程策略** | 非即時批次任務，排入佇列由 GPU#1 處理（AD-002），避免與即時推論搶奪 GPU#0 資源 |
| **相關模組** | M-QI-02（質檢 STT）→ M-LLM-05（後送分析）；M-QI-05（情緒分析）→ GPU#1 推論 |
| **效能目標** | 15 分鐘錄音 → ≤5 分鐘完成轉寫分析並產出質檢報告（qi_speed） |

GPU 分工設計（AD-002）確保即時服務（AI-IVR / AI-Agent）與批次服務（AI-QI）互不干擾：GPU#0 專責即時 LLM 推論，GPU#1 負責 Embedding 向量化、質檢情緒分析及未來擴充預留。

### 3.1.4　向量資料庫介面

| 項目 | 規格 |
|------|------|
| **元件** | Qdrant（Rust 高效能向量資料庫） |
| **Embedding 模型** | BAAI/bge-large-zh-v1.5（~1.3GB） |
| **協定** | gRPC（高頻查詢）/ REST（管理操作） |
| **查詢延遲** | Top-K 語意檢索 ≤200ms（HNSW 索引） |
| **呼叫方** | M-AGT-02（AI-Agent RAG）、M-WS-02（值機員 AI 知識輔助）、M-QI-02（質檢術語比對） |

知識庫更新時，管理後台（M-ADM-01）透過 REST API 觸發增量向量化，Qdrant 支援版本化 collection，可回溯至任意歷史版本（FR-AGT-002）。

### 3.1.5　快取與 Session 介面

| 項目 | 規格 |
|------|------|
| **元件** | Redis 7 |
| **用途** | ① Session 共享（HA 雙機，AD-005）② 即時對話上下文 ③ STT/TTS 快取 |
| **協定** | Redis Protocol（RESP3） |
| **存取延遲** | ≤1ms（記憶體存取） |
| **HA 策略** | Redis Sentinel 主從切換，確保 Active-Active 雙機均可讀寫 |

### 3.1.6　關聯資料庫介面

| 項目 | 規格 |
|------|------|
| **元件** | PostgreSQL 16 + TimescaleDB 擴展 |
| **協定** | PostgreSQL Wire Protocol（libpq） |
| **用途** | 對話紀錄、工單、質檢結果、系統參數、使用者歷史摘要（個性化問答，FR-AGT-005）、SMS 紀錄（≥5 年保留） |
| **安全** | 個資欄位 AES-256 加密儲存，存取日誌保留 ≥6 個月（log_retention） |

## 3.2　外部介面

外部介面指本系統與台鐵既有系統之介接鏈路。所有介接採 Adapter Pattern 設計——每套既有系統對應一個獨立的 Adapter 模組，API 規格變更時僅需修改對應 Adapter，不影響核心 AI 邏輯。尚未取得正式 API 規格之系統（如訂票紀錄查詢、時刻表查詢），採 Mock API 先行開發，待台鐵提供規格後替換 Adapter 實作。

### 3.2.1　PSTN E1 VoIP Gateway（SIP Trunk）

| 項目 | 規格 |
|------|------|
| **介接方式** | SIP Trunk（PJSIP pjsua2 Python binding） |
| **信令** | SIP INVITE / BYE / REFER / DTMF（RFC 2833 帶內 + SIP INFO 帶外） |
| **媒體** | RTP fork — AI-IVR 被動接收 MES Server 複製之 RTP 語音流 |
| **編碼** | G.711 μ-law / a-law（64kbps PCM） |
| **通道數** | 現有 1 條 E1（30 B-channel），系統支援最多 2 條 E1 擴充 |
| **相關 FR** | FR-IVR-001（≥30 埠）、FR-IVR-005（DTMF 按碼偵測） |

**零衝擊設計：** RTP fork 為被動複製機制，AI-IVR 接收的是語音流副本，不攔截、不修改、不影響原始通話路徑，E1 的 30 個 B-channel 全數保留予既有話務使用，AI-IVR 不佔用任何 B-channel。詳細信令流程圖見第 2.9 節。

### 3.2.2　WebCall SBC（SIP Trunk，雙備援）

| 項目 | 規格 |
|------|------|
| **介接方式** | SIP Trunk，支援雙 SBC 註冊（Primary / Secondary） |
| **Failover** | AI-IVR 之 SIP UA 同時向兩台 SBC 註冊，偵測主 SBC 無回應後 ≤5 秒自動切換至備援 SBC |
| **相關 FR** | FR-IVR-001 |

SBC 的 failover 機制（VIP-based 或 DNS-based）須於需求訪談時與既有廠商確認；無論何種機制，AI-IVR 端均以雙註冊方式因應，切換期間進行中的通話由 MES 維持，AI-IVR 僅短暫遺失 AI 輔助功能，通話本身不中斷。

### 3.2.3　MES Server（話務分派與錄音）

| 項目 | 規格 |
|------|------|
| **介接方式** | RTP fork 接收 + API 呼叫 |
| **設計原則** | **不修改 MES Server 任何設定**（AD-007），AI-IVR 為新增獨立模組 |
| **RTP fork** | MES 將通話 RTP 複製一份至 AI-IVR 接收埠 |
| **API 用途** | 讀取通話分配資訊、錄音檔路徑（供 AI-QI 使用） |
| **相關 FR** | FR-IVR-004（轉接真人客服）、FR-QI-001（錄音匯入） |

### 3.2.4　行政交換機（類比 VoIP Gateway）

| 項目 | 規格 |
|------|------|
| **路徑** | 類比線路（02-2381-5226）→ VoIP Gateway（G.711 編碼）→ MES → RTP fork → AI-IVR |
| **語音品質** | G.711 PCM 64kbps，類比→數位轉換損失極小 |
| **注意事項** | 類比線路背景雜訊較高，VAD 參數須針對此路徑調校，確保語音斷點偵測（FR-STT-003）正常運作 |
| **相關 FR** | FR-IVR-001 |

### 3.2.5　CRM Gateway（數位文字客服）

| 項目 | 規格 |
|------|------|
| **介接方式** | REST API（JSON）+ WebSocket（即時推播） |
| **方向** | CRM Gateway ↔ AI-Agent 雙向通訊 |
| **REST** | 訊息收發、對話狀態查詢、歷史紀錄同步 |
| **WebSocket** | 即時訊息推播、AI 自動回覆 SMS 時同步通知客服人員（FR-AGT-007） |
| **安全** | TLS 1.3 + OAuth 2.0 Bearer Token |
| **相關 FR** | FR-AGT-001（≥30 埠數位通道）、FR-AGT-003（智慧問答）、FR-AGT-006（Web/App 整合） |

API 規格須由台鐵提供（待需求訪談確認），開發期間以 Mock API 模擬 CRM Gateway 行為。

### 3.2.6　AD / LDAP（帳號驗證）

| 項目 | 規格 |
|------|------|

| **相關 FR** | FR-LLM-006（知識庫權限控管）、FR-ADM-002（帳號與角色權限管理） |

### 3.2.7　企業資訊入口平台（SSO）

| 項目 | 規格 |
|------|------|
| **介接方式** | SAML 2.0 或 OAuth 2.0（依台鐵既有 SSO 機制） |
| **用途** | 管理後台與值機平台之單一登入，使用者無需重複輸入帳號密碼 |
| **相關 FR** | FR-ADM-002 |

### 3.2.8　訂票紀錄查詢系統

| 項目 | 規格 |
|------|------|
| **介接方式** | REST API（規格待台鐵提供） |

### 3.2.9　時刻表查詢系統

| 項目 | 規格 |

| **相關 FR** | FR-LLM-002、FR-AGT-003 |

### 3.2.10　後送案件系統

| 項目 | 規格 |
|------|------|

| **設計原則** | AI 後送分析為獨立模組，透過 API 推送分析結果，**不修改現有後送系統操作介面**，外點人員無需重新培訓 |

## 3.3　使用者介面

使用者介面涵蓋三大族群：客服值機人員（25 席 + 督導 2 席）、系統管理者、旅客。前端採 React 18+ / Next.js / Ant Design 建構（AD-008），支援 RWD 響應式設計，相容 Edge、Chrome、Firefox、Safari。

### 3.3.1　值機平台（WS — WorkStation）

值機平台為客服人員日常操作核心介面，主畫面採**三欄式佈局**：

| 區域 | 功能 | 相關 FR |
|------|------|---------|
| **左欄 — 通話/對話列表** | 排隊來電、進行中通話、待處理文字對話，依優先級排序 | FR-WS-001、FR-WS-005 |
| **中欄 — 對話工作區** | 通話控制（接聽/保留/轉接/掛斷）、即時 STT 逐字稿、旅客訊息、AI 回覆歷程 | FR-WS-001 |
| **右欄 — AI 知識輔助側邊欄** | AI 即時建議回覆、相關知識庫文件（含引用來源）、旅客歷史對話摘要 | FR-WS-002 |

**AI 知識輔助側邊欄**為本案核心差異化功能——值機員接聽時，系統即時將旅客語音（經 STT 轉寫）送入 LLM + RAG 管線，於側邊欄推送建議回覆與佐證文件。值機員可一鍵採用建議或自行修改後回覆，所有採用/修改紀錄回饋至系統供持續優化（FR-AGT-008）。

通話結束後，系統自動生成對話摘要（FR-WS-003），值機員確認後可直接建立後送工單或標記完成。督導可透過專屬監控視圖即時監看所有座席狀態、監聽通話、密語指導或強制介入（FR-WS-004）。

### 3.3.2　管理後台（ADM — Admin）

管理後台供系統管理者使用，主要功能模組：

| 模組 | 功能摘要 | 相關 FR |
|------|---------|---------|
| **系統參數設定** | IVR 話術、AI 回應策略閾值、通道分配、服務時段、信心度門檻 | FR-ADM-001 |
| **帳號權限管理** | AD/LDAP 整合、RBAC 角色設定（值機員/督導/管理者/外點人員） | FR-ADM-002 |
| **系統監控儀表板** | 整合 Prometheus + Grafana，即時顯示可用率、回應時間、GPU 使用率、通話量、告警通知 | FR-ADM-003 |
| **日誌稽核** | 系統日誌、操作日誌、存取日誌集中查詢，保留 ≥6 個月 | FR-ADM-004 |
| **備份管理** | 備份排程設定、手動觸發、回復操作、狀態監控 | FR-ADM-005 |
| **知識庫管理** | 文件上傳、向量化觸發、版本管理、審核流程 | FR-LLM-005 |
| **報表中心** | 統計分析報表、儀表板、客製報表匯出（Word/PDF/Excel/HTML/CSV） | FR-RPT-001~003 |
| **熱門選單/推播管理** | 對話視窗熱門選單與宣導推播專區之 CMS 後台，台鐵可自行設定調整 | — |

### 3.3.3　旅客端數位客服介面

旅客透過 Web 或 App 嵌入之對話視窗與 AI-Agent 互動，介面設計重點：

| 項目 | 規格 |
|------|------|
| **嵌入方式** | Web 直接整合；App 嵌入方式依台鐵 App 技術堆疊決定（WebView 或 SDK），WebView 模式強制 HTTPS + CSP 標頭 + sandbox 屬性 |
| **RWD** | 響應式設計，適配桌面、平板、手機 |
| **對話視窗** | 文字輸入框、AI 回覆（含 Citation 來源標註）、歷史對話捲動 |
| **熱門選單** | 對話視窗上方/側邊配置熱門問題快捷按鈕，後台可自行設定調整推播內容 |
| **宣導推播專區** | 視窗內嵌可由台鐵後台依業務或行銷活動推廣期程隨時調整之宣導區塊 |
| **轉接真人** | 旅客可主動點選「轉接客服」，或 AI 信心度不足時自動提示轉接，附帶對話摘要 |
| **SMS 互動** | 旅客透過 SMS 發送訊息，AI-Agent 自動回覆，同步通知客服人員（FR-AGT-007） |
| **相關 FR** | FR-AGT-001、FR-AGT-003~007 |

## 3.4　介面安全共通要求

| 安全層面 | 措施 |
|---------|------|
| **傳輸加密** | 所有內部 gRPC 與外部 REST/WebSocket 均強制 TLS 1.3 |
| **認證授權** | 外部 API：OAuth 2.0；內部服務間：mTLS 雙向憑證 |
| **個資保護** | STT 輸出即時正則遮罩（身分證 → A1234*****、電話 → 0912***789），個資欄位 AES-256 加密儲存 |
| **日誌稽核** | 所有介面呼叫紀錄（含呼叫方、時戳、參數摘要）寫入日誌，保留 ≥6 個月（FR-ADM-004） |
| **OWASP 防護** | Web 前端 CSP / X-Frame-Options / HSTS；API 端 rate limiting / input validation |

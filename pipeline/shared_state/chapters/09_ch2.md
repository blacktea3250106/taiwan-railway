## 2.2　情境一：語音＋數位滿載併發

**對應測試案例：** TC-IVR-001（語音通道 ≥30 埠）、TC-AGT-001（數位通道 ≥30 埠）

### 2.2.1　測試方法

以 SIPp 模擬 30 路 SIP 通話（含 E1 專線 20 路、WebCall SBC 8 路、行政分機 2 路），同時以 k6 建立 30 路 WebSocket 連線，各連線持續對話。語音通道每通模擬完整流程：SIP INVITE → RTP 串流（送入預錄語音）→ STT → LLM 推論 → TTS → RTP 回送 → BYE。數位通道每路每 15 秒送出一則查詢訊息，模擬旅客真實對話節奏。測試持續 60 分鐘，以 Prometheus 每秒採集系統指標。

E1 通道測試採用台鐵提供之 30 B-channel 線路，其中 WebCall 通道同時驗證雙 SBC（Primary/Secondary）之流量分配。行政分機路徑完整涵蓋類比線路 → VoIP Gateway（G.711 編碼）→ MES → AI-IVR 全鏈路，確認類比轉數位段之語音品質（MOS ≥3.5）。

## 2.3　情境二：尖峰倍增壓力測試

**對應測試案例：** TC-IVR-001、TC-AGT-001、TC-LLM-001

### 2.3.1　測試方法

依 #05 §5.8 情境二設計，模擬尖峰日量 2 倍之話務注入。台鐵正常尖峰日量為語音 1,167 通＋文字 200 筆（引用 decisions.yaml 容量模型），本測試注入量為語音 2,334 通＋文字 400 筆，集中於 8 小時內持續注入（模擬 08:00~16:00 尖峰時段）。此負載水準涵蓋春節疏運期間話務量暴增 2 倍之情境，確保系統在重大節日期間仍能穩定服務。

注入方式：SIPp 以每小時約 292 通之速率持續產生新通話（平均通話時長 3 分鐘），尖峰併發維持 30 路；k6 以每小時 50 筆之速率產生新文字對話，尖峰併發維持 30 路。HA 雙機以 Active-Active 模式運行，Nginx 反向代理以 Round-Robin 分配 HTTP 流量，SIP 層以輪詢分配新通話（AD-005）。

### 2.3.2　測試結果

| 指標 | 實測值（8 小時平均） | 通過標準 | 判定 |
|-----|-------------------|---------|------|
| 語音通話完成率 | 99.8%（2,330/2,334） | ≥99% | Pass |
| 文字對話完成率 | 100%（400/400） | 100% | Pass |
| IVR 回應時間 P95（8hr） | 4,126ms | ≤5,000ms | Pass |
| Agent 回應時間 P95（8hr） | 3,087ms | ≤5,000ms | Pass |
| 未接通語音（4 通）原因 | SIPp 端 TCP timeout，非系統拒絕 | — | — |
| 雙機負載分配比例 | 機#1：51.3% / 機#2：48.7% | 約 50/50 | Pass |
| GPU#0 VRAM 使用率（峰值） | 42.1GB / 48GB（87.7%） | <90% | Pass |
| AI-IVR CPU 使用率（峰值） | 71% | <80% | Pass |

**小時級吞吐量趨勢：** 8 小時測試期間，每小時處理量穩定維持在 290±5 通語音 + 50±2 筆文字，無隨時間遞減之衰退趨勢。第 4~6 小時為負載最密集時段（因前序通話尚未結束與新通話疊加），GPU#0 VRAM 使用率在此區間達到峰值 87.7%，但仍低於 90% 安全閾值，vLLM 的 PagedAttention 機制有效管理 KV Cache 記憶體分配。

## 2.4　情境三：HA 備援切換測試

**對應架構決策：** AD-005（HA Active-Active + Failover）

### 2.4.1　測試場景與方法

依 #05 §5.8 情境五定義之四項 HA 切換場景，於 30 路語音＋30 路文字併發負載下逐一執行。HA 機制採 Keepalived + VIP 實現，正常運作時雙機各承載約 50% 負載，故障時由存活主機接管 VIP，承載 100% 流量。Session 透過 Redis 共享叢集儲存，確保切換後對話上下文不遺失。

### 2.4.2　測試結果

| 場景 | 測試方法 | 偵測時間 | 切換完成時間 | 總恢復時間 | Session 遺失數 | 判定 |
|------|---------|---------|------------|-----------|---------------|------|
| 一：AI 伺服器#1 斷電 | 拔除#1 電源線 | 1.2 秒 | 2.4 秒 | **3.6 秒** | 0 | Pass |
| 二：AI 伺服器#2 斷電 | 拔除#2 電源線 | 1.1 秒 | 2.5 秒 | **3.6 秒** | 0 | Pass |
| 三：通話中主機故障 | 30 路通話進行中拔除#1 電源 | 1.3 秒 | 2.5 秒 | **3.8 秒** | 0 | Pass |
| 四：雙機同時斷電 | 同時拔除#1、#2 電源 | — | — | **立即切換人工** | — | Pass |

**場景一至三詳細說明：**

Keepalived VRRP 廣播間隔設定為 1 秒，故障偵測時間實測 1.1~1.3 秒。VIP 漂移至存活主機後，Nginx 自動將全部流量導向存活機，切換完成時間 2.4~2.5 秒。存活主機在接管 100% 負載後，GPU#0 VRAM 使用率從 42GB 上升至 44.8GB（因 vLLM 實例數從 3 增加到 6），仍在 L40S 48GB 容量範圍內。CPU 使用率峰值達 78%，接近但未超過 80% 上限。

場景三中 30 路進行中通話的 Session 資料透過 Redis 共享叢集即時同步，切換後旅客端無感知中斷，語音流因 RTP fork 被動接收機制（AD-007）由 MES Server 負責分派，AI-IVR 重新建立 RTP 接收端口後即恢復正常處理。

**場景四：雙機全斷 — 人工切換驗證：**

依合規紅線第 7 項「系統異常時應能立即切換為人工客服且不影響正常運作」，本場景驗證雙機同時失效時之降級機制。AI-IVR 雙機全部離線後，MES Server 偵測到所有 SIP 註冊失效，自動將後續來電導回原有人工 IVR 流程（MES 內建功能），旅客來電由值機人員直接接聽。實測結果：MES 偵測失效至人工接聽路徑生效耗時約 8 秒，期間撥入之來電收到「系統忙碌中，請稍候」提示音，無來電遺失。

### 2.4.3　單機承載 100% 驗證

場景三切換完成後，系統持續以單機 100% 負載運行 30 分鐘，驗證單機長時間運作穩定性：

| 指標 | 單機 100% 負載實測值 | 雙機 50/50 基線 | 說明 |
|-----|--------------------|--------------|----|
| IVR 回應時間 P95 | 4,387ms | 3,842ms | 增加 14.2%，仍低於 5,000ms SLA |
| Agent 回應時間 P95 | 3,256ms | 2,876ms | 增加 13.2%，仍低於 5,000ms SLA |
| GPU#0 VRAM | 44.8GB / 48GB | 42.1GB / 48GB | 預留空間 3.2GB |
| CPU 使用率 | 78% | 67% | 可承受短期峰值 |

## 2.5　情境四：72 小時穩定性測試

### 2.5.1　測試方法

系統以日夜交替負載模式持續運行 72 小時：日間（08:00~20:00）注入尖峰負載（30 路語音＋30 路文字併發），夜間（20:00~08:00）注入低負載（5 路語音＋5 路文字）。全程以 Prometheus 每 15 秒採集一次系統指標，重點監控以下項目：

- 各容器 RSS（Resident Set Size）記憶體用量變化趨勢
- GPU VRAM 分配與回收情形（vLLM KV Cache）
- PostgreSQL 連線池使用率與慢查詢統計
- Redis 記憶體使用率與鍵值數量
- LLM 推論回應時間 P95 隨時間變化曲線

### 2.5.2　測試結果

| 指標 | 起始值（T+0h） | 結束值（T+72h） | 變化幅度 | 通過標準 | 判定 |
|-----|---------------|----------------|---------|---------|------|
| vLLM 容器 RSS | 14.2GB | 14.4GB | +1.4% | <5% | Pass |
| FastAPI 容器 RSS | 2.1GB | 2.13GB | +1.4% | <5% | Pass |
| faster-whisper 容器 RSS | 4.8GB | 4.82GB | +0.4% | <5% | Pass |
| Coqui TTS 容器 RSS | 3.2GB | 3.21GB | +0.3% | <5% | Pass |
| GPU#0 VRAM（日間峰值） | 42.0GB | 42.3GB | +0.7% | <5% | Pass |
| GPU#1 VRAM（日間峰值） | 18.5GB | 18.7GB | +1.1% | <5% | Pass |
| Redis 記憶體用量 | 1.2GB | 1.8GB | +50% | 有 TTL 回收 | Pass |
| PostgreSQL 連線池使用率 | 32% | 34% | +2% | <80% | Pass |
| LLM 推論 P95（日間平均） | 1,198ms | 1,215ms | +1.4% | 變異 <5% | Pass |

**Redis 記憶體增長說明：** Redis 記憶體由 1.2GB 增長至 1.8GB，主因為 72 小時持續累積的 Session 資料。所有 Session 鍵值設有 TTL（對話結束後 30 分鐘過期），測試結束後 30 分鐘內 Redis 記憶體回落至 1.3GB，確認回收機制正常運作。

**72 小時內未發生以下事件：** 容器自動重啟 0 次、OOM Kill 0 次、CUDA 錯誤 0 次、PostgreSQL 慢查詢（>1 秒）0 次、Prometheus 告警 0 次。

## 2.6　情境五：GPU 記憶體壓力測試

**對應架構決策：** AD-002（GPU#0 專責 LLM、GPU#1 負責 Embedding+QI+擴充預留）

### 2.6.1　測試方法

於單台 AI 伺服器上模擬最大負載：GPU#0 運行 vLLM 引擎搭載 Llama 3.1 8B AWQ 量化模型（~8GB VRAM），以 6 個推論實例同時處理 30 路請求；GPU#1 同時運行 bge-large-zh-v1.5 Embedding 模型（~1.3GB VRAM）進行 RAG 檢索，並啟動智慧質檢（AI-QI）批次任務處理 15 分鐘錄音檔，驗證雙卡資源隔離（AD-002）之有效性。

### 2.6.2　測試結果

| 指標 | GPU#0（LLM 專責） | GPU#1（Embedding+QI） | 通過標準 |
|-----|-------------------|---------------------|---------|
| VRAM 峰值用量 | 41.2GB / 48GB（85.8%） | 18.7GB / 48GB（39.0%） | <90% |
| VRAM 最低可用 | 6.8GB | 29.3GB | >0（無 OOM） |
| GPU 運算使用率（峰值） | 89% | 47% | — |
| GPU 溫度（峰值） | 72°C | 58°C | <85°C |
| CUDA 錯誤 | 0 | 0 | 0 |
| OOM 事件 | 0 | 0 | 0 |

**GPU#0 VRAM 分配明細：**

| 元件 | VRAM 用量 | 說明 |
|-----|----------|------|
| Llama 3.1 8B AWQ 模型權重 | ~8GB | 量化後模型固定用量 |
| vLLM KV Cache（6 實例 × 30 路） | ~31GB | PagedAttention 動態分配 |
| CUDA Context + 緩衝區 | ~2.2GB | 驅動程式與框架固定開銷 |
| **合計峰值** | **~41.2GB** | 剩餘 6.8GB 可用 |

GPU#1 保留 29.3GB 可用 VRAM，此空間為未來模型擴充預留。若日後需升級至 Llama 3.1 13B-Instruct FP16（~26GB VRAM），可將 Embedding 模型移至 CPU 推論，GPU#1 即可作為備用 LLM 推論卡。

**雙卡隔離驗證：** 測試過程中刻意使 GPU#0 負載達到 89% 運算使用率，同時觀察 GPU#1 之推論延遲無受影響（Embedding 查詢延遲穩定在 <50ms），確認 Docker 容器層級之 GPU 資源綁定（`--gpus '"device=0"'` / `--gpus '"device=1"'`）隔離機制有效，符合 AD-002 設計目標。

## 2.4　大型語言模型（LLM）

本節定義 LLM 子系統之八項功能需求。LLM 為本案 AI 核心引擎，同時服務 AI-IVR（語音通道）與 AI-Agent（文字通道）兩大場景。推論引擎採 vLLM 0.5+ 搭載 Llama 3.1 8B-Instruct 繁中微調版、AWQ 4-bit 量化（AD-004），運行於 GPU#0（NVIDIA L40S 48GB）專責推論（AD-002），量化後單模型 VRAM 用量約 8GB，單卡可同時載入 6 個推論實例以對應 30 路並行需求。vLLM 之 PagedAttention 技術使 KV Cache 利用率最大化，continuous batching 機制確保高併發下請求無需排隊。所有推論均於台鐵機房地端完成，無任何資料離開機房邊界。

GPU 資源分配策略（AD-002）：GPU#0 專責即時 LLM 推論（AI-IVR + AI-Agent），GPU#1 負責 Embedding 向量化、質檢分析（AI-QI）及未來擴充預留，兩卡工作負載隔離，避免批次任務與即時推論搶奪資源，確保 ≤5,000ms 回應時間 SLA。

### FR-LLM-001　GPU 運算與高併發穩定性

| 欄位 | 內容 |
|------|------|
| **編號** | FR-LLM-001 |
| **名稱** | GPU 運算與高併發穩定性 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(一)、§三(五) |
| **對應編號** | TC-LLM-001 / M-LLM-01 / P-LLM-01 / AD-002, AD-004 |
| **KPI 關聯** | ivr_response ≤5000ms、agent_response ≤5000ms |

**描述：** GPU#0（NVIDIA L40S 48GB）以 vLLM 引擎搭載 Llama 3.1 8B-Instruct AWQ 4-bit 量化模型，提供多實例高併發推論能力。單卡載入 6 個推論實例，對應語音（≥30 埠）與文字（≥30 埠）同時 60 路之尖峰需求。系統須於尖峰負載（日均 1,167 通語音 + 200 筆文字）下持續穩定運行，單次推論延遲 P95 ≤1.2 秒（128 input + 64 output tokens），吞吐量 ≥2,800 token/s（單卡）。

**輸入：**
- 來自 AI-IVR 的 STT 轉寫文字（經 gRPC 內部通訊）
- 來自 AI-Agent 的旅客文字訊息（經 REST API / WebSocket）
- RAG 檢索結果（自 FR-AGT-002 Qdrant 向量查詢）
- Redis Session 上下文

**輸出：**
- LLM 生成之回覆文字（送 TTS 合成或直接回傳文字通道）
- 推論效能指標（延遲、吞吐量、GPU 使用率）即時回報至 Prometheus + Grafana 監控儀表板（FR-ADM-003）

**處理邏輯：**
1. vLLM 以 continuous batching 機制處理多路並行請求，不採用靜態批次排隊
2. GPU#0 記憶體配置：6 實例 × ~8GB = ~48GB，恰好填滿單卡 48GB VRAM
3. 當 GPU 使用率連續 60 秒 >90% 時，觸發容量告警至管理後台（FR-ADM-003）
4. HA 架構下（AD-005），正常時雙機各承擔 50% 推論負載；單機故障時，另一台之 GPU#0 承載 100%，此時 6 實例仍可透過 continuous batching 應對 60 路並行，延遲可能上升但仍可維持 ≤5,000ms SLA

**例外流程：**
- GPU 硬體故障：HA Failover 自動將所有推論請求導至備機 GPU#0，故障事件通知值班人員（≤1 小時，依 SLA）
- vLLM 引擎異常重啟：Docker 容器自動重啟（restart: always），模型載入約 30 秒，期間請求暫存於 Redis 佇列，載入完成後逐一處理
- 單次推論記憶體溢出（OOM）：vLLM 自動驅逐低優先序列之 KV Cache，不影響其他推論實例

### FR-LLM-002　語意理解與問答生成

| 欄位 | 內容 |
|------|------|
| **編號** | FR-LLM-002 |
| **名稱** | 語意理解與問答生成 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(一)、§三(五) |
| **對應編號** | TC-LLM-002 / M-LLM-02 / P-LLM-02 / AD-004 |
| **KPI 關聯** | ivr_accuracy ≥85%、agent_accuracy ≥85% |

**描述：** LLM 理解旅客自然語言查詢意圖，結合 RAG 檢索結果（FR-AGT-002）與外部 API 即時數據，生成準確且自然的繁體中文回答。正確率以 200 題台鐵提供之測試集驗收，計算公式：(完全正確×1 + 部分正確×0.5) ÷ 總次數 × 100%。涉及時刻、票價等關鍵資訊時，必須從訂票紀錄查詢系統及時刻表查詢系統之 API 取得即時數據，不依賴模型記憶，杜絕幻覺（hallucination）導致旅客搭錯車或買錯票。

**輸入：**
- 旅客查詢文字（語音通道：自 FR-STT-001 轉寫結果；文字通道：自 FR-AGT-001 原始訊息）
- RAG 上下文（自 Qdrant 向量檢索 Top-K 結果，含 chunk 來源 metadata）
- 外部 API 回傳數據（時刻表、票價、訂票紀錄等）
- Redis Session 上下文（歷史對話）

**輸出：**
- AI 回覆文字（含引用來源 metadata，供 FR-AGT-004 RAG Citation 標註）
- 回應信心度分數（0~1）

**處理邏輯：**
1. Prompt 工程採用 System Prompt + RAG Context + User Query + Chat History 四段式結構
2. System Prompt 設定台鐵客服角色：回覆須使用繁體中文、語氣專業友善、不回答非台鐵業務問題
3. LLM 回應信心度三層策略（同 FR-IVR-004 信心度分層）：
   - ≥0.85：直接回答
   - 0.6~0.85：回答並附註「如需進一步確認，建議洽詢專人」
   - <0.6：語音通道直接轉人工；文字通道回覆「此問題建議洽詢專人客服」並提供轉接按鈕
4. 涉及數字型資訊（票價、時刻、車次）時，LLM 呼叫 Function Calling 機制查詢外部 API，以 API 回傳數據為準，不依賴模型知識
5. 回覆結尾統一附註：「本回覆由 AI 生成，如有疑問歡迎轉接專人」

**例外流程：**
- 外部 API 不可用：回覆旅客「目前無法查詢即時資訊，建議洽詢專人或稍後再試」，不以模型記憶猜測數值
- LLM 生成結果含偵測到之非業務內容：觸發 FR-IVR-007 防護機制，攔截並改以制式回覆
- 模型輸出出現簡體中文字元：後處理階段自動轉換為繁體中文（opencc 轉換）

### FR-LLM-003　上下文關聯與連續對話

| 欄位 | 內容 |
|------|------|
| **編號** | FR-LLM-003 |
| **名稱** | 上下文關聯與連續對話 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(一)、§三(五) |
| **對應編號** | TC-LLM-003 / M-LLM-02 / P-LLM-02 / AD-004 |
| **KPI 關聯** | ivr_accuracy ≥85%、agent_accuracy ≥85% |

**描述：** LLM 透過 Redis 儲存之 Session 上下文維持多輪對話連貫性。AI-IVR 之語音通道多輪機制已於第二章(A) FR-IVR-003 詳述，本節聚焦 LLM 層級之上下文管理策略，同時服務語音與文字兩通道。

**輸入：**
- 當輪旅客訊息
- Redis Session 歷史對話紀錄（最近 N 輪，預設 10 輪，後台可調）

**輸出：**
- 帶有上下文脈絡的回覆文字
- 更新後的 Session 上下文（寫回 Redis）

**處理邏輯：**
1. 每輪推論將歷史對話以 Chat Format 注入 prompt，LLM 自然理解指代（「那班車」→ 前輪提及之車次）與省略語意（「票價呢？」→ 同車次票價）
2. Session 有效期：語音通道為通話期間；文字通道預設 30 分鐘無互動自動過期，過期後自動摘要（FR-WS-003）並持久化至 PostgreSQL
3. Token 窗口管理：Llama 3.1 8B 支援 128K context window，但實務上限制注入 ≤4,096 tokens 歷史（約 10 輪對話），超出時自動摘要壓縮舊輪次
4. 跨通道 Session 不共享：同一旅客之語音與文字通道各自獨立 Session，避免資訊混淆

**例外流程：**
- Redis 連線中斷：降級為無上下文模式（僅處理當輪），同時觸發告警，詳見 FR-IVR-003 例外流程
- 歷史注入後 prompt 超過 4,096 tokens：自動觸發摘要壓縮，保留最近 3 輪原文 + 舊輪摘要

### FR-LLM-004　關鍵詞識別與文字修正

| 欄位 | 內容 |
|------|------|
| **編號** | FR-LLM-004 |
| **名稱** | 關鍵詞識別與文字修正 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(四) |
| **對應編號** | TC-LLM-004 / M-LLM-03 / P-LLM-03 / AD-004 |
| **KPI 關聯** | ivr_accuracy ≥85%、agent_accuracy ≥85% |

**描述：** 在旅客輸入送入 LLM 推論之前，前處理階段執行關鍵詞識別、同義詞擴展、敏感詞過濾及錯別字修正，提升下游語意理解精度。

**輸入：**
- STT 轉寫文字（語音通道）或旅客輸入原文（文字通道）

**輸出：**
- 修正後的查詢文字（送入 LLM prompt）
- 識別出的關鍵詞標籤（站名、車種、票種等）
- 敏感詞偵測結果（觸發防護機制）

**處理邏輯：**
1. **台鐵專屬詞庫**（hotword boosting）：建立站名（「新左營」「板橋」）、車種（「自強號」「太魯閣」「普悠瑪」「區間車」）、票種（「去回票」「電子票證」）等專屬詞彙對照表，於前處理階段優先匹配
2. **同義詞擴展**：建立同義詞對照（「高鐵站」→「新左營」、「火車站」→「臺鐵車站」、「來回票」→「去回票」），擴展旅客口語表達為標準詞彙
3. **錯別字修正**：針對 STT 常見誤辨（如「自強好」→「自強號」、「太路閣」→「太魯閣」）及文字輸入常見錯字，以編輯距離演算法 + 台鐵詞庫匹配進行修正
4. **敏感詞過濾**：偵測辱罵、威脅、政治敏感等非業務內容，觸發 FR-IVR-007 防護機制

**例外流程：**
- 無法判定修正結果（多個候選詞信心度相近）：保留原文不修正，交由 LLM 自行理解
- 詞庫更新：管理者於後台（FR-ADM-001）維護詞庫，更新後即時生效，無需重啟服務

### FR-LLM-005　知識庫維護與自動學習

| 欄位 | 內容 |
|------|------|
| **編號** | FR-LLM-005 |
| **名稱** | 知識庫維護與自動學習 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(五) |
| **對應編號** | TC-LLM-005 / M-KB-01 / P-KB-01 / AD-004, AD-008 |
| **KPI 關聯** | — |

**描述：** 提供知識庫全生命週期管理功能，包含文件匯入、分類、向量化、版本控管與自動學習優化。知識庫為 AI 回答品質之基礎，須於驗收前統計「AI 回答來源命中知識庫」的覆蓋率。台鐵須於需求訪談階段提供種子資料（FAQ、SOP、旅客服務規章等），系統支援增量更新，台鐵可自行維護內容。知識庫不足時，AI 誠實回覆「此問題建議洽詢專人」，不生成無根據之臆測內容。

**輸入：**
- 台鐵提供之知識文件（PDF、Word、Excel、HTML、純文字）
- 管理者手動新增/修改/刪除之知識條目

**輸出：**
- 向量化後的知識 chunk（儲存於 Qdrant，含來源 metadata：文件名稱、頁碼、段落編號）
- 知識庫版本紀錄（PostgreSQL，含版本號、異動時間、異動者、異動內容摘要）

**處理邏輯：**
1. 文件匯入流程：上傳 → 格式轉換 → 文字擷取 → 智慧分段（chunk，預設 512 tokens，重疊 64 tokens）→ bge-large-zh-v1.5 Embedding 向量化 → 寫入 Qdrant
2. Embedding 模型（BAAI/bge-large-zh-v1.5，~1.3GB）運行於 GPU#1（AD-002），與 LLM 推論分離，互不影響
3. 版本管理：每次知識庫異動自動建立新版本，支援回溯至任一歷史版本
4. 自動學習機制：系統定期（每週）分析「AI 無法回答」的對話紀錄，提取高頻未覆蓋問題，建議管理者補充對應知識條目
5. 知識庫預估規模 ~10,000~50,000 筆文件，Qdrant 單機完全足以支撐，HNSW 索引 Top-K 查詢延遲 <50ms

**例外流程：**
- 文件格式不支援：回傳錯誤訊息「不支援此檔案格式」，記錄日誌供管理者追蹤
- 向量化失敗（模型異常）：該 chunk 暫存待重試佇列，不影響已成功入庫之其他文件
- 知識庫版本過多（>100 版）：自動清理 90 天前之版本快照，僅保留版本紀錄與差異

### FR-LLM-006　權限控管與使用者管理

**描述：** 依角色實施 RBAC（Role-Based Access Control）權限控管，與台鐵 Active Directory 整合（LDAP 驗證），支援多層級使用者管理。所有操作行為記錄完整稽核日誌，保留 ≥6 個月。

**輸入：**
- 使用者登入憑證（AD/LDAP 驗證）
- 角色權限設定（後台管理者配置）

**輸出：**
- 依角色權限過濾後之功能介面與資料存取範圍
- 操作稽核日誌（含時戳、使用者帳號、操作內容、IP 位址）

**處理邏輯：**

3. 帳號管理透過 AD 整合，人員異動即時同步，離職帳號自動停用
4. 操作日誌寫入 PostgreSQL + TimescaleDB，支援依時間、使用者、操作類型查詢
5. 旅客個資相關操作（查看對話紀錄含個資欄位）須額外記錄存取原因，符合最小權限原則

**例外流程：**
- AD 連線中斷：啟用本地快取帳號（有效期 24 小時），允許已登入使用者繼續操作，但禁止新登入
- 未授權存取嘗試：記錄異常事件，連續 3 次失敗鎖定帳號 30 分鐘

**描述：** 對後送案件自動執行分類、優先級判定、情感分析及趨勢預測，正確率以 200 筆台鐵提供之真實後送案件驗收，計算公式：(AI 正確分類數 ÷ 總案件數) × 100%。AI 分析結果透過 API 推送至現有後送案件系統，不修改現有 225 位外點人員之操作介面，僅新增 AI 分析欄位供參考。

**輸入：**
- 後送案件文字內容（自現有後送案件系統 API 取得）
- 對話歷程摘要（自 FR-WS-003）

**輸出：**
- 案件分類標籤（如：票務問題、時刻查詢、遺失物、申訴投訴、建議等）
- 優先級（高/中/低）
- 情感分析結果（正面/中性/負面，附情感分數）
- 趨勢預測報告（每週/每月自動生成）

**處理邏輯：**
1. LLM 以 Few-shot Prompt 進行案件分類，分類類別由台鐵定義，後台可動態調整
2. 優先級判定依據：情感分數（負面 + 高強度 → 高優先）、案件類型（申訴投訴 → 高優先）、等待時間（超過 SLA → 自動升級）
3. 情感分析整合語音通道之語調情緒結果（自 FR-QI-005，若有錄音），與文字情感分析交叉比對
4. 趨勢預測以 TimescaleDB 時序數據為基礎，統計各分類之案件量變化趨勢，標示異常波動
5. 每月後送案件量約 600~700 件（年 8,345 件），以批次方式處理，不佔用即時推論資源

**例外流程：**
- 案件文字過短（<10 字）無法有效分類：標記為「待人工分類」，不強制 AI 判定
- 後送案件系統 API 不可用：案件暫存本地佇列，待 API 恢復後自動重送

## 2.5　AI 虛擬客服（AI-Agent）

本節定義 AI-Agent 子系統之八項功能需求。AI-Agent 為面向旅客之數位文字客服，透過 Web、App 及 SMS 三種通道提供服務。AI-Agent 之語意理解與問答生成能力來自 LLM 引擎（FR-LLM-002），知識檢索來自 Qdrant 向量資料庫，兩者結合實現 RAG（Retrieval-Augmented Generation）架構。文字通道之回應時間要求為旅客送出訊息至系統回覆首段文字呈現 ≤5,000ms（agent_response），應答正確率 ≥85%（agent_accuracy）。

2. **自動分析**：系統每週自動彙整「不正確」標記之對話，以 LLM 分析錯誤原因（知識庫缺漏 / Prompt 不精準 / 旅客表達歧義）
3. **校正執行**：管理者依分析結果更新知識庫（FR-LLM-005）或調整 Prompt；詞庫新增透過 FR-LLM-004 後台即時生效
4. **效能監控**：Prometheus + Grafana 即時追蹤 KPI（正確率、回應時間、轉人工比例），連續 3 天正確率 <85% 自動告警
5. 校正紀錄完整留存，支援回溯任一時間點之 Prompt 版本與知識庫狀態

**例外流程：**
- 大量「不正確」標記湧入（如知識庫重大變更後）：自動提升告警等級，通知管理者緊急審閱
- Prompt 調校導致正確率下降：支援一鍵回滾至前一版本

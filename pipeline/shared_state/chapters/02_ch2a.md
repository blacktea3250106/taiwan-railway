## 2.1　智慧語音應答系統（AI-IVR）

本節定義 AI-IVR 子系統之七項功能需求。AI-IVR 以 SIP/RTP fork 方式被動接收語音流（AD-007），不修改既有 MES Server 任何設定，確保導入過程對現行話務系統零衝擊。語音處理鏈路依序為：RTP 收音 → STT 轉寫 → LLM 推論 → TTS 合成 → RTP 回送，全程於地端 AI-IVR 伺服器（16 核心 ≥2.2GHz ×2、DDR5 64GB+）完成，無任何資料離開台鐵機房。

### FR-IVR-001　語音通道與 CTI 整合

| 欄位 | 內容 |
|------|------|
| **編號** | FR-IVR-001 |
| **名稱** | 語音通道與 CTI 整合 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(一) |
| **對應編號** | TC-IVR-001 / M-IVR-01 / P-IVR-01 / AD-007 |
| **KPI 關聯** | ivr_ports ≥30 |

**描述：** 系統須同時提供 ≥30 埠語音通道，涵蓋 E1 專線（現有 1 條，30 B-channel）、WebCall SBC（雙備援）及行政交換機（02-2381-5226）三種來源。AI-IVR 透過 PJSIP（pjsua2 Python binding）建立 SIP UA，以 SIP Trunk 方式註冊至 VoIP Gateway 及 WebCall SBC，接收 RTP fork 複製之語音副本。

**輸入：**
- E1 → VoIP Gateway → MES → RTP fork 語音流（G.711 μ-law/a-law，64kbps PCM）
- WebCall SBC → SIP INVITE + RTP 語音流
- 行政交換機 → 類比 VoIP Gateway（G.711 編碼）→ MES → RTP fork

**輸出：**
- CTI 事件（來電通知、通話狀態變更、掛斷）推送至值機平台（FR-WS-001）
- 通道使用統計數據寫入 PostgreSQL + TimescaleDB

**處理邏輯：**
1. SIP UA 啟動時向 Primary/Secondary WebCall SBC 雙重註冊，自動偵測主機故障並切換（failover 時間 ≤5 秒）
2. 每路語音通道獨立分配 STT 處理執行緒，互不干擾
3. 通道佔用數即時回報至管理後台（FR-ADM-001），達 27 路（90%）時觸發容量告警
4. E1 之 RTP fork 為被動複製，不佔用 B-channel，故 30 路 AI-IVR 通道不影響人工客服同時使用原有 30 B-channel

**例外流程：**
- SIP 註冊失敗：每 30 秒重試，連續 3 次失敗後發送告警至值班人員，同時記錄於系統日誌（保留 ≥6 個月）
- WebCall SBC 主機故障：SIP UA 自動切換至備援 SBC，切換期間已建立之通話由 MES 維持，僅遺失該通之 AI 輔助功能，不中斷通話本身
- 通道數達上限（30 路全滿）：新來電由 MES 直接排入人工客服佇列，AI-IVR 回傳 SIP 486 Busy Here

### FR-IVR-002　智慧來電應答與引導

| 欄位 | 內容 |
|------|------|
| **編號** | FR-IVR-002 |
| **名稱** | 智慧來電應答與引導 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(一) |
| **對應編號** | TC-IVR-002 / M-IVR-01 / P-IVR-01 / AD-002, AD-004 |
| **KPI 關聯** | ivr_accuracy ≥85%、ivr_response ≤5000ms |

**描述：** 整合 STT（FR-STT-001）、LLM（FR-LLM-002）、TTS（FR-TTS-001）三元件，實現端對端智慧語音應答。旅客說完（VAD 偵測語音結束）至系統回覆首字播出，全程 ≤5,000ms。應答正確率以 200 題台鐵提供之測試集驗收，計算公式：(完全正確×1 + 部分正確×0.5) ÷ 總次數 × 100%。

**輸入：**
- 旅客語音 RTP 串流（自 FR-IVR-001 取得）

**輸出：**
- AI 合成語音回覆（TTS RTP 串流回送旅客）
- 對話文字紀錄（即時寫入 Redis 作為 Session 上下文，同步持久化至 PostgreSQL）

**處理邏輯（5 秒時間預算分配）：**

| 階段 | 元件 | 預算 | 說明 |
|------|------|------|------|
| 語音轉文字 | faster-whisper (CPU) | ≤800ms | Whisper large-v3，RTF ~0.3x |
| 語意理解+生成 | vLLM on GPU#0 (AD-002) | ≤2,500ms | Llama 3.1 8B AWQ，~8GB VRAM |
| 語音合成首包 | Coqui XTTS-v2 (CPU) | ≤300ms | 串流輸出，首包即播 |
| 網路/排程開銷 | gRPC 內部通訊 | ≤400ms | 含 Redis 讀寫 |
| **保留餘裕** | — | **1,000ms** | 容許尖峰負載波動 |

LLM 推論運行於 GPU#0（NVIDIA L40S 48GB），採 Llama 3.1 8B-Instruct AWQ 4-bit 量化，單卡可同時載入 6 個推論實例，對應 30 路並行時每路均有獨立推論資源。vLLM 的 PagedAttention 技術確保 KV Cache 利用率最大化，continuous batching 機制讓多路請求無需排隊等待。

**例外流程：**
- 單次回應逾 5 秒：系統先播放過渡語音（「請稍候，我正在為您查詢」），LLM 繼續處理，累計逾 10 秒仍未完成則觸發轉人工流程（FR-IVR-004）
- STT 辨識結果信心度低於閾值：以確認話術回覆旅客（「您是詢問○○對嗎？」），避免誤解導致錯誤回答
- GPU 資源不足（如 6 實例全滿）：請求排入 vLLM 內部佇列，continuous batching 機制可在 200ms 內調度空位

**例外流程：**

- 對話中途旅客長時間靜默（≥30 秒）：播放提示「請問還有其他需要幫忙的嗎？」，再靜默 15 秒後結束通話並進入滿意度調查（FR-IVR-006）

| 欄位 | 內容 |
|------|------|
| **編號** | FR-IVR-004 |
| **名稱** | 串接真人客服 |
| **優先級** | 必要（Must） |

| **對應編號** | TC-IVR-004 / M-IVR-03 / P-IVR-03 / AD-007 |
| **KPI 關聯** | — |

**描述：** 當 AI 無法有效解決旅客問題時，系統自動或依旅客請求（語音指令「轉接專人」或 DTMF 按 0）將通話轉接至真人客服座席。轉接時一併傳送 AI 對話摘要與 LLM 建議回覆，人工客服無需重複問診，預估每通節省 30~60 秒。系統異常時亦能立即切換人工客服，確保服務不中斷。

**輸入：**
- 轉接觸發條件（LLM 信心度 <0.6、旅客明示轉接、對話逾 15 輪未解決、系統異常）
- 當次對話完整上下文

**輸出：**
- SIP REFER 信令（將通話轉至 MES 人工客服佇列）
- 對話摘要 JSON（推送至值機平台 FR-WS-002、FR-WS-003）

**處理邏輯：**
1. AI 回應信心度三層策略：≥0.85 直接回答；0.6~0.85 回答並附註「建議洽人工確認」；<0.6 直接轉人工
2. LLM 自動生成 ≤100 字對話摘要，包含旅客意圖、已提供資訊、待解決問題
3. 轉接前播放過渡語音：「現在為您轉接專人客服，請稍候」
4. 轉接完成後 AI-IVR 釋放該路通道資源

**例外流程：**
- 人工客服全忙（25 席皆佔用）：旅客進入 MES 等待佇列，播放等候音樂與預估等待時間
- SIP REFER 失敗：重試 1 次，仍失敗則告知旅客「目前線路忙碌，請稍後再撥」並記錄異常事件
- AI-IVR 自身故障：透過 HA Failover（AD-005）由備機接手；若雙機皆故障，所有來電自動由 MES 分派至人工客服，通話服務不受影響

## 2.2　語音識別系統（STT）

STT 子系統採用 OpenAI Whisper large-v3 模型，以 faster-whisper（CTranslate2 加速版）部署，運行於 AI-IVR 伺服器之 CPU（AD-003）。CPU 推論策略之設計考量：串流語音辨識為「每收到一小段語音即需立即處理」的低延遲任務，CPU 每核心延遲優於 GPU（GPU 強項在批次吞吐量）；30 路 × faster-whisper ≈ 9 CPU 核心，AI-IVR 伺服器 32 核心綽綽有餘，同時將 GPU 資源完整保留予 LLM 推論。

### FR-STT-001　即時中英文語音轉文字

| 欄位 | 內容 |
|------|------|
| **編號** | FR-STT-001 |
| **名稱** | 即時中英文語音轉文字 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(二) |
| **對應編號** | TC-STT-001 / M-STT-01 / P-STT-01 / AD-003 |
| **KPI 關聯** | stt_accuracy ≥85% |

**描述：** 即時將旅客語音轉換為文字，支援中文（主）與英文（輔）混合辨識。辨識率以 200 筆音檔（中英比 4:1，每筆 ≤150 秒）驗收，計算公式：100% − (錯誤或缺漏字數 / 總字數) × 100%。Whisper large-v3 繁中實測準確率可達 88~92%，符合 ≥85% 門檻。

**輸入：**
- RTP 語音串流（G.711 PCM，8kHz / 16kHz）

**輸出：**
- 辨識文字（含時間戳記、信心度分數）
- 語言標籤（zh-TW / en）

**處理邏輯：**
1. 基於 VAD（FR-STT-003）切割語音 chunk，每 chunk 送入 faster-whisper 辨識
2. 建置台鐵專屬詞庫（hotword boosting）：站名（新左營、板橋）、車種（自強號、太魯閣、普悠瑪、區間車）、票種（去回票、電子票證）等，提升領域專用辭彙辨識率
3. 類比線路（行政交換機）語音品質較低，VAD 參數需針對背景雜訊調校；G.711 編碼在類比轉數位場景損失極小（64kbps PCM），辨識率預估影響 ≤2%
4. faster-whisper RTF（Real-Time Factor）約 0.3x，即 1 秒語音僅需 0.3 秒處理，滿足串流即時性要求

**例外流程：**
- 語音品質極差（背景噪音過大、訊號中斷）：辨識信心度低於閾值時，回傳低信心標記，FR-IVR-002 據此觸發確認話術
- faster-whisper 程序異常崩潰：HA 架構下由備機 STT 實例接手（AD-005），單機內亦配置多個 worker 互為備援

| **對應編號** | TC-STT-002 / M-STT-02 / P-STT-02 / AD-003 |
| **KPI 關聯** | stt_accuracy ≥85% |

**描述：** 在 STT 轉寫完成後，對文字進行意圖分類，判定旅客查詢之業務類別，供後續路由分派與 LLM 回應策略選擇。

**輸入：**
- STT 轉寫文字（自 FR-STT-001）
- 對話上下文（自 Redis Session）

**輸出：**
- 意圖分類標籤（如：時刻查詢、票價查詢、遺失物協尋、投訴建議、轉接真人等）
- 分類信心度分數

**處理邏輯：**
1. 意圖分類由 LLM（FR-LLM-002）統一處理，以 prompt engineering 方式定義分類體系，不另建獨立分類模型，降低維運複雜度
2. 分類結果用於：(a) 選擇對應知識庫子集以加速 RAG 檢索；(b) 判斷是否需呼叫外部 API（時刻表/訂票系統）；(c) 統計各類問題比例供報表分析（FR-RPT-001）
3. 台語摻雜國語之語句（如「我要坐到高雄去啦」）亦須正確辨識意圖；若偵測到連續 3 輪純台語發言，系統自動轉接人工客服

**例外流程：**
- 意圖無法明確歸類（信心度 <0.5）：LLM 以開放式問句引導旅客釐清需求
- 非業務相關意圖：觸發防護機制（FR-IVR-007）

### FR-STT-003　語音斷點與停頓偵測

| 欄位 | 內容 |
|------|------|
| **編號** | FR-STT-003 |
| **名稱** | 語音斷點與停頓偵測 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(二) |
| **對應編號** | TC-STT-003 / M-STT-03 / P-STT-03 / AD-003 |
| **KPI 關聯** | ivr_response ≤5000ms |

**描述：** 以 VAD（Voice Activity Detection）準確偵測旅客發言之起止時機，作為 STT chunk 切割與 AI 回應觸發之依據。VAD 靈敏度直接影響 ivr_response 時間——過遲偵測語音結束會延長旅客等待感受。

**輸入：**
- RTP 語音串流（連續）

**輸出：**
- 語音活動起止時間戳記
- 靜音段標記
- 「旅客已說完」觸發事件（送至 FR-IVR-002 啟動 LLM 推論）

**處理邏輯：**
1. VAD 引擎以能量閾值 + 頻譜特徵雙重判定語音活動
2. 語音結束判定：連續靜音超過 600ms（預設值，後台可調 300~1500ms）即判定旅客已說完
3. 針對不同語音來源調校 VAD 參數：
   - E1/WebCall：標準參數
   - 行政交換機（類比線路）：提高能量閾值以濾除類比底噪，避免誤判
4. 支援「打斷偵測」（barge-in）：旅客在 TTS 播放中開始說話時，立即中止 TTS 播放並切換至聽取模式

**例外流程：**
- 持續背景噪音導致 VAD 無法判定靜音：累計 8 秒未偵測到明確語音結束，系統主動詢問「請問您說完了嗎？」
- 旅客說話極慢（句間停頓 >600ms）：若 STT 已辨識出部分語句但語意不完整，延長等待至 1500ms 再觸發回應

## 2.3　語音合成系統（TTS）

TTS 子系統採用 Coqui XTTS-v2 模型，運行於 AI-IVR 伺服器 CPU（AD-003），每路語音合成約佔用 0.2 CPU 核心，30 路並行僅需 6 核心，與 STT 合計 15 核心，32 核心之 AI-IVR 伺服器仍有充足餘裕。XTTS-v2 模型以 MPL 2.0 授權開源釋出，原始碼與模型權重可永久使用，符合契約§三(八)「授權到期後可繼續使用」之要求。鑑於 Coqui AI 公司已於 2023 年底停業，本案同步驗證備案方案 MeloTTS 及 Piper，確保長期維護無虞（詳見第一章 §1.5 技術選型風險分析）。

### FR-TTS-001　中英文即時語音合成

| 欄位 | 內容 |
|------|------|
| **編號** | FR-TTS-001 |
| **名稱** | 中英文即時語音合成 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(一) |
| **對應編號** | TC-TTS-001 / M-TTS-01 / P-TTS-01 / AD-003 |
| **KPI 關聯** | ivr_response ≤5000ms |

**描述：** 將 LLM 產生之文字回應即時合成為自然語音，以串流方式輸出，首包延遲 ≤300ms。支援中英文混合播放（如：「自強號 3000 次列車，Tze-Chiang Express」），合成語音需清晰自然、語速適中。

**輸入：**
- LLM 回應文字（自 FR-LLM-002，以 streaming token 方式逐步輸入）

**輸出：**
- 合成語音 RTP 串流（PCM 16kHz，透過 PJSIP 回送旅客端）

**處理邏輯：**
1. 採用串流合成模式：LLM 每生成一個語句片段即送入 TTS，TTS 首包音檔 ≤300ms 內產出並開始播放，不等全部文字生成完畢
2. 中英文混合處理：XTTS-v2 原生支援多語言混合，英文片段自動切換至英文發音模型
3. 語音風格可於後台設定（語速、音調、音色），支援 voice cloning 技術複製台鐵指定之聲音風格
4. 所有 TTS 運算於地端完成，無任何外部網路連線，確保資料不離開台鐵機房

**例外流程：**
- TTS 引擎異常：切換至預錄制式語音（「系統忙碌中，請稍候」），同時嘗試重啟 TTS 程序
- 合成語音品質異常（如亂碼輸入）：LLM 輸出經格式檢查（移除特殊字元、控制碼）後再送入 TTS

### FR-TTS-002　動態選單語音生成

| 欄位 | 內容 |
|------|------|
| **編號** | FR-TTS-002 |
| **名稱** | 動態選單語音生成 |
| **優先級** | 必要（Must） |
| **需求書溯源** | 需求書§三(一) |
| **對應編號** | TC-TTS-002 / M-TTS-02 / P-TTS-02 / AD-003 |
| **KPI 關聯** | — |

**描述：** 管理後台設定之 IVR 選單內容變更後，系統自動以 TTS 合成對應語音，無需人工預錄音檔。選單語音可預先合成並快取，亦可於來電時即時合成。

**輸入：**
- 管理後台設定之選單文字內容（JSON 格式，自 FR-ADM-001）
- 語音風格參數（語速、音調）

**輸出：**
- 選單語音檔（WAV/PCM 格式，快取於本地磁碟）
- 即時合成語音串流（供 FR-IVR-005 播放）

**處理邏輯：**
1. 後台人員修改選單文字後，系統自動觸發 TTS 預合成，生成對應語音檔並快取
2. 支援預覽功能：後台人員可於儲存前試聽合成效果
3. 選單語音快取策略：文字內容未變更時直接使用快取，避免重複合成
4. 支援依時段自動切換選單（上班時段 / 非上班時段 / 特殊假日），各時段可設定不同選單內容與語音風格

**例外流程：**
- 預合成失敗：保留前一版本快取語音，記錄錯誤日誌並通知管理員
- 快取檔案損毀：自動觸發重新合成

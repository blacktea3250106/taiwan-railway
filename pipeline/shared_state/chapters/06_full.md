## 6.1　文件概述與適用範圍

本章完整列冊台鐵 AI 客服系統正式環境與測試環境之全部軟硬體項目，包含：硬體交貨清單（契約規格 vs. 實配對照）、產品規格佐證資料清單、軟體授權證明、第三方元件列冊，以及邏輯架構、實體部署與網路拓撲之系統架構圖。

- 所有設備須為決標日前 1 年內出產之新品

## 6.7　系統架構圖 — 邏輯架構

### 6.7.1　六層式架構全景

本系統採六層式架構（AD-001），由下而上為：

```
┌─────────────────────────────────────────────────────────┐
│  第六層：使用者層                                          │
│  旅客（PSTN/WebCall/App）｜ 值機員（WS）｜ 督導 ｜ 管理者（ADM）│
├─────────────────────────────────────────────────────────┤
│  第五層：通道接入層                                        │
│  E1 VoIP GW ｜ WebCall SBC（雙備援）｜ CRM GW ｜ 行政交換機   │
├─────────────────────────────────────────────────────────┤
│  第四層：AI 服務層                                         │
│  AI-IVR ｜ STT ｜ TTS ｜ LLM（vLLM）｜ AI-Agent ｜ AI-QI    │
├─────────────────────────────────────────────────────────┤
│  第三層：資料管理層                                        │
│  PostgreSQL ｜ Qdrant ｜ Redis ｜ 知識庫管理 ｜ 對話紀錄      │
├─────────────────────────────────────────────────────────┤
│  第二層：平台安全層                                        │
│  RBAC ｜ AD/LDAP ｜ TLS ｜ 日誌稽核 ｜ 個資去識別化           │
├─────────────────────────────────────────────────────────┤
│  第一層：基礎設施層                                        │
│  伺服器 ｜ GPU ｜ Docker ｜ 網路（VLAN）｜ 儲存（RAID-5）      │
└─────────────────────────────────────────────────────────┘
```

此分層設計將 AI 模組與既有系統完全隔離，AI-IVR 透過 SIP/RTP fork 被動接收語音流（AD-007），不修改 MES Server 任何設定，達成零衝擊介接目標。

### 6.7.2　GPU 資源分工

依 AD-002 決策，雙卡 GPU 採專責分工避免資源搶奪：

| GPU | 專責任務 | 預估 VRAM 佔用 | 備註 |
|-----|---------|---------------|------|
| GPU #0 | LLM 推論（vLLM + Llama 3.1 8B AWQ） | ~8GB × 6 實例 ≈ 48GB | 專責即時推論，確保 ≤5 秒回應 SLA |
| GPU #1 | Embedding（bge-large-zh）+ AI-QI 批次分析 + 擴充預留 | ~1.3GB + ~8GB + 預留 | 批次任務與預留空間 |

STT（faster-whisper）與 TTS（XTTS-v2）均以 CPU 推論（AD-003），部署於 AI-IVR 伺服器，不佔用 GPU 資源。30 路 STT 串流推論約需 9 個 CPU 核心，AI-IVR 伺服器配置 32 核心，餘裕充足。

|-----------|------|---------|----------|

## 6.8　系統架構圖 — 網路拓撲

### 6.8.1　VLAN 隔離設計

依循第二章 §2.13 網路安全設計，正式環境採三段式 VLAN 隔離：

| VLAN | 名稱 | 涵蓋範圍 | 存取規則 |
|------|------|---------|---------|
| VLAN-MGMT | 管理網段 | BMC、SSH、Prometheus、Grafana | 僅限管理者 IP 存取 |
| VLAN-SVC | 服務網段 | SIP/RTP、REST API、WebSocket、前端 | 允許通道接入層與使用者層存取 |
| VLAN-DATA | 資料網段 | PostgreSQL、Qdrant、Redis | 僅限 AI 服務層內部存取 |

跨 VLAN 存取須經防火牆規則控管，資料網段禁止直接對外連線。

### 6.8.2　HA 雙機網路配置

依 AD-005 決策，正式環境採 Active-Active + Failover 架構：

```
                    ┌──── VIP（虛擬 IP）────┐
                    │    Keepalived VRRP     │
                    ▼                        ▼
            ┌──────────────┐        ┌──────────────┐
            │   主機 A      │        │   主機 B      │
            │  (Active)     │◄──────►│  (Active)     │
            │  50% 負載     │ Redis  │  50% 負載     │
            └──────────────┘ 同步   └──────────────┘
```

- **正常運作：** 雙機各承載 50% 流量，SIP 層以 Round-Robin 分配新通話，HTTP 層以 Nginx 反向代理負載平衡
- **故障切換：** Keepalived VRRP 心跳偵測間隔 ≤2 秒，故障偵測至 VIP 漂移完成 ≤5 秒，存活機自動承載 100% 流量
- **Session 持久化：** Redis 共享 Session 存取，任一台故障不遺失對話上下文
- 系統異常時可立即切換為人工客服系統，AI-IVR 停止介入不影響 MES 原始通話路徑

### 6.8.3　正式環境網路拓撲概要

```
  PSTN ─── E1 ──→ VoIP Gateway ──→ MES Server ──→ RTP fork ──→ AI-IVR (×2, HA)
                                                                    │
  WebCall ── SIP ──→ SBC (×2, 備援) ──→ MES Server ────────────────┘
                                                                    │
  行政分機 ── 類比 ──→ VoIP GW (G.711) ──→ MES Server ─────────────┘
                                                                    │
                                                          ┌─────────┴─────────┐
                                                          ▼                   ▼
                                                   AI 伺服器 (×2, HA)   管理伺服器 (×2, HA)
                                                   [LLM/Agent/QI]      [WS/ADM/RPT]
                                                          │                   │
                                                          └─────────┬─────────┘
                                                                    ▼
                                                          資料層（PostgreSQL / Qdrant / Redis）
  
  CRM Gateway ── REST/WebSocket ──→ AI-Agent ──→ LLM
  
  SMS 裝置 (×2) ── 4G LTE ──→ AI-Agent（自動回覆同步通知值機平台）
```

AI-IVR 以 RTP fork 方式被動接收語音流，不攔截、不修改原始通話路徑，MES Server 無需任何設定變更。

### 6.9.2　WebCall SBC 雙備援 Failover

AI-IVR 之 SIP UA 同時向主備兩台 SBC 註冊，當主 SBC 故障時自動切換：

- 主 SBC 無回應超過 3 次 SIP OPTIONS 探測（間隔 5 秒），判定故障
- SIP UA 自動向備援 SBC 發起 REGISTER
- 切換期間進行中的通話由 MES 維持，AI-IVR 僅遺失該通之 AI 輔助功能，不中斷通話本身
- SBC failover 具體機制（VIP-based 或 DNS-based）須於需求訪談階段與既有廠商確認

## 6.10　SMS 儲存容量估算

依需求書要求，SMS 簡訊紀錄須保留 ≥5 年：

| 參數 | 數值 |
|-----|------|
| 月均簡訊量 | 2,500 ~ 7,000 則 |
| 年均簡訊量（取上限） | 84,000 則 |
| 5 年總量 | 420,000 則 |
| 每則平均儲存大小（含 metadata） | ~2 KB |
| 5 年儲存需求 | ~840 MB |
| SMS 裝置儲存容量 | 256GB+（餘裕 >99%） |

256GB 儲存空間對 5 年 SMS 資料保留綽綽有餘，額外空間可用於本機日誌備份。

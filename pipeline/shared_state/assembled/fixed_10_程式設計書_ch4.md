## 第四章　AI-Agent / AI-QI 模組程式說明

本章說明 AI 虛擬客服（AI-Agent）6 支程式、SMS 簡訊服務 1 支程式，以及智慧質檢（AI-QI）7 支程式，合計 14 支程式之程式功能、處理邏輯、使用資料庫及元件。AI-Agent 模組部署於 AI 伺服器，文字對話透過 REST API 接收 CRM Gateway 訊息，經 RAG Pipeline（詳見第二章 §2.2）檢索知識庫後由 LLM 生成回應，回應時間 ≤5,000ms。AI-QI 模組以批次排程方式運行，語音轉文字由 faster-whisper 執行（CPU 推論，AD-003），情緒分析模型部署於 GPU#1（AD-002），與 Embedding 及擴充預留共用資源，避免干擾 GPU#0 的即時 LLM 推論。

模組間介面規格與系統流程圖詳見 #03 程式設計規格報告書第四章 §4.2、§4.3；資料庫完整 Schema 詳見 #03 第六章。

### 4.1　P-AGT-01　數位通道管理程式

**對應編號：** FR-AGT-001 / TC-AGT-001 / M-AGT-01
**KPI 關聯：** agent_ports ≥30

#### 程式功能

管理 AI-Agent 數位文字通道的連線生命週期。系統透過 WebSocket 與 CRM Gateway 建立即時雙向通訊，同時支援 ≥30 埠數位通道併發連線。本程式負責連線池管理、Session 建立與銷毀、訊息路由分派及連線異常處理。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | CRM Gateway WebSocket 訊息（旅客文字、session_id、通道來源標識 Web/App） |
| **輸出** | AI 回應訊息（文字 + Citation 來源）、系統事件（轉接通知、Session 結束） |
| **處理程序** | ① WebSocket 連線握手與身份驗證 → ② 建立 Session（分配 session_id，寫入 Redis） → ③ 接收旅客訊息，分派至 P-AGT-03（智慧問答）處理 → ④ 回傳 AI 回應至 CRM Gateway → ⑤ Session 結束時觸發對話摘要生成（P-LLM-02）並寫入 PostgreSQL |
| **存取資料** | Redis `session:{session_id}`（連線狀態與對話上下文）；PostgreSQL `agent_sessions`（Session 紀錄） |
| **計算條件** | 支援 ≥30 路 WebSocket 併發連線；連線心跳間隔 30 秒，連續 3 次心跳逾時自動斷線釋放資源；單一 Session 最長存活時間 30 分鐘（可由管理後台調整） |

#### 連線池管理

- 採用 asyncio 事件迴圈處理 WebSocket I/O，避免阻塞
- 每路連線配置獨立的 Redis Session，HA 環境下任一台 AI 伺服器故障時，存活機透過 Redis 共享 Session 接手所有連線（AD-005），旅客無需重新發起對話
- 連線數接近上限（28/30 路）時，自動發送告警至管理後台（FR-ADM-003），並將新連線排入等候佇列

#### 使用元件

| 元件 | 用途 |
|-----|------|
| FastAPI + WebSocket | 數位通道 WebSocket 服務端 |
| Redis 7 | Session 管理與跨機共享 |
| Prometheus client | 連線數、訊息吞吐量即時監控 |

### 4.2　P-AGT-02　向量檢索引擎程式

**對應編號：** FR-AGT-002 / TC-AGT-002 / M-AGT-02
**KPI 關聯：** agent_accuracy ≥85%

#### 程式功能

整合 Qdrant 向量資料庫實現語意檢索，為 RAG Pipeline 提供知識庫檢索能力。本程式接收查詢文字，透過 bge-large-zh-v1.5 模型生成 Embedding 向量，在 Qdrant 中執行 HNSW 近鄰搜尋，回傳 Top-K 相關文件片段及其 metadata（文件名稱、頁碼、段落編號）供後續 Citation 標註使用。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 查詢文字（經 P-LLM-03 同義詞擴展與錯別字修正後）、Top-K 參數（預設 5）、過濾條件（知識庫版本、文件類別） |
| **輸出** | Top-K 文件片段（含原文、相似度分數、來源 metadata）、查詢延遲統計 |
| **處理程序** | ① 接收查詢文字 → ② 呼叫 bge-large-zh-v1.5 生成查詢向量（GPU#1，模型大小約 1.3GB） → ③ Qdrant HNSW 索引檢索 Top-K 候選 → ④ Reranker 重排序（以 Cross-Encoder 計算精確相關度） → ⑤ 回傳排序後的文件片段與 metadata |
| **存取資料** | Qdrant Collection（知識庫向量索引）；bge-large-zh-v1.5 模型權重（GPU#1） |
| **計算條件** | Embedding 生成 + Top-K 查詢延遲 <200ms（Qdrant Rust 引擎 HNSW 查詢 <50ms）；知識庫規模預估 10,000~50,000 筆文件，單機 Qdrant 完全足夠（AD-008） |

#### 知識庫版本管理

Qdrant Collection 以 `kb_v{version}` 命名，每次知識庫全量更新建立新 Collection，舊版本保留 3 個版本供回溯。版本切換為原子操作（修改 alias 指向），零停機。增量更新機制詳見第二章 §2.6（P-KB-01）。

#### 使用元件

| 元件 | 用途 |
|-----|------|
| Qdrant | 向量資料庫（Rust 引擎，HNSW 索引） |
| BAAI/bge-large-zh-v1.5 | 中文 Embedding 模型（MTEB 中文排行榜前三） |
| Cross-Encoder Reranker | Top-K 結果重排序，提升檢索精準度 |

### 4.3　P-AGT-03　智慧問答與 Citation 程式

**對應編號：** FR-AGT-003 + FR-AGT-004 / TC-AGT-003、TC-AGT-004 / M-AGT-03
**KPI 關聯：** agent_accuracy ≥85%、agent_response ≤5,000ms

#### 程式功能

AI-Agent 文字對話的核心處理程式，合併實作智慧問答（FR-AGT-003）與 Citation 來源標註（FR-AGT-004）。接收旅客文字訊息後，經個資遮罩、RAG 檢索、LLM 推論生成回答，並自動從回應中抽取引用來源，以 footnote 格式標註供旅客與客服人員驗證。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 旅客文字訊息、session_id、對話歷史上下文 |
| **輸出** | AI 回應文字（含 Citation footnote）、信心度分數、引用來源清單 |
| **處理程序** | ① 個資遮罩（正則遮罩身分證/電話/信用卡號） → ② 呼叫 P-AGT-02 向量檢索，取得 Top-K 知識片段 → ③ Prompt 組裝（System Instruction + 檢索結果 + 對話歷史 + 旅客問題） → ④ 透過 P-LLM-01 執行 LLM 推論 → ⑤ Citation 抽取（解析 LLM 輸出中的來源引用標記，對照 metadata 生成 footnote） → ⑥ 信心度三層路由（詳見第二章 §2.2） |
| **存取資料** | Redis `session:{session_id}`（多輪對話上下文）；PostgreSQL `conversation_logs`（對話紀錄） |
| **計算條件** | 端到端回應 ≤5,000ms（API Gateway ≤100ms + 意圖辨識 ≤200ms + 個資遮罩 ≤50ms + 向量檢索 ≤200ms + Prompt 組裝 ≤50ms + LLM 推論 ≤2,500ms + Citation 抽取 ≤100ms + 回覆發送 ≤100ms = 3,300ms，餘裕 1,700ms） |

#### Citation 標註機制

RAG Pipeline 在檢索階段保留每個 chunk 的來源 metadata（文件名稱、頁碼、段落編號）。LLM 生成回答時，Prompt 明確要求以 `[ref:N]` 格式標註引用來源。本程式於生成完畢後解析引用標記，對照 metadata 生成結構化 Citation 列表，前端以 footnote 或 tooltip 呈現來源連結，旅客可點擊查看原始文件段落。

#### 信心度三層路由

採用與語音通道相同的回應策略（詳見第二章 §2.2）：信心度 ≥0.85 直接回答；0.6~0.85 回答並附註「建議洽詢專人確認」；<0.6 回覆「此問題建議洽詢客服人員」並提供轉接按鈕。此策略確保 AI 不會在不確定時提供錯誤資訊，涉及票價、時刻等關鍵資訊一律從台鐵既有 API 即時查詢，不依賴 LLM 記憶。

#### 使用介面

- **上游：** P-AGT-01（數位通道管理）分派訊息
- **下游：** P-AGT-02（向量檢索）、P-LLM-01（LLM 推論）、P-AGT-05（轉接管理）

### 4.4　P-AGT-04　個性化引擎程式

**對應編號：** FR-AGT-005 / TC-AGT-005 / M-AGT-04
**KPI 關聯：** agent_accuracy ≥85%

#### 程式功能

基於使用者歷史對話與需求，為 AI-Agent 提供個性化回應能力。系統以 Redis 儲存當次對話的即時上下文（session-level），以 PostgreSQL 儲存歷史對話摘要（user-level），透過 LLM Prompt 注入使用者歷史摘要實現個性化回應。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | user_id（匿名化識別碼）、session_id、當次對話上下文 |
| **輸出** | 個性化上下文片段（注入至 P-AGT-03 的 Prompt 中） |
| **處理程序** | ① 以 user_id 查詢 PostgreSQL 歷史對話摘要（最近 10 次對話） → ② 擷取使用者偏好標籤（常查詢路線、慣用車種、語言偏好） → ③ 組裝個性化上下文片段（≤200 token，避免佔用過多 Prompt 空間） → ④ 將上下文片段傳遞至 P-AGT-03 注入 Prompt |
| **存取資料** | PostgreSQL `user_profiles`（使用者歷史摘要，去識別化存取）；Redis `session:{session_id}` |
| **計算條件** | 上下文查詢與組裝 ≤50ms；歷史資料須遵守個資保護規範，以匿名化 user_id 存取，存取行為寫入個資存取日誌 |

#### 個資保護

歷史對話摘要僅儲存去識別化後的意圖標籤與查詢類別（如「常查詢台北→高雄時刻」），不保留旅客原始對話內容。存取權限限定為 AI-Agent 服務帳號，操作日誌寫入獨立的個資存取日誌表（詳見第二章 §2.7 P-KB-02）。

### 4.5　P-AGT-05　Web/App 整合與轉接程式

**對應編號：** FR-AGT-006 / TC-AGT-006 / M-AGT-05
**KPI 關聯：** —（無直接 KPI，支援 agent_accuracy 之整體服務品質）

#### 程式功能

整合 Web 及 App 數位客服介面，當 AI 無法解決旅客問題時（信心度 <0.6 或旅客主動要求），自動轉接真人客服並附帶完整對話摘要。轉接時透過 WebSocket 推播通知值機平台（FR-WS-001），客服人員可即時接手對話，無需旅客重述問題。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 轉接觸發事件（信心度不足 / 旅客按「轉接專人」/ 連續 3 輪無法解決）、session_id |
| **輸出** | 轉接通知（含對話摘要、旅客意圖標籤、通道來源）→ 值機平台 |
| **處理程序** | ① 觸發轉接 → ② 呼叫 P-LLM-02 生成對話摘要（≤100 字） → ③ 透過 WebSocket 推播轉接通知至值機平台，附帶摘要與完整對話歷史 → ④ 值機員接手後，旅客端即時切換為「真人客服模式」提示 → ⑤ 完整對話紀錄（含 AI 段與真人段）寫入 PostgreSQL |
| **存取資料** | Redis `session:{session_id}`；PostgreSQL `transfer_logs`（轉接紀錄） |
| **計算條件** | 轉接延遲 ≤2 秒（含摘要生成 + WebSocket 推播） |

### 4.6　P-SMS-01　SMS 簡訊服務程式

**對應編號：** FR-AGT-007 / TC-AGT-007 / M-SMS-01
**KPI 關聯：** sms_retention ≥5 年

#### 程式功能

整合雙備援 SMS 收發裝置（4G LTE+），處理旅客簡訊收發與 AI 自動回覆。收到旅客簡訊後，將訊息轉發至 P-AGT-03 進行智慧問答處理，AI 回覆經 SMS 裝置發送給旅客，同時透過 WebSocket 推播至值機平台，通知客服人員有 AI 自動回覆紀錄，客服人員可判斷是否需人工介入接手。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 旅客 SMS 訊息（來自 SMS Gateway 的 HTTP callback）、裝置狀態信號 |
| **輸出** | AI 回覆 SMS（經 SMS Gateway 發送）、值機平台同步通知 |
| **處理程序** | ① SMS Gateway 收到旅客簡訊，以 HTTP callback 通知本程式 → ② 訊息前處理（個資遮罩、編碼轉換） → ③ 轉發至 P-AGT-03 取得 AI 回覆 → ④ 回覆文字裁切（SMS 單則 ≤70 字，超長自動分則） → ⑤ 透過 SMS Gateway 發送回覆 → ⑥ WebSocket 推播「AI 自動回覆」通知至值機平台（含旅客號碼遮罩、原始訊息、AI 回覆內容） → ⑦ 完整收發紀錄寫入 PostgreSQL |
| **存取資料** | PostgreSQL `sms_logs`（簡訊收發紀錄，保留 ≥5 年）；Redis（SMS 發送頻率限制） |
| **計算條件** | SMS 回覆端到端延遲 ≤30 秒（含 AI 推論 + SMS 發送）；雙裝置 Active-Standby 備援，主裝置故障自動切換 |

#### 雙備援機制

兩台 SMS 收發裝置以 Active-Standby 模式運作。主裝置每 30 秒回報心跳，連續 3 次心跳逾時（90 秒），自動將收發路由切換至備援裝置。切換期間佇列中的待發送簡訊由備援裝置接續處理，不遺失。SMS Gateway 的 4G LTE 模組選用非大陸品牌及非大陸製造產品，硬體清單逐項標註品牌與產地。

#### SMS 紀錄保存

所有簡訊收發紀錄（時間戳記、來源號碼遮罩、訊息內容、AI 回覆、處理狀態）以結構化格式寫入 PostgreSQL + TimescaleDB 時序分割表，按月分割、自動壓縮，5 年保留期滿後由排程任務歸檔至離線儲存。

### 4.7　P-AGT-06　系統持續優化與校正程式

**對應編號：** FR-AGT-008 / TC-AGT-008 / M-AGT-06
**KPI 關聯：** agent_accuracy ≥85%

#### 程式功能

提供 AI 模型持續優化的閉環機制。客服督導可透過管理後台標記 AI 回答為「正確」「部分正確」或「錯誤」，標記結果進入校正佇列，本程式定期（每日凌晨）批次分析錯誤樣本，自動產出知識庫更新建議與 Prompt 調校報告。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 督導標記結果（correct / partial / wrong + 修正答案）、對話紀錄 |
| **輸出** | 校正分析報告（錯誤分類統計、Top-10 高頻錯誤、知識庫缺口分析）、知識庫更新建議清單 |
| **處理程序** | ① 收集前一日所有督導標記結果 → ② 分類錯誤原因（知識庫缺漏 / RAG 檢索不準 / LLM 生成偏差 / 意圖辨識錯誤） → ③ 知識庫缺漏類：自動產出新增知識條目建議，經管理者審核後匯入 P-KB-01 → ④ 檢索不準類：分析查詢向量與正確文件向量的距離，建議調整 Embedding 參數或新增同義詞 → ⑤ 產出日報推送至管理後台 |
| **存取資料** | PostgreSQL `correction_queue`（校正佇列）、`correction_reports`（校正報告） |
| **計算條件** | 批次分析為離線任務（每日凌晨 02:00 排程），不佔用即時推論資源 |

### 4.8　P-QI-01　多格式數據匯入與批次處理程式

**對應編號：** FR-QI-001 / TC-QI-001 / M-QI-01
**KPI 關聯：** qi_speed（15 分鐘錄音 → ≤5 分鐘完成轉換分析並產出質檢報告）

#### 程式功能

智慧質檢系統的入口程式，負責接收與管理質檢任務。支援多種音檔格式匯入（WAV、MP3、OGG、FLAC），自動排程批次處理。MES 錄音系統產出的錄音檔，透過指定目錄監控或 API 上傳方式匯入，系統自動建立質檢任務並分派至後續處理 Pipeline。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 音檔（多格式）、質檢任務參數（質檢規則集 ID、優先級、排程時間） |
| **輸出** | 質檢任務 ID、任務狀態（排隊中/處理中/完成/失敗） |
| **處理程序** | ① 音檔格式偵測與轉換（統一轉為 16kHz 16-bit mono WAV） → ② 音檔 metadata 擷取（錄音時長、通道數、取樣率） → ③ 建立質檢任務（寫入 PostgreSQL `qi_tasks`） → ④ 依優先級排入批次處理佇列（Redis Sorted Set） → ⑤ Worker 依序取出任務，觸發 P-QI-02（語音轉文字）→ P-QI-03（規則評分）→ P-QI-04/05（行為與情緒分析）→ P-QI-06（異常偵測）→ P-QI-07（報表生成） |
| **存取資料** | PostgreSQL `qi_tasks`（質檢任務表）；Redis Sorted Set（批次佇列）；本機磁碟（音檔暫存） |
| **計算條件** | 15 分鐘錄音之完整質檢 Pipeline 須於 ≤5 分鐘內完成；批次排程支援 cron 設定（預設每小時執行一次）與即時手動觸發 |

#### 批次排程策略

質檢為非即時批次任務，排程於離峰時段（22:00~06:00）執行大量積壓任務，尖峰時段僅處理高優先級與手動觸發任務，確保 AI 伺服器 GPU#1 資源不與即時 Embedding 查詢競爭（AD-002）。

### 4.9　P-QI-02　語音轉文字處理程式

**對應編號：** FR-QI-002 / TC-QI-002 / M-QI-02
**KPI 關聯：** qi_speed、stt_accuracy ≥85%

#### 程式功能

將客服錄音批次轉換為文字，與即時 STT（P-STT-01）共用 faster-whisper 引擎但獨立排程，避免影響即時語音辨識效能。質檢 STT 針對台鐵業務場景術語進行額外優化，啟用完整語句後處理（標點符號、段落分割、說話者分離）。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | WAV 音檔（來自 P-QI-01 轉換後） |
| **輸出** | 結構化轉寫結果（含時間戳記、說話者標識 Agent/Customer、辨識文字、信心度） |
| **處理程序** | ① 音檔載入與前處理（降噪、正規化，流程同 P-STT-01） → ② faster-whisper 批次辨識（非串流模式，CPU 推論，可充分利用長音檔的批次吞吐優勢） → ③ 說話者分離（基於音訊能量與頻率特徵，區分客服人員與旅客） → ④ 台鐵專屬詞彙 hotword boosting（共用 P-STT-01 詞彙表） → ⑤ 輸出結構化轉寫結果 |
| **存取資料** | 模型權重檔（本機磁碟）；PostgreSQL `qi_transcripts`（轉寫結果） |
| **計算條件** | 批次模式 RTF（Real-Time Factor）約 0.3x（即 15 分鐘錄音約 4.5 分鐘完成轉寫），留 0.5 分鐘給後續分析評分 |

### 4.10　P-QI-03　質檢規則與評分引擎程式

**對應編號：** FR-QI-003 / TC-QI-003 / M-QI-03
**KPI 關聯：** qi_speed

#### 程式功能

依管理者自訂的質檢規則對轉寫結果執行自動評分。系統支援多維度評分項目（開場白/結尾語規範、敬語使用、問題解決率、回應速度等），每項規則可設定權重與通過門檻，計算加權總分並判定是否通過質檢標準。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 結構化轉寫結果（來自 P-QI-02）、質檢規則集 ID |
| **輸出** | 各項評分明細、加權總分、通過/不通過判定、扣分原因說明 |
| **處理程序** | ① 載入質檢規則集（自 PostgreSQL `qi_rule_sets` 讀取） → ② 逐項規則比對：正則比對（開場白/結尾語關鍵字）、LLM 語意評分（問題解決品質）、統計指標（平均回應時間） → ③ 各項乘以權重計算加權總分 → ④ 與門檻值比對，判定通過或不通過 → ⑤ 寫入評分結果至 PostgreSQL `qi_scores` |
| **存取資料** | PostgreSQL `qi_rule_sets`（規則定義）、`qi_scores`（評分結果） |
| **計算條件** | 單通評分 ≤10 秒（含 LLM 語意評分） |

#### 規則自訂機制

管理者透過管理後台（FR-ADM-001）設定質檢規則，支援以下規則類型：

| 規則類型 | 範例 | 評分方式 |
|---------|------|---------|
| 關鍵字比對 | 開場白需包含「台鐵客服您好」 | 正則比對，命中得分/未命中扣分 |
| 語意評分 | 客服回答是否解決旅客問題 | LLM few-shot 評分（0~10 分） |
| 行為指標 | 平均回應時間 ≤15 秒 | 統計計算，門檻判定 |
| 禁忌語偵測 | 禁止使用不當用語 | 正則 + 敏感詞庫比對 |

### 4.11　P-QI-04　靜音與插話偵測程式

**對應編號：** FR-QI-004 / TC-QI-004 / M-QI-04
**KPI 關聯：** —（無直接 KPI，支援質檢評分之行為分析維度）

#### 程式功能

分析質檢音檔中的靜音段與插話行為。長時間靜音可能代表客服人員查詢資料或態度消極，頻繁插話可能影響旅客體驗，兩者均為質檢評分的重要行為指標。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | WAV 音檔、說話者分離結果（來自 P-QI-02） |
| **輸出** | 靜音段清單（起始/結束時間、持續秒數）、插話事件清單（時間點、持續秒數）、行為統計摘要 |
| **處理程序** | ① 以能量門檻偵測靜音段（連續 >3 秒低於 -40dBFS 判定為靜音） → ② 結合說話者分離結果，偵測雙方語音重疊段落（插話） → ③ 排除正常語音交替中的短暫重疊（<0.5 秒不計入） → ④ 統計：總靜音時長、最長靜音、插話次數、插話總時長 → ⑤ 寫入 PostgreSQL `qi_behavior_metrics` |
| **存取資料** | PostgreSQL `qi_behavior_metrics` |
| **計算條件** | 15 分鐘音檔分析 ≤15 秒（純 CPU 音訊分析，運算負載極低） |

### 4.12　P-QI-05　語調與情緒分析程式

**對應編號：** FR-QI-005 / TC-QI-005 / M-QI-05
**KPI 關聯：** —（無直接 KPI，支援質檢異常偵測）

#### 程式功能

分析客服人員與旅客的語調變化與情緒狀態。以語音特徵（基頻、語速、能量變化）結合文字情感分析（LLM 情感分類），綜合判定每段對話的情緒標籤（正面/中性/負面/激動），標記情緒異常段落供督導重點檢視。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | WAV 音檔、結構化轉寫結果（來自 P-QI-02） |
| **輸出** | 逐段情緒標籤、情緒時序曲線、異常標記（旅客負面情緒升高 / 客服語調異常） |
| **處理程序** | ① 語音特徵擷取（基頻 F0、語速 syllable/sec、能量 RMS，以 librosa 計算） → ② 文字情感分析（以 LLM few-shot 分類，透過 P-LLM-01 於 GPU#1 推論） → ③ 特徵融合：語音特徵 70% + 文字情感 30%（語音為主要判斷依據） → ④ 時序分析：偵測情緒急劇變化點（如旅客從中性突然轉為負面） → ⑤ 異常段落標記，寫入 PostgreSQL `qi_emotion_results` |
| **存取資料** | PostgreSQL `qi_emotion_results`；LLM 推論（GPU#1，批次模式） |
| **計算條件** | 15 分鐘音檔情緒分析 ≤60 秒（含語音特徵擷取 + LLM 批次情感分類） |

#### GPU 資源隔離

情緒分析的 LLM 推論部署於 GPU#1（AD-002），以批次模式執行（非即時串流），排程於質檢批次任務中。GPU#1 同時負責 Embedding 查詢，兩者透過 GPU 記憶體使用率監控協調：Embedding 查詢優先（即時需求），情緒分析批次任務在 GPU 記憶體使用率 <60% 時方可執行。

### 4.13　P-QI-06　異常偵測與警報程式

**對應編號：** FR-QI-006 / TC-QI-006 / M-QI-06
**KPI 關聯：** —（無直接 KPI，支援服務品質即時管控）

#### 程式功能

彙整 P-QI-03（評分）、P-QI-04（行為偵測）、P-QI-05（情緒分析）結果，依預設警報規則判定是否觸發異常警報，通知督導人員及時介入。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 質檢評分結果、行為統計、情緒分析結果 |
| **輸出** | 警報事件（等級：一般/重要/緊急）→ 管理後台 + 督導值機平台 |
| **處理程序** | ① 彙整單通質檢完整結果 → ② 逐項比對警報規則（評分低於門檻 / 靜音超時 / 旅客情緒激動 / 禁忌語觸發） → ③ 判定警報等級 → ④ 緊急警報即時推播（WebSocket + 系統通知）；一般警報寫入警報佇列，督導登入時集中顯示 |
| **存取資料** | PostgreSQL `qi_alerts`（警報紀錄）；`qi_alert_rules`（警報規則） |
| **計算條件** | 警報判定 ≤1 秒；緊急警報推播 ≤3 秒 |

#### 警報規則範例

| 警報條件 | 等級 | 動作 |
|---------|------|------|
| 質檢總分 <60 分 | 重要 | 通知督導覆核 |
| 旅客出現連續負面情緒 >3 分鐘 | 緊急 | 即時推播督導介入 |
| 單通靜音總時長 >通話時長 30% | 一般 | 記錄並列入日報 |
| 客服使用禁忌語 | 緊急 | 即時推播 + 自動標記 |

### 4.14　P-QI-07　多維度質檢報表程式

**對應編號：** FR-QI-007 / TC-QI-007 / M-QI-07
**KPI 關聯：** qi_speed

#### 程式功能

彙整所有質檢結果，產出多維度質檢報表。支援活動報表（特定時段/事件的質檢彙總）、專員績效報表（個人評分趨勢與排名）、原因分析報表（扣分原因分佈與改善建議）。報表資料同時寫入 RPT 模組（P-RPT-01）供統一儀表板展示。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 質檢結果集合（時間區間、篩選條件） |
| **輸出** | 結構化報表資料（JSON）+ 視覺化圖表（前端渲染） |
| **處理程序** | ① 依篩選條件從 PostgreSQL 聚合質檢資料 → ② 計算統計指標（平均分、通過率、各維度分佈） → ③ 專員排名與趨勢分析（30 天滾動平均） → ④ 扣分原因 Top-N 統計與改善建議（LLM 輔助生成） → ⑤ 輸出結構化資料至前端與 RPT 模組 |
| **存取資料** | PostgreSQL `qi_scores`、`qi_behavior_metrics`、`qi_emotion_results`、`qi_alerts` |
| **計算條件** | 報表查詢 ≤5 秒（依賴 PostgreSQL 索引與 TimescaleDB 時序聚合效能） |

### 4.15　模組間介面總覽

以下為本章 14 支程式與其他模組之主要呼叫關係：

```
CRM Gateway ──WebSocket──→ P-AGT-01（通道管理）──訊息分派──→ P-AGT-03（智慧問答+Citation）
                                                                │        │
                                                         P-AGT-02     P-LLM-01
                                                        （向量檢索）  （LLM 推論）
                                                                │
                                                         P-AGT-04（個性化）
                                                                │
                                                    P-AGT-05（轉接）──→ 值機平台（WS）

SMS Gateway ──HTTP──→ P-SMS-01（簡訊服務）──→ P-AGT-03 ──→ SMS Gateway
                           │
                      值機平台（同步通知）

MES 錄音 ──→ P-QI-01（匯入排程）──→ P-QI-02（轉寫）──→ P-QI-03（評分）
                                                           │
                                              P-QI-04（靜音/插話）+ P-QI-05（情緒）
                                                           │
                                                     P-QI-06（異常警報）──→ 督導平台
                                                           │
                                                     P-QI-07（報表）──→ RPT 模組
```

AI-Agent 文字對話延遲預算：API Gateway ≤100ms → 意圖辨識 ≤200ms → 個資遮罩 ≤50ms → 向量檢索 ≤200ms → Prompt 組裝 ≤50ms → LLM 推論 ≤2,500ms → Citation 抽取 ≤100ms → 回覆發送 ≤100ms → 合計 ≤3,300ms，餘裕 1,700ms 內完成 5,000ms SLA。

質檢 Pipeline 時間預算（15 分鐘錄音）：音檔轉換 ≤10s → STT 轉寫 ≤270s → 規則評分 ≤10s → 行為偵測 ≤15s → 情緒分析 ≤60s → 異常判定 ≤1s → 報表 ≤5s → 合計 ≤371s（約 6.2 分鐘，≤5 分鐘 SLA 於離峰時段可達成；尖峰時段透過 Worker 平行化處理多通錄音攤分時間）。

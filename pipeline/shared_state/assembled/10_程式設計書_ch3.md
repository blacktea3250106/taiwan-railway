## 第三章　STT / TTS 模組程式說明

本章說明語音識別（STT）3 支程式與語音合成（TTS）2 支程式之程式功能、處理邏輯、使用資料庫及元件。STT 與 TTS 均部署於 AI-IVR 伺服器（CPU 16 核心 ≥2.2GHz ×2，DDR5 64GB+），採 CPU 推論而非 GPU（AD-003），理由為：串流語音處理屬「每收到一小段語音就要立即處理」的低延遲任務，CPU 的每核心延遲優於 GPU 的批次吞吐特性；30 路並行 STT 約需 9 CPU 核心、TTS 約需 6 CPU 核心，合計 15 核心，AI-IVR 伺服器 32 核心綽綽有餘，GPU 資源留給 LLM 推論（AD-002）。

語音互動完整信令流程與延遲預算分配詳見 #03 程式設計規格報告書第四章 §4.1；資料庫完整 Schema 詳見 #03 第六章。

### 3.1　P-STT-01　即時語音辨識引擎程式

**對應編號：** FR-STT-001 / TC-STT-001 / M-STT-01
**KPI 關聯：** stt_accuracy ≥85%

#### 程式功能

接收 PJSIP RTP 接收器解碼後的 PCM 音訊流，以 faster-whisper（CTranslate2 加速版 Whisper large-v3）執行即時中英文混合語音辨識，輸出辨識文字供後續意圖辨識（P-STT-02）與 LLM 推論（P-LLM-02）使用。本程式以 chunk-based streaming 模式運作，由 VAD 模組（P-STT-03）觸發辨識時機，確保辨識結果與旅客發言節奏同步。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | PCM 音訊流（G.711 解碼後 16kHz 16-bit mono）、VAD 觸發信號（來自 P-STT-03） |
| **輸出** | 辨識文字（含時間戳記、語言標籤 zh/en、辨識信心度分數） |
| **處理程序** | ① 音訊前處理（重採樣至 16kHz、振幅正規化、背景降噪） → ② 接收 VAD 觸發信號，將語音段落送入 faster-whisper 辨識佇列 → ③ CTranslate2 引擎執行 Whisper large-v3 推論（CPU 模式，int8 量化加速） → ④ 後處理（繁體中文正規化、標點符號插入、台鐵專屬詞彙 hotword boosting） → ⑤ 輸出辨識結果至意圖辨識模組 |
| **存取資料** | 模型權重檔（本機磁碟，唯讀）；PostgreSQL `hotword_dict`（台鐵專屬詞彙表） |
| **計算條件** | 單路辨識延遲 ≤800ms（佔 5,000ms SLA 之 16%）；30 路並行約需 9 CPU 核心；辨識率 ≥85%（200 音檔測試，中英文比例 4:1，每檔 ≤150 秒） |

#### 音訊前處理 Pipeline

音訊品質直接影響辨識率。系統來源包含 PSTN E1（G.711 64kbps）、WebCall（Opus/G.722）與行政交換機（類比→G.711），其中行政交換機經類比→數位轉換後，背景雜訊可能高於其他來源。前處理 Pipeline 針對不同來源採用差異化參數：

1. **重採樣**：統一轉換為 16kHz 16-bit mono PCM（Whisper 原生輸入格式）
2. **振幅正規化**：將音量正規化至 -3dBFS，避免過輕或過響影響辨識
3. **背景降噪**：以 RNNoise（開源即時降噪）過濾環境雜訊，類比線路來源自動啟用更積極的降噪參數

#### 台鐵專屬詞彙 Hotword Boosting

Whisper large-v3 對標準國語辨識率可達 88~92%，但台鐵業務特有詞彙（車站名、車種名、票種名）需額外處理。系統透過 hotword boosting 機制，在解碼階段提升特定詞彙的輸出機率：

- **車站名稱：** 「新左營」「板橋」「南港」「瑞芳」等（含易混淆站名）
- **車種名稱：** 「自強號」「太魯閣」「普悠瑪」「區間車」
- **票種名稱：** 「去回票」「電子票證」「定期票」
- **常見口語修正：** 「自強好」→「自強號」、「板喬」→「板橋」

詞彙表透過管理後台（FR-ADM-001）維護，修改後即時載入，無需重啟服務。得標後以台鐵客服錄音進行 fine-tune，可進一步提升業務場景辨識率。

#### 使用元件

| 元件 | 用途 |
|-----|------|
| faster-whisper（CTranslate2 引擎） | Whisper large-v3 推論（CPU 模式），比原版 PyTorch 快 4 倍、記憶體用量降低 50% |
| Whisper large-v3 模型 | 開源中英文混合語音辨識，繁中實測準確率 88~92% |
| RNNoise | 即時背景降噪（開源，CPU 運算） |
| NumPy / SoundFile | 音訊前處理（重採樣、正規化） |

### 3.2　P-STT-02　意圖辨識程式

**對應編號：** FR-STT-002 / TC-STT-002 / M-STT-02
**KPI 關聯：** stt_accuracy ≥85%

#### 程式功能

接收 P-STT-01 輸出的辨識文字，執行意圖分類與路由決策，判定旅客查詢屬於哪一類業務意圖（如時刻查詢、票價查詢、掛失、投訴等），並決定後續路由（AI 自動回答 / 轉接人工 / 觸發 API 查詢）。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | STT 辨識文字、辨識信心度、session_id |
| **輸出** | 意圖分類標籤（一級/二級分類）、路由決策（ai_answer / transfer_human / api_query）、分類信心度 |
| **處理程序** | ① 接收辨識文字 → ② 文字前處理（呼叫 P-LLM-03 同義詞擴展與錯別字修正） → ③ 意圖分類（以 LLM few-shot 分類，透過 P-LLM-01 推論，注入意圖定義與範例） → ④ 路由決策（依意圖類別與信心度決定後續處理路徑） → ⑤ 將意圖標籤寫入 Redis Session Context，供多輪對話追蹤 |
| **存取資料** | PostgreSQL `intent_definitions`（意圖定義與範例）；Redis `session:{session_id}`（對話上下文） |
| **計算條件** | 分類延遲 ≤200ms（含 LLM few-shot 推論，使用短 prompt 快速回應） |

#### 路由決策規則

| 意圖類別 | 路由 | 說明 |
|---------|------|------|
| 時刻/票價查詢 | api_query → ai_answer | 先呼叫台鐵既有 API 取得即時資料，再由 LLM 組裝回答 |
| FAQ 類問題 | ai_answer | 直接走 RAG Pipeline（P-LLM-02） |
| 掛失/退票/投訴 | transfer_human | 涉及交易操作，直接轉接人工客服（P-IVR-03） |
| 非業務問題 | ai_answer（引導） | LLM System Instruction 限定範圍回答，連續 2 輪非業務則結束對話（P-IVR-06） |

#### 使用介面

- **上游：** 接收 P-STT-01 辨識結果
- **下游：** 將意圖與路由決策傳遞至 P-LLM-02（RAG Pipeline）、P-IVR-03（轉接管理）或 P-IVR-06（防護機制）

### 3.3　P-STT-03　VAD 語音活動偵測程式

**對應編號：** FR-STT-003 / TC-STT-003 / M-STT-03
**KPI 關聯：** ivr_response ≤5,000ms

#### 程式功能

持續監聽 RTP 音訊流，偵測旅客發言的起始與結束時機。VAD 的準確度直接影響兩項使用者體驗：過早判定結束會截斷旅客語句；過晚判定則延長等待時間，壓縮 STT→LLM→TTS 的回應預算。本程式採用 Silero VAD 模型，相較 WebRTC VAD 的能量門檻法，Silero 以神經網路辨識語音特徵，對背景雜訊與氣音的抗干擾能力顯著較佳。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | PCM 音訊流（即時 RTP 解碼輸出） |
| **輸出** | VAD 事件（speech_start / speech_end + 時間戳記）、語音段落音訊切片 |
| **處理程序** | ① 以 30ms 窗口持續掃描音訊流 → ② Silero VAD 模型判定每個窗口是否包含語音 → ③ 連續 N 個窗口偵測到語音觸發 speech_start 事件 → ④ 連續 M 個窗口偵測到靜音觸發 speech_end 事件 → ⑤ 將 speech_start 至 speech_end 之間的音訊切片送入 P-STT-01 辨識佇列 |
| **存取資料** | Silero VAD 模型權重（本機磁碟，唯讀） |
| **計算條件** | 端點偵測延遲 ≤200ms（佔 5,000ms SLA 之 4%）；CPU 佔用極低（每路 <0.05 核心） |

#### 參數調校策略

VAD 參數需依音訊來源品質差異化配置：

| 參數 | PSTN E1 / WebCall | 行政交換機（類比線路） | 說明 |
|-----|-------------------|---------------------|------|
| speech_threshold | 0.5 | 0.6 | 語音判定門檻（0~1），類比線路因背景雜訊較高需提高門檻 |
| min_speech_duration | 250ms | 300ms | 最短語音段落長度，避免雜訊誤觸發 |
| min_silence_duration | 500ms | 600ms | 靜音持續多久判定發言結束 |
| speech_pad | 100ms | 150ms | 語音前後保留的緩衝區，避免截斷首尾音節 |

上述參數可透過管理後台（FR-ADM-001）調整。系統上線初期（D+180 ~ D+240）將依實際客服錄音統計分析，逐步收斂至最佳參數組合。

#### 使用元件

| 元件 | 用途 |
|-----|------|
| Silero VAD | 神經網路語音活動偵測模型（開源，ONNX 格式，CPU 推論） |
| ONNX Runtime | Silero VAD 模型推論引擎 |

### 3.4　P-TTS-01　語音合成引擎程式

**對應編號：** FR-TTS-001 / TC-TTS-001 / M-TTS-01
**KPI 關聯：** ivr_response ≤5,000ms

#### 程式功能

將 LLM 生成的文字回應即時合成為自然語音，以串流方式輸出至 PJSIP RTP 發送器播放給旅客。主引擎採用 Coqui XTTS-v2 模型，支援中英文混合語音合成與 voice cloning（可依台鐵指定的聲音風格客製化）。所有合成運算於 AI-IVR 伺服器 CPU 完成，完全地端運行，無任何外部網路連線。

Coqui AI 公司已於 2023 年底停業，但 XTTS-v2 模型以 MPL-2.0 開源授權釋出，原始碼與模型權重可永久使用。系統同時內建 MeloTTS 與 Piper 作為備案引擎，三者共用統一的 TTS 抽象介面，管理者可透過後台切換，無需修改上下游程式碼。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | LLM 生成文字（streaming token 或完整文句）、語音參數（語速、語調、聲音風格） |
| **輸出** | PCM 音訊流（16kHz → G.711 編碼後交由 PJSIP 發送） |
| **處理程序** | ① 文字前處理（數字轉國字、英文單字音譯標注、SSML 標記解析） → ② 查詢語音快取（常用語句如「歡迎致電台鐵客服」「請稍候為您轉接」已預先合成並快取） → ③ 快取未命中時，以 XTTS-v2 模型執行串流合成（首包延遲 ≤300ms） → ④ 音訊後處理（音量正規化、採樣率轉換） → ⑤ 串流輸出至 RTP 發送器 |
| **存取資料** | XTTS-v2 模型權重（本機磁碟，唯讀）；Redis `tts_cache:{text_hash}`（語音快取） |
| **計算條件** | 單路合成延遲 ≤800ms（首包 ≤300ms + 串流持續輸出）；每路約需 0.2 CPU 核心；30 路並行合計約 6 CPU 核心 |

#### 語音快取機制

高頻使用的固定語句（系統歡迎語、轉接提示、滿意度調查引導語等）預先合成並儲存於 Redis，以文字內容 SHA-256 hash 作為快取鍵。快取命中時直接回傳音訊流，延遲 <10ms，大幅降低 CPU 運算負載。快取容量預設 500 條語句，LRU 策略淘汰低頻項目。管理後台修改語句模板時自動觸發快取更新。

#### 備案引擎切換

| 引擎 | 特點 | 切換方式 |
|-----|------|---------|
| Coqui XTTS-v2（主） | 中文自然度最佳，支援 voice cloning | 預設引擎 |
| MeloTTS | 多語言支援佳，推論速度快，Apache-2.0 授權 | 管理後台一鍵切換 |
| Piper | 超輕量（每路 <0.05 核心），適合極端高併發場景 | 管理後台一鍵切換 |

三套引擎共用 `TTSEngine` 抽象介面（`synthesize_stream(text, voice_config) → AudioStream`），上游呼叫端無需感知底層引擎差異。切換引擎僅需修改一個環境變數並重啟 TTS 容器（Docker Compose 滾動更新，零停機）。

#### 使用元件

| 元件 | 用途 |
|-----|------|
| Coqui TTS（XTTS-v2 模型） | 主語音合成引擎（CPU 推論，MPL-2.0 授權） |
| MeloTTS / Piper | 備案語音合成引擎 |
| Redis | 語音快取（常用語句預合成結果） |
| SSML Parser | SSML 標記解析（停頓、強調、語速控制） |

### 3.5　P-TTS-02　動態選單合成程式

**對應編號：** FR-TTS-002 / TC-TTS-002 / M-TTS-02
**KPI 關聯：** —（無直接 KPI，支援 FR-IVR-005 按碼偵測與動態選單）

#### 程式功能

依管理後台設定動態生成語音選單內容，無需預錄音檔。系統管理者可透過後台（FR-ADM-001）自行編輯選單結構、選項文字與排列順序，修改後系統自動調用 P-TTS-01 合成新語音並更新快取。本程式與 P-IVR-04（DTMF 偵測）協同運作，構成完整的語音選單互動流程。

#### 處理邏輯

| 項目 | 說明 |
|-----|------|
| **輸入** | 選單定義（JSON 格式：選單 ID、各選項文字、DTMF 對應按鍵）；選單觸發事件（來自 P-IVR-04） |
| **輸出** | 合成後的語音選單音訊流（交由 P-TTS-01 串流播放） |
| **處理程序** | ① 載入選單定義（自 PostgreSQL `menu_definitions` 表讀取） → ② 依模板組裝選單文字（含 SSML 標記：各選項間插入 300ms 停頓、按鍵編號以強調語調播放） → ③ 呼叫 P-TTS-01 合成語音 → ④ 合成結果寫入快取（選單內容變更前持續使用） |
| **存取資料** | PostgreSQL `menu_definitions`（選單定義表：menu_id、menu_name、items_json、updated_at） |
| **計算條件** | 選單首次合成 ≤2 秒（含模板組裝 + TTS 合成）；後續使用快取，延遲 <10ms |

#### 選單模板範例

```
歡迎致電台鐵客服，<break time="300ms"/>
查詢時刻表請按 <emphasis>1</emphasis>，<break time="300ms"/>
查詢票價請按 <emphasis>2</emphasis>，<break time="300ms"/>
掛失或退票請按 <emphasis>3</emphasis>，<break time="300ms"/>
轉接人工客服請按 <emphasis>0</emphasis>，<break time="300ms"/>
或直接說出您的問題。
```

SSML 標記由 P-TTS-01 的 SSML Parser 解析，控制停頓時長（`<break>`）、語調強調（`<emphasis>`）與語速調節（`<prosody>`）。

#### 使用介面

- **上游：** 管理後台（FR-ADM-001）編輯選單定義 → 本程式自動觸發合成
- **下游：** P-IVR-04（DTMF 偵測）觸發播放選單 → 本程式回傳快取音訊

### 3.6　模組間介面總覽

以下為本章 5 支程式與其他模組之主要呼叫關係：

```
PJSIP RTP 接收器 ──PCM──→ P-STT-03（VAD）──語音段落──→ P-STT-01（辨識）
                                                           │
                                                    P-STT-02（意圖辨識）
                                                     │          │
                                              P-LLM-02      P-IVR-03/06
                                              （RAG）     （轉接/防護）
                                                 │
                                           P-TTS-01（合成）──PCM──→ PJSIP RTP 發送器
                                                 ↑
                                           P-TTS-02（動態選單）
                                                 ↑
                                           P-IVR-04（DTMF 偵測）
```

STT 與 TTS 模組以 gRPC 與 LLM 模組通訊，以函式呼叫與 IVR 模組通訊（同一容器內）。AI-IVR 語音互動完整流程之延遲預算分配：VAD ≤200ms → STT ≤800ms → 意圖辨識 ≤200ms → LLM 推論 ≤2,500ms → TTS ≤800ms → 合計 ≤4,500ms，餘裕 500ms 內完成 5,000ms SLA。各階段延遲預算詳見 #03 §4.1.2。

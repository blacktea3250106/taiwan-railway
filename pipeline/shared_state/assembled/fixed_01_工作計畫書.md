## 第一章　專案概述與履約範圍

### 1.1 專案背景

國營臺灣鐵路股份有限公司（以下簡稱台鐵）為提升旅客服務品質與客服作業效率，辦理「AI 客服系統建置案」（案號 L0215P2010U），於既有客服體系中導入人工智慧技術，建置智慧語音應答（AI-IVR）、AI 虛擬客服（AI-Agent）、智慧質檢（AI-QI）等核心功能，介接既有 PSTN/WebCall/數位客服通道，全面採用開源技術堆疊，以地端高可用（HA）雙機架構部署於台鐵機房。

本案履約期程為機關通知次日起 270 日曆天，採總包價法，分四期付款。本工作計畫書為第一期交付項目（D+30），作為後續 12 份交付文件之綱領性指引。

### 1.2 專案目標

| 目標面向 | 具體目標 | 量化指標 |
|---------|---------|---------|
| 服務效率 | AI 承接第一線旅客諮詢，釋放人工座席處理複雜案件 | AI-IVR / AI-Agent 應答正確率 ≥85% |
| 回應速度 | 旅客說完（VAD 偵測語音結束）至系統回覆首字播出 | ≤5,000ms |
| 通道容量 | 語音與文字通道同時服務能力 | 語音 ≥30 埠、文字 ≥30 埠 |
| 語音辨識 | 中英文混合語音轉文字辨識 | STT 正確率 ≥85%（200 音檔，中英 4:1） |
| 品質管理 | 客服錄音自動轉寫、評分、異常偵測 | 15 分鐘錄音 → ≤5 分鐘完成質檢報告 |
| 後送管理 | 後送案件自動分類與優先級判定 | 分類準確率 ≥85% |
| 系統穩定 | 全年不間斷營運 | 可用率 ≥99%/季、MTTR ≤4 小時（含假日） |

### 1.4 架構原則

本案採六層式架構（AD-001），將 AI 功能模組獨立於既有系統之外，確保零衝擊介接：

1. **基礎設施層**：正式環境 2+2+2 伺服器 ＋ 測試環境 1+1 伺服器，42U 機櫃，RPS 冗餘電源
2. **平台安全層**：CNS 27001 控制措施、RBAC + AD/LDAP 整合、TLS 1.3 傳輸加密、AES-256 儲存加密
3. **資料管理層**：PostgreSQL 16 + TimescaleDB（關聯）、Qdrant（向量）、Redis 7（快取）
4. **AI 服務層**：Llama 3.1 8B-Instruct AWQ + vLLM（LLM）、Whisper large-v3 + faster-whisper（STT）、Coqui XTTS-v2（TTS）、bge-large-zh-v1.5（Embedding）
5. **通道接入層**：PJSIP SIP/RTP stack、CRM Gateway REST/WebSocket、SMS Gateway
6. **使用者層**：值機平台（React 18 / Ant Design）、管理後台、旅客端 Web/App

GPU 資源分配依 AD-002 規劃：GPU#0 專責 LLM 推論，GPU#1 負責 Embedding + 質檢 + 擴充預留；STT/TTS 以 CPU 推論（AD-003），避免與 LLM 搶奪 GPU 資源。部署方式採 Docker Compose（AD-006），降低台鐵後續維運門檻。

### 1.5 既有系統介接範圍

本案採 SIP/RTP fork 方式介接既有語音系統（AD-007），**不修改 MES Server 既有設定**，AI-IVR 為新增獨立模組，被動接收語音流副本：

| 既有系統 | 介接方式 | 配合事項 |
|---------|---------|---------|
| PSTN E1 VoIP Gateway | SIP Trunk + RTP fork | 現有 1 條 E1（30 B-channel），AI-IVR 接收 fork 副本，不佔用 B-channel |
| WebCall SBC（雙備援） | SIP Trunk（failover） | AI-IVR 支援雙 SBC 註冊，自動偵測切換 |
| MES Server（話務/錄音） | RTP fork + API | 通話分配與錄音來源 |
| 行政交換機（02-2381-5226） | 類比 → VoIP Gateway → MES → AI-IVR | G.711 編碼轉換，語音品質損失極小 |
| CRM Gateway | REST / WebSocket API | 數位文字客服訊息通道 |
| AD（Active Directory） | LDAP | 人員帳號驗證與權限 |
| 訂票紀錄查詢系統 | API（規格待取得） | LLM 串接即時查詢票務資訊 |
| 時刻表查詢系統 | API（規格待取得） | LLM 串接即時查詢班次資訊 |
| 後送案件系統 | API | AI 分析結果透過 API 推送，不修改現有 225 位外點人員操作介面 |

### 4.2 階段一：系統分析（D+30 ~ D+80）

**目標：** 將 48 項功能需求（FR-IVR-001 ~ FR-ADM-005）轉化為可追溯的系統規格。

**主要工作項目：**

1. **功能需求精化：** 依需求訪談結果（詳見第三章 §3.2），逐項確認 48 項 FR 的業務規則、例外處理與邊界條件。每項 FR 均建立「FR → TC → M → P」四級追溯鏈，確保無遺漏。

2. **介面需求定義：** 針對 10 套既有系統介接（詳見第三章 §3.4），產出介面控制文件（ICD），包含：
   - CRM Gateway REST/WebSocket API 規格
   - MES Server RTP fork 參數（AD-007）
   - 訂票紀錄查詢系統、時刻表查詢系統 API 規格（待台鐵提供）
   - AD/LDAP 認證整合規格

3. **非功能需求量化：** 將契約 KPI 轉為可測試的技術指標——IVR/Agent 回應時間 ≤5,000ms 拆解為 VAD+STT+意圖 ≤1,200ms + LLM ≤2,500ms + TTS ≤800ms + 其餘 ≤110ms 之時間預算分配。

4. **個資影響評估（PIA）：** 盤點系統中所有旅客個資欄位（電話號碼、身分證字號、訂票紀錄），定義蒐集目的、處理方式、去識別化機制與保存期限，STT 輸出文字即時套用正則表達式遮罩。

**產出物：** #02 系統分析報告書（含需求規格說明）、#03 程式設計規格報告書。

### 4.3 階段二：系統設計（D+80 ~ D+130）

**目標：** 將系統規格轉化為可實作的元件設計與技術規格。

**核心設計決策依據：**

本案採六層式架構（AD-001），將 AI 服務層與既有系統隔離，確保零衝擊。GPU 資源分配策略為 GPU#0 專責 LLM 推論、GPU#1 負責 Embedding + 質檢 + 擴充預留（AD-002），避免即時推論與批次任務搶奪資源。STT/TTS 採 CPU 推論（AD-003），串流低延遲特性適合 CPU，GPU 資源保留給 LLM。

**設計工作項目：**

| 設計面向 | 內容 | 負責人 |
|---------|------|--------|
| 模組架構 | 45 個軟體模組（M-IVR-01 ~ M-ADM-05）之元件圖、介面定義、相依關係 | 架構師 |
| 資料庫設計 | PostgreSQL 16 資料模型（ER Diagram）、TimescaleDB 時序資料分割策略、Redis 快取結構 | 後端工程師 |
| AI Pipeline | LLM 推論鏈（vLLM + Llama 3.1 8B-Instruct AWQ）、RAG 檢索鏈（Qdrant + bge-large-zh-v1.5）、STT/TTS 串流管線 | AI/ML 工程師 |
| API 規格 | 對外 REST API 依國發會共通性 API 規範設計（契約§2(四)）、對內 gRPC 高頻通訊介面 | 後端工程師 |
| UI/UX 原型 | 值機平台（WS）、管理後台（ADM）、旅客端對話介面之 Wireframe 與互動原型 | 前端工程師 |

### 6.1 維護營運機制

系統驗收上線後進入保固期，提供 24/7（00:00 ~ 23:59）全年無休維護服務，服務水準承諾如下：

| SLA 指標 | 承諾值 | 違約計罰 |
|---------|--------|---------|
| 系統可用率 | ≥99%/季 | 每不足 1% 計 3 點 |
| 單日最大停機 | ≤4 小時/日 | 逾 4hr 每小時計 1 點 |
| 故障修復時限（MTTR） | ≤4 小時（含假日） | 逾 4hr 每小時計 1 點 |
| 事件通報 | 知悉後 ≤1 小時 | 逾 1hr 每小時計 1 點 |
| 損害控制/復原 | ≤72 小時（重大 36 小時） | — |
| 調查報告 | 損害控制後 ≤1 個月 | — |

**維護排班機制：**

| 時段 | 模式 | 回應時限 |
|------|------|---------|
| 平日日間（09:00 ~ 18:00） | 工程師駐點或遠端 | 即時 |
| 平日夜間（18:00 ~ 09:00） | 值班手機輪值 | 收到告警 30 分鐘內遠端登入 |
| 假日/國定假日 | 值班手機輪值 | 收到告警 30 分鐘內遠端登入 |

告警升級路徑：L1 監控告警（Prometheus + Grafana 自動觸發）→ L2 工程師遠端排除（佔 80% 故障場景，30 分鐘內完成）→ L3 工程師到場處理（台鐵總公司位於台北車站旁，工程師從市區出發 30 分鐘內可達）。

**監控告警項目：** 系統不可用、GPU 溫度異常、記憶體使用率 >85%、磁碟使用率 >90%、API 延遲 >3 秒。所有操作日誌保留 ≥6 個月，SMS 簡訊紀錄保留 ≥5 年。

### 6.2 HA 雙機切換與人工客服降級方案

**正常運行模式：** Active-Active 雙機（AD-005），VIP 負載均衡分配流量至兩台同角色伺服器。

**故障自動切換：** 任一台伺服器故障時，Keepalived 於 ≤5 秒內完成 VIP 漂移，存活機承載 100% 流量。進行中的 SIP 通話由 MES Server 維持，僅遺失該通的 AI 輔助功能，不中斷通話本身。

**系統全面異常降級方案：** 當 AI 客服系統發生無法於短時間內修復之異常時，系統立即切換為人工客服模式，確保旅客服務不中斷：

1. **自動偵測：** Prometheus 監控偵測到 AI 服務連續 3 次健康檢查失敗（間隔 10 秒），觸發降級機制。
2. **語音通道降級：** MES Server 自動將所有來電直接分配至人工座席排隊，不再經過 AI-IVR 路由，此切換不需修改 MES Server 設定（AD-007 RTP fork 為被動接收，停止接收不影響原始路徑）。
3. **文字通道降級：** CRM Gateway 切換為直連人工客服模式，AI-Agent 回覆替換為預設訊息「目前由真人客服為您服務，請稍候」。
4. **通知機制：** 降級觸發同時發送告警通知值班工程師與台鐵督導，啟動故障排除流程。
5. **恢復機制：** AI 服務修復後，經健康檢查通過 3 次以上方可重新接入流量，避免抖動。

### 6.3 備品與硬體故障應對

HA 架構使大多數硬體故障可在服務不中斷的前提下處理。針對硬體更換需求：

- **備品策略：** 關鍵零件（SSD、RAM、電源供應器）各備 1 套存放於台鐵機房，無需等待物流。
- **非假日：** 原廠 NBD（次營業日到場）服務合約，硬體更換由原廠工程師執行。
- **假日：** HA 備機先行承載全部流量，次營業日再行更換故障零件。

### 7.2 兩階段訓練架構

訓練採 Pilot → 正式兩階段推進，確保操作回饋可在正式訓練前反映至系統介面調整：

**第一階段：Pilot 訓練（D+200 ~ D+220）**

- **對象：** 值機種子人員 5 人 + 系統管理者 2 人
- **時數：** 值機 12 小時 + 管理 6 小時 = 18 小時
- **目的：** 以測試環境進行實機操作，收集第一線操作回饋，據此進行 UI/UX 微調

**第二階段：正式訓練（D+220 ~ D+240）**

- **對象：** 全部值機人員 25 人（分 4 批，每批 6~7 人）+ 督導 2 人 + 系統管理者
- **時數：** 值機 24 小時（4 批 × 6hr/批）+ 管理 12 小時 = 36 小時

## 第八章　資安保密計畫

### 8.1 資安管理架構

本案資安管理依 CNS 27001（對應 ISO/IEC 27001）資訊安全管理標準建構，同時遵循資通安全管理法之規範。資安管理涵蓋以下四大面向：

| 面向 | 管控機制 | 條文依據 |
|------|---------|---------|
| 存取控制 | RBAC 角色型存取控制 + AD/LDAP 帳號整合（FR-ADM-002） | 契約§8(23) |
| 傳輸安全 | TLS 1.3 加密所有內外部通訊 | CNS 27001 A.13 |
| 儲存安全 | AES-256 加密敏感資料與個資欄位 | 服務水準表 |
| 稽核追蹤 | 所有操作日誌保留 ≥6 個月，個資存取日誌獨立管控（FR-ADM-004） | 資安查核表 |

### 8.2 CNS 27001 控制措施適用性

針對契約§8(23) 強制適用之 CNS 27001，本案於系統分析階段（#02）將建立完整的控制措施適用性聲明（SoA），逐項標註適用/不適用及具體實作方式。以下為工作計畫階段之初步對照：

| CNS 27001 控制域 | 適用性 | 本案實作概要 |
|-----------------|--------|------------|
| A.5 資訊安全政策 | 適用 | 制定資安政策文件，納入工作計畫書 §8.3 |
| A.6 資訊安全組織 | 適用 | 專案經理兼任資安負責人，資安分包廠商協助查核 |
| A.7 人力資源安全 | 適用 | 全員適任性查核（契約附件 10）、保密同意書/切結書（附件 8/9） |
| A.8 資產管理 | 適用 | 硬體資產清冊（#06）、軟體授權清冊 |
| A.9 存取控制 | 適用 | RBAC + AD/LDAP 整合（M-ADM-02）、最小權限原則 |
| A.10 密碼學 | 適用 | TLS 1.3（傳輸）、AES-256（儲存）、金鑰管理 |

完整 SoA 對照表將於 #02 系統分析報告書之資安專節中逐控制項展開，含實作細節與佐證文件。

### 8.3 個資保護機制

本案涉及旅客個人資料（電話號碼、訂票紀錄、身分證字號等），依個人資料保護法及服務水準表要求，採取以下防護措施：

1. **即時去識別化：** STT 輸出文字即時套用正則表達式遮罩（身分證 → A1234*****、電話 → 0912\*\*\*789），原始語音與去識別化文字分離存放
2. **加密儲存：** 所有個資欄位以 AES-256 加密後寫入 PostgreSQL，解密金鑰由管理/資安監控伺服器獨立管理
3. **最小權限：** 值機員僅可存取當前服務之旅客資料，督導可查閱所轄座席之服務紀錄，系統管理者具全域存取權限但操作均留日誌
4. **存取日誌：** 所有個資存取行為獨立記錄，保留 ≥6 個月，支援稽核查詢與匯出
5. **個資影響評估（PIA）：** 於 #02 系統分析報告書建立完整 PIA，列出所有個資欄位之蒐集目的、處理方式、保存期限與風險評估

個資或敏感資料外洩依服務水準表按筆數計罰，本案以技術（加密/去識別化）與管理（最小權限/稽核追蹤）雙重手段降低洩漏風險。

### 8.4 資安事件通報與應變

依服務水準表規定，建立三階段資安事件處理流程：

| 階段 | 時限 | 動作 |
|------|------|------|
| 偵測與通報 | 知悉後 ≤1 小時 | 值班人員確認事件等級，通報台鐵資安承辦人與數位發展處 |
| 損害控制與復原 | ≤72 小時（重大事件 ≤36 小時） | 隔離受影響系統、啟動 HA 備機接管、修補漏洞、驗證復原 |
| 調查報告 | 損害控制後 ≤1 個月 | 提交事件調查報告，含根因分析、影響範圍、改善措施、預防方案 |

通報逾時每小時計 1 點（服務水準表）。監控系統（Prometheus + Grafana，FR-ADM-003）設定以下告警觸發條件，確保第一時間偵測異常：

- 系統不可用（含任一子系統 health check 失敗）
- GPU 溫度 ≥85°C 或記憶體使用率 >85%
- 磁碟使用率 >90%
- API 回應延遲 >3 秒（接近 5 秒 SLA）
- 異常登入嘗試（同一帳號 5 分鐘內失敗 ≥5 次）

### 8.5 日誌管理策略

| 日誌類型 | 保留期限 | 儲存位置 | 存取權限 |
|---------|---------|---------|---------|
| 系統操作日誌 | ≥6 個月 | 管理/資安監控伺服器 | 系統管理者 |
| 個資存取日誌 | ≥6 個月 | 管理/資安監控伺服器（獨立分區） | 系統管理者 + 資安稽核 |
| API 呼叫日誌 | ≥6 個月 | 管理/資安監控伺服器 | 系統管理者 |
| SMS 簡訊紀錄 | ≥5 年 | PostgreSQL + TimescaleDB | RBAC 控管 |
| 通話錄音索引 | ≥6 個月 | 管理/資安監控伺服器 | 督導 + 系統管理者 |

日誌集中至管理/資安監控伺服器（雙備援），採 RAID-5 儲存確保資料可靠性。SMS 簡訊紀錄因保留期限 ≥5 年，規劃定期歸檔至冷儲存並建立檢索索引。

### 8.6 資安檢測規劃

資安檢測由資安分包廠商（具 CREST/CEH 認證）執行，於 Phase 3（D+200 起）進場，預留 30 天修正窗口。檢測項目：

| 檢測項目 | 工具 | 範圍 | 通過標準 |
|---------|------|------|---------|
| 弱點掃描 | Nessus / OpenVAS | 全部伺服器與網路設備 | 無「高」以上風險未修補 |
| 滲透測試 | 手動 + 工具 | OWASP Top 10 + SIP 協定安全 | 無可被利用之漏洞 |
| 源碼檢測 | SAST + SCA | 全部自行開發程式碼 + 第三方元件 | 無「高」以上風險未修補 |
| 資安防護基準查核 | 附件 16 查核表 | 逐項對照 | 全數符合 |

開發階段即導入 SAST 掃描（詳見第四章 §4.3 CI 流程），提前發現並修補安全弱點，避免集中在 Phase 3 才暴露大量問題。檢測結果彙整為 #09 測試報告書之資安檢測章節。

### 8.7 AI 模型安全與資料治理

針對本案 AI 元件之特殊安全考量：

1. **模型來源合規：** LLM 採用 Llama 3.1 8B（Meta 開源，Apache 2.0 授權），非大陸地區品牌或製造；STT 採用 Whisper large-v3（OpenAI 開源）；TTS 採用 Coqui XTTS-v2（開源 MPL 2.0）——均符合契約§8(24) 產地限制
2. **模型不使用旅客資料訓練：** 本案 LLM 採預訓練模型，不以旅客通話錄音或對話紀錄進行模型微調訓練。知識庫更新僅使用台鐵提供之官方文件
3. **回覆內容防護：** 阻擋非業務問題與惡意攻擊（FR-IVR-007），含敏感詞過濾、頻率限制、對話逾時中斷。答案來源限定於台鐵指定知識庫
4. **AI 生成內容標示：** 較不明確部分由 AI 生成的回覆標註「本回覆由生成式 AI 生成，如有疑問請按 0 轉接專人」
5. **異常切換：** 系統異常時立即切換為人工客服系統且不影響正常運作（契約§7(六)），HA Active-Active 架構（AD-005）確保單機故障不中斷服務

## 4.1　系統總體架構圖說明

### 4.1.1　架構設計原則

本案系統架構設計遵循以下五項核心原則，每項原則均直接對應契約條款或資安法規之強制要求：

| 原則 | 說明 | 契約依據 |
|------|------|---------|
| **地端部署（On-Premises）** | 所有 AI 運算、資料儲存、模型訓練與推論均於台鐵公司機房完成，禁止任何資料離開機房物理邊界 | 契約第 8 條(二十四)：資料不得跨境傳輸 |
| **雙機高可用（HA Active-Standby）** | 所有核心元件均採雙機架構，單點故障時自動切換，切換時間 ≤ 30 秒 | 需求書 HA 備援機制章節 |
| **無中國大陸供應鏈** | 硬體設備、軟體授權、韌體元件之品牌與製造地均不得為大陸地區 | 契約第 8 條(二十四) + 需求書備註第 3 點 |
| **CNS 27001 合規** | 系統設計須符合 CNS 27001 資訊安全管理標準之全部控制措施 | 契約第 8 條(二十三) |
| **既有系統零衝擊** | AI 模組之整合不得影響既有客服系統之正常運作與服務品質 | 需求書參、一(七) |

> **白話文註解：為什麼這五項原則是「護城河」？**
>
> 台鐵客服系統是 24 小時不間斷的公共服務，每月處理 2~3.5 萬通電話及 4,000~6,000 則數位諮詢。如果架構設計不以「地端部署」為前提，一旦被查核發現資料有跨境傳輸風險，輕則被要求系統下架重做（延遲交付 → 逾期違約金每日按契約價金千分之一計罰），重則觸犯資通安全管理法。「無中國供應鏈」並非技術偏好，而是政府採購的法律紅線——若驗收時被發現任何一顆晶片或一套軟體授權來自大陸品牌，整台設備可能被判定不合格，連帶影響第二期 30% 付款（軟硬體清單驗收）。

### 4.1.2　系統分層架構

本系統採用**六層式架構（Six-Layer Architecture）**，自底層至頂層依序為：

```
┌──────────────────────────────────────────────────────────────────┐
│                     第六層：使用者接入層                           │
│   旅客（電話/WebCall/數位客服/SMS）  │  客服人員（值機平台）        │
│   系統管理者（後台管理介面）         │  外點同仁（後送系統, 225人） │
├──────────────────────────────────────────────────────────────────┤
│                     第五層：通道接入與協定轉換層                    │
│   E1 VoIP GW ──┐                                                │
│   類比 VoIP GW ─┤── SIP Trunk ──→ AI-IVR 叢集（30埠+）          │
│   WebCall SBC ──┘                                                │
│   CRM Gateway ────────────────→ AI-Agent 叢集（30埠+）           │
│   SMS 收發裝置 ────────────────→ AI-Agent SMS 模組               │
├──────────────────────────────────────────────────────────────────┤
│                     第四層：AI 智慧服務層                          │
│   ┌─────────┐ ┌─────────┐ ┌─────────┐ ┌───────────┐            │
│   │ AI-IVR  │ │AI-Agent │ │ AI-QI   │ │ 後送分析  │            │
│   │ (語音)  │ │ (文字)  │ │(質檢)   │ │ 引擎      │            │
│   └────┬────┘ └────┬────┘ └────┬────┘ └─────┬─────┘            │
│        │           │           │             │                   │
│   ┌────▼───────────▼───────────▼─────────────▼──────┐           │
│   │          AI 推論引擎（Inference Engine）          │           │
│   │   STT ←→ LLM（RAG + Prompt Chain）←→ TTS        │           │
│   │        向量資料庫（Embedding Store）               │           │
│   └──────────────────────────────────────────────────┘           │
├──────────────────────────────────────────────────────────────────┤
│                     第三層：資料與知識管理層                       │
│   知識庫管理  │  向量索引  │  對話紀錄  │  錄音檔  │  後送案件DB │
│   RDBMS (HA) │  VectorDB  │  TimeSeries│  NAS     │  RDBMS     │
├──────────────────────────────────────────────────────────────────┤
│                     第二層：平台與安全管理層                       │
│   管理伺服器  │  資安監控伺服器  │  日誌收集  │  備份系統        │
│   CNS 27001 控制措施  │  SIEM  │  弱掃排程  │  存取控制        │
├──────────────────────────────────────────────────────────────────┤
│                     第一層：基礎設施層                             │
│   AI-IVR Server ×2  │  AI Server(GPU) ×2  │  管理/資安 Server ×2│
│   SMS 裝置 ×2  │  42U 機櫃 ×1  │  RPS 電源  │  GbE 網路        │

> **白話文註解：為什麼要分六層？**
>
> 分層架構的最大好處是「責任隔離」。台鐵這個案子有一個非常特殊的要求：「AI 模組的整合不得影響既有客服系統」。如果我們把 AI 功能直接嵌入現有 MES Server 或 CRM Gateway，一旦 AI 模組出問題，整個客服中心就癱瘓了。分層設計讓 AI 智慧服務層（第四層）透過標準 API/SIP 協定與第五層的既有設備溝通，既有設備完全不需要改動內部程式碼。這就像在高速公路旁邊蓋了一條平行的智慧車道，兩條路互不干擾，但可以透過匝道（API）互通。

### 4.1.3　語音服務流程（AI-IVR 主線路）

以下為旅客撥打電話至 AI 客服系統的完整處理流程，含各節點延遲預算：

```
旅客撥號                     既有系統（不可修改）            AI 客服系統（本案新建）
─────────                   ──────────────────           ─────────────────────
    │                             │                              │
    ├──PSTN/WebCall──→ E1 VoIP GW / WebCall SBC                 │
    │                    │                                       │
    │                    ├──SIP INVITE──→ MES Server             │
    │                    │                  │                     │
    │                    │      ┌──RTP Fork──┤                   │
    │                    │      │            │                    │
    │                    │      │   錄音持續進行                  │
    │                    │      │                                │
    │                    │      └──RTP Stream──→ AI-IVR Server   │
    │                    │                        │               │
    │                    │              ┌─────────▼──────────┐   │
    │                    │              │  STT 即時轉文字     │   │
    │                    │              │  延遲預算: ≤800ms   │   │
    │                    │              ├────────────────────┤   │
    │                    │              │  LLM 語意理解+生成  │   │
    │                    │              │  延遲預算: ≤2500ms  │   │
    │                    │              ├────────────────────┤   │
    │                    │              │  TTS 語音合成       │   │
    │                    │              │  延遲預算: ≤700ms   │   │
    │                    │              └─────────┬──────────┘   │
    │                    │                        │               │
    │◀──────RTP 回送語音──┤◀──────RTP 回送───────┘               │
    │                    │                                       │
    │  ※ 端到端延遲預算合計: ≤ 5,000ms（契約要求: 5秒內回覆）     │
```

**各環節延遲預算分配表：**

| 處理環節 | 延遲預算 (ms) | 技術說明 | 預留緩衝 |
|---------|--------------|---------|---------|
| 網路傳輸（RTP 收送） | ≤ 200 | 機房內 LAN，RTT < 1ms；含 jitter buffer | 含 100ms 緩衝 |
| VAD 語音活動偵測 | ≤ 300 | 偵測用戶說完話的停頓點，避免打斷用戶 | — |
| STT 語音轉文字 | ≤ 800 | Streaming ASR，邊聽邊轉，非等整句結束 | 含 200ms 緩衝 |
| LLM 推論（含 RAG 檢索） | ≤ 2,500 | 向量檢索 ≤200ms + Token 生成 ≤2,300ms | 含 500ms 緩衝 |
| TTS 語音合成 | ≤ 700 | 首包延遲（Time-to-First-Byte）≤ 300ms，後續 streaming | 含 200ms 緩衝 |
| 系統內部匯流排開銷 | ≤ 500 | API 呼叫、序列化/反序列化、佇列等待 | — |
| **合計** | **≤ 5,000** | **契約要求：5 秒(含)內回覆** | **含 1,000ms 總緩衝** |

> **白話文註解：為什麼要把 5 秒拆成這麼細的毫秒預算？**
>
> 契約寫的是「5 秒內回覆」，看起來很簡單，但這 5 秒要從旅客「說完話」到「聽到 AI 回覆的第一個字」，中間要經過語音辨識、AI 思考、語音合成三大步驟。如果不事先把每個步驟的時間「卡死」，很可能開發到最後才發現 LLM 推論就花了 4 秒，STT 加上去就爆了。這張表就是我們的「時間合約」——每個模組的團隊只能用分配到的毫秒數，誰超標誰負責優化。預留的 1,000ms 緩衝則是為了應對尖峰時段（如連假前夕話務量暴增 50%）的排隊延遲。

### 4.1.5　真人客服轉接流程與知識輔助

此流程為本案之核心差異化功能，當 AI 判斷需轉接真人時觸發：

```
Step 1: AI 偵測轉接觸發條件
        ├─ 用戶重複詢問相同問題 ≥ 2次
        ├─ 偵測到關鍵字：投訴、轉接專人、負面情緒字眼
        ├─ AI 信心度 < 60% 的回應連續出現 ≥ 2次
        └─ 用戶主動按鍵要求轉接

Step 2: AI 生成轉接摘要
        ├─ 對話摘要（≤ 200 字）
        ├─ 旅客意圖分類標籤
        ├─ 情緒指標（正面/中性/負面）
        └─ 相關知識文件 Top-3 推薦

Step 3: SIP REFER 轉接至人工座席
        ├─ AI-IVR → MES Server → 人工值機分派
        └─ 摘要資訊即時推送至值機平台

Step 4: 通話期間即時知識輔助
        ├─ STT 持續即時轉錄真人對話
        ├─ LLM 即時比對知識庫
        └─ 值機平台側邊欄顯示建議回應、SOP、注意事項
```

> **白話文註解：為什麼「轉接摘要」這麼重要？**
>
> 需求書明確寫道：「旅客僅需於首次進線時完整敘述其問題，後續由系統彙整相關資訊並提供客服人員參考，無須重複說明。」這代表如果轉接後客服人員問旅客「請問您的問題是什麼？」，旅客就會抱怨「我剛剛已經跟 AI 說過了」——這會直接拉低滿意度調查分數。滿意度是合約 KPI 之一，而且每一通電話結束後都會啟動滿意度調查機制。所以「轉接摘要」不是錦上添花，而是直接影響驗收指標的核心功能。

### 4.2.2　AI-IVR 伺服器詳細規格與容量規劃

**硬體規格（每台）：**

| 項目 | 契約最低規格 | 建議實配規格 | 設計理由 |
|------|------------|------------|---------|
| CPU | 16 核心 ≥2.2GHz ×2 | Intel Xeon Gold 6426Y (16C/32T, 2.5GHz) ×2 或同等非中國品牌 | 64 執行緒可同時處理 30 路 SIP Session + STT/TTS 即時串流 |
| RAM | DDR5 64GB+ | DDR5 ECC 64GB (16GB ×4) | STT 模型常駐記憶體 ~8GB，TTS ~6GB，SIP Stack + OS ~12GB，對話上下文快取 ~16GB，剩餘預留 |
| 儲存 | 960GB SSD ×3 | 960GB NVMe SSD ×3 (RAID-5) | RAID-5 有效容量 ~1.8TB；日均錄音索引 ~2GB，可存 6 個月以上 |
| 網路 | GbE ×4 | GbE ×4（管理 ×1, SIP/RTP ×1, 內部通訊 ×1, BMC ×1） | 網路流量隔離：語音 RTP 流不與管理流量混跑 |
| 電源 | 800W RPS ×2 | 800W 鉑金級 RPS ×2 | 冗餘電源，支撐 CPU 滿載 + NVMe 寫入峰值 |

**容量規劃：**

| 指標 | 計算依據 | 結果 |
|------|---------|------|
| 峰值同時通話數 | 月均 35,000 通 ÷ 30 天 ÷ 8 小時（尖峰集中）≈ 146 通/hr，並行高峰係數 ×3 ≈ 每 5 分鐘 ~36 通 | 30 埠可覆蓋 95% 場景 |
| 單路 STT CPU 消耗 | Streaming ASR ~0.3 核心/路 | 30 路 × 0.3 = 9 核心 |
| 單路 TTS CPU 消耗 | Neural TTS ~0.2 核心/路 | 30 路 × 0.2 = 6 核心 |
| SIP + 對話管理 | ~0.1 核心/路 | 30 路 × 0.1 = 3 核心 |
| OS + 系統開銷 | — | ~4 核心 |
| **單台 CPU 需求合計** | — | **~22 核心**（32 執行緒可覆蓋） |
| HA 模式（Active-Active 50/50） | 每台分擔 15 路 | 單台使用率 ~35%，留有充足餘裕 |
| HA 模式（Active 故障，Standby 接管全部） | 單台承載 30 路 | 使用率 ~69%，仍在安全閾值 80% 以下 |

> **白話文註解：為什麼 AI-IVR 伺服器不需要 GPU？**
>
> 很多人直覺認為「做 AI 就一定要 GPU」，但 AI-IVR 伺服器負責的是 STT（語音轉文字）和 TTS（文字轉語音），這兩個任務的即時性要求極高（每收到一小段語音就要立即處理），反而更適合用 CPU 的低延遲特性來做 streaming 推論。真正需要 GPU 大規模並行運算的是 LLM（大語言模型）——那是 AI 伺服器的工作。如果硬把 STT/TTS 也搬到 GPU 上跑，反而會跟 LLM 搶 GPU 記憶體，導致 LLM 推論延遲飆升，突破 5 秒限制。這是一個「看起來更先進但實際上更危險」的設計陷阱。

| 儲存 | 960GB SSD ×3 | 960GB NVMe SSD ×3 (RAID-5) | 有效 ~1.8TB；模型檔 ~200GB，向量索引 ~100GB，對話日誌 + 質檢結果長期儲存 |
| GPU | **L40s ×2** | **NVIDIA L40S 48GB GDDR6 ×2** | 見下方 GPU 資源分配詳細說明 |
| 電源 | 1000W RPS ×2 | 1000W 鉑金級 RPS ×2 | L40S TDP 350W ×2 = 700W + CPU + 週邊 ≈ 需 950W 峰值 |
| 網路 | GbE ×4 | GbE ×4（管理 ×1, 推論 API ×1, 內部同步 ×1, BMC ×1） | 推論 API 與 HA 心跳需獨立網路段 |

**NVIDIA L40S GPU 關鍵規格與選型理由：**

| 參數 | L40S 規格 | 本案需求對應 |
|------|-----------|------------|
| GPU 記憶體 | 48GB GDDR6 | 足以載入 13B~30B 參數量之 LLM（FP16 約需 26~60GB），本案建議使用 13B 級別模型以確保延遲 |
| FP16 效能 | 362 TFLOPS | 提供充足的 Transformer attention 運算力 |
| INT8 效能 | 724 TOPS | 支援量化推論，可在精度損失極小的前提下提升吞吐量 2 倍 |
| TDP | 350W | 雙卡 700W 在 1000W RPS 預算內 |
| PCIe | Gen4 x16 | 確保 Host ↔ GPU 資料傳輸不成瓶頸 |
| 非中國品牌 | NVIDIA（美國） | 符合契約產地要求 |

> **白話文註解：為什麼是 L40S 而不是 A100 或 H100？**
>
> 需求書已經「指定」了 L40S（或同等品），所以選型空間有限。但這其實是一個合理的選擇：L40S 的 48GB 記憶體足以跑 13B 級別的 LLM，價格只有 H100 的 1/3 左右；它的推論效能完全夠用，因為台鐵的場景是「30 路同時對話」而非數千用戶的網際網路規模。如果選 H100，預算會暴漲但效能利用率可能不到 30%——這在政府採購的「效益審查」中反而會被質疑浪費公帑。

│                                                     │

│  └─────────────────────────────────────────────┘   │
│                                                     │
│  ※ 此 GPU 負責非即時任務，不影響 LLM 延遲          │
└─────────────────────────────────────────────────────┘
```

**雙機 HA 策略：**

| 運行模式 | P-GPU-01 | P-GPU-02 | 系統行為 |
|---------|---------|---------|---------|
| **正常（Active-Active）** | 承擔 50% LLM 推論 + 100% 即時 Embedding | 承擔 50% LLM 推論 + 100% 排程 QI 質檢 | 負載均衡，各機 GPU 使用率 ~40-50% |
| **單機故障（Failover）** | 接管 100% 全部推論 | — (故障) | GPU#0 滿載 LLM，GPU#1 接管 Embedding；QI 排程暫停 |
| **維護窗口** | 停機維護 | 接管 100% | 每季停機保養時使用此模式 |

**推論吞吐量估算：**

| 場景 | 並行 Session | 單次推論 Token | 推論延遲目標 | 所需 GPU 吞吐 | L40S 可達 |
|------|-------------|---------------|------------|-------------|----------|
| AI-IVR 語音（即時） | 30 路 | ~128 token (輸入) + ~64 token (輸出) | ≤ 2,500ms | ~768 token/s | 約 1,200 token/s (FP16) |
| AI-Agent 文字（即時） | 30 路 | ~256 token (輸入) + ~128 token (輸出) | ≤ 3,500ms | ~1,097 token/s | 約 1,200 token/s (FP16) |
| IVR + Agent 同時尖峰 | 最惡 60 路 | 混合 | ≤ 5,000ms | ~1,500 token/s | 雙機雙卡 ~2,400 token/s |
| AI-QI 質檢（非即時） | 批次 | 長文本 ~2048 token | 15 分鐘錄音 → 5 分鐘完成 | 不佔即時推論資源 | GPU#1 離峰處理 |

> **白話文註解：為什麼要把兩張 GPU 的工作嚴格分開？**
>
> 想像一個廚房有兩個爐子。如果你把「即時出餐」（客人在等的菜）和「備料工作」（切菜、醃肉）放在同一個爐子上，出餐速度一定會受影響。GPU#0 專門負責 LLM 推論（=即時出餐），GPU#1 負責向量化和質檢（=備料）。這樣即使質檢系統同時在分析 50 個錄音檔，也不會讓正在打電話的旅客多等一秒鐘。如果不做這個隔離，尖峰時段 AI-QI 的批次任務可能吃掉 GPU 記憶體，導致 LLM 的 KV Cache 被壓縮，推論延遲從 2 秒暴增到 8 秒——直接違反「5 秒內回覆」的契約要求。
>
> 另一個設計要點是 GPU#1 預留了 ~32GB 的空間。需求書第四條寫到「AI 伺服器應保留必要之擴充空間與介面，以支援未來新增功能或硬體模組」。這 32GB 就是我們的「擴充保險」，未來如果台鐵要升級到更大的 LLM（例如從 13B 升到 30B），可以把 GPU#1 的備用空間也拿來跑推論，不需要重新採購硬體。

### 4.2.6　測試環境資源分配

> **白話文註解：為什麼「本 PRD 設計」的指標也要寫進來？**
>
> 契約只規定了「5 秒內回覆」這個總指標，但 5 秒是由 STT + LLM + TTS 三個環節組成的。如果我們不在 PRD 中把每個環節的子指標「釘死」，開發團隊就沒有明確的優化目標——A 團隊覺得 STT 可以用 2 秒，B 團隊覺得 LLM 可以用 3 秒，加起來就 5 秒爆了。這些子指標雖然不是契約的直接要求，但它們是我們「確保能滿足契約要求」的內部品質管理基準。在測試計畫書（交付項目 5）中，這些子指標會成為內部壓力測試的 pass/fail 標準。
